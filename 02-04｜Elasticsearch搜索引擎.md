## 〇、零基础动手上手（面试够用版）

> 如果你对 ES 完全陌生，先别急着背下面 65 道题。这一节给你一个"能跑起来、能查出来"的最小闭环，外加三张图把三个最核心的原理讲成人话。目标：面试时能说清楚"为什么快、为什么刚写搜不到、数据怎么分片"，并且真能写出基础 DSL。

### 〇.1 一句话定位（你已经在用的认知）

ES 就是**架在 MySQL/MongoDB 前面的一层"搜索读模型"**：主库负责写和权威数据，ES 负责把数据重新组织成"好搜"的结构，专门扛复杂查询。它快，不是因为它比数据库聪明，而是因为它**提前把"哪些词出现在哪些文档里"算好存起来了**（倒排索引）。

### 〇.2 最小动手闭环（照抄就能跑）

**第 1 步：一行命令起 ES（Docker）**

```bash
docker run -d --name es -p 9200:9200 -p 9300:9300 \
  -e "discovery.type=single-node" \
  -e "xpack.security.enabled=false" \
  docker.elastic.co/elasticsearch/elasticsearch:8.13.4
```

等 30 秒，访问 `http://localhost:9200` 看到 JSON 就成功了。

**第 2 步：建一个索引 + 指定字段类型（Mapping）**

```bash
curl -X PUT "localhost:9200/articles" -H "Content-Type: application/json" -d '{
  "mappings": {
    "properties": {
      "title":   { "type": "text" },      # 全文检索
      "author":  { "type": "keyword" },   # 精确过滤/排序
      "status":  { "type": "keyword" },
      "created": { "type": "date" }
    }
  }
}'
```

**第 3 步：灌 3 条数据（Bulk 批量写入）**

```bash
curl -X POST "localhost:9200/articles/_bulk" -H "Content-Type: application/json" -d '
{"index":{"_id":1}}
{"title":"Go 语言并发模型详解","author":"张三","status":"published","created":"2026-01-01"}
{"index":{"_id":2}}
{"title":"Elasticsearch 倒排索引原理","author":"李四","status":"published","created":"2026-02-01"}
{"index":{"_id":3}}
{"title":"MySQL 索引与查询优化","author":"张三","status":"draft","created":"2026-03-01"}
'
```

**第 4 步：写 3 个最核心的查询**

```bash
# ① 全文检索（match）：搜 "倒排索引" 相关文章
curl -X POST "localhost:9200/articles/_search" -H "Content-Type: application/json" -d '{
  "query": { "match": { "title": "倒排索引" } }
}'

# ② 精确过滤（term + filter）：只看张三发的、已发布的
curl -X POST "localhost:9200/articles/_search" -H "Content-Type: application/json" -d '{
  "query": {
    "bool": {
      "must":   { "match": { "title": "语言" } },
      "filter": [
        { "term": { "author": "张三" } },
        { "term": { "status": "published" } }
      ]
    }
  }
}'

# ③ 组合（bool）：标题含"优化"，且状态已发布
curl -X POST "localhost:9200/articles/_search" -H "Content-Type: application/json" -d '{
  "query": {
    "bool": {
      "must":   { "match": { "title": "优化" } },
      "filter": { "term": { "status": "published" } }
    }
  }
}'
```

跑完这四步，你就已经"会用" ES 了：建索引、灌数据、写 match / term / bool 查询。

**第 5 步：感受两个关键现象（面试常考）**

- **刚写入搜不到？** 写入默认要等约 1 秒 `refresh` 才生成可搜 Segment。立刻查可能查不到 → 这就是"近实时（NRT）"。强制立刻可见：`curl -X POST "localhost:9200/articles/_refresh"`。
- **text 字段用 term 查不到？** `term` 不做分词，而 `title` 是 text（被切成了多个词）。对 `author`（keyword）用 term 能查到；对 `title` 用 term 查"语言"则查不到 → 这就是 text vs keyword 的本质区别。

### 〇.3 三张图：把核心原理讲成人话

**图 1：倒排索引为什么快（搜索的灵魂）**

正排（数据库默认）：一行一行扫，找"倒排索引"这个词要遍历所有文章。

```text
正排（慢）：
  文档1: "Go 语言并发模型详解"
  文档2: "Elasticsearch 倒排索引原理"   ← 要扫到这行才知道它含"倒排索引"
  文档3: "MySQL 索引与查询优化"
  （查询 = 逐行扫描，O(N)）

倒排（快）：
  词项         -> 出现的文档
  "倒排索引"   -> [文档2]
  "语言"       -> [文档1, 文档3]
  "优化"       -> [文档3]
  （查询 = 直接查字典拿到文档列表，O(1)~O(logN)）
```

ES 就是把"词项 → 文档列表"这张大字典提前建好。搜"倒排索引"时，直接拿到文档2，不用扫全表。

**图 2：近实时（NRT）——为什么刚写搜不到**

```text
写入请求
  -> 进内存缓冲（Lucene In-memory Buffer）
  -> 同时写 Translog（防崩溃丢失）
  -> [默认 1 秒] 一次 refresh
  -> 生成新的 Segment（不可变小文件）
  -> 文档变得"可搜索"
  -> [默认 30 秒 / 512MB] 一次 flush
  -> 提交点 + 清理 Translog，真正落盘

关键：写入成功 ≠ 立刻可搜。中间隔着一次 refresh。
```

**图 3：数据怎么分片（Shard 路由）**

```text
写入文档 id=100
  -> shard = hash(100) % 主分片数(假设=3)
  -> 落在 shard 1
  -> 主分片 shard 1 写入
  -> 复制到 shard 1 的副本

查询"倒排索引"
  -> 协调节点广播到所有分片
  -> 每个分片本地查、返回 TopK
  -> 协调节点合并成全局 TopK
  -> 返回

分片作用：① 数据分布到多节点（能扩容） ② 并行查询（更快） ③ 副本容灾
```

## 一、Elasticsearch 学习地图与资深回答框架

### 1. Elasticsearch 的知识图谱主线是什么？

**一句话结论：** 以“可重建的搜索读模型”为顶层定位，沿着文档建模、写入索引、分布式检索、可靠同步、集群治理和业务落地逐层展开。

```text
业务需求与边界
  -> 文档与 Mapping
  -> Analyzer 与倒排索引
  -> Query / Filter / 排序 / 聚合
  -> Primary / Replica / Segment / Translog
  -> Shard 路由与分布式归并
  -> 数据同步与索引重建
  -> 性能容量与生产治理
  -> 业务选型和项目表达
```

复习时始终追问五件事：

1. **业务目标：** 是全文召回、组合筛选、相关性排序，还是统计分析？
2. **数据模型：** 哪些字段检索、过滤、排序、聚合，哪些字段只展示？
3. **链路原理：** 写入如何生成 Segment，搜索如何跨分片归并？
4. **工程代价：** 实时性、一致性、资源、复杂度分别牺牲什么？
5. **生产闭环：** 如何容量评估、监控、降级、重建和复盘？

### 2. 资深后端如何回答 ES 面试题？

采用“结论 -> 原理 -> 方案 -> 边界 -> 落地”的金字塔结构。

**示例：为什么不能把 ES 当主数据库？**

- **结论：** ES 应作为可重建的搜索读模型，交易事实仍由 MySQL 或 MongoDB 持有。
- **原理：** ES 为搜索与分析优化，依赖近实时刷新、分片副本和最终一致同步，不擅长跨文档强事务与复杂约束。
- **方案：** 主库先提交，通过 Outbox、CDC 或 Kafka 传播变更，消费者幂等更新 ES。
- **边界：** 短暂延迟、重复和乱序必须由版本、对账与重建兜底。
- **落地：** 监控同步延迟、Bulk 失败、版本差异和索引健康，ES 故障时降级而不阻塞交易。

### 3. 如何回答 ES 场景设计题？

按六步展开：

1. 明确搜索目标、数据量、QPS、P99、结果新鲜度和可用性目标。
2. 确认主数据源与 ES 的职责边界。
3. 设计索引、Mapping、Analyzer、路由、分页与排序。
4. 设计全量构建、增量同步、幂等、乱序和删除处理。
5. 估算分片、节点、磁盘、Heap、查询与写入吞吐。
6. 给出限流、降级、灰度、回滚、对账、快照和故障恢复方案。

### 4. ES 高频问题应如何串联复习？

| 层次 | 核心问题 | 必须讲出的关键词 |
|---|---|---|
| 定位 | 为什么用 ES | 搜索读模型、近实时、可重建 |
| 建模 | Mapping 怎么设计 | text/keyword、nested、dynamic、Doc Values |
| 原理 | 为什么搜索快 | 倒排索引、FST、Posting、BKD、页缓存 |
| 查询 | DSL 怎么优化 | filter、bool、分页、聚合、Profile |
| 写入 | 为什么刚写搜不到 | refresh、Segment、Translog |
| 集群 | 分片怎么规划 | 路由、Primary、Replica、恢复成本 |
| 一致性 | DB 如何同步 ES | Outbox/CDC、Kafka、幂等、版本、对账 |
| 治理 | 线上慢怎么查 | 证据、DSL、分片、线程池、Heap、磁盘 |
| 实战 | 项目如何表达 | 约束、权衡、指标、灰度、收益 |

## 二、使用场景、能力边界与组件选型

### 5. Elasticsearch 最适合解决什么问题？

**结论：** 适合大规模文档的全文搜索、多条件过滤、相关性排序、聚合分析和地理位置检索。

典型场景：

- 商品、内容、素材、工单和运营后台搜索。
- 日志、指标、链路等可观测数据分析。
- 标签、时间、状态、关键词组合筛选。
- 自动补全、模糊匹配、同义词和多语言检索。
- 搜索结果上的分桶、趋势和 TopN 聚合。

判断依据不是“数据量大”，而是查询模型是否具有**检索性**：召回条件复杂、排序相关、组合过滤多、关系 Join 少。

### 6. 什么场景不适合使用 ES？

- 强事务、唯一约束、外键和跨实体一致性是核心要求。
- 高频单行点更新远多于搜索，且每次更新都要求立即可见。
- 复杂多表 Join 和临时关系查询占主导。
- 数据不能从其他事实源恢复，又缺少严格备份恢复体系。
- 数据规模和查询都很简单，数据库索引已经满足 SLA。

**资深表达：** ES 的引入会增加同步链路、容量成本、数据一致性和运维复杂度；只有搜索收益超过这些成本时才值得使用。

### 7. ES、MySQL、MongoDB、Redis 和 Kafka 如何分工？

| 组件 | 核心职责 | 不应承担的主要职责 |
|---|---|---|
| MySQL | 事务、约束、关系模型、权威业务事实 | 大规模全文相关性检索 |
| MongoDB | 灵活文档、聚合根、变化较快的业务模型 | 复杂全文召回和全局强事务 |
| Redis | 热点缓存、计数、限流、短期状态 | 大规模持久搜索和唯一事实源 |
| Kafka | 事件传播、削峰、解耦、重放 | 随机查询和业务事实存储 |
| Elasticsearch | 搜索、过滤、排序、聚合、派生读模型 | 交易主库和跨系统强一致事务 |

典型链路：

```text
MySQL / MongoDB（权威数据）
  -> Outbox / CDC
  -> Kafka（可靠传播与重放）
  -> Elasticsearch（搜索读模型）
  -> Redis（热点结果或短期缓存）
```

### 8. ES 与 MySQL 全文索引如何选择？

**MySQL 优先：** 数据量不大、搜索条件简单、事务一致性更重要、团队不希望增加新基础设施。

**ES 优先：** 需要相关性评分、复杂 Analyzer、多字段召回、模糊匹配、高亮、聚合或水平扩展。

不要只比较单次耗时，还要比较：功能上限、数据同步成本、查询稳定性、扩容路径和故障恢复复杂度。

### 9. ES 与 MongoDB 查询如何选择？

MongoDB 适合作为业务文档的主存储，擅长按确定字段和索引查询完整聚合文档；ES 擅长把文档拆成可检索词项后进行召回、评分和聚合。

常见组合是 MongoDB 保存素材或任务完整文档，ES 只保存检索、过滤、排序和列表展示需要的字段。详情页按 ID 回查 MongoDB，避免 ES 文档无限膨胀。

## 三、核心对象、请求链路与分布式检索

### 10. ES 的核心对象如何串起来？

```text
Cluster
  -> Node（master-eligible / data / ingest / coordinating）
  -> Index
  -> Primary Shard + Replica Shard
  -> Lucene Index
  -> Segment
  -> Document -> Field -> Term
```

- Index 是逻辑数据集合。
- Shard 是数据分布、并行和故障恢复的基本单元，本质是 Lucene Index。
- Segment 是不可变的倒排索引文件集合。
- Replica 是 Primary 的完整副本，用于容灾并可分担搜索。
- 协调节点负责请求分发、结果归并和 Fetch。

### 11. 一条写入请求如何执行？

1. 客户端请求任意节点，该节点成为协调节点。
2. 根据 `_id` 或自定义 routing 计算目标 Primary Shard。
3. 请求转发到 Primary，完成校验、版本控制和 Lucene 内存缓冲写入，同时记录 Translog。
4. Primary 将操作并行转发到 Replica。
5. 满足活跃分片和确认条件后向客户端返回。
6. 后续 refresh 生成可搜索 Segment；flush 推进提交点并轮转 Translog；merge 合并小 Segment。

**关键边界：** 写入返回成功、持久化完成和搜索可见是三个不同概念。

### 12. 一条搜索请求如何执行？

搜索通常分为 `query then fetch` 两阶段：

1. 协调节点解析索引和路由，选择每个目标 Shard 的 Primary 或 Replica 副本。
2. Query Phase 在各 Shard 本地执行查询和排序，返回 DocID、score、sort values 等候选结果。
3. 协调节点合并各 Shard 的局部 TopK，形成全局 TopK。
4. Fetch Phase 向命中的 Shard 拉取 `_source` 或指定字段。
5. 协调节点组装最终响应。

因此，分片过多、`from + size` 过大、返回文档过重和复杂聚合都会放大 CPU、网络和协调节点内存。

### 13. ES 如何决定文档属于哪个 Shard？

概念上可理解为：

```text
shard = hash(routing) % number_of_primary_shards
```

默认 routing 是 `_id`。自定义 routing 可以让同一租户或业务实体落到同一 Shard，查询时指定 routing 可减少扇出。

风险是 routing key 分布不均会制造热点；设计前必须评估基数、数据倾斜、最大租户和未来迁移能力。

### 14. Primary 和 Replica 分别解决什么？

- Primary 负责该 Shard 写入操作的排序和复制协调。
- Replica 提供冗余副本，Primary 故障时可提升为新 Primary，也可承接搜索。
- Replica 增加能提升读能力和可用性，但会增加写复制、磁盘和恢复成本。
- Replica 不是备份；误删除会同步到所有副本，仍需 Snapshot。

### 15. 协调节点为什么可能成为瓶颈？

它承担请求广播、候选结果维护、全局排序、聚合归并和响应序列化。大分页、高基数聚合、跨大量 Shard 查询或返回大 `_source` 时，协调节点会承受大量 Heap、CPU 和网络压力。

治理方法：减少目标 Shard、限制结果窗口、使用 `search_after`、压缩返回字段、限制聚合桶、隔离重查询，并为专用协调节点做容量评估。

## 四、Mapping、Analyzer、倒排索引与相关性

### 16. ES 为什么搜索快？

**结论：** ES 通过“写时建立索引、读时按词项定位”的方式，把全文扫描变成有序词典和 Posting List 的集合运算。

关键结构：

- **倒排索引：** Term -> 包含该 Term 的文档列表。
- **Term Dictionary/FST：** 快速定位词项并压缩公共前缀。
- **Posting List：** 保存 DocID、词频、位置等信息，可跳跃和压缩。
- **Doc Values：** 面向列的磁盘结构，服务排序、聚合和脚本访问。
- **BKD Tree：** 支持数值、日期和地理字段的范围与空间查询。
- **Filesystem Cache：** Lucene 大量利用操作系统页缓存。

代价是写放大、磁盘空间、近实时可见和 Mapping 设计复杂度。

### 17. text 和 keyword 有什么区别？

| 类型 | 索引方式 | 典型用途 |
|---|---|---|
| `text` | 经过 Analyzer 切分为多个 Term | 全文召回、相关性评分 |
| `keyword` | 整体作为一个精确值 | 过滤、排序、聚合、去重 |

名称常设计为多字段：`name` 用 `text` 搜索，`name.keyword` 用于精确过滤或排序。ID、状态、枚举、手机号等通常使用 `keyword`，不要误设为 `text`。

### 18. Analyzer 的执行链路是什么？

```text
Character Filter -> Tokenizer -> Token Filter
```

- Character Filter 在分词前清洗或映射字符。
- Tokenizer 决定如何切词。
- Token Filter 做小写、停用词、词干、同义词等处理。

索引时 Analyzer 决定存什么 Term；查询时 Search Analyzer 决定拿什么 Term 去匹配。两者不一致可能是业务需要，也可能导致完全搜不到。

### 19. 中文分词、同义词和停用词如何设计？

- 先根据业务语料定义召回目标，不要只比较分词器名称。
- 用 `_analyze` 对真实词库测试切词结果。
- 索引侧尽量稳定，搜索侧同义词更新更灵活。
- 同义词要区分等价词和单向扩展，防止召回爆炸。
- 停用词可能误删品牌、型号中的有效字符。
- 上线前用离线查询集评估召回率、准确率和 NDCG 等指标。

### 20. Mapping 设计有哪些关键原则？

1. 先列出查询矩阵：每个字段是否检索、过滤、排序、聚合、展示。
2. 明确字段类型，避免依赖无限制动态映射。
3. 控制字段数量，防止 Mapping Explosion。
4. 精确字段用 keyword/numeric/date，全文字段用 text。
5. 排序聚合使用 Doc Values；大文本不参与检索时关闭索引。
6. 只保存搜索需要的数据，详情可回查主库。
7. 重要 Mapping 变更通过新索引 + Alias 切换，不在旧索引上冒险修改。

### 21. dynamic mapping 有什么风险？

动态字段可能把日期识别成字符串、把 ID 识别成数值，或者因用户自定义 key 产生数十万字段，导致集群状态膨胀、Heap 压力和查询异常。

生产中对核心索引应使用显式 Mapping；变化字段可用 `dynamic: strict/false`、`flattened`、键值数组或独立扩展集合承载。

### 22. object、nested 和父子关系如何选择？

- `object` 会把对象数组扁平化，无法保持数组元素之间的字段关联。
- `nested` 把每个子对象作为隐藏 Lucene 文档，可保证同一子对象内的组合条件，但增加写入、查询和存储成本。
- `join` 父子关系允许独立更新，但查询代价高、路由限制多，应谨慎使用。
- 高频查询场景常优先反范式冗余，接受通过同步更新换取读性能。

### 23. query 和 filter 有什么区别？

- Query Context 判断“匹配度多高”，参与 `_score`。
- Filter Context 判断“是否匹配”，不计算评分，更适合状态、租户、时间、权限等精确条件，也更有缓存机会。

常见设计是 `bool.must` 放全文召回，`bool.filter` 放结构化约束，既表达清晰又减少无意义评分。

### 24. match、term、match_phrase 和 multi_match 如何选择？

- `match` 会分析查询文本，适合 `text` 全文检索。
- `term` 不分析输入，适合 keyword、数值、状态等精确值。
- `match_phrase` 要求词项位置满足短语关系，召回更精确但成本更高。
- `multi_match` 用于多字段召回，可按字段设置 boost。

最常见错误是在 text 字段上用 term 查自然语言，或对 keyword 字段用 match 后误判结果。

### 25. BM25 相关性评分如何理解？

评分核心考虑：查询词在当前文档中的频率、在全局文档中的稀有程度、字段长度归一化和字段 boost。它适合文本相关性，但不是完整业务排序。

生产排序往往是：

```text
文本相关性
  + 质量分
  + 新鲜度衰减
  + 热度/转化率
  + 业务置顶和风控约束
```

需要用离线标注集和线上 A/B 测试验证，不能只凭开发者感觉调 boost。

## 五、Query DSL、分页、排序与聚合设计

### 26. bool 查询如何组织复杂业务条件？

- `must`：必须匹配且参与评分。
- `filter`：必须满足但不评分。
- `should`：提高召回或评分，可配 `minimum_should_match`。
- `must_not`：排除条件。

资深回答要补充：先限制租户、状态和时间范围，再做全文召回；将稳定过滤与评分逻辑分离；防止用户输入直接拼接脚本或任意 DSL。

### 27. 深分页为什么慢，如何治理？

`from + size` 要求每个 Shard 取出本地前 `from + size` 条，再由协调节点全局归并。页数越深，CPU、Heap 和网络成本越大。

方案选择：

- 普通交互分页：限制最大页深。
- 下一页浏览：`search_after` + 稳定唯一排序键。
- 一致性遍历：PIT + `search_after`。
- 离线全量导出：使用适合批量遍历的机制并限速，避免冲击在线集群。
- 运营确需随机跳页：评估产品改造或预计算，不能无限放开结果窗口。

### 28. search_after 为什么需要稳定排序？

它以“上一页最后一条的 sort values”为下一页游标。若排序值不唯一或翻页期间索引刷新导致视图变化，会出现重复或漏数据。

通常使用业务排序字段加唯一 ID 作为 tie-breaker，并结合 PIT 固定搜索视图。游标应编码查询条件版本，防止用户更换条件后继续使用旧游标。

### 29. 排序和字段返回如何优化？

- 排序字段使用 keyword/numeric/date，并确保有 Doc Values。
- 不需要评分时关闭 `track_scores`，大结果集谨慎使用 `_score` 排序。
- 只返回必要 `_source` 字段，或使用 `docvalue_fields/stored_fields`。
- 避免脚本排序和运行时字段进入高 QPS 主链路。
- 固定过滤查询可使用 preference 改善缓存局部性，但要考虑热点。

### 30. 聚合查询为什么容易拖垮集群？

聚合要在各 Shard 建立局部桶并在协调节点归并。高基数字段、大时间范围、过多 Shard、嵌套聚合和过大的 `size` 会放大内存与网络。

治理方法：

- 先通过 filter 和时间范围缩小数据集。
- 在 keyword/numeric/date 字段上聚合。
- 限制桶数量和层级，使用 composite aggregation 做可分页桶遍历。
- 报表与在线搜索隔离索引、节点或计算链路。
- 对固定历史数据利用请求缓存或预聚合。

### 31. 如何用 Profile、Explain 和慢日志定位查询问题？

- 慢日志先找到具体索引和 DSL。
- Profile 分析查询在各 Lucene 阶段的耗时，但本身有开销，只用于受控诊断。
- Explain 查看单文档为什么得到某个评分，不用于大规模性能分析。
- 同时结合 `_nodes/hot_threads`、线程池拒绝、Shard 分布、Heap、GC 和磁盘延迟判断是否为资源问题。

### 32. 商品或内容搜索如何设计？

1. 主库保存商品、内容和库存等权威数据。
2. ES 文档冗余标题、类目、品牌、标签、状态、时间和展示字段。
3. Analyzer 负责召回；filter 负责租户、状态、权限和上下架。
4. 排序综合相关性、质量、新鲜度和业务规则。
5. 库存、价格等高频强一致字段可在召回后回查，避免持续打写 ES。
6. 无结果时做纠错、同义词、热门内容或类目降级。

## 六、写入、Refresh、Translog、Merge 与数据同步

### 33. refresh、flush 和 merge 分别解决什么？

| 机制 | 主要作用 | 核心代价 |
|---|---|---|
| Refresh | 将内存缓冲形成新 Segment，使文档可搜索 | 频繁执行产生大量小 Segment |
| Flush | 建立持久化提交点并轮转/清理 Translog | 磁盘 IO 和短时资源消耗 |
| Merge | 合并 Segment、回收删除空间、减少查询开销 | CPU、磁盘 IO 和写放大 |

大批量导入可临时调大 refresh interval，控制 Replica 和 Bulk 并发；导入完成后恢复设置并校验，不能把临时配置遗留在线上。

### 34. ES 如何保证单次写入可靠性？

- Primary 对操作排序并写入本地结构与 Translog。
- 操作复制到 Replica，确认策略决定返回时需要多少活跃副本。
- Translog 支持节点崩溃后的操作恢复。
- Primary 故障后由满足条件的副本接管。

这些机制只保证 ES 集群内部的可靠性边界，不保证 MySQL/MongoDB 与 ES 之间的原子一致，也不能替代 Snapshot。

### 35. Bulk API 如何正确使用？

- 以“批次字节数、单条大小、并发数、服务端拒绝率”共同调优，不能只追求条数。
- HTTP 200 不代表每个 item 成功，必须逐条解析失败项。
- 对 429、临时网络错误做有界指数退避；Mapping 错误等永久失败进入隔离队列。
- 文档 ID 使用稳定业务 ID，更新操作必须幂等。
- 生产者速度超过 ES 承载能力时要背压，不能无限堆积内存。

### 36. 更新和删除为什么会产生写放大？

Lucene Segment 不可变。更新通常是“标记旧文档删除 + 写入新文档”，删除也先记录删除标记，空间在后续 merge 才回收。因此高频更新、大文档和嵌套文档会放大磁盘、Merge 和网络成本。

设计上应减少无意义更新、拆分高频变化字段、合并批量变更，并避免把实时计数器直接放入 ES。

### 37. MySQL/MongoDB 到 ES 如何保证最终一致？

推荐链路：

```text
业务事务提交权威数据
  -> Outbox 或 CDC 可靠产生事件
  -> Kafka 持久化、削峰和重放
  -> ES Consumer 按业务 ID 幂等 upsert/delete
  -> 失败重试 + 隔离队列
  -> 定时对账 + 全量重建
```

关键点：

- 消息带实体 ID、事件 ID、业务版本、操作类型和发生时间。
- 消费者可回查权威源，避免旧消息携带的快照覆盖新状态。
- 使用外部版本或业务版本拒绝乱序旧写。
- 删除 404 可按目标状态视为幂等成功。
- 对账比较数量、版本、关键字段和抽样业务结果。

### 38. 为什么不建议业务线程直接双写 DB 和 ES？

DB 成功而 ES 失败、ES 成功而 DB 回滚、网络超时结果未知都会产生不一致。跨两个系统无法靠普通本地事务原子提交。

如果规模较小只能同步双写，也应以 DB 为事实源，记录待同步状态或 Outbox，并允许异步补偿；不能把一次 HTTP 成功当作最终一致性证明。

### 39. 如何安全地重建索引？

1. 创建带版本的新索引和明确 Mapping。
2. 从权威源按稳定游标全量导入，限速并逐条检查 Bulk 失败。
3. 同时记录或消费增量变更，解决全量期间的数据变化。
4. 校验总量、版本、关键字段、抽样搜索和性能。
5. 原子切换 Alias，灰度观察并保留回切路径。
6. 稳定后再删除旧索引，删除前确认 Snapshot 和保留周期。

## 七、集群、分片、副本、高可用与灾备

### 40. 分片数量如何设计？

分片数同时决定并行度、单分片大小、集群元数据、恢复速度和查询扇出。

评估维度：

- 预计主分片数据量和增长周期。
- 单分片搜索与写入吞吐压测结果。
- 节点故障后副本恢复时间。
- 单次查询命中的 Shard 数。
- 租户或时间维度的路由与生命周期。

原则是先通过真实数据压测得到单 Shard 能力，再反推数量；不要迷信固定 GB 经验值，也不要为未来无限预留小分片。

### 41. 分片过多和过少分别有什么问题？

**过多：** 集群状态和 Heap 开销大、查询广播与归并重、线程池竞争、恢复任务多、小 Segment 多。

**过少：** 单 Shard 过大、迁移恢复慢、并行度不足、热点难以拆分，达到上限后扩展困难。

治理包括 rollover、shrink/split、reindex、按时间或租户拆索引，但每种操作都要评估停机、磁盘和双写成本。

### 42. 节点角色如何规划？

- Master-eligible 节点负责集群元数据和选主，不应承担重查询。
- Data 节点持有 Shard 并执行读写，可按热温冷数据分层。
- Ingest 节点执行预处理管道，复杂处理应考虑独立计算链路。
- Coordinating-only 节点可隔离请求归并，但不是无限扩容的万能层。

小集群不必过度拆角色；规模扩大后再根据资源隔离和故障域规划。

### 43. ES 如何处理节点故障和脑裂风险？

集群通过 Master 维护成员与 Shard Allocation，Primary 丢失后从合格 Replica 中提升新 Primary，并重新补齐副本。生产上应：

- 部署奇数个独立故障域的 Master-eligible 节点。
- 配置合理发现机制，不允许不同环境误组成集群。
- 通过 allocation awareness 分散 Primary 和 Replica。
- 网络分区期间优先保证多数派安全，避免双主写入。
- 恢复后检查未分配 Shard、数据路径和版本兼容性。

### 44. Yellow、Red 分别意味着什么？

- Yellow：所有 Primary 已分配，但至少一个 Replica 未分配，通常仍可服务但容灾下降。
- Red：至少一个 Primary 未分配，相关数据不可用或不完整。

排查不能停在颜色，要用 allocation explain 找到具体原因：节点离线、磁盘水位、分配规则、版本、损坏、同故障域约束或副本数超过节点数。

### 45. Snapshot 为什么仍然必要？

Replica 只能抵抗节点故障，无法抵抗误删除、错误更新、逻辑污染和整个集群损坏。Snapshot 应存放在独立可靠仓库，并定期做恢复演练。

灾备设计要明确 RPO、RTO、索引优先级、恢复顺序、密钥权限和跨区域带宽。若 ES 可从主库重建，也要估算全量重建耗时是否满足 RTO。

### 46. ILM、Rollover 和数据分层解决什么问题？

对日志和时序数据，可按大小、文档数或时间 rollover，随后进入热、温、冷、冻结和删除阶段。它将容量治理制度化，避免单索引无限增长。

业务搜索索引未必适合纯时间切分；应根据查询是否跨周期、更新是否频繁和 Alias 路由决定。

## 八、性能优化、容量规划与 Golang 工程实践

### 47. ES 容量如何估算？

```text
原始数据量
  x 索引膨胀系数
  x（1 + Replica 数）
  x 增长周期
  + Merge/恢复临时空间
  + 安全余量
```

索引膨胀系数必须用真实 Mapping、Analyzer 和样本导入测量。节点规划还要同时满足：

- 磁盘水位与故障恢复余量。
- 单节点 Heap 和操作系统页缓存。
- 峰值搜索、写入、Merge 和 Snapshot IO。
- N-1 节点故障下的容量和 SLA。

### 48. Heap 为什么重要，为什么也不能给得无限大？

Heap 承担集群元数据、查询、聚合、缓存和索引缓冲；过小会频繁 GC 或触发 Circuit Breaker。过大则延长 GC，并挤占 Lucene 依赖的操作系统页缓存。

应根据节点角色和实测调优，重点观察 Heap 使用趋势、老年代、GC 暂停、Breaker、字段数量、分片数量和高基数聚合，而不是只看某一刻百分比。

### 49. 如何系统优化查询性能？

按层次处理：

1. **业务层：** 缩小时间和租户范围，限制分页、聚合和返回字段。
2. **DSL 层：** filter 替代无意义评分，避免脚本、通配前缀和复杂 nested。
3. **Mapping 层：** 正确字段类型、Doc Values、冗余查询字段。
4. **Shard 层：** 减少查询扇出，修复热点和不合理分片。
5. **资源层：** 检查线程池、CPU、Heap、GC、磁盘和网络。
6. **架构层：** 搜索与报表隔离、预计算、缓存或独立集群。

### 50. 如何系统优化写入性能？

- 使用合适 Bulk 大小和有限并发。
- 批量导入时降低 refresh 频率，必要时临时调整 Replica，完成后恢复。
- 简化 Analyzer、Ingest Pipeline、脚本和高成本 Mapping。
- 减少大文档、高频局部更新和 nested 更新。
- 保证磁盘低延迟、节点负载均衡和足够 Merge 余量。
- 监控 write/bulk 线程池拒绝、Merge、Refresh、Translog、GC 和磁盘水位。

### 51. Golang ES 客户端如何治理？

- 所有请求使用 `context` deadline，禁止无界等待。
- 复用 HTTP Transport，配置连接池、空闲连接和 TLS。
- 对查询与写入设置不同超时、并发和重试策略。
- 只对明确可重试且幂等的请求退避重试，避免重试风暴。
- Bulk Worker 必须有有界队列、背压、逐项错误处理和优雅停机。
- 记录 index、operation、status、took、attempt、trace_id，但避免打印敏感全文和巨大 DSL。

### 52. 如何对 ES 做压测？

使用接近生产的数据分布、Mapping 和查询混合，不只测单一简单 DSL。分别测稳态、峰值、批量导入、节点故障和数据恢复。

输出至少包括吞吐、P50/P95/P99、超时率、拒绝率、CPU、Heap、GC、磁盘延迟、Merge、缓存命中、Shard 热点和 N-1 场景结果。

## 九、生产治理、安全与故障排查

### 53. ES 监控体系应该覆盖什么？

| 层次 | 核心指标 |
|---|---|
| 业务 | 搜索成功率、无结果率、结果新鲜度、同步延迟 |
| 接口 | QPS、P95/P99、超时、错误码、降级率 |
| 查询 | 慢日志、目标 Shard、分页深度、聚合桶、返回大小 |
| 写入 | Bulk 成功率、逐项失败、吞吐、积压、重试 |
| 集群 | Health、未分配 Shard、节点数、主节点变化 |
| 资源 | CPU、Heap、GC、磁盘水位/延迟、网络、线程池拒绝 |
| 数据 | 主库/ES 数量、版本、关键字段和删除状态差异 |

### 54. ES 查询慢或接口超时如何排查？

1. 确认影响范围、开始时间、索引、租户、DSL 和 trace。
2. 判断 ES span 是否真的慢，排除客户端连接池、网络和业务等待。
3. 检查慢日志、分页、聚合、脚本、nested、返回字段和查询 Shard 数。
4. 检查 CPU、Heap、GC、线程池拒绝、热点线程、磁盘和 Segment Merge。
5. 对比发布、流量、索引重建、Kafka 重放和报表任务时间线。
6. 先限流重查询、关闭非核心聚合、缩小范围或走降级，再做 Mapping/索引改造。

### 55. 写入慢、Bulk 失败或同步积压如何排查？

先把链路拆开：事件产生 -> Kafka lag -> Consumer -> Bulk -> Primary/Replica -> Refresh/Merge。

重点检查：Bulk item 错误、429 拒绝、Mapping 冲突、批次和并发、refresh、Replica、磁盘水位、Merge、GC、热点 Shard。止血时降低消费者并发、暂停全量重建和低优先级同步，保护权威主库，不能盲目加消费者把 ES 打垮。

### 56. Heap、GC、Circuit Breaker 或 OOM 如何处理？

立即限制高基数聚合、深分页、大结果集和脚本查询，暂停大规模导入。然后定位是字段/分片膨胀、聚合桶、缓存、Ingest、Mapping Explosion 还是持续 Merge。

扩 Heap 只能暂时缓解；根治通常是缩小查询、修正 Mapping、减少 Shard、隔离负载或扩 Data Node，同时给操作系统保留页缓存。

### 57. ES 与主库数据不一致怎么查？

```text
主库是否提交成功？
  -> Outbox/CDC 是否产生事件？
  -> Kafka 是否积压或丢在错误 Topic？
  -> Consumer 是否失败、乱序或被永久错误阻塞？
  -> ES Bulk item 是否失败？
  -> Alias 是否指向正确索引？
  -> 查询是否因 refresh 尚不可见？
```

恢复时以主库为准，按版本幂等回放或局部重建；大规模回灌必须限速。修复后补对账、告警和故障注入测试。

### 58. ES 安全和多租户如何治理？

- 强制认证、TLS、最小权限和凭证轮换。
- 禁止业务账号拥有删除集群、修改安全设置等高危权限。
- 租户隔离可使用共享索引 + tenant_id、按大租户独立索引或独立集群，依据规模和合规选择。
- 应用层必须强制注入租户过滤，不能依赖前端传参自觉。
- 限制查询复杂度、结果窗口、脚本、通配索引和批量导出。
- 审计索引删除、Alias 切换、Snapshot 恢复和权限变更。

### 59. 生产故障的标准止血与复盘框架是什么？

**止血顺序：** 控制入口 -> 隔离重请求 -> 保护主库和 ES -> 恢复关键 Shard/节点 -> 分批恢复同步 -> 校验业务结果。

**证据最小集：** 时间线、Index/Alias、Shard、节点、DSL、业务参数、错误响应、trace、线程池、Heap/GC、磁盘、发布和上下游指标。

**复盘输出：** 直接原因、触发条件、放大机制、为何未提前发现、恢复为何耗时，以及监控、容量、限流、变更流程和演练的改进项。

## 十、业务架构与项目实战

### 60. 广告素材库为什么引入 ES？

**背景：** 素材和广告任务需要按名称、项目、类型、媒体、状态、创建人和时间组合查询，并支持关键词召回和排序。继续在 MongoDB/MySQL 堆叠条件会扩大主库索引和资源压力。

**方案：**

- MySQL/MongoDB 保留权威业务数据。
- Kafka 传播变更，ES 保存列表检索字段和必要展示字段。
- `resource_name` 使用 text + keyword，ID、状态、类型和租户使用 keyword，时间使用 date。
- 查询强制租户和权限过滤，详情回查主库。
- ES 故障时保留基础列表或热门结果降级，不影响素材上传与投放主链路。

**价值：** 查询能力和主库稳定性都得到提升，同时承认同步延迟和额外治理成本。

### 61. MongoDB/MySQL 到 ES 的同步项目如何表达？

先讲问题：直接双写存在失败窗口、消息重复乱序和删除遗漏。

再讲方案：主库提交后由 Outbox/CDC 产生 Kafka 事件，消费者按实体 ID 回查最新状态，通过 Bulk 幂等 upsert/delete；事件包含业务版本，旧版本拒绝覆盖；永久失败进入隔离队列，对账任务扫描版本差异。

最后讲指标：同步 P99 延迟、Bulk item 失败率、Kafka lag、对账差异数、重建耗时和故障恢复 RTO。

### 62. ES 查询灰度迁移如何设计？

1. 保留 Mongo/MySQL 与 ES 双查询路径。
2. 通过用户、租户或流量比例灰度。
3. 影子请求对比数量、关键 ID、排序和耗时，不把影子结果返回用户。
4. 对可接受的近实时差异设置窗口，避免误报。
5. 监控 ES 错误和同步延迟，支持快速回切。
6. 全量后保留一段观察期，再下线旧查询和冗余索引。

这体现的是迁移能力，而不只是“会写 DSL”。

### 63. 如何设计一个高可用的内容搜索服务？

- API 层做参数校验、租户权限、复杂度限制和 deadline。
- Query Builder 将业务条件映射为稳定 DSL，禁止任意透传。
- ES 索引按内容类型和生命周期规划，热点查询可短时缓存。
- 排序融合相关性、时效、质量和风控状态。
- ES 超时时返回热门内容、类目结果或上次成功缓存。
- 主库写入不依赖 ES；同步链路可重试、可对账、可重建。
- 以 P99、无结果率、点击率、同步延迟和降级率评估系统。

### 64. 如何回答“你在 ES 项目中解决过什么线上问题”？

使用 STAR + 技术闭环：

- **场景：** 某运营列表 P99 上升并伴随 ES CPU 飙高。
- **任务：** 在不影响投放主链路的情况下恢复查询并定位根因。
- **行动：** 用 trace 和慢日志定位到深分页叠加高基数聚合；先限制分页、关闭非核心聚合并按租户限流，再用 search_after、预聚合和索引字段改造；补充 DSL 指纹和桶数量监控。
- **结果：** 用真实数字说明 P99、错误率、资源使用和最大可承载 QPS 的改善。
- **复盘：** 说明为什么原容量模型漏掉该查询，以及如何通过评审、压测和预算防止复发。

### 65. 如何用两分钟回答 ES 架构题？

> 我把 ES 定位为可重建的搜索读模型，而不是交易主库。建模时从查询矩阵出发，明确全文、过滤、排序、聚合和展示字段，用 text/keyword、nested 和 Analyzer 控制召回与成本。写入上由 MySQL 或 MongoDB 先提交，通过 Outbox/CDC 和 Kafka 可靠传播，消费者按业务 ID 与版本幂等更新 ES，并用对账和新索引加 Alias 支持重建。查询链路重点控制分片扇出、深分页、高基数聚合和返回体；集群按真实压测规划 Shard、Replica、Heap、磁盘和 N-1 容量。线上通过业务指标、慢日志、线程池、Heap/GC、磁盘和同步延迟分层监控，故障时限流重请求、保护主库、分批恢复并校验数据。项目效果最终用搜索 P99、成功率、同步延迟、差异率和业务转化来衡量。
