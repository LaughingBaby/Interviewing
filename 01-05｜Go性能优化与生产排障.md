# 技术｜Go 性能优化与生产排障面试题

> 面向 9 年以上 Golang Web 后端业务研发岗位。本篇不把 pprof 当答案本身，而是建立从业务现象、系统指标、Runtime 证据到代码热点的完整诊断链，并强调止血、恢复和防复发。

---

## 一、生产诊断地图、SLO 与排障框架

### 1. Go 生产排障的知识主线是什么？

```text
业务影响与时间线
  -> RED / USE / Business Invariant
  -> CPU / Heap / Goroutine / Block / Mutex / Trace
  -> HTTP/RPC / Pool / DB / Redis / MQ
  -> Code / Config / Release / Traffic
  -> Limit / Degrade / Rollback / Recover
  -> Load Test / Capacity / Postmortem
```

### 2. 标准排障六步是什么？

**结论先行：标准排障不是从日志里猜根因，而是按“影响—证据—假设—止血—修复—验证”建立可证伪的闭环。**

1. **定义影响：**确认开始时间、错误率/P99、业务量、租户、区域、版本以及数据正确性。
2. **保存证据：**记录时间线，保留 Metrics、日志、Trace、Profile、发布和配置 Diff；重启前先判断是否会丢失现场。
3. **建立假设：**把问题归到入口过载、本地 CPU/内存/锁、资源池、网络或下游，并记录支持和反驳证据。
4. **安全止血：**选择限流、降级、回滚、隔离或扩容，明确压力会转移到哪里、如何撤销。
5. **根因修复：**用最小改动验证单一假设，补回归测试、资源上限和观测。
6. **恢复复盘：**小流量探测、渐进放流、积压限速回放、业务对账，最后形成有 Owner 的行动项。

排障过程中持续更新时间线和假设状态，避免换班或跨团队协作时重复试错。

**面试一句话：**我先量化影响并保存现场，再用证据提出可证伪假设；止血必须可回滚且不转嫁压力，修复后用灰度、长稳和业务对账证明恢复。

### 3. 为什么先看业务 SLI 而不是机器指标？

CPU 高可能是健康高吞吐，CPU 低也可能全部等待下游。先看成功率、P99、吞吐、积压、数据正确性和用户影响，再用资源指标解释原因。

### 4. RED 和 USE 分别是什么？

- RED：请求 Rate、Error、Duration。
- USE：资源 Utilization、Saturation、Errors。

Go 服务还要关注 Goroutine、GC、Heap Goal、线程、FD、池等待和队列。饱和比使用率更能解释排队与尾延迟。

### 5. 如何建立故障时间线？

对齐告警开始、发布、配置、流量、扩容、定时任务、GC、依赖异常和人工操作。所有事件使用统一时区；保留新旧版本差异和第一条异常，而不是从大量重复错误开始猜。

### 6. 止血时最容易犯什么错？

盲目重启丢失证据、无限扩容打爆下游、调大超时增加在途、开启更多重试放大流量、未经确认清缓存/重置 Offset。每个动作先判断压力会转移到哪里和如何回滚。

## 二、Metrics、Logs、Trace、pprof 与 Runtime Trace

### 7. Go 服务最小监控集是什么？

- 请求 QPS、成功率、状态码、P50/P95/P99。
- Goroutine、CPU、RSS、Heap、GC CPU/Pause。
- HTTP/gRPC 连接与请求阶段。
- DB/Redis 等 Pool InUse/Wait/Timeout。
- MQ Lag、Queue Length、Reject、Oldest Age。
- 发布版本、实例、租户和业务不变量。

### 8. 常用 pprof Profile 分别解决什么？

- CPU：采样执行热点。
- Heap：当前存活与历史分配热点。
- Goroutine：当前栈和等待状态。
- Block：Channel、Select、锁等阻塞累计。
- Mutex：锁竞争等待。
- Threadcreate：线程创建来源。

| 现象 | 首选 Profile | 关键视角 |
|---|---|---|
| CPU 高或吞吐下降 | CPU | `cum` 看调用链累计成本，`flat` 看函数自身热点 |
| 分配率/GC CPU 高 | Heap `alloc_space/alloc_objects` | 谁持续制造短命对象 |
| GC 后存活堆上涨 | Heap `inuse_space/inuse_objects` | 谁持有仍可达对象 |
| Goroutine 持续增长 | Goroutine | Stack 聚类、等待状态和退出路径 |
| Channel/锁等待 | Block | 累计阻塞位置，注意采样率 |
| Mutex 竞争 | Mutex | 持锁方造成的等待累计 |

Profile 是采样窗口内的证据，不自动等于根因。必须记录版本、流量和采样时段，并与正常基线或修复后 Profile 比较。Runtime Trace 更适合解释调度时间线和并行度问题。

**面试一句话：**CPU 看执行热点，alloc 看累计分配，inuse 看当前滞留，Goroutine/Block/Mutex 看等待与竞争；选择 Profile 要由现象驱动，并和同负载基线比较。

### 9. `alloc_space` 和 `inuse_space` 有什么区别？

alloc_space 看累计分配，适合高分配/GC 压力；inuse_space 看当前仍存活，适合内存占用和滞留。一个函数累计分配很高但对象短命，未必是泄漏；存活持续增长更值得追踪。

### 10. pprof 的 Top、List、Web 和 Diff 如何使用？

Top 找主要成本，List 映射源码行，Web/火焰图看调用路径，Diff 比较变更前后或故障/正常。分析必须带采样时段、流量和版本，不能拿不同负载 Profile 直接下结论。

### 11. 生产开启 pprof 有什么风险？

暴露内部路径和 Stack；CPU/Trace/高采样 Block 产生开销；Heap Dump 可能很大并含敏感数据。应内网鉴权、短时采集、限制并发、保存到安全位置，不直接公网开放。

### 12. Runtime Trace 什么时候比 pprof 更有用？

CPU 不高但延迟大、Goroutine 长期等待、并行度不足、Syscall/Netpoller/GC 交互复杂时。它展示调度时间线；热点计算仍用 CPU Profile，存活对象用 Heap。

### 13. 分布式 Trace 与 Runtime Trace 有什么区别？

分布式 Trace 看跨服务业务调用和依赖耗时；Runtime Trace 看单进程调度、阻塞、GC 和 P。先用前者找到慢服务，再用后者解释进程内部为什么慢。

### 14. 日志怎样帮助性能诊断？

结构化记录 operation、duration、queue_wait、pool_wait、attempt、dependency、result、version 和 Trace ID。慢日志按采样/阈值记录，不在高峰打印完整请求和大对象导致二次故障。

## 三、Benchmark、压测、基线与性能方法论

### 15. 性能优化的正确顺序是什么？

**结论先行：性能优化的核心是用数据证明瓶颈在哪里、改动为何有效，以及是否把成本转移到了别处。**

```text
业务目标/SLO -> 可重复基线 -> Profile 定位 -> 单一假设
  -> 最小改动 -> Benchmark -> 端到端压测 -> 灰度 -> 回归门禁
```

目标应具体到请求模型、实例规格、P99、错误率和成本边界。局部 Benchmark 变快不代表服务变快：减少 CPU 可能增加内存，批量变大可能提高吞吐却恶化尾延迟，缓存可能把压力转移到 GC。因此验证需同时覆盖 CPU、B/op、allocs/op、RSS、GC、吞吐和 P99。

**面试一句话：**我先定义业务 SLO 和基线，用 Profile 找热点，再用最小改动验证假设；Benchmark 证明局部，压测与灰度证明系统收益，最后固化回归门禁。

### 16. 微基准容易犯哪些错误？

包含 Setup、结果被编译消除、数据过小、只跑一次、并行度失真、缓存热度与生产不同、忽略分配和方差。使用 `benchstat` 等方式多次比较，并固定环境。

### 17. 如何设计服务压测？

使用真实请求比例、对象大小、热点分布和依赖延迟；覆盖稳态、突发、长稳、阶梯、故障和恢复；记录入口与下游全部指标。压测环境容量和网络要已知，不能把压测机瓶颈归因服务。

### 18. 开环和闭环压测有什么区别？

闭环客户端等待响应再发，请求变慢时自动降低到达率，可能掩盖过载；开环按目标速率到达，更能观察排队和崩溃点。两者结合，报告排队和协调遗漏问题。

### 19. 为什么必须看 P99？

平均值掩盖排队、GC、锁和慢依赖。多级串行/扇出系统中尾延迟会累积；P99 对用户超时和容量更相关。同时给出样本量，低 QPS P99 波动大。

### 20. 如何建立性能回归门禁？

对稳定核心函数保存 Benchmark，对关键接口保存同环境负载基线；设置合理容差；性能敏感变更附 Profile 和前后数据。微小噪声不阻塞，显著回退需解释。

### 21. PGO 适合什么时候使用？

有代表性 CPU Profile、CPU 热路径稳定、编译与部署可复现时。PGO 可能改善内联和布局，但不替代算法、分配和 IO 优化；升级 Go 或流量变化后重新评估。

## 四、CPU 飙高、热点函数、GC 与锁竞争

### 22. Go 服务 CPU 飙高如何排查？

**结论先行：先判断 CPU 上升是流量驱动、单位请求成本上升、GC/系统开销，还是 Cgroup Throttling，再选择 Profile 和止血动作。**

1. 对齐异常时间，比较单实例与全局、版本、QPS、请求类型和数据大小。
2. 拆 User/System/Throttled Time，检查 GC CPU、分配率、Goroutine 和 Run Queue。
3. 在异常窗口采集 CPU Profile，并与正常同负载基线 Diff；用 Top 找主成本、List/火焰图定位调用路径。
4. 若 Go Profile 解释不了高 System CPU，继续检查 Syscall、小包网络、日志 IO、Futex、Page Fault、线程和 CGO。
5. 验证错误重试、序列化/压缩、正则、热循环、锁自旋、后台任务和日志暴增等放大源。

止血可限流、关闭非核心计算、回滚或扩容，但扩容前确认数据库和第三方容量；若瓶颈是重试风暴，扩容会继续放大下游流量。修复后在相同请求模型下比较 CPU/request、P99 和 Profile。

**面试一句话：**CPU 排查先区分流量、单位成本、GC、System 和 Throttling，再用同窗口 Profile 建证据链；修复后以 CPU/request、P99 和热点消失共同验证。

### 23. CPU 高但 QPS 没涨有哪些原因？

新版本算法退化、错误重试、日志暴增、GC 因分配上升、锁竞争/自旋、后台任务、健康检查、数据分布变化和死循环。用版本、操作维度和 Profile 区分。

### 24. System CPU 高说明什么？

可能大量 Syscall、网络、小包、文件日志、线程调度、锁 Futex、Page Fault 或 CGO。结合 syscall/网络指标、Thread、Trace 和系统工具，不只看 Go CPU Profile。

### 25. JSON/Proto 序列化为什么可能成为热点？

大对象、反射、重复转换、无用字段、Buffer 分配和日志二次序列化。优先减少响应字段和次数、分页/批量、缓存结果，再考虑代码生成或 Pool。

### 26. GC CPU 高如何处理？

判断是分配速率高还是活对象大：alloc Profile、Heap Goal、GC 周期和 Mark Assist。减少临时对象、反复 String/Byte、过大 Batch 和高指针结构；限制缓存；谨慎 Pool；最后调整 GOGC/Memory Limit。

### 27. 锁竞争如何定位？

Mutex Profile 找等待累计热点，Block Profile/Trace 看阻塞，CPU Profile 看自旋；检查临界区内 IO、全局锁、热点 Key 和读写锁误用。优化可拆分状态、分片锁、缩短临界区或不可变快照。

### 28. 为什么加 RWMutex 可能更慢？

读锁本身有原子/管理成本，写者会阻塞新读，读临界区太短时并行收益不足；高核下共享计数器还可能竞争。用真实读写比例 Benchmark。

### 29. CPU Throttling 如何影响 P99？

容器达到 CPU Quota 后在周期内被限制，平均 CPU 看似未满但请求暂停，Run Queue 和延迟上升。看 Cgroup Throttled Time、Quota、GOMAXPROCS 和实例请求分布。

## 五、内存上涨、GC、对象滞留与 OOM

### 30. 内存问题必须区分哪些概念？

累计分配、当前 Heap 存活、Runtime 保留、已归还、线程栈、CGO/mmap、Page Cache 和 RSS。VSS 通常不代表物理压力；容器 OOM 看 RSS/Working Set 与 Limit。

### 31. 内存持续上涨如何排查？

**结论先行：内存上涨先区分累计分配快、GC 后活对象增长、Runtime 保留未归还、Goroutine/线程栈，还是 CGO/mmap 等非 Go 内存，再决定工具。**

| 证据 | 更可能的方向 |
|---|---|
| `alloc_space` 高、GC 后 Heap 稳定 | 短命对象多，导致 GC 压力 |
| 多轮 GC 后 `inuse_space` 仍增长 | 缓存、Map、队列、Slice 或 Closure 持有 |
| Goroutine 与 Heap 同涨 | 阻塞 Goroutine 持有栈和对象 |
| HeapAlloc 下降但 RSS 不降 | HeapIdle、线程栈或尚未 Scavenge |
| RSS 上涨而 Runtime 指标稳定 | CGO、mmap、共享库或其他外部内存 |

对齐流量、发布和 GC 周期，在多个时间点采集 Heap Profile Diff，按对象类型和调用路径寻找增长，并联查 Goroutine、缓存 Entry/Bytes、队列最老年龄和 Batch 大小。受控触发 GC 只能辅助分类，不能作为修复。

修复后要做长稳测试，证明 GC 后 Heap、RSS、Goroutine 和队列回到稳定平台，而不是只看一次下降。

**面试一句话：**我先用 GC 后水位和 Runtime/RSS 差异分类，再用多时点 Heap Diff 找持有路径；修复必须通过长稳证明水位不再随时间增长。

### 32. Go 有“内存泄漏”吗？

GC 语言仍会逻辑泄漏：对象被全局 Map、缓存、Goroutine、Timer、Closure、Slice 底层数组等可达引用持有。GC 只能回收不可达对象，不知道业务已不需要。

### 33. Slice 内存滞留怎样在 Profile 中表现？

少量小 Slice 可能使大底层数组存活，Profile 显示大 `[]byte` 分配仍 inuse，保留路径来自解析/缓存函数。长期保留小片段时 Clone/Copy，并限制单对象大小。

### 34. Map/缓存增长如何治理？

设置容量、TTL、淘汰和租户配额；监控 Entry/Bytes/Hit/Eviction；值包含大引用时清理；并发安全和定时清理不阻塞主路径。无界 `sync.Map` 仍是无界缓存。

### 35. Goroutine 泄漏为什么导致内存上涨？

每个 G 持有栈、等待对象、Closure 和调用链引用；泄漏数量增长会使 Heap 和调度元数据一起增加。Heap 与 Goroutine Profile 联合分析，不能只优化对象。

### 36. RSS 高但 HeapAlloc 低有哪些原因？

HeapIdle 尚未 Released、Goroutine/OS Thread 栈、CGO 分配、mmap、Race Detector、共享库和内核 Buffer。看 MemStats/runtime metrics、线程、CGO 指标和进程映射。

### 37. OOMKilled 前为什么不一定有 Go Panic？

内核/Cgroup 直接杀进程，Runtime 无机会输出。保留容器指标、事件、Heap/GC 趋势和退出码；设置低于 Limit 的 GOMEMLIMIT 与告警，但为非 Go 内存留余量。

### 38. 调低 GOGC 为什么可能让系统更差？

GC 更频繁，Heap 降低但 CPU/Mark Assist 增加，P99 上升；活对象过大时收益有限。若 Memory Limit 过低还会 Thrash。调参必须与流量和分配优化一起压测。

### 39. `sync.Pool` 为什么可能造成反优化？

对象 Reset 不完整导致引用滞留，异常大 Buffer 被长期复用，池操作和类型断言有成本，GC 可清空导致命中不稳定。只复用高频、构造成本明显、大小受控的临时对象。

## 六、Goroutine、Channel、锁、死锁与泄漏排查

### 40. Goroutine 数量持续上涨如何定位？

按时间抓 Goroutine Profile Diff，查看增长 Stack 和状态；对照 Queue、连接、超时和发布；找到创建点和退出条件。重点状态是 chan send/receive、select、semacquire、IO wait。

### 41. 大量 Goroutine 处于 IO Wait 正常吗？

长连接服务可能正常，但若随请求积压增长且无 Deadline，则是风险。结合活跃连接、请求数、连接池、最老等待和回落情况判断。

### 42. Channel Send 阻塞怎样处理？

确认 Receiver 是否退出、消费变慢、Buffer 是否满和 Context 是否可取消。止血降低生产、扩消费前保护下游；根治有界队列、Select ctx、Owner 关闭和队列指标。

### 43. Channel Receive 阻塞怎样处理？

确认 Sender 生命周期、是否忘记关闭、是否发送到另一个实例/Channel、nil Channel 是否被启用。Pipeline 每级必须有取消和关闭协议。

### 44. 如何排查死锁？

Goroutine Dump 看等待图，Mutex/Block Profile 找热点；检查锁顺序、锁内 Channel/IO、回调重入和 WaitGroup。全局死锁 Runtime 可能 Fatal，部分业务死锁则只表现接口挂起。

### 45. Timer/Ticker 泄漏如何识别？

高频循环创建 `time.After`、长期 Ticker 未 Stop、AfterFunc Closure 持有对象。看 Heap 分配路径、Goroutine/Timer 指标和代码；复用 Timer 并明确停止/Reset。

### 46. 如何证明修复了 Goroutine 泄漏？

构造原始触发场景长稳压测，观察请求结束后 Goroutine 回到稳定基线；Profile 中增长 Stack 消失；队列和内存回落；加可重复回归测试和泄漏阈值。

### 47. Race Detector 没报错能否证明并发正确？

不能，它只检测执行过的冲突内存访问；死锁、逻辑竞态和未覆盖路径仍存在。结合同步设计、状态机测试、压力调度和代码 Review。

## 七、HTTP/RPC、连接池、DNS 与网络故障

### 48. 接口 P99 上升如何拆阶段？

**结论先行：P99 上升要先把端到端耗时拆成排队、本地处理、资源池等待和网络/下游阶段，再判断是哪一段的分布发生变化。**

```text
入口/LB 排队 -> Middleware/Handler -> DB/HTTP/gRPC Pool Wait
  -> DNS -> Connect -> TLS -> 下游排队与处理
  -> TTFB -> Body Read -> 本地序列化与 Write
```

分布式 Trace 用于找到慢 Span，`httptrace` 和客户端指标进一步拆 DNS、Connect、TLS、连接复用与 TTFB；Runtime Trace/Block/Mutex 用于解释进程内调度与等待。还要按版本、实例、Endpoint、租户和响应大小切分，避免平均值掩盖局部热点。

P99 是结果，不是根因。若 Pool Wait 上升，继续判断是池太小、连接泄漏、事务变长还是下游慢；若下游 Span 正常，则检查入口排队、CPU Throttling、GC Assist 和响应写出。

**面试一句话：**我先用 Trace 把 P99 拆成本地、池等待和下游，再用 httptrace 或 Runtime 工具细分；每个慢阶段继续追到资源饱和或代码路径。

### 49. `context deadline exceeded` 如何排查？

检查入口原始 Deadline、当前剩余预算、在哪层创建子 Context、客户端/服务端 Span、下游是否收到取消、Pool Wait 和重试。不要直接调大超时。

### 50. HTTP 连接池耗尽有什么表现？

请求等待可用连接、P99 上升但下游 QPS未增、Goroutine 卡在 Transport 获取连接。原因包括 Body 未关闭、MaxConns 过低、下游慢和每请求建 Client。

### 51. 数据库连接池耗尽如何处理？

看 InUse、Idle、WaitCount/Duration、慢 SQL 和事务时长。短期限流重请求/回滚；根治关闭 Rows、缩短事务、优化 SQL、合理池预算。盲目加连接可能压垮数据库。

### 52. 大量 TIME_WAIT 是什么原因？

频繁短连接或未复用 Keep-Alive。检查 Client/Transport 生命周期、服务端 Connection Header、NAT/LB 和端口范围。TIME_WAIT 是 TCP 正常机制，不直接在系统层粗暴消除。

### 53. DNS 故障怎样表现？

偶发 Dial Timeout、解析耗时高、部分实例使用旧 IP。检查 Resolver、TTL、连接复用、搜索域和网络；保存分阶段 Trace。服务发现变更后要让旧连接逐步轮换。

### 54. TLS 握手慢或失败如何排查？

证书过期/链、SNI、时钟、Cipher、握手频率、CPU 和网络。复用连接降低握手；监控 Handshake Duration/Error；证书轮换前验证客户端信任。

### 55. Connection Reset、EOF 和 Broken Pipe 如何理解？

可能对端主动关闭、LB Idle Timeout、服务发布、网络中断或本方写已关闭连接。结合请求是否幂等、服务器日志和连接生命周期分类；不能一律重试写请求。

### 56. 如何排查 FD 耗尽？

看进程 FD 数/上限和类型，检查连接/文件/Body 泄漏、长连接增长和日志文件。止血限流和回收异常实例；修复 Close、池、超时并设置 FD 告警，调高上限不是根治。

## 八、依赖变慢、队列积压、过载与级联故障

### 57. 如何判断是 Go 服务慢还是下游慢？

Trace 分解本地处理、Pool Wait 和依赖 Span；服务 CPU/Queue 与下游 P99 交叉验证。客户端超时不代表下游没执行，写操作还要查询最终状态。

### 58. Redis 变慢时如何保护数据库？

入口限流、短时本地缓存/过期数据、热点请求合并、关闭非核心查询；限制回源并发，防穿透。恢复后渐进回流和预热，不瞬间让所有 Key 回源。

### 59. Kafka 积压如何计算恢复时间？

```text
净消化速率 = 当前消费速率 - 新增速率
恢复时间 = backlog / 净消化速率
```

若净值不为正，增加分区/消费者或降低生产；扩消费前确认 DB/ES 容量。看最老消息年龄，不只 Lag 数。

这里的消费速率必须是当前下游容量约束下可持续的“有效完成速率”，不是 Consumer 拉取速率。若净消化速率不为正，单纯等待永远无法恢复；需要降低生产、提高单条处理效率，或在分区数允许且 DB/ES 有余量时增加消费者。

恢复计划还要考虑昼夜流量、失败重试、Poison Message、Rebalance 和消息成本差异。恢复时限制回放并发，避免积压流量与实时流量共同打爆下游。

**面试一句话：**恢复时间等于积压除以“可持续消费速率减新增速率”；净速率不为正就无法恢复，扩消费前必须验证分区和下游余量，并用最老消息年龄衡量真实影响。

### 60. Poison Message 如何影响 Go Consumer？

同一消息快速失败重试，占满 Worker/Partition。分类永久错误，有限重试后隔离到 DLQ，记录业务 ID 和 Cause；不能跳过后假装业务完整，需人工/补偿流程。

### 61. 重试风暴如何识别？

入口 QPS 未变但下游调用和 attempt 增长，错误与延迟同步恶化。按 attempt 指标和 Trace 识别多层重试；止血关闭/降低重试，启用熔断和 Retry Budget。

### 62. 如何识别过载而不是单纯慢代码？

到达率超过服务率，Queue/Pool Wait 和在途持续增长，P99 先升后错误；CPU/下游可能饱和。过载需要 Admission Control 和 Backpressure，仅优化一个函数不够。

### 63. 级联故障的止血顺序是什么？

**结论先行：级联故障止血的目标是先阻断放大环，再保护最难恢复的事实数据和核心写链路。**

1. 在最靠近入口处做 Admission Control，限制新增在途。
2. 关闭或压低多层重试，阻断一次请求变成多次下游调用。
3. 熔断已经慢或失败的依赖，让线程、连接和 Deadline 尽快释放。
4. 降级非核心读、推荐和报表等功能，保留核心状态写入。
5. 隔离租户、平台或优先级，避免一个故障域耗尽全部资源。
6. 保护数据库、队列等事实存储，暂停危险回放和批任务。
7. 从底层依赖向上游小流量恢复，限制缓存回源、积压回放和重试速度。

每个动作都要观察错误率、在途、队列、连接池和下游负载。盲目扩容、调大超时或全量重启，可能让更多请求同时压向已经过载的下游。

**面试一句话：**级联故障先控入口和重试放大，再熔断、降级与隔离，优先保护事实存储；恢复按依赖从下到上渐进放流。

### 64. 第三方 API 全面异常怎么办？

按平台熔断并隔离，任务持久排队而非占 Goroutine；用户看到“处理中/平台异常”；暂停快速重试；监控配额和恢复；平台恢复后按账号限速回放，防二次限流。

## 九、容量、发布、恢复、复盘与稳定性建设

### 65. Go 服务实例容量如何估算？

单实例安全 QPS 来自真实混合压测并受 CPU、内存、连接和下游限制；实例数按峰值/安全 QPS乘冗余，并验证 N-1。Little's Law 用 `并发≈QPS×延迟` 估算连接和 Goroutine。

### 66. 如何做长稳压测？

持续数小时/天保持代表性流量，观察 GC 后 Heap、RSS、Goroutine、FD、连接、Queue 和 P99 是否稳定；周期任务和缓存淘汰必须覆盖。短峰值压测发现不了泄漏。

### 67. 灰度发布监控哪些指标？

按 version 比较 QPS、错误码、P99、CPU、RSS、GC、Goroutine、池等待、依赖调用倍数和业务转化；设置自动停止阈值。灰度样本量不足时延长而非盲目全量。

### 68. 发布后内存缓慢上涨如何处理？

冻结放量，比较新旧 Heap/Goroutine Diff 和代码变更；若接近 Limit 先回滚保存 Profile；判断缓存预热还是持续增长。回滚后确认对象和实例是否恢复，不只看错误率。

### 69. 服务恢复后为什么不能立即全量放流？

缓存冷、连接未建、积压待消化、GC/IO 仍高。小比例探测，预热，限制积压回放和重试，逐级放量；监控恢复斜率和业务一致性。

### 70. 一份有效 Postmortem 包含什么？

**结论先行：有效 Postmortem 的目的不是找责任人，而是解释系统为何允许故障发生、扩大并延迟恢复，并把结论转成可验证的改进。**

- **影响：**用户、业务、数据、区域、持续时间和 SLO 消耗。
- **时间线：**统一时区记录发布、首个异常、检测、决策、止血和恢复。
- **根因链：**区分触发条件、直接根因、放大机制和潜伏条件。
- **检测与响应：**为什么告警早或晚，哪些证据缺失，哪些动作有效或有副作用。
- **恢复验证：**流量、积压、数据一致性和业务指标如何确认正常。
- **行动项：**每项有 Owner、期限、优先级和验收指标，覆盖预防、检测、缓解与恢复。

“某开发写了 Bug”不是完整根因，还要回答测试、Review、灰度、容量、限流或回滚为什么没有挡住。行动项避免只写“加强测试/提高意识”，而应落到具体门禁、告警、资源上限或演练。

**面试一句话：**Postmortem 要还原影响与时间线，拆出触发、根因和放大机制，复盘检测与恢复，并把改进落实为有 Owner、期限和验收指标的系统行动项。

### 71. 如何建立 Go 生产运行手册？

每类告警写确认命令、Dashboard、Profile 采集、止血选项、风险、升级联系人、恢复和数据校验。命令定期演练，链接和权限可用；不把危险操作作为第一步。

### 72. 稳定性建设优先级怎样排？

先核心 SLO和业务不变量，再 Deadline/有界资源/幂等，之后监控与灰度，最后自动扩缩和高级调优。没有基础限流和回滚时先上复杂自愈通常风险更大。

## 十、真实项目、优化案例与资深表达

### 73. 素材同步服务内存突增如何回答？

背景是全量补偿一次加载大量素材并每条启动 Goroutine。先限制作业和回滚；Heap/Goroutine Profile 证明大 Slice 与在途任务；改为游标分页、有界 Worker、按字节 Bulk、任务状态落库；用峰值 RSS、GC CPU、P99 和完成时间对比。

### 74. 外部 SDK 造成 CPU 和 Race 如何回答？

发布后 CPU 高且偶发并发异常。Race 定位共享 Map，CPU Profile 显示锁/重试或序列化；Adapter 隔离 SDK，账号级实例/锁，限制重试，推动升级；回归运行 Race 和负载测试，并按平台监控。

### 75. Kafka Consumer 积压如何回答？

拆分 Poll、业务处理、DB/ES Bulk 和 Offset；发现无限并发导致 DB Pool Wait，吞吐反而下降。按 Partition/Key 建有界 Worker，批量写，限制 DB 并发，毒消息隔离；计算净消化速率，渐进恢复并验证幂等。

### 76. 如何表达一次 CPU 优化？

必须给证据链：业务 P99/QPS -> CPU Profile 热点 -> 假设 -> Benchmark -> 改动 -> 压测/灰度。例：减少重复 JSON 序列化和无用字段后，说明 CPU、B/op、P99 和实例数变化，而不是只说“用了 sync.Pool”。

### 77. 如何表达一次内存优化？

区分累计分配和存活；Heap Diff 找到大 Slice/缓存/Goroutine持有；通过分页、Clone 小片段、缓存上限和取消修复；展示 GC 后 Heap、RSS、Goroutine、GC CPU 和 OOM 余量。

### 78. 如何用两分钟回答 Go 线上排障？

> 我先确认业务影响、时间线和发布/流量变化，同时保存 Trace、日志、Runtime 和依赖证据；根据风险限流、降级或回滚，避免盲目重启和重试。接口慢先用 Trace 拆本地、池等待和下游，再按现象选工具：CPU Profile 查计算，Heap 的 alloc/inuse 区分分配与滞留，Goroutine/Block/Mutex 查泄漏和竞争，Runtime Trace 查调度、IO 和 GC。内存还要区分 Heap、Runtime保留、CGO 与 RSS；连接问题看 DNS、建连、TLS、Body Close 和 Pool Wait。修复后长稳和灰度验证 P99、资源、队列和业务不变量，渐进恢复积压。最后复盘触发与放大机制，把改进落到容量、门禁、监控、预案和故障演练。
