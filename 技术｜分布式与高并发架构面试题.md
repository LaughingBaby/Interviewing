## 一、分布式当前能力全景：一致性、并发、稳定性与容量

> 先掌握分布式系统常见问题域：一致性、高并发、幂等、限流降级、容量和可观测性。

### 1. CAP 和 BASE 怎么理解？

**结论先行：**
CAP 描述分布式系统在网络分区下无法同时保证一致性和可用性；BASE 是互联网系统常用的最终一致设计思想。

**分层展开：**

**CAP**
- C：一致性。
- A：可用性。
- P：分区容忍性。
- 分布式系统必须考虑 P，发生分区时只能在 C 和 A 中取舍。

**BASE**
- Basically Available：基本可用。
- Soft State：软状态。
- Eventually Consistent：最终一致。

**工程理解**
- 交易资金链路偏一致性。
- 内容展示、推荐、计数偏可用性。
- 订单履约常用最终一致 + 补偿。

**面试追问：CAP 是不是只能三选二？**
更准确地说，在网络分区发生时，需要在一致性和可用性之间取舍。平时系统可以同时表现出较好的一致性和可用性。

### 2. 分布式事务有哪些方案？

**结论先行：**
常见方案有 2PC/TCC/Saga/本地消息表/事务消息，互联网业务最常用的是最终一致 + 补偿。

**分层展开：**

**2PC**
- 强一致倾向。
- 协调者控制提交。
- 阻塞和性能成本高。

**TCC**
- Try/Confirm/Cancel。
- 适合资源预留。
- 业务侵入大。

**Saga**
- 长事务拆成多个本地事务。
- 每步有补偿动作。
- 适合流程型业务。

**本地消息表**
- 本地事务写业务数据和事件。
- 异步投递 MQ。
- 消费端幂等和补偿。

**面试追问：订单支付适合哪种？**
通常使用本地事务 + MQ + 幂等 + 对账补偿。资金核心部分由支付系统或数据库事务保证，跨系统状态推进用最终一致。

### 3. 幂等怎么设计？

**结论先行：**
幂等是让同一个请求执行一次和执行多次结果一致，是分布式重试、MQ 消费、支付回调和接口防重的基础。

**分层展开：**

**幂等来源**
- 客户端重试。
- RPC 超时重试。
- MQ 重复消费。
- 支付渠道重复回调。

**实现方式**
- 唯一请求号。
- 数据库唯一约束。
- 状态机流转校验。
- 去重表。
- Redis 短期防重。

**关键原则**
- 幂等要靠最终写入点保证。
- 不能只依赖前端按钮置灰。
- 幂等记录要考虑过期和容量。

**面试追问：下单接口怎么做幂等？**
客户端生成或服务端发放幂等号，同一用户同一幂等号只能创建一个订单，数据库唯一约束兜底，重复请求返回已有订单。

### 4. 重试、超时和幂等是什么关系？

**结论先行：**
超时触发重试，重试要求幂等；没有幂等保护的写操作不能随意重试。

**分层展开：**

**超时**
- 防止调用无限等待。
- 释放资源。
- 给上游明确失败边界。

**重试**
- 应对临时故障。
- 可能放大流量。
- 要配合退避和预算。

**幂等**
- 保证重复请求不会产生重复副作用。
- 是安全重试的前提。

**工程建议**
- 查询可重试。
- 创建、扣款、扣库存要先做幂等。
- 多层重试要统一治理。

**面试追问：接口超时但下游成功了怎么办？**
通过幂等号或业务单号查询最终状态。不能简单再次创建，要按状态机推进或返回已有结果。

### 5. 热点数据怎么治理？

**结论先行：**
热点治理要先识别热点，再通过缓存、本地副本、分片、限流和隔离降低单点压力。

**分层展开：**

**热点类型**
- 热点 key。
- 热点用户。
- 热点商品。
- 热点接口。
- 热点数据库行。

**识别方式**
- 业务埋点。
- 中间件统计。
- Redis hotkey。
- DB 行锁等待。
- 接口 P99 异常。

**治理方案**
- 本地缓存。
- key 分片。
- 读写分离。
- 请求合并。
- 热点隔离。
- 限流降级。

**面试追问：热点库存行怎么处理？**
可以用 Redis 预扣、库存分段、队列串行化、活动分片和最终 DB 对账，避免所有请求打同一行。

### 6. 限流算法有哪些？

**结论先行：**
常见限流算法有固定窗口、滑动窗口、漏桶、令牌桶，互联网接口最常用令牌桶和滑动窗口。

**分层展开：**

**固定窗口**
- 简单。
- 窗口边界有突刺。

**滑动窗口**
- 更平滑。
- 存储和计算成本更高。

**漏桶**
- 固定速率流出。
- 平滑但不支持突发。

**令牌桶**
- 固定速率生成令牌。
- 支持一定突发。

**面试追问：秒杀入口用什么限流？**
通常多级限流：网关令牌桶、用户维度滑动窗口、库存维度队列或令牌，配合风控。

### 7. 系统雪崩怎么防？

**结论先行：**
防雪崩要避免单点过载和故障扩散，核心是隔离、限流、熔断、降级、缓存和异步削峰。

**分层展开：**

**隔离**
- 线程池隔离。
- 连接池隔离。
- 服务隔离。
- 资源隔离。

**保护**
- 超时。
- 限流。
- 熔断。
- 降级。

**削峰**
- MQ。
- 排队。
- 批处理。

**恢复**
- 灰度恢复。
- 缓存预热。
- 避免重试风暴。

**面试追问：故障恢复为什么要慢慢恢复？**
瞬间恢复可能让积压请求和缓存未命中同时打到系统，造成二次雪崩。

## 二、核心原理分析：一致性取舍、最终一致、重试放大、热点与雪崩

> 这一章回答“系统为什么会不稳定”。

### 9. 秒杀系统怎么设计？

**结论先行：**
秒杀系统核心是把流量挡在前面，把库存扣减做成可控的短链路，并通过异步化和补偿保证最终结果。

**分层展开：**

**流量层**
- CDN 静态化。
- 网关限流。
- 用户维度防刷。
- 验证码或排队。

**缓存层**
- 活动和商品预热。
- Redis 扣减预库存。
- 热点 key 分片。

**业务层**
- 一人一单幂等。
- 库存原子扣减。
- 下单异步化。
- 订单状态处理中。

**数据层**
- MQ 削峰。
- DB 最终落库。
- 库存对账补偿。

**面试追问：Redis 扣减成功就代表购买成功吗？**
不代表。Redis 扣减只是预占，最终还要订单创建、支付、库存落库和对账。用户侧可以展示排队中或处理中。

### 10. 如何设计高并发读系统？

**结论先行：**
高并发读系统要用多级缓存、读模型冗余、热点治理、异步更新和降级兜底来减少对主存储的压力。

**分层展开：**

**缓存体系**
- CDN。
- 本地缓存。
- Redis。
- DB。

**读模型**
- 按查询场景冗余数据。
- 避免复杂 join。
- 热点数据预计算。

**一致性**
- 接受短暂不一致。
- 事件驱动更新。
- 版本号或更新时间校验。

**稳定性**
- 热 key 拆分。
- 本地缓存兜底。
- 限流和降级。

**面试追问：读多写少是不是一定适合缓存？**
大多数适合，但要看数据大小、更新频率、一致性要求和命中率。低频冷数据缓存价值不高。

### 11. 如何设计高并发写系统？

**结论先行：**
高并发写系统要控制写入入口、拆分热点、异步化、批量化，并保证幂等和最终一致。

**分层展开：**

**入口控制**
- 限流。
- 队列削峰。
- 请求合并。

**数据拆分**
- 分库分表。
- 按业务维度分片。
- 热点资源单独处理。

**写入优化**
- 批量写。
- 异步写。
- 事件驱动。
- 减少事务范围。

**正确性**
- 幂等。
- 状态机。
- 去重。
- 补偿和对账。

**面试追问：异步写会不会丢数据？**
可能，所以要有可靠消息、持久化队列、失败重试和补偿。不能把异步等同于不关心结果。

## 三、高并发与业务系统设计实战

> 这一章回答“怎么设计系统”：订单、Feed、计数等通用母题。

### 12. 订单系统如何设计？

**结论先行：**
订单系统核心是状态机、幂等、数据一致性和可追溯，不能只把订单表设计出来。

**分层展开：**

**核心模型**
- 订单主表。
- 订单明细。
- 支付单。
- 履约单。
- 状态流转表。

**主链路**
- 创建订单。
- 锁定库存。
- 发起支付。
- 支付回调。
- 履约发货。
- 售后退款。

**稳定性**
- 下单幂等。
- 支付回调幂等。
- MQ 异步推进。
- 定时补偿。
- 对账。

**面试追问：订单为什么要状态机？**
状态机能限制非法流转，保证重复消息和乱序消息不会把订单推进到错误状态。

### 13. Feed 流系统怎么设计？

**结论先行：**
Feed 流核心是在写扩散和读扩散之间取舍，关注用户关系规模、读写比例、延迟和存储成本。

**分层展开：**

**写扩散**
- 发布时推送到粉丝收件箱。
- 读很快。
- 大 V 写入成本高。

**读扩散**
- 读取时拉取关注人的内容并聚合排序。
- 写轻。
- 读复杂且延迟高。

**混合模式**
- 普通用户写扩散。
- 大 V 读扩散。
- 热点内容缓存。

**面试追问：小红书类关注流怎么做？**
通常关注流可混合推拉，推荐流更多依赖召回、排序和缓存。业务要区分关注关系和推荐内容。

### 14. 计数系统怎么设计？

**结论先行：**
计数系统要根据实时性和准确性选择方案，高并发点赞浏览常用 Redis 聚合 + 异步落库 + 定期对账。

**分层展开：**

**强实时计数**
- Redis INCR。
- 定期刷 DB。
- 展示读 Redis。

**高吞吐计数**
- 本地聚合。
- MQ 异步。
- 批量落库。

**准确性**
- 幂等事件。
- 去重。
- 对账修正。

**面试追问：浏览量需要强一致吗？**
通常不需要。浏览量更关注趋势和低成本，可接受短暂误差和延迟落库。

## 四、红果定向与项目表达

> 这一章服务你的目标岗位，把短剧内容、广告投放、播放页和埋点链路讲成系统设计能力。

### 15. 如何设计一个短剧内容发布系统？

**业务目标：**
支持短剧剧集、章节、封面、字幕、素材、标签、简介等内容从创建、审核、灰度到发布，既要支撑运营后台，也要保证 C 端播放页读取稳定。

**核心对象：**
- 剧集：series。
- 章节：episode。
- 素材：video、cover、subtitle。
- 审核记录：audit_record。
- 发布记录：publish_record。
- 灰度配置：experiment/config。

**写链路：**
1. 运营或创作者提交草稿。
2. 服务端校验基础字段和素材完整性。
3. 进入审核流程。
4. 审核通过后生成待发布版本。
5. 发布时更新主状态，并发送 MQ 通知搜索、推荐、缓存、CDN 预热等下游。

**读链路：**
- C 端只读取已发布版本。
- 热门剧集详情、章节列表走 Redis 缓存。
- 播放地址可短 TTL 缓存，避免过期 URL 长期有效。
- 内容搜索走 ES。

**一致性：**
- DB 中的发布状态是权威状态。
- 搜索、推荐、缓存通过 MQ 最终一致。
- 下游失败进入补偿任务。
- 发布/下线操作要幂等。

**稳定性：**
- 发布链路和 C 端读链路隔离。
- 审核、转码、索引更新异步化。
- 缓存设置随机 TTL，热点内容加互斥重建。
- 下游推荐或搜索异常时，播放页仍能读取基础内容。

### 16. 如何设计一个短剧广告投放平台？

**业务目标：**
把短剧素材、媒体账号、像素、落地页、广告任务、数据回流和投放复盘串成一个闭环，支撑运营高效创建和管理广告。

**核心模块：**
- 素材库：视频、图片、标题、标签、项目、违规状态。
- 账号和资产：媒体账号、像素、落地页、授权状态。
- 广告任务：Campaign、AdGroup/AdSet、Ad。
- 异步执行：任务拆分、媒体 API 调用、失败重试。
- 数据回流：曝光、点击、转化、花费、ROI/ROAS。
- 工单协作：投放组分配、绩效归档、复盘。

**存储选型：**
- MongoDB：广告任务和媒体平台配置，适合半结构化、多平台差异。
- MySQL：投放工单、分配规则、绩效统计，适合事务和结构化关系。
- Redis：媒体 API 限流、Cron 锁、缓存版本。
- Kafka：广告创建、素材同步、通知、ES 更新。
- ES：素材检索和广告任务复杂查询。

**关键稳定性设计：**
- 媒体 API 调用必须限流，避免平台破频。
- 广告创建异步化，前台不阻塞等待外部平台。
- 消费者幂等，避免重复创建广告。
- ES 查询灰度迁移，保留 Mongo 兜底。
- 失败任务保留平台错误码和请求上下文，支持重试。

**面试亮点：**
这个系统不是一个后台 CRUD，而是外部平台不稳定、任务最终一致、数据回流延迟、运营强依赖的复杂业务平台。

### 17. 如何设计短剧播放页的高并发读链路？

**读请求内容：**
- 剧集基础信息。
- 章节列表。
- 当前章节播放地址。
- 用户播放进度。
- 推荐内容。
- 运营配置和实验信息。

**架构设计：**
- CDN 承载视频文件和静态资源。
- 播放页基础信息通过服务端 API 聚合。
- 热门剧集和章节列表放 Redis。
- 用户播放进度写多读少，可异步写入或批量写入。
- 曝光、点击、完播、停留时长等埋点走 MQ/日志链路。

**降级策略：**
- 推荐服务超时时返回热门内容。
- 实验配置读取失败时走默认配置。
- 播放进度写失败不影响播放。
- 评论/互动模块异常不影响主播放链路。

**容量估算：**
先估 DAU、峰值在线、播放页 QPS、热门剧集占比、缓存命中率、埋点量。播放页读链路和埋点写链路要分开估算，因为埋点通常远高于页面请求。

### 18. 如何设计广告/内容埋点回流链路？

**业务目标：**
收集曝光、点击、播放、完播、广告转化、素材效果等事件，用于推荐、投放复盘、BI 报表和增长实验。

**链路：**
1. 客户端或服务端产生事件。
2. 接入层做参数校验、限流和采样。
3. 写入 Kafka。
4. 实时消费更新短周期指标。
5. 离线链路做归因、聚合和报表。
6. 结果回流到投放平台或推荐系统。

**关键问题：**
- 事件重复：使用 event_id 或业务唯一键幂等。
- 事件乱序：按 event_time 处理，允许迟到窗口。
- 高峰削峰：Kafka 分区、批量发送、异步落库。
- 数据质量：字段校验、版本管理、异常值监控。
- 隐私合规：敏感信息脱敏，按地区策略处理。

**项目映射：**
枫叶互动的广告数据回流、前端埋点、BI 报表和投放复盘，本质就是这类链路。回答时可以强调你熟悉“投放动作 -> 媒体平台数据 -> 内部 BI -> 运营决策”的闭环。

### 19. 架构题如何体现资深度？

**不要只画模块，要主动讲取舍：**
- 为什么这个数据放 MongoDB，而不是 MySQL？
- 为什么查询上 ES，但写入不直接写 ES？
- 为什么广告创建要异步化？
- 为什么 Redis 锁不能代替业务幂等？
- 为什么发布链路和 C 端读链路要隔离？
- 哪些地方允许最终一致，哪些地方必须强一致？

**高分回答结构：**
1. 明确业务目标和约束。
2. 拆读链路、写链路、异步链路、补偿链路。
3. 说明存储选型和数据边界。
4. 说明高可用、降级、限流、幂等。
5. 说明监控指标和故障预案。
6. 最后讲方案边界和后续演进。

**结论先行：**
容量评估要根据峰值 QPS、接口耗时、资源使用、数据增长和冗余系数计算，并通过压测验证。

**分层展开：**

**流量估算**
- 日活、峰值系数。
- 核心接口 QPS。
- 突发活动放大倍数。

**资源估算**
- CPU。
- 内存。
- DB 连接。
- Redis QPS。
- MQ 分区和消费能力。

**压测验证**
- 单机基准。
- 集群压测。
- 稳定性压测。
- 故障演练。

**面试追问：压测只看 QPS 够吗？**
不够。还要看 P99、错误率、资源水位、下游压力、恢复能力和数据正确性。

## 五、面试方法论、速记与回答模板

> 本模块统一收纳“怎么答、怎么组织、怎么临场展开”的内容；前面章节负责知识本体，这里负责面试表达。

### 方法 1：资深后端如何回答架构设计题？

**结论先行：**
架构设计题不要直接堆技术组件，要先明确目标和约束，再拆读写链路、数据模型、一致性、稳定性和可观测性。

**分层展开：**

**目标澄清**
- QPS、峰值、数据量。
- 一致性要求。
- P99 延迟。
- 可用性目标。
- 成本和上线周期。

**链路设计**
- 读链路。
- 写链路。
- 异步链路。
- 补偿链路。

**稳定性**
- 限流、熔断、降级。
- 多级缓存。
- 幂等和重试。
- 故障隔离。

**治理**
- 监控、日志、Trace。
- 压测和容量评估。
- 灰度和回滚。

**面试追问：架构设计最怕什么？**
最怕没有约束直接给方案。资深工程师应该先问清楚业务目标和边界，再做取舍。
