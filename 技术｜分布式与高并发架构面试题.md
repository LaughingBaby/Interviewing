## 二、分布式与高并发 当前能力全景：能力边界、核心模型与工程用法

> 本模块回答“它能做什么、怎么正确使用”。先掌握能力边界，再进入底层原理。

### 1. CAP 和 BASE 怎么理解？

**结论先行：**
CAP 描述分布式系统在网络分区下无法同时保证一致性和可用性；BASE 是互联网系统常用的最终一致设计思想。

**分层展开：**

**CAP**
- C：一致性。
- A：可用性。
- P：分区容忍性。
- 分布式系统必须考虑 P，发生分区时只能在 C 和 A 中取舍。

**BASE**
- Basically Available：基本可用。
- Soft State：软状态。
- Eventually Consistent：最终一致。

**工程理解**
- 交易资金链路偏一致性。
- 内容展示、推荐、计数偏可用性。
- 订单履约常用最终一致 + 补偿。

**面试追问：CAP 是不是只能三选二？**
更准确地说，在网络分区发生时，需要在一致性和可用性之间取舍。平时系统可以同时表现出较好的一致性和可用性。

### 2. 分布式事务有哪些方案？

**结论先行：**
常见方案有 2PC/TCC/Saga/本地消息表/事务消息，互联网业务最常用的是最终一致 + 补偿。

**分层展开：**

**2PC**
- 强一致倾向。
- 协调者控制提交。
- 阻塞和性能成本高。

**TCC**
- Try/Confirm/Cancel。
- 适合资源预留。
- 业务侵入大。

**Saga**
- 长事务拆成多个本地事务。
- 每步有补偿动作。
- 适合流程型业务。

**本地消息表**
- 本地事务写业务数据和事件。
- 异步投递 MQ。
- 消费端幂等和补偿。

**面试追问：订单支付适合哪种？**
通常使用本地事务 + MQ + 幂等 + 对账补偿。资金核心部分由支付系统或数据库事务保证，跨系统状态推进用最终一致。

**延伸：Redis、MySQL、MongoDB 事务如何选？**

| 场景 | 首选方案 | 原因 |
| --- | --- | --- |
| Redis 内部多条命令要连续执行 | `MULTI/EXEC` 或 Lua | Redis 事务只保证命令队列连续执行，复杂判断用 Lua 更合适 |
| 订单、库存、账务、权限等结构化强一致写入 | MySQL 本地事务 | InnoDB 有 undo/redo、MVCC、锁和约束，适合强一致 OLTP |
| 文档聚合根内部状态变更 | MongoDB 单文档原子更新 | 单文档天然原子，成本低，符合文档建模 |
| 少量跨 Mongo 文档/集合一致变更 | MongoDB 多文档事务 | 能补充跨文档原子提交，但要短事务、可重试 |
| DB 写成功后发 MQ | Outbox/本地消息表或事务消息 | DB 和 MQ 不在一个本地事务里，要显式建模最终一致 |
| DB 写成功后删 Redis 缓存 | Cache Aside + 重试/MQ/binlog/Change Stream | Redis 事务无法覆盖数据库事务，缓存只能最终一致 |

**面试表达：**
我会先判断事务边界在哪里。如果只是 Redis 内部状态，用 Lua 或 `MULTI/EXEC`；如果最终正确性落在关系型数据上，用 MySQL 事务、唯一约束、条件更新和状态机；如果是文档模型，优先通过 MongoDB 单文档原子更新解决，少量跨集合再用事务。跨 DB、Redis、MQ、第三方 API 时，不追求一个大事务，而是用本地事务 + 事件 + 幂等 + 补偿实现最终一致。

### 3. 幂等怎么设计？

**结论先行：**
幂等是让同一个请求执行一次和执行多次结果一致，是分布式重试、MQ 消费、支付回调和接口防重的基础。

**分层展开：**

**幂等来源**
- 客户端重试。
- RPC 超时重试。
- MQ 重复消费。
- 支付渠道重复回调。

**实现方式**
- 唯一请求号。
- 数据库唯一约束。
- 状态机流转校验。
- 去重表。
- Redis 短期防重。

**关键原则**
- 幂等要靠最终写入点保证。
- 不能只依赖前端按钮置灰。
- 幂等记录要考虑过期和容量。

**面试追问：下单接口怎么做幂等？**
客户端生成或服务端发放幂等号，同一用户同一幂等号只能创建一个订单，数据库唯一约束兜底，重复请求返回已有订单。

### 4. 重试、超时和幂等是什么关系？

**结论先行：**
超时触发重试，重试要求幂等；没有幂等保护的写操作不能随意重试。

**分层展开：**

**超时**
- 防止调用无限等待。
- 释放资源。
- 给上游明确失败边界。

**重试**
- 应对临时故障。
- 可能放大流量。
- 要配合退避和预算。

**幂等**
- 保证重复请求不会产生重复副作用。
- 是安全重试的前提。

**工程建议**
- 查询可重试。
- 创建、扣款、扣库存要先做幂等。
- 多层重试要统一治理。

**面试追问：接口超时但下游成功了怎么办？**
通过幂等号或业务单号查询最终状态。不能简单再次创建，要按状态机推进或返回已有结果。

### 5. 热点数据怎么治理？

**结论先行：**
热点治理要先识别热点，再通过缓存、本地副本、分片、限流和隔离降低单点压力。

**分层展开：**

**热点类型**
- 热点 key。
- 热点用户。
- 热点商品。
- 热点接口。
- 热点数据库行。

**识别方式**
- 业务埋点。
- 中间件统计。
- Redis hotkey。
- DB 行锁等待。
- 接口 P99 异常。

**治理方案**
- 本地缓存。
- key 分片。
- 读写分离。
- 请求合并。
- 热点隔离。
- 限流降级。

**面试追问：热点库存行怎么处理？**
可以用 Redis 预扣、库存分段、队列串行化、活动分片和最终 DB 对账，避免所有请求打同一行。

### 6. 限流算法有哪些？

**结论先行：**
常见限流算法有固定窗口、滑动窗口、漏桶、令牌桶和并发数限制。互联网接口最常用令牌桶和滑动窗口；系统设计时还要回答“限流放在哪里”：本机、网关、Redis 全局、MQ 队列、数据库条件更新都可以做，但解决的问题不一样。

**分层展开：**

**固定窗口**
- 简单。
- 窗口边界有突刺。
- 适合粗粒度、低成本计数。

**滑动窗口**
- 更平滑。
- 存储和计算成本更高。
- 适合精确限制最近 N 秒/N 分钟请求数。

**漏桶**
- 固定速率流出。
- 平滑但不支持突发。
- 适合保护慢下游，偏排队削峰。

**令牌桶**
- 固定速率生成令牌。
- 支持一定突发。
- 适合接口入口限流和网关限流。

**并发数限制**
- 限制同时执行中的请求或任务数。
- 适合保护连接池、worker pool、第三方 API 并发。

**限流组件如何选**

| 位置/组件 | 解决什么问题 | 优点 | 缺点 |
| --- | --- | --- | --- |
| Go 本地限流 | 单实例自我保护、SDK 客户端保护 | 最快，无网络开销 | 多实例不共享额度 |
| Nginx/Envoy/API Gateway | 入口 IP、URI、租户、API 配额 | 靠近入口，统一治理 | 业务细粒度较弱 |
| Redis 全局限流 | 多实例共享用户/账号/API 额度 | 灵活，适合外部 API 配额 | Redis 热 key 和网络开销 |
| Sentinel/服务治理库 | QPS、并发、热点参数、熔断降级 | 和服务保护结合紧密 | 跨语言/跨实例能力依赖框架 |
| MQ/任务队列 | 削峰填谷，控制消费速度 | 可靠排队，保护下游 | 增加延迟，不是实时拒绝 |
| DB 条件更新 | 强一致额度扣减 | 和业务事务一致 | 性能重，不适合高 QPS 入口 |

**工程组合**
- 第一层：网关粗限流，挡住异常流量和攻击。
- 第二层：服务本地限流，保护自身 goroutine、连接池和下游。
- 第三层：Redis 全局限流，控制跨实例共享额度。
- 第四层：MQ 削峰，把突发任务转成平稳消费。
- 第五层：DB 条件更新，兜住库存、余额、强配额的最终正确性。

**面试追问：秒杀入口用什么限流？**
通常多级限流：网关令牌桶、用户维度滑动窗口、库存维度队列或令牌，配合风控。

**面试追问：外部广告 API 调用用什么限流？**
一般用本地并发控制 + Redis 全局滑动窗口。比如按平台、广告账号、接口做 `1s/5m/1d` 多层窗口，避免多个服务实例同时调用导致整体破频；平台返回限流错误后，还要动态降速和冷却。

## 三、核心能力原理分析：一致性、缓存、锁与并发控制

> 本模块回答“为什么这样设计、代价是什么”。复杂原理要结合一致性取舍、缓存副本和锁机制理解。

### 专题沉淀：缓存一致性与多级缓存实战

**定位：**
这部分是 Redis「缓存与数据库一致性」的上位方法论。Redis 文档负责讲 Cache Aside 在 Redis 中怎么落地；这里负责讲分布式系统里多副本一致性如何取舍。

**核心结论：**
缓存与数据库很难低成本做到绝对强一致。生产上通常把数据库作为事实源，把 Redis、本地缓存、ES、读模型都看成派生副本，通过失效、重建、重试、补偿、版本号和 TTL 实现最终一致或有界不一致。

```text
事实源 DB
   │
   ├── Redis：共享缓存
   ├── L1 本地缓存：热点副本
   ├── ES：检索读模型
   └── 报表/数仓：分析读模型
```

**一致性目标分级：**

| 目标 | 适合场景 | 常见方案 |
|---|---|---|
| 强一致 | 余额、支付、库存最终扣减、权限安全 | DB 事务、条件更新、锁、状态机、关键读绕过缓存 |
| 最终一致 | 用户资料、商品详情、内容详情、配置展示 | Cache Aside、MQ/CDC 补偿、TTL 兜底 |
| 有界不一致 | 本地缓存、多级缓存、热点读模型 | 短 TTL、主动失效、版本号、写后读主 |

**Cache Aside 主流程：**

```text
读：先读缓存，miss 查 DB，回填缓存
写：先更新 DB，提交成功后删除缓存
补偿：删除失败重试 / MQ / Outbox / CDC / TTL
```

**为什么不是先删缓存再写 DB？**
先删缓存会制造空窗期，并发读可能在 DB 更新前读到旧值并回填缓存，最后 DB 是新值，缓存反而是旧值。

**为什么不是更新 DB 后直接更新缓存？**
并发写可能乱序覆盖；缓存可能是复杂聚合结果，一次写库影响多个缓存 key；删除缓存是幂等动作，更适合重试和补偿。

**延迟二次删除的定位：**
延迟二次删除不是强一致方案，而是清理“旧读请求晚回填”的补偿手段。如果计算出来要延迟 5 到 12 秒，它仍然能把旧缓存从 TTL 级别缩短到秒级，但已经不适合作为敏感业务的主方案。写后读敏感场景应配合读主、版本号、Lua 防旧覆盖或 CDC。

**删除缓存失败怎么办：**

- 同步有限重试，避免拖垮写接口。
- MQ 异步重试，删除操作天然幂等。
- Outbox，把业务变更和缓存失效事件放进同一个本地事务。
- MySQL binlog / Mongo Change Stream / CDC 统一传播失效事件。
- TTL 最终兜底，但不能只靠 TTL。

**多级缓存一致性：**

```text
读 L1 本地缓存
  └── miss 读 Redis
        └── miss 读 DB

写 DB 成功
  ├── 删除 Redis
  └── 发布失效事件，让所有实例删除 L1
```

本地缓存分散在多个进程，不能像 Redis 一样删一个共享 key 就结束。重要失效事件更适合可靠 MQ、Redis Stream 或 CDC；Redis Pub/Sub 可用于允许短暂不一致的场景，但必须配合短 TTL。

**高并发重建保护：**

- 单机用 `singleflight` 合并同 key 回源。
- 跨实例用 Redis 锁做互斥重建。
- 超热点可用逻辑过期，先返回旧值，后台异步刷新。
- 一致性更高时，缓存 value 带 version，旧版本不允许覆盖新版本。

**面试 1 分钟回答：**
我会先说明数据库是事实源，缓存是派生副本。读链路用 Cache Aside，miss 查 DB 再回填；写链路先提交 DB，再删除缓存。删除失败不能只打日志，要进入重试、MQ、Outbox 或 CDC 补偿，TTL 只是兜底。并发读可能在 DB 更新前读到旧值，并在删除缓存后回填旧缓存，可以用延迟二次删除、版本号或写后读主降低风险。多级缓存还要广播 L1 失效，并用短 TTL 控制不一致窗口。

### 专题沉淀：锁机制与并发控制实战

**定位：**
这部分是 Redis「分布式锁」的上位方法论。Redis 文档保留 Redis 锁的实现细节；这里负责横向比较 Go、MySQL、MongoDB、Redis、Kafka、ES、etcd/ZooKeeper 中的并发控制。

**核心结论：**
锁不是业务正确性的魔法开关。锁本质是所有权与访问顺序协议，只能减少并发进入者；最终正确性必须由数据落点处的原子规则、幂等、唯一约束、状态机和补偿兜底。

```text
锁 + 原子条件更新 + 幂等 + 唯一约束 + 版本号 + fencing token + 补偿
```

**不要把所有并发控制都叫锁：**

| 类型 | 典型机制 | 适合场景 | 关键风险 |
|---|---|---|---|
| 悲观锁 | Go Mutex、MySQL `for update`、Redis 锁、etcd/ZK 锁 | 冲突频繁、临界区短 | 等待、死锁、吞吐下降 |
| 乐观锁 | version、CAS、ES `seq_no + primary_term` | 冲突较少、可重试 | 冲突后要重试或合并 |
| 原子条件更新 | `update ... where status=?` | 库存扣减、任务领取、状态抢占 | 只能保护数据落点内部 |
| 幂等 | 唯一索引、幂等号、去重表 | 重试、重复消息、重复回调 | 幂等键设计和容量治理 |
| 串行化 | Kafka 按 key 分区、单队列消费 | 同一业务 key 顺序处理 | 吞吐受分区限制 |
| 租约 Lease | Redis TTL、etcd lease、ZK session | 跨进程短时所有权 | 过期后旧持有者可能继续执行 |

**跨组件对比：**

| 组件 | 并发控制机制 | 适合解决的问题 | 不适合承担的职责 |
|---|---|---|---|
| Go | Mutex、RWMutex、atomic、channel | 单进程 goroutine 共享内存 | 跨进程协调 |
| MySQL | 行锁、gap lock、next-key lock、事务、唯一约束 | 强一致写入、库存、订单状态 | 高延迟跨系统互斥 |
| MongoDB | 单文档原子更新、事务、write conflict retry | 文档状态流转、任务抢占 | 大量跨集合强事务 |
| Redis | `SET NX PX` + Lua + TTL 租约 | 缓存重建、任务抢占、防重复提交 | 金融级强一致 |
| Kafka | 分区所有权、消费组、幂等生产、事务 | 按 key 串行事件处理 | 通用互斥锁 |
| ES | `seq_no + primary_term` 乐观并发 | 防止旧文档覆盖新文档 | 事实源强一致 |
| etcd/ZK | 共识、lease/session、watch、顺序节点 | Leader 选举、严格协调 | 高 QPS 业务热路径 |

**Redis 锁的正确定位：**
Redis 锁适合短时、容错型互斥，例如缓存重建、定时任务抢占、接口防抖和非核心资源串行化。它不能替代数据库事务、幂等和条件更新。

**fencing token 的意义：**
owner token 只能证明“这把锁是不是我的”，不能防止锁过期后的旧客户端继续写。fencing token 是单调递增令牌，下游只接受更大的 token，用来拒绝旧持锁者。

**生产选型口诀：**

```text
单进程共享内存：Go 锁
数据最终正确：DB 条件更新 / 事务 / 唯一约束
短时跨实例互斥：Redis 锁
强协调和选主：etcd / ZooKeeper
事件顺序处理：Kafka 分区串行化
旧文档覆盖防护：ES 乐观并发
重复执行兜底：幂等和状态机
```

**面试 1 分钟回答：**
我不会把锁当成万能方案。锁解决的是并发进入临界区的问题，但不能解决网络超时、消息重复、锁过期后旧客户端继续写、数据库部分提交这些问题。生产上会先判断并发冲突发生在哪里：单进程用 Go 锁，数据写入正确性优先用数据库条件更新和事务，跨实例短时互斥才用 Redis 锁，强协调用 etcd/ZK，消息顺序用 Kafka 分区。最终还要通过幂等、唯一约束、版本号、fencing token 和补偿机制保证正确性。

### 7. 秒杀系统怎么设计？

**结论先行：**
秒杀系统核心是把流量挡在前面，把库存扣减做成可控的短链路，并通过异步化和补偿保证最终结果。

**分层展开：**

**流量层**
- CDN 静态化。
- 网关限流。
- 用户维度防刷。
- 验证码或排队。

**缓存层**
- 活动和商品预热。
- Redis 扣减预库存。
- 热点 key 分片。

**业务层**
- 一人一单幂等。
- 库存原子扣减。
- 下单异步化。
- 订单状态处理中。

**数据层**
- MQ 削峰。
- DB 最终落库。
- 库存对账补偿。

**面试追问：Redis 扣减成功就代表购买成功吗？**
不代表。Redis 扣减只是预占，最终还要订单创建、支付、库存落库和对账。用户侧可以展示排队中或处理中。

### 8. 如何设计高并发读系统？

**结论先行：**
高并发读系统要用多级缓存、读模型冗余、热点治理、异步更新和降级兜底来减少对主存储的压力。

**分层展开：**

**缓存体系**
- CDN。
- 本地缓存。
- Redis。
- DB。

**读模型**
- 按查询场景冗余数据。
- 避免复杂 join。
- 热点数据预计算。

**一致性**
- 接受短暂不一致。
- 事件驱动更新。
- 版本号或更新时间校验。

**稳定性**
- 热 key 拆分。
- 本地缓存兜底。
- 限流和降级。

**面试追问：读多写少是不是一定适合缓存？**
大多数适合，但要看数据大小、更新频率、一致性要求和命中率。低频冷数据缓存价值不高。

### 9. 如何设计高并发写系统？

**数据拆分**

**写入优化**
- 批量写。
- 异步写。

**正确性**
- 幂等。
- 状态机。

**面试追问：异步写会不会丢数据？**

### 11. Feed 流系统怎么设计？

**结论先行：**
Feed 流核心是在写扩散和读扩散之间取舍，关注用户关系规模、读写比例、延迟和存储成本。

**分层展开：**

**写扩散**
- 发布时推送到粉丝收件箱。
- 读很快。
- 大 V 写入成本高。

**读扩散**
- 读取时拉取关注人的内容并聚合排序。
- 写轻。
- 读复杂且延迟高。

**混合模式**
- 普通用户写扩散。
- 大 V 读扩散。
- 热点内容缓存。

**面试追问：小红书类关注流怎么做？**
通常关注流可混合推拉，推荐流更多依赖召回、排序和缓存。业务要区分关注关系和推荐内容。

### 12. 计数系统怎么设计？

**结论先行：**
计数系统要根据实时性和准确性选择方案，高并发点赞浏览常用 Redis 聚合 + 异步落库 + 定期对账。

**分层展开：**

**强实时计数**
- Redis INCR。
- 定期刷 DB。
- 展示读 Redis。

**高吞吐计数**
- 本地聚合。
- MQ 异步。
- 批量落库。

**准确性**
- 幂等事件。
- 去重。
- 对账修正。

**面试追问：浏览量需要强一致吗？**
通常不需要。浏览量更关注趋势和低成本，可接受短暂误差和延迟落库。

## 六、项目实战表达：把分布式与高并发讲成项目能力

### 15. 如何设计一个短剧内容发布系统？

**业务目标：**
支持短剧剧集、章节、封面、字幕、素材、标签、简介等内容从创建、审核、灰度到发布，既要支撑运营后台，也要保证 C 端播放页读取稳定。

**核心对象：**
- 剧集：series。
- 章节：episode。
- 素材：video、cover、subtitle。
- 审核记录：audit_record。
- 发布记录：publish_record。
- 灰度配置：experiment/config。

**写链路：**
1. 运营或创作者提交草稿。
2. 服务端校验基础字段和素材完整性。
3. 进入审核流程。
4. 审核通过后生成待发布版本。
5. 发布时更新主状态，并发送 MQ 通知搜索、推荐、缓存、CDN 预热等下游。

**读链路：**
- C 端只读取已发布版本。
- 热门剧集详情、章节列表走 Redis 缓存。
- 播放地址可短 TTL 缓存，避免过期 URL 长期有效。
- 内容搜索走 ES。

**一致性：**
- DB 中的发布状态是权威状态。
- 搜索、推荐、缓存通过 MQ 最终一致。
- 下游失败进入补偿任务。
- 发布/下线操作要幂等。

**稳定性：**
- 发布链路和 C 端读链路隔离。
- 审核、转码、索引更新异步化。
- 缓存设置随机 TTL，热点内容加互斥重建。
- 下游推荐或搜索异常时，播放页仍能读取基础内容。

### 16. 如何设计一个短剧广告投放平台？

**业务目标：**
把短剧素材、媒体账号、像素、落地页、广告任务、数据回流和投放复盘串成一个闭环，支撑运营高效创建和管理广告。

**核心模块：**
- 素材库：视频、图片、标题、标签、项目、违规状态。
- 账号和资产：媒体账号、像素、落地页、授权状态。
- 广告任务：Campaign、AdGroup/AdSet、Ad。
- 异步执行：任务拆分、媒体 API 调用、失败重试。
- 数据回流：曝光、点击、转化、花费、ROI/ROAS。
- 工单协作：投放组分配、绩效归档、复盘。

**存储选型：**
- MongoDB：广告任务和媒体平台配置，适合半结构化、多平台差异。
- MySQL：投放工单、分配规则、绩效统计，适合事务和结构化关系。
- Redis：媒体 API 限流、Cron 锁、缓存版本。
- Kafka：广告创建、素材同步、通知、ES 更新。
- ES：素材检索和广告任务复杂查询。

**关键稳定性设计：**
- 媒体 API 调用必须限流，避免平台破频。
- 广告创建异步化，前台不阻塞等待外部平台。
- 消费者幂等，避免重复创建广告。
- ES 查询灰度迁移，保留 Mongo 兜底。
- 失败任务保留平台错误码和请求上下文，支持重试。

**面试亮点：**
这个系统不是一个后台 CRUD，而是外部平台不稳定、任务最终一致、数据回流延迟、运营强依赖的复杂业务平台。

### 17. 如何设计短剧播放页的高并发读链路？

**读请求内容：**
- 剧集基础信息。
- 章节列表。
- 当前章节播放地址。
- 用户播放进度。
- 推荐内容。
- 运营配置和实验信息。

**架构设计：**
- CDN 承载视频文件和静态资源。
- 播放页基础信息通过服务端 API 聚合。
- 热门剧集和章节列表放 Redis。
- 用户播放进度写多读少，可异步写入或批量写入。
- 曝光、点击、完播、停留时长等埋点走 MQ/日志链路。

**降级策略：**
- 推荐服务超时时返回热门内容。
- 实验配置读取失败时走默认配置。
- 播放进度写失败不影响播放。
- 评论/互动模块异常不影响主播放链路。

**容量估算：**
先估 DAU、峰值在线、播放页 QPS、热门剧集占比、缓存命中率、埋点量。播放页读链路和埋点写链路要分开估算，因为埋点通常远高于页面请求。

### 18. 如何设计广告/内容埋点回流链路？

**业务目标：**
收集曝光、点击、播放、完播、广告转化、素材效果等事件，用于推荐、投放复盘、BI 报表和增长实验。

**链路：**
1. 客户端或服务端产生事件。
2. 接入层做参数校验、限流和采样。
3. 写入 Kafka。
4. 实时消费更新短周期指标。
5. 离线链路做归因、聚合和报表。
6. 结果回流到投放平台或推荐系统。

**关键问题：**
- 事件重复：使用 event_id 或业务唯一键幂等。
- 事件乱序：按 event_time 处理，允许迟到窗口。
- 高峰削峰：Kafka 分区、批量发送、异步落库。
- 数据质量：字段校验、版本管理、异常值监控。
- 隐私合规：敏感信息脱敏，按地区策略处理。

**项目映射：**
枫叶互动的广告数据回流、前端埋点、BI 报表和投放复盘，本质就是这类链路。回答时可以强调你熟悉“投放动作 -> 媒体平台数据 -> 内部 BI -> 运营决策”的闭环。

### 19. 架构题如何体现资深度？

**不要只画模块，要主动讲取舍：**
- 为什么这个数据放 MongoDB，而不是 MySQL？
- 为什么查询上 ES，但写入不直接写 ES？
- 为什么广告创建要异步化？
- 为什么 Redis 锁不能代替业务幂等？
- 为什么发布链路和 C 端读链路要隔离？
- 哪些地方允许最终一致，哪些地方必须强一致？

**高分回答结构：**
1. 明确业务目标和约束。
2. 拆读链路、写链路、异步链路、补偿链路。
3. 说明存储选型和数据边界。
4. 说明高可用、降级、限流、幂等。
5. 说明监控指标和故障预案。
6. 最后讲方案边界和后续演进。

**结论先行：**
容量评估要根据峰值 QPS、接口耗时、资源使用、数据增长和冗余系数计算，并通过压测验证。

**分层展开：**

**流量估算**
- 日活、峰值系数。
- 核心接口 QPS。
- 突发活动放大倍数。

**资源估算**
- CPU。
- 内存。
- DB 连接。
- Redis QPS。
- MQ 分区和消费能力。

**压测验证**
- 单机基准。
- 集群压测。
- 稳定性压测。
- 故障演练。

**面试追问：压测只看 QPS 够吗？**
不够。还要看 P99、错误率、资源水位、下游压力、恢复能力和数据正确性。
