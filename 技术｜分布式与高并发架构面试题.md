# 技术｜分布式与高并发架构面试题

> 面向 9 年+ Golang Web 后端业务岗。
> 重点体现复杂业务系统设计能力：高并发读写、分布式一致性、幂等、限流降级、热点治理和业务架构取舍。

## 一、分布式架构总框架

### 1. 资深后端如何回答架构设计题？

**结论先行：**
架构设计题不要直接堆技术组件，要先明确目标和约束，再拆读写链路、数据模型、一致性、稳定性和可观测性。

**分层展开：**

**目标澄清**
- QPS、峰值、数据量。
- 一致性要求。
- P99 延迟。
- 可用性目标。
- 成本和上线周期。

**链路设计**
- 读链路。
- 写链路。
- 异步链路。
- 补偿链路。

**稳定性**
- 限流、熔断、降级。
- 多级缓存。
- 幂等和重试。
- 故障隔离。

**治理**
- 监控、日志、Trace。
- 压测和容量评估。
- 灰度和回滚。

**面试追问：架构设计最怕什么？**
最怕没有约束直接给方案。资深工程师应该先问清楚业务目标和边界，再做取舍。

## 二、一致性与分布式事务

### 2. CAP 和 BASE 怎么理解？

**结论先行：**
CAP 描述分布式系统在网络分区下无法同时保证一致性和可用性；BASE 是互联网系统常用的最终一致设计思想。

**分层展开：**

**CAP**
- C：一致性。
- A：可用性。
- P：分区容忍性。
- 分布式系统必须考虑 P，发生分区时只能在 C 和 A 中取舍。

**BASE**
- Basically Available：基本可用。
- Soft State：软状态。
- Eventually Consistent：最终一致。

**工程理解**
- 交易资金链路偏一致性。
- 内容展示、推荐、计数偏可用性。
- 订单履约常用最终一致 + 补偿。

**面试追问：CAP 是不是只能三选二？**
更准确地说，在网络分区发生时，需要在一致性和可用性之间取舍。平时系统可以同时表现出较好的一致性和可用性。

### 3. 分布式事务有哪些方案？

**结论先行：**
常见方案有 2PC/TCC/Saga/本地消息表/事务消息，互联网业务最常用的是最终一致 + 补偿。

**分层展开：**

**2PC**
- 强一致倾向。
- 协调者控制提交。
- 阻塞和性能成本高。

**TCC**
- Try/Confirm/Cancel。
- 适合资源预留。
- 业务侵入大。

**Saga**
- 长事务拆成多个本地事务。
- 每步有补偿动作。
- 适合流程型业务。

**本地消息表**
- 本地事务写业务数据和事件。
- 异步投递 MQ。
- 消费端幂等和补偿。

**面试追问：订单支付适合哪种？**
通常使用本地事务 + MQ + 幂等 + 对账补偿。资金核心部分由支付系统或数据库事务保证，跨系统状态推进用最终一致。

### 4. 幂等怎么设计？

**结论先行：**
幂等是让同一个请求执行一次和执行多次结果一致，是分布式重试、MQ 消费、支付回调和接口防重的基础。

**分层展开：**

**幂等来源**
- 客户端重试。
- RPC 超时重试。
- MQ 重复消费。
- 支付渠道重复回调。

**实现方式**
- 唯一请求号。
- 数据库唯一约束。
- 状态机流转校验。
- 去重表。
- Redis 短期防重。

**关键原则**
- 幂等要靠最终写入点保证。
- 不能只依赖前端按钮置灰。
- 幂等记录要考虑过期和容量。

**面试追问：下单接口怎么做幂等？**
客户端生成或服务端发放幂等号，同一用户同一幂等号只能创建一个订单，数据库唯一约束兜底，重复请求返回已有订单。

### 5. 重试、超时和幂等是什么关系？

**结论先行：**
超时触发重试，重试要求幂等；没有幂等保护的写操作不能随意重试。

**分层展开：**

**超时**
- 防止调用无限等待。
- 释放资源。
- 给上游明确失败边界。

**重试**
- 应对临时故障。
- 可能放大流量。
- 要配合退避和预算。

**幂等**
- 保证重复请求不会产生重复副作用。
- 是安全重试的前提。

**工程建议**
- 查询可重试。
- 创建、扣款、扣库存要先做幂等。
- 多层重试要统一治理。

**面试追问：接口超时但下游成功了怎么办？**
通过幂等号或业务单号查询最终状态。不能简单再次创建，要按状态机推进或返回已有结果。

## 三、高并发读写设计

### 6. 秒杀系统怎么设计？

**结论先行：**
秒杀系统核心是把流量挡在前面，把库存扣减做成可控的短链路，并通过异步化和补偿保证最终结果。

**分层展开：**

**流量层**
- CDN 静态化。
- 网关限流。
- 用户维度防刷。
- 验证码或排队。

**缓存层**
- 活动和商品预热。
- Redis 扣减预库存。
- 热点 key 分片。

**业务层**
- 一人一单幂等。
- 库存原子扣减。
- 下单异步化。
- 订单状态处理中。

**数据层**
- MQ 削峰。
- DB 最终落库。
- 库存对账补偿。

**面试追问：Redis 扣减成功就代表购买成功吗？**
不代表。Redis 扣减只是预占，最终还要订单创建、支付、库存落库和对账。用户侧可以展示排队中或处理中。

### 7. 如何设计高并发读系统？

**结论先行：**
高并发读系统要用多级缓存、读模型冗余、热点治理、异步更新和降级兜底来减少对主存储的压力。

**分层展开：**

**缓存体系**
- CDN。
- 本地缓存。
- Redis。
- DB。

**读模型**
- 按查询场景冗余数据。
- 避免复杂 join。
- 热点数据预计算。

**一致性**
- 接受短暂不一致。
- 事件驱动更新。
- 版本号或更新时间校验。

**稳定性**
- 热 key 拆分。
- 本地缓存兜底。
- 限流和降级。

**面试追问：读多写少是不是一定适合缓存？**
大多数适合，但要看数据大小、更新频率、一致性要求和命中率。低频冷数据缓存价值不高。

### 8. 如何设计高并发写系统？

**结论先行：**
高并发写系统要控制写入入口、拆分热点、异步化、批量化，并保证幂等和最终一致。

**分层展开：**

**入口控制**
- 限流。
- 队列削峰。
- 请求合并。

**数据拆分**
- 分库分表。
- 按业务维度分片。
- 热点资源单独处理。

**写入优化**
- 批量写。
- 异步写。
- 事件驱动。
- 减少事务范围。

**正确性**
- 幂等。
- 状态机。
- 去重。
- 补偿和对账。

**面试追问：异步写会不会丢数据？**
可能，所以要有可靠消息、持久化队列、失败重试和补偿。不能把异步等同于不关心结果。

### 9. 热点数据怎么治理？

**结论先行：**
热点治理要先识别热点，再通过缓存、本地副本、分片、限流和隔离降低单点压力。

**分层展开：**

**热点类型**
- 热点 key。
- 热点用户。
- 热点商品。
- 热点接口。
- 热点数据库行。

**识别方式**
- 业务埋点。
- 中间件统计。
- Redis hotkey。
- DB 行锁等待。
- 接口 P99 异常。

**治理方案**
- 本地缓存。
- key 分片。
- 读写分离。
- 请求合并。
- 热点隔离。
- 限流降级。

**面试追问：热点库存行怎么处理？**
可以用 Redis 预扣、库存分段、队列串行化、活动分片和最终 DB 对账，避免所有请求打同一行。

## 四、典型业务系统

### 10. 订单系统如何设计？

**结论先行：**
订单系统核心是状态机、幂等、数据一致性和可追溯，不能只把订单表设计出来。

**分层展开：**

**核心模型**
- 订单主表。
- 订单明细。
- 支付单。
- 履约单。
- 状态流转表。

**主链路**
- 创建订单。
- 锁定库存。
- 发起支付。
- 支付回调。
- 履约发货。
- 售后退款。

**稳定性**
- 下单幂等。
- 支付回调幂等。
- MQ 异步推进。
- 定时补偿。
- 对账。

**面试追问：订单为什么要状态机？**
状态机能限制非法流转，保证重复消息和乱序消息不会把订单推进到错误状态。

### 11. Feed 流系统怎么设计？

**结论先行：**
Feed 流核心是在写扩散和读扩散之间取舍，关注用户关系规模、读写比例、延迟和存储成本。

**分层展开：**

**写扩散**
- 发布时推送到粉丝收件箱。
- 读很快。
- 大 V 写入成本高。

**读扩散**
- 读取时拉取关注人的内容并聚合排序。
- 写轻。
- 读复杂且延迟高。

**混合模式**
- 普通用户写扩散。
- 大 V 读扩散。
- 热点内容缓存。

**面试追问：小红书类关注流怎么做？**
通常关注流可混合推拉，推荐流更多依赖召回、排序和缓存。业务要区分关注关系和推荐内容。

### 12. 计数系统怎么设计？

**结论先行：**
计数系统要根据实时性和准确性选择方案，高并发点赞浏览常用 Redis 聚合 + 异步落库 + 定期对账。

**分层展开：**

**强实时计数**
- Redis INCR。
- 定期刷 DB。
- 展示读 Redis。

**高吞吐计数**
- 本地聚合。
- MQ 异步。
- 批量落库。

**准确性**
- 幂等事件。
- 去重。
- 对账修正。

**面试追问：浏览量需要强一致吗？**
通常不需要。浏览量更关注趋势和低成本，可接受短暂误差和延迟落库。

## 五、稳定性与容量

### 13. 限流算法有哪些？

**结论先行：**
常见限流算法有固定窗口、滑动窗口、漏桶、令牌桶，互联网接口最常用令牌桶和滑动窗口。

**分层展开：**

**固定窗口**
- 简单。
- 窗口边界有突刺。

**滑动窗口**
- 更平滑。
- 存储和计算成本更高。

**漏桶**
- 固定速率流出。
- 平滑但不支持突发。

**令牌桶**
- 固定速率生成令牌。
- 支持一定突发。

**面试追问：秒杀入口用什么限流？**
通常多级限流：网关令牌桶、用户维度滑动窗口、库存维度队列或令牌，配合风控。

### 14. 系统雪崩怎么防？

**结论先行：**
防雪崩要避免单点过载和故障扩散，核心是隔离、限流、熔断、降级、缓存和异步削峰。

**分层展开：**

**隔离**
- 线程池隔离。
- 连接池隔离。
- 服务隔离。
- 资源隔离。

**保护**
- 超时。
- 限流。
- 熔断。
- 降级。

**削峰**
- MQ。
- 排队。
- 批处理。

**恢复**
- 灰度恢复。
- 缓存预热。
- 避免重试风暴。

**面试追问：故障恢复为什么要慢慢恢复？**
瞬间恢复可能让积压请求和缓存未命中同时打到系统，造成二次雪崩。

### 15. 容量评估怎么做？

**结论先行：**
容量评估要根据峰值 QPS、接口耗时、资源使用、数据增长和冗余系数计算，并通过压测验证。

**分层展开：**

**流量估算**
- 日活、峰值系数。
- 核心接口 QPS。
- 突发活动放大倍数。

**资源估算**
- CPU。
- 内存。
- DB 连接。
- Redis QPS。
- MQ 分区和消费能力。

**压测验证**
- 单机基准。
- 集群压测。
- 稳定性压测。
- 故障演练。

**面试追问：压测只看 QPS 够吗？**
不够。还要看 P99、错误率、资源水位、下游压力、恢复能力和数据正确性。
