# 技术｜消息队列MQ面试题

> 面向 9 年+ Golang Web 后端业务岗。
> 覆盖 Kafka、RocketMQ、RabbitMQ 的通用面试重点，强调可靠性、顺序性、重复消费、积压治理和业务一致性。

## 一、MQ 面试总框架

### 1. 为什么系统需要消息队列？

**结论先行：**
消息队列主要解决异步解耦、削峰填谷、最终一致和事件驱动问题，但会引入延迟、重复消费、消息丢失和运维复杂度。

**分层展开：**

**核心价值**
- 异步：缩短主链路耗时。
- 解耦：生产者不直接依赖消费者。
- 削峰：平滑突发流量。
- 广播：同一事件驱动多个业务。

**典型场景**
- 下单后发券、积分、通知。
- 支付成功后履约。
- 缓存删除补偿。
- 日志和行为数据采集。

**引入代价**
- 一致性从强一致变最终一致。
- 消费失败要重试和补偿。
- 消息积压会影响业务时效。
- 排查链路更复杂。

**面试追问：什么时候不应该用 MQ？**
强同步返回、强事务、低 QPS 简单调用、消费者结果必须立即影响主流程时，不应为了架构复杂度强行上 MQ。

## 二、可靠性与幂等

### 2. 如何保证消息不丢失？

**结论先行：**
消息不丢失要从生产端、Broker、消费端三段保证：生产确认、Broker 持久化和副本、消费成功后再 ack。

**分层展开：**

**生产端**
- 发送消息等待 broker ack。
- 失败重试。
- 本地消息表或 outbox 保证业务数据和消息一致。

**Broker 端**
- 消息持久化。
- 多副本复制。
- 合理配置 ack 策略。
- 监控 ISR 或副本状态。

**消费端**
- 处理成功后提交 offset 或 ack。
- 失败不确认，进入重试。
- 消费逻辑幂等。

**面试追问：做到绝对不丢失了吗？**
工程上是尽量降低概率并可追溯补偿。跨系统绝对不丢失需要完整事务和审计，成本很高。

### 3. 如何处理重复消费？

**结论先行：**
MQ 通常只能保证至少一次投递，重复消费是常态，业务消费者必须设计幂等。

**分层展开：**

**重复来源**
- 生产者重试。
- Broker 重投。
- 消费成功但 ack 失败。
- rebalance 后重复拉取。

**幂等方案**
- 唯一业务 ID。
- 去重表。
- 状态机判断。
- 数据库唯一键。
- Redis setnx 短期去重。

**工程建议**
- 幂等要落在最终写入资源上。
- 去重记录要有生命周期。
- 不要只依赖内存去重。

**面试追问：Redis 去重够安全吗？**
适合短期防抖和低风险场景。核心资金、订单状态应使用数据库唯一约束或状态机，Redis 故障不能影响正确性。

### 4. 如何保证消息顺序？

**结论先行：**
消息顺序要保证同一业务维度的消息进入同一个分区或队列，并由单消费者串行处理。

**分层展开：**

**生产端**
- 按业务 key 选择分区，例如 order_id。
- 同一 key 的消息发送到同一队列。

**Broker 端**
- 单分区内有序。
- 多分区之间不保证全局顺序。

**消费端**
- 同一分区单线程或有序消费。
- 失败消息会阻塞后续消息，要有策略。

**工程取舍**
- 全局有序吞吐低。
- 局部有序更常见。
- 乱序可用状态机和版本号兜底。

**面试追问：订单消息需要全局有序吗？**
通常不需要，只需要同一订单维度有序。全局有序代价太高。

### 5. 消息积压怎么处理？

**结论先行：**
消息积压先止血定位生产速度和消费速度差距，再扩容消费者、优化消费逻辑、临时跳过非核心处理或重建消费组。

**分层展开：**

**定位**
- 看积压量和增长速度。
- 看生产 QPS 和消费 QPS。
- 查消费错误率和耗时。
- 查下游 DB/Redis 是否慢。

**止血**
- 扩容消费者。
- 提高并发消费。
- 暂停非核心消息。
- 降级慢依赖。

**根因治理**
- 优化消费 SQL。
- 批量处理。
- 拆分 topic。
- 增加分区。
- 消费失败进入死信。

**面试追问：只扩消费者一定有用吗？**
不一定。如果瓶颈在分区数、顺序消费、下游数据库或锁竞争，盲目扩消费者没有收益，甚至会压垮下游。

## 三、事务、一致性与业务设计

### 6. 本地事务和发送消息如何保证一致？

**结论先行：**
常用方案是本地消息表或 outbox：业务数据和消息记录在同一个数据库事务中提交，再由异步任务投递 MQ。

**分层展开：**

**问题**
- 先写 DB 后发 MQ，发 MQ 可能失败。
- 先发 MQ 后写 DB，DB 可能失败。
- 本地事务无法直接覆盖 MQ。

**本地消息表**
- 业务表和消息表同事务写入。
- 后台任务扫描未发送消息。
- 发送成功后标记状态。
- 失败重试。

**outbox 模式**
- 本质也是把事件写入本地持久存储。
- 由投递器可靠发送。
- 可结合 binlog 订阅。

**面试追问：事务消息能完全替代本地消息表吗？**
不一定。RocketMQ 事务消息能解决一部分问题，但业务回查、状态管理和补偿仍要设计。本地消息表更通用可控。

### 7. 最终一致怎么设计？

**结论先行：**
最终一致不是简单异步，而是要有可靠事件、幂等消费、失败重试、补偿任务、对账和可观测性。

**分层展开：**

**核心链路**
- 主事务落库。
- 可靠发布事件。
- 消费者幂等处理。
- 失败进入重试或死信。

**补偿机制**
- 定时扫描异常状态。
- 对账任务。
- 人工修复入口。
- 告警和审计。

**业务边界**
- 明确不一致窗口。
- 明确用户可见状态。
- 核心资金链路要更严格。

**面试追问：最终一致的用户体验怎么处理？**
通过中间态、处理中状态、轮询或通知来表达异步处理过程，不能让用户看到互相矛盾的状态。

### 8. 延迟消息适合什么场景？

**结论先行：**
延迟消息适合订单超时关闭、支付超时检查、任务重试、预约通知等需要未来某个时间触发的场景。

**分层展开：**

**常见实现**
- RocketMQ 延迟消息。
- Kafka 延迟 topic 或时间轮。
- Redis ZSet 延迟队列。
- 定时任务扫描。

**设计注意**
- 延迟精度要求。
- 消息量和峰值。
- 失败重试和幂等。
- 业务状态二次校验。

**工程建议**
- 延迟触发后不要直接执行危险动作。
- 先查当前业务状态。
- 避免用户已支付但订单被关闭。

**面试追问：订单超时关闭为什么要二次查状态？**
因为延迟消息可能晚到、重复到或订单状态已变化。关闭前必须确认订单仍处于待支付状态。

## 四、Kafka、RocketMQ、RabbitMQ 选型

### 9. Kafka 为什么吞吐高？

**结论先行：**
Kafka 吞吐高主要来自顺序写磁盘、page cache、批量发送、零拷贝和分区并行。

**分层展开：**

**存储层**
- 日志追加写。
- 顺序 IO。
- 充分利用 page cache。

**传输层**
- 批量生产和消费。
- sendfile 零拷贝。
- 压缩减少网络传输。

**架构层**
- topic 分区。
- consumer group 并行消费。
- 多 broker 横向扩展。

**面试追问：Kafka 适合低延迟单条消息吗？**
Kafka 更擅长高吞吐日志流。低延迟单条消息也能做，但要在 batch、linger、acks 等参数上取舍。

### 10. RocketMQ 有什么特点？

**结论先行：**
RocketMQ 在业务消息场景能力强，支持事务消息、延迟消息、顺序消息和较完善的重试死信机制。

**分层展开：**

**优势**
- 事务消息。
- 延迟消息。
- 顺序消息。
- 消费重试和死信。
- 适合电商交易场景。

**注意点**
- Broker、NameServer、消费组治理要规范。
- 事务消息回查逻辑要可靠。
- 延迟级别和精度要看版本能力。

**面试追问：RocketMQ 事务消息原理是什么？**
先发送半消息，执行本地事务，再提交或回滚消息；如果生产者异常，Broker 会回查本地事务状态。

### 11. RabbitMQ 适合什么场景？

**结论先行：**
RabbitMQ 适合传统业务队列、复杂路由和较低吞吐的可靠消息场景，不是大数据日志流的首选。

**分层展开：**

**优势**
- Exchange 路由灵活。
- ack、重试、死信机制成熟。
- 适合任务队列和业务通知。

**限制**
- 极高吞吐不如 Kafka。
- 集群和镜像队列治理复杂。
- 大量积压时性能可能下降明显。

**面试追问：如何选择 Kafka、RocketMQ、RabbitMQ？**
日志流和大吞吐选 Kafka；交易业务、延迟事务消息偏 RocketMQ；复杂路由和传统任务队列可选 RabbitMQ。

## 五、线上治理

### 12. 消费者失败重试怎么设计？

**结论先行：**
重试要区分临时失败和永久失败，设置最大次数、退避策略、死信队列和人工处理机制。

**分层展开：**

**临时失败**
- 网络超时。
- 下游短暂不可用。
- 数据还未就绪。

**永久失败**
- 参数非法。
- 业务状态不允许。
- 数据格式错误。

**治理**
- 指数退避。
- 最大重试次数。
- 死信队列。
- 告警和工单。
- 幂等消费。

**面试追问：为什么不能无限重试？**
无限重试会阻塞队列、放大故障、拖垮下游。不可恢复错误要尽快进入死信和人工处理。

### 13. MQ 如何监控？

**结论先行：**
MQ 监控要覆盖生产、Broker、消费、业务结果四层，不能只看消息是否发送成功。

**分层展开：**

**生产侧**
- 发送成功率。
- 发送耗时。
- 重试次数。

**Broker**
- topic 积压。
- 分区/队列状态。
- 磁盘、网络、复制延迟。

**消费侧**
- 消费 QPS。
- 消费耗时。
- 失败率。
- lag。

**业务侧**
- 订单状态是否推进。
- 补偿任务数量。
- 死信数量。

**面试追问：MQ 发送成功就代表业务成功吗？**
不代表。发送成功只代表事件进入 MQ，最终业务是否完成取决于消费者处理、幂等、补偿和下游状态。

## 六、你的项目版 MQ 高频追问

### 15. 枫叶互动广告平台里 MQ 用在哪些场景？

**推荐回答：**

在枫叶互动广告投放平台里，Kafka 主要承担异步解耦、削峰、失败重试和最终一致性：

- 广告创建：前台提交任务后，后台消费者异步调用媒体 API。
- 素材同步：素材入库或修改后，异步同步到媒体账号和 ES。
- ES 更新：MongoDB 权威数据变更后，通过 Kafka 更新 ES 索引。
- 导出任务：大报表导出异步化，避免 HTTP 请求长时间阻塞。
- 通知任务：素材违规、任务失败、报表预警等通知异步发送。
- 埋点/日志：前端操作、投放行为、平台返回日志进入异步链路。

**一句话总结：**
MQ 在这个项目里不是“为了用 Kafka”，而是把外部 API 调用、搜索索引、通知、导出这些不适合同步阻塞用户操作的链路拆出去。

### 16. 广告创建消息如何保证幂等？

**业务问题：**
广告创建调用媒体 API，消息可能因为生产重试、消费失败、消费者重平衡而重复投递。如果不做幂等，可能重复创建广告，造成投放事故。

**幂等设计：**
- 消息里带 `msg_id`、`task_id`、`trace_id`、`msg_type`。
- 广告任务表/Mongo 文档中记录任务状态。
- 消费前检查任务状态，只有 `pending` 或可重试状态才能进入 `publishing`。
- 状态更新使用条件更新，例如 `pending -> publishing`。
- 媒体平台支持幂等 key 时传业务唯一 key；不支持时用本地任务状态和平台广告 ID 兜底。
- 消费成功后记录平台返回 ID，后续重复消息看到已成功则直接 ack。

**追问：如果调用媒体 API 成功，但更新本地状态失败怎么办？**
这属于典型不确定状态。需要保留请求 trace 和业务 key，后续通过平台查询接口反查是否创建成功。如果能查到平台广告 ID，就修正本地状态；查不到再进入重试或人工处理。

### 17. DB/Mongo 到 ES 的消息同步如何保证最终一致？

**链路：**
1. 业务更新 MongoDB 中的素材或广告任务。
2. 发送 Kafka 消息，消息携带数据 ID、操作类型、版本号和 trace_id。
3. 消费者读取权威数据源，构造 ES document。
4. 使用 ES Bulk API upsert/delete。
5. 失败记录重试次数和错误原因。
6. fallback 任务定时扫描未同步或版本落后的数据进行补偿。

**为什么消费者要回查 DB/Mongo，而不是完全相信消息体？**
消息可能乱序或延迟。消费者回查权威数据源，可以拿到最新状态，减少旧消息覆盖新数据的风险。删除消息可以特殊处理，删除 ES 时 404 视为成功，保证幂等。

### 18. 消息积压如何结合 360 项目回答？

**业务背景：**
360 家庭防火墙设备事件上报存在突发流量，峰值事件会造成消费堆积，影响 App 推送、设备状态更新和数据统计。

**排查步骤：**
- 看 consumer lag，是所有分区都积压还是局部分区积压。
- 看消费者实例数、消费耗时、错误率和重试次数。
- 看下游 DB/Redis/HTTP 是否变慢。
- 看消息 key 是否导致分区倾斜。

**处理方案：**
- 临时扩容消费者，但不能超过分区数太多。
- 慢逻辑异步化，例如 App 推送从主消费流程拆出去。
- 多 topic 隔离不同优先级事件，避免低优先级拖慢主链路。
- 消费端批量处理和批量写 DB。
- 对失败消息进入重试队列或死信队列，避免阻塞主消费。

**面试总结：**
消息积压不能只说“加机器”，要先判断瓶颈在消费端 CPU、下游存储、外部接口、分区倾斜还是失败重试。

### 19. 如何设计 Kafka 消息信封？

**推荐结构：**

```json
{
  "version": "1.0",
  "trace_id": "trace-xxx",
  "source": "ad-api",
  "msg_id": "uuid",
  "msg_type": "resource_es_upsert",
  "msg_time": 1730000000,
  "data": {},
  "extra": {}
}
```

**字段作用：**
- `version`：兼容消息结构升级。
- `trace_id`：串联一次业务操作。
- `source`：定位生产方。
- `msg_id`：消费幂等和排查。
- `msg_type`：同 topic 多类型消息分发。
- `msg_time`：判断延迟和过期消息。
- `data`：业务数据。
- `extra`：灰度、重试、扩展信息。

**追问：消息结构升级怎么办？**
新增字段保持向后兼容，消费者按 version 做兼容解析；不要随意删除或改名老字段。重要消息可以保留 schema 校验或灰度消费。

**结论先行：**
支付成功事件要以支付单状态变更为准，通过本地事务记录事件，MQ 发布，消费者幂等推进订单、履约、通知等后续流程。

**分层展开：**

**生产端**
- 支付回调幂等。
- 更新支付单成功。
- 同事务写 outbox 事件。
- 异步投递 MQ。

**消费端**
- 按支付单号或订单号幂等。
- 校验当前状态。
- 推进订单状态。
- 触发履约和通知。

**补偿**
- outbox 未发送重试。
- 消费失败重试。
- 定时对账支付渠道和本地状态。

**面试追问：支付回调重复来了怎么办？**
用支付流水号或支付单号做唯一约束，状态机判断已成功则直接返回成功，保证回调幂等。
