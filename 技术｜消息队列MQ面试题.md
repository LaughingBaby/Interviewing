# 技术｜消息队列 MQ 面试题

> 面向 9 年以上 Golang Web 后端工程师。本文以 Kafka 为主线，按照“场景与边界 → 核心模型 → 生产 → 存储复制 → 消费 → 一致性 → 性能容量 → 治理排障 → 业务架构”的知识图谱组织。
>
> 回答统一遵循金字塔结构：**结论先行 → 核心机制 → 完整流程 → 工程取舍 → 生产实践 → 面试追问**。

## 一、Kafka 学习地图与资深面试回答框架

### 1. Kafka 的知识图谱主线是什么？

**结论先行：**

Kafka 是一个以 Partition 为并行和顺序边界、以副本机制保证可用性、以 Consumer Group 扩展消费能力、以 Offset 管理消费进度的分布式日志系统。

```text
场景与边界
  -> 核心模型
  -> Producer
  -> Broker
  -> Consumer
  -> 可靠性与一致性
  -> 性能与容量
  -> 治理与排障
  -> 业务架构与选型
```

**核心关系：**

```text
吞吐上限     <- Partition 数量、批量、压缩、Broker 资源
局部顺序     <- 相同业务 Key 进入同一 Partition
写入可靠性   <- acks + ISR + min.insync.replicas + 幂等 Producer
消费可靠性   <- 业务成功后提交 Offset + 消费端幂等
水平扩展     <- Partition 在 Broker 和 Consumer 之间分摊
最终一致     <- 可靠事件 + 幂等 + 重试 + 补偿 + 对账
```

### 2. 资深后端如何回答 Kafka 问题？

**结论先行：**

不要只背配置或定义。先回答它解决什么问题，再解释实现机制，最后说明失效边界、工程取舍和线上证据。

| 题型 | 回答顺序 |
|---|---|
| 概念题 | 定义 → 对象关系 → 适用边界 → 常见误区 |
| 原理题 | 数据结构 → 正常流程 → 异常流程 → 设计代价 |
| 配置题 | 配置作用 → 参数联动 → 吞吐/延迟/可靠性取舍 → 监控验证 |
| 排障题 | 影响面 → 指标和证据 → 定向止血 → 根因 → 恢复验证 |
| 架构题 | 业务目标 → 事件模型 → 正确性 → 容量 → 治理 → 降级补偿 |

**面试追问：高级岗位和中级岗位的回答差别是什么？**

中级岗位通常能说明 API 和常见方案；高级岗位还要能确定业务边界、量化容量、处理故障、设计补偿，并说明为什么没有选择另一种方案。

## 二、使用场景与能力边界

### 3. Kafka 解决什么问题？

**结论先行：**

Kafka 主要解决高吞吐事件流中的异步解耦、削峰填谷、多订阅分发、最终一致和数据重放问题，但会引入延迟、重复、积压和运维复杂度。

**核心能力：**

- 异步：把通知、索引、导出等非主链路任务移出同步请求。
- 解耦：生产者只发布稳定事件，不感知每个下游实现。
- 削峰：Broker 持久化突发流量，消费者按可承受速度处理。
- 多订阅：不同 Consumer Group 独立处理同一事件流。
- 重放：按 Offset 重新消费，重建索引、读模型或修复数据。
- 事件驱动：业务状态变化通过事件推进后续流程。

**工程取舍：**

- 同步强一致变为可控的不一致窗口。
- 调用栈变成跨服务异步链路，定位需要消息身份和全链路追踪。
- Broker 能承接积压，但不能代替容量规划和下游保护。
- 至少一次投递下，重复是正常现象，消费者必须幂等。

### 4. 什么场景不适合使用 Kafka？

**结论先行：**

结果必须立即返回、需要跨外部系统强原子事务、消息量很小但路由复杂、要求精确单条定时触发时，不应只因为“解耦”就引入 Kafka。

**典型边界：**

| 场景 | 原因 | 更合理的方向 |
|---|---|---|
| 登录鉴权结果 | 主流程必须同步获得结果 | RPC/本地校验 |
| 扣款与账务记账 | 需要严格业务正确性 | 本地事务、状态机、对账；事件只做后续驱动 |
| 任意条件历史查询 | Kafka 按日志位置读取 | 数据库、搜索或分析存储 |
| 精确到时执行单条任务 | 原生模型不是定时队列 | 调度系统、时间轮或专用延迟能力 |
| 小流量复杂路由 | 分区日志模型不擅长丰富路由 | 按团队现有消息体系评估 |

**面试追问：Kafka 能长期保存数据，为什么不是数据库？**

它擅长按 Partition 和 Offset 顺序读写，不提供数据库式任意索引、关联查询、约束和事务模型。长期保留日志不等于适合作为业务查询源。

## 三、核心数据模型

### 5. Kafka 的核心对象如何串起来？

**结论先行：**

Topic 表达一类事件，Partition 是存储、并行和顺序边界，Record 是日志记录，Offset 是分区内位置，Consumer Group 是一个逻辑订阅者，Replica 提供数据冗余。

```text
Cluster
  ├── Broker 1: Topic A / Partition 0 Leader
  ├── Broker 2: Topic A / Partition 1 Leader + Partition 0 Follower
  └── Broker 3: Topic A / Partition 2 Leader + 其他 Follower

Producer --Key--> Topic A / Partition N / Record(offset)
                                      |
                                      +--> Consumer Group X
                                      +--> Consumer Group Y
```

| 对象 | 职责 | 常见误区 |
|---|---|---|
| Broker | 保存分区副本并处理请求 | Broker 数量不等于消费并发度 |
| Topic | 事件流的逻辑分类 | Topic 不是越细越好 |
| Partition | 存储分片、并行度和顺序边界 | 跨 Partition 不保证全局顺序 |
| Record | Key、Value、Header、Timestamp 等组成的记录 | 大文件不应直接放入消息体 |
| Offset | Record 在 Partition 中的位置 | 不是全局消息 ID，也不是业务成功证明 |
| Replica | Partition 的冗余副本 | 副本存在不代表始终同步 |
| Consumer Group | 一个逻辑订阅方，组内分摊 Partition | 不同业务不应为节省资源共用 Group |

### 6. Partition 为什么是 Kafka 最核心的设计？

**结论先行：**

Partition 同时决定生产吞吐、消费并行度、局部顺序、数据分布和故障恢复粒度，因此分区设计本质上是业务正确性与容量设计的交点。

```text
同一 Consumer Group 中：
一个 Partition 同一时刻只归一个 Consumer 成员
有效消费并行度 <= Partition 数量
Consumer 数量 > Partition 数量时，多余成员空闲
```

**工程取舍：**

- 分区少：并行能力不足，但顺序和治理简单。
- 分区多：吞吐上限更高，但文件、元数据、复制、选举和 Rebalance 成本增加。
- 扩分区会改变 Key 到 Partition 的映射；对依赖同一 Key 历史顺序的业务要提前评估。

**面试追问：分区数增加后，相同 Key 仍然有序吗？**

扩容前后的哈希取模结果可能不同，新旧消息可能位于不同 Partition。单个 Partition 内仍有序，但跨扩容时点不能自动保证该 Key 的全历史顺序。

### 7. 一条消息的完整生命周期是什么？

**完整流程：**

```text
1. 业务生成事件和稳定业务 Key
2. Producer 序列化并选择 Partition
3. 消息进入客户端批次，由 Sender 发往 Leader
4. Leader 追加日志，Follower 拉取复制
5. 满足 acks 条件后 Producer 收到结果
6. Consumer Group 为成员分配 Partition
7. Consumer Fetch/Poll 消息并执行幂等业务
8. 业务成功后提交下一条待消费 Offset
9. 监控记录端到端延迟、失败、Lag 和业务结果
10. 失败消息进入重试、死信、补偿或对账链路
```

**生产实践：**

消息必须带 `msg_id`、业务 ID、事件类型、Schema 版本、发生时间和 `trace_id`，才能把生产、存储、消费和业务结果串成可审计链路。

## 四、Producer 生产机制

### 8. Producer 的完整发送链路是什么？

**结论先行：**

Producer 不是业务线程直接逐条写 Broker，而是经过序列化、分区、客户端聚批和异步网络发送；吞吐、延迟、可靠性与内存压力都在这条链路上相互制约。

```text
业务线程
  -> Serializer
  -> Partitioner
  -> RecordAccumulator（按 Topic-Partition 聚合批次）
  -> Sender（网络线程）
  -> Broker Leader
  -> ISR 副本复制
  -> 按 acks 条件返回 ACK
```

**核心机制：**

1. Serializer 把 Key 和 Value 转成字节。
2. Partitioner 根据显式分区、Key 或客户端策略选择 Partition。
3. RecordAccumulator 为每个 Topic-Partition 聚合批次。
4. Sender 选择已就绪批次，经网络连接发给目标 Leader。
5. Leader 追加日志并根据可靠性配置等待副本。
6. 成功回调或可重试错误进入重试；不可重试错误返回业务。

**面试追问：业务发送方法返回成功意味着什么？**

只代表达到了 Producer 配置所定义的 Broker 确认条件，不代表消费者业务完成，更不代表外部数据库已最终一致。

### 9. `acks`、ISR 和 `min.insync.replicas` 如何联动？

**结论先行：**

可靠业务通常使用 `acks=all`，并让 `min.insync.replicas` 至少为 2、Replication Factor 通常为 3；这样在同步副本不足时拒绝写入，用可用性换数据安全。

| 配置 | 含义 | 主要风险 |
|---|---|---|
| `acks=0` | 不等待 Broker 确认 | 客户端几乎无法感知丢失 |
| `acks=1` | Leader 写入后确认 | Leader 在副本追平前故障可能丢数据 |
| `acks=all` | 等待当前 ISR 达到确认要求 | ISR 不足时写入失败，延迟也更高 |
| `min.insync.replicas` | `acks=all` 时要求的最少同步副本 | 过低保护不足，过高降低故障时可写性 |

**生产实践：**

不能只看 `acks`。还要联合监控 ISR、Under Replicated Partitions、生产错误和副本延迟，并明确故障时选择“拒绝写入”还是“降级可靠性”。

### 10. Producer 如何通过批量和压缩提高吞吐？

**结论先行：**

`batch.size` 决定单批容量，`linger.ms` 允许等待更多消息形成批次，压缩减少网络和磁盘字节；更大批次通常提高吞吐，但会增加延迟和客户端内存占用。

| 配置 | 影响 | 调优判断 |
|---|---|---|
| `batch.size` | 单 Partition 批次上限 | 小批次浪费请求，大批次占内存 |
| `linger.ms` | 等待聚批时间 | 提升批量率，但增加端到端延迟 |
| `compression.type` | 网络、Broker 存储和 CPU | 按消息类型和 CPU/网络瓶颈压测 |
| `buffer.memory` | Producer 总缓冲区 | 满时业务线程会阻塞或失败 |
| `max.block.ms` | 获取元数据或缓冲空间最多阻塞多久 | 防止请求线程无限等待 |

**工程取舍：**

调优必须同时观察平均批次大小、压缩率、发送延迟、缓冲等待、错误率、Broker CPU 和网络，不能只追求更高 TPS。

### 11. 幂等 Producer 和事务 Producer 分别解决什么？

**结论先行：**

幂等 Producer 解决单个 Producer 会话内因网络重试产生的重复写；事务 Producer 进一步原子提交多个 Kafka 分区写入及消费位点，但都不等于业务请求全局幂等。

**核心机制：**

- 幂等写通过 Producer 身份、序列号和 Broker 端校验识别重试重复。
- `max.in.flight.requests.per.connection` 会影响并发请求和重试顺序；使用客户端推荐的幂等配置组合，避免手工参数破坏保证。
- 事务通过 `transactional.id` 识别 Producer，异常实例会被 fencing。
- 消费者使用 `read_committed` 才会屏蔽已中止事务的数据。

**边界：**

用户重复点击、服务重复调用、进程重启后的业务重复、消费者重复执行，仍需业务幂等键、唯一约束和状态机处理。

### 12. Producer 如何处理热点 Key、背压和优雅停机？

**结论先行：**

热点 Key 会让单个 Partition 成为瓶颈；缓冲区耗尽说明 Broker 或网络跟不上生产速度。解决时要先区分业务顺序要求，再决定拆 Key、限流还是扩容。

**生产实践：**

- 必须有序：保留稳定 Key，优化单分区消费与下游写入，必要时按更细业务实体拆分。
- 不要求严格顺序：Key 加可控散列后缀，消费端再聚合。
- Producer 设置发送超时、错误分类、重试上限和熔断，不能在请求线程无限重试。
- 监控缓冲区等待、批次大小、发送错误、Throttle 时间和热点分区。
- Go 服务停机时先停止接收新请求，等待已接收发送结果，Flush/Close 后退出；失败事件应有 Outbox 或补偿来源。

**面试追问：为什么不能看到发送失败就无限重试？**

无限重试会占满内存和连接，放大 Broker 故障，并拖住业务线程。重试必须受投递超时和业务时效约束，可靠事件还要有持久化补偿来源。

## 五、Broker 存储、复制与集群架构

### 13. Kafka 的日志文件如何组织？

**结论先行：**

每个 Partition 是一份追加日志，目录下按 Segment 分段；每个 Segment 由数据文件和稀疏索引组成，便于滚动、删除、恢复和定位。

```text
topic-a-0/
  00000000000000000000.log
  00000000000000000000.index
  00000000000000000000.timeindex
  00000000001234567890.log
  00000000001234567890.index
  00000000001234567890.timeindex
```

| 文件 | 作用 |
|---|---|
| `.log` | 顺序保存 Record Batch |
| `.index` | 相对 Offset 到文件物理位置的稀疏索引 |
| `.timeindex` | 时间戳到相对 Offset 的稀疏索引 |

**查找流程：**

```text
目标 Offset
  -> 根据 Segment 起始 Offset 定位 Segment
  -> 在稀疏 .index 中找到不大于目标值的最近索引项
  -> 跳到 .log 对应物理位置
  -> 顺序扫描到目标 Record Batch
```

### 14. Kafka 为什么吞吐高？

**结论先行：**

高吞吐来自顺序追加、Page Cache、批量、压缩、零拷贝和 Partition 并行的共同作用，不是“磁盘一定比内存快”。

**核心机制：**

- 顺序追加减少随机寻道和复杂原地更新。
- Page Cache 由操作系统统一缓存热日志，避免应用维护巨大对象堆。
- Producer 和 Consumer 都以 Record Batch 为主要传输单位。
- 压缩按批次执行，减少网络与磁盘字节。
- 文件向网络发送可利用 `sendfile` 等零拷贝路径，减少用户态拷贝和上下文切换。
- Partition 让不同 Broker、磁盘和消费者并行工作。

**工程取舍：**

批量和压缩会消耗 CPU 并增加等待；零拷贝路径也受加密等条件影响。真实性能必须通过端到端压测和系统指标验证。

### 15. Retention 和 Log Compaction 有什么区别？

**结论先行：**

Retention 按时间或空间删除旧 Segment；Log Compaction 按 Key 保留较新的值。前者适合事件日志，后者适合可重建的状态流。

```text
行为事件、审计记录、操作流水 -> cleanup.policy=delete
账户最新配置、实体最新状态   -> cleanup.policy=compact
需要状态快照且限制历史窗口   -> compact,delete
```

**边界：**

- Compact 是后台异步过程，不保证重复 Key 立即只剩一条。
- Value 为空通常表示 tombstone，消费者必须处理删除语义。
- Compact 不等于数据库唯一约束，也不能解决业务重复执行。
- Retention 删除以 Segment 为单位，不是消息到点立即逐条删除。

### 16. Leader、Follower、ISR、LEO 和 HW 是什么？

**结论先行：**

Leader 处理分区读写，Follower 拉取复制；LEO 表示副本日志末端位置，ISR 是保持同步的副本集合，HW 是默认可见的高水位边界。

```text
Leader:    0 1 2 3 4 5 6   LEO=7
Follower1: 0 1 2 3 4 5 6   LEO=7
Follower2: 0 1 2 3 4 5     LEO=6

ISR = {Leader, Follower1}
HW  = 已被 ISR 可靠推进的可见边界
```

**生产实践：**

- Replication Factor 表示副本目标数量，不代表所有副本当前健康。
- ISR 缩小要结合磁盘延迟、网络、GC、Broker 负载和副本分布排查。
- 副本要跨故障域分布，避免同机架故障同时损失多份数据。

### 17. Leader 故障后如何恢复？

**结论先行：**

Controller 从可选副本中选举新 Leader，客户端刷新元数据后恢复请求；可靠性取决于是否从 ISR 选举，以及故障前消息是否达到提交条件。

**异常流程：**

```text
Broker 失联
  -> Controller 感知副本和 Broker 状态变化
  -> 为受影响 Partition 选择新 Leader
  -> 更新元数据并传播
  -> Producer/Consumer 刷新元数据
  -> Follower 继续追赶并恢复 ISR
```

**工程取舍：**

允许非 ISR 副本参与非干净选举可以提高极端故障下的可用性，但可能丢失已在旧 Leader、尚未同步的数据。核心事件通常优先一致性，不轻易启用这种降级。

### 18. KRaft 解决什么问题？

**结论先行：**

KRaft 使用 Kafka 自身的元数据日志和 Raft 风格仲裁管理 Controller 元数据，统一集群元数据体系，减少外部协调组件和双系统运维复杂度。

**核心角色：**

- Controller Quorum 维护 Broker、Topic、Partition、副本和配置等元数据。
- Active Controller 处理元数据变更，其他 Controller 参与仲裁并保持日志同步。
- Broker 注册并接收元数据变化，负责实际数据读写和复制。

**生产实践：**

- Controller 与数据 Broker 的资源、故障域和监控要明确规划。
- 重点监控 Controller 选举、元数据日志同步、请求延迟和 Broker 注册状态。
- 升级或迁移必须按部署版本支持的官方路径演练，不能把元数据切换当成普通配置变更。

**面试追问：Controller 管元数据，是否也承载业务消息？**

职责不同。Controller 管理集群元数据和选举，业务 Record 仍由对应 Partition Leader 所在 Broker 处理。

## 六、Consumer 与消费组

### 19. Consumer Group、Offset 和消费并行度是什么关系？

**结论先行：**

一个 Consumer Group 表示一个逻辑订阅者，组内成员分摊 Partition；Offset 按 Group、Topic、Partition 记录恢复位置，有效并行度不超过 Partition 数量。

```text
Topic: 6 Partitions
Group A: 3 Consumers -> 每个成员约 2 个 Partition
Group B: 6 Consumers -> 每个成员约 1 个 Partition
Group C: 8 Consumers -> 2 个成员空闲
```

**Offset 误区：**

- 提交的是下一条待消费位置。
- Offset 前进不等于业务数据已经正确写入。
- 先提交后处理可能丢业务；先处理后提交可能重复。
- 不同 Group 的 Offset 互相独立。

### 20. Consumer 的 Fetch、Poll 和业务处理如何配合？

**结论先行：**

Consumer 从 Broker 批量 Fetch，应用通过 Poll 取得记录并处理；单批大小、处理耗时、提交时机和心跳存活必须统一设计。

**核心参数：**

| 配置 | 作用 | 风险 |
|---|---|---|
| `fetch.min.bytes` | Broker 累积到一定数据再返回 | 提高吞吐但可能增加等待 |
| `fetch.max.wait.ms` | Fetch 最长等待时间 | 与最小字节共同决定拉取延迟 |
| `max.poll.records` | 单次交给应用的最大记录数 | 太大可能处理超时 |
| `max.poll.interval.ms` | 两次 Poll 的最大间隔 | 超过后成员可能被移出组 |
| `session.timeout.ms` | 会话失效时间 | 太小易抖动，太大故障发现慢 |
| `heartbeat.interval.ms` | 心跳频率 | 必须和会话超时协调 |

**生产实践：**

先用消费耗时分位数确定批量上限，再压测吞吐。不要靠无限增大 Poll 间隔掩盖慢 SQL、外部 API 或锁竞争。

### 21. Consumer Group 如何完成分区分配？

**结论先行：**

成员找到 Group Coordinator 后加入组，协调出成员和订阅信息，再同步分配结果；成员、订阅或 Partition 变化会触发重新分配。

```text
FindCoordinator
  -> JoinGroup：上报成员和订阅
  -> 选出 Group Leader/确定协议
  -> 计算 Partition Assignment
  -> SyncGroup：下发分配结果
  -> 稳定消费和心跳
```

| 策略 | 特点 | 适用判断 |
|---|---|---|
| Range | 按 Topic 连续范围分配 | 多 Topic 时可能倾斜 |
| RoundRobin | 跨成员轮询分配 | 较均衡，但变更可能移动较多 Partition |
| Sticky | 尽量均衡且保留原分配 | 降低状态和缓存迁移 |
| Cooperative Sticky | 增量协作式迁移 | 减少全量停止，但客户端策略必须兼容 |

### 22. 为什么会发生 Rebalance，如何降低影响？

**结论先行：**

Rebalance 是成员或 Partition 变化后的所有权重分配；它本身不是故障，但频繁发生会导致暂停、重复处理和状态迁移。

**常见原因：**

- 实例扩缩容、重启、发布或 OOM。
- 心跳超时、网络抖动、长时间 GC。
- 单批处理超过 `max.poll.interval.ms`。
- Topic 增加 Partition 或订阅变化。
- Coordinator 故障。

**治理方式：**

- 控制单批记录数和最慢处理时间。
- 使用 Cooperative Sticky 降低全量迁移。
- 稳定实例可设置 `group.instance.id` 使用静态成员，减少短暂重启引发的抖动。
- Rebalance 回调中暂停拉取、保存可安全提交的 Partition 进度、释放分区级资源。
- 发布时优雅退出，避免所有实例同时离组。

### 23. Go Consumer 如何设计 Worker Pool？

**结论先行：**

Go 中可以把拉取与业务处理解耦，但必须保留 Partition 所有权和 Offset 连续提交语义；不能把消息随意扔进全局 Worker Pool 后按完成顺序提交。

```text
Poll Loop
  -> 按 Topic-Partition 分发
  -> Partition Worker / 有界队列
  -> 幂等业务处理
  -> 记录连续完成水位
  -> 仅提交连续安全 Offset
```

**工程取舍：**

- 要求同 Key 有序：同 Partition 串行或按 Key 路由到固定 Worker。
- 不要求顺序：可并行处理，但必须跟踪空洞，不能因后面的消息先完成就越过失败 Offset 提交。
- Worker 队列必须有界；满时 Pause 对应 Partition 或降低拉取，形成背压。
- 下游调用设置超时、并发上限、退避和熔断。
- 停机时停止 Poll，等待在途任务，提交连续完成水位，再离组。

### 24. 毒消息如何处理？

**结论先行：**

毒消息不能无限阻塞 Partition，也不能无审计跳过。应区分暂时失败和永久失败，使用有界重试、死信、告警和人工修复闭环。

**处理流程：**

```text
解析/业务失败
  -> 分类：临时错误 / 永久错误 / 未知错误
  -> 临时错误：退避后进入重试 Topic
  -> 永久错误：死信 Topic + 告警
  -> 未知错误：有限次重试后转死信
  -> 修复后以隔离 Group 重放
```

## 七、可靠性、顺序与一致性

### 25. 如何保证消息尽量不丢失？

**结论先行：**

消息可靠性必须覆盖生产、Broker、消费和业务补偿四层。任何单一配置都不能证明端到端业务不丢。

| 层次 | 核心措施 |
|---|---|
| 生产 | Outbox/可靠来源、`acks=all`、幂等、有限重试、发送结果审计 |
| Broker | 多副本、合理最小同步副本、跨故障域、ISR 和磁盘监控 |
| 消费 | 业务成功后提交 Offset、幂等、失败重试、优雅停机 |
| 业务 | 状态扫描、对账、补偿、死信处理、可控重放 |

**工程判断：**

工程上追求的是低丢失概率、可发现、可追溯、可补偿。对支付和资产类事件，业务账本和对账系统比“MQ 发成功了”更接近最终正确性证明。

### 26. 为什么会重复消费，如何保证幂等？

**结论先行：**

至少一次投递下，重复是正常结果。幂等应落在最终业务资源，而不是只在消费进程内记忆消息 ID。

**重复来源：**

- Producer 超时重试。
- 消费成功但 Offset 提交失败。
- Rebalance、进程崩溃或网络抖动后重新拉取。
- 业务上游重复产生同一事件。
- 人工重放或补偿扫描。

**幂等手段：**

- 数据库唯一约束：适合创建类操作。
- 状态机条件更新：如 `pending -> paid`，不允许状态倒退。
- 业务幂等键：订单号、支付流水号、任务 ID。
- 版本号：只接受比当前版本新的事件。
- 消费记录表：记录消息 ID、结果和处理时间。
- Redis 去重：适合短期低风险优化，不能作为核心正确性的唯一依赖。

### 27. 如何保证消息顺序？

**结论先行：**

Kafka 只保证单 Partition 内的追加顺序。业务通常需要同一实体局部有序，而不是所有消息全局有序。

```text
order_id / campaign_id / device_id
  -> 稳定 Key
  -> 同一 Partition
  -> Partition 内按 Offset 消费
  -> 消费端串行或连续水位提交
```

**异常兜底：**

- 消息携带业务版本或事件时间。
- 更新使用 `where version < incoming_version` 条件。
- 旧消息到达时丢弃或转审计，不覆盖新状态。
- 失败消息若转入重试 Topic，会改变与后续消息的时间关系；严格顺序业务应阻塞该业务 Key、使用状态机，或把失败实体隔离处理。

**面试追问：全局有序怎么实现？**

可以只使用一个 Partition 并单线程消费，但吞吐和可用性扩展受限。除非业务真的要求全局顺序，否则优先设计局部顺序。

### 28. At-most-once、At-least-once 和业务 Exactly-once 怎么理解？

| 语义 | 典型提交顺序 | 结果 |
|---|---|---|
| At-most-once | 先提交 Offset，再处理 | 可能丢，不轻易重复 |
| At-least-once | 先处理，成功后提交 | 不轻易丢，可能重复 |
| Kafka 内部 Exactly-once | 事务写入 + 位点原子提交 + `read_committed` | Kafka 事务边界内一次可见 |

**关键边界：**

Kafka 事务不能原子覆盖 MySQL、MongoDB、Elasticsearch、Redis 或外部 HTTP API。所谓**业务 Exactly-once**仍要依靠业务幂等、数据库约束、状态机、Outbox、补偿和对账实现可证明的最终结果。

### 29. 本地事务和 Kafka 消息如何保证一致？

**结论先行：**

常用方案是 Transactional Outbox：业务数据和事件记录在同一个本地数据库事务中提交，再由独立 Publisher 或 CDC 投递 Kafka。

```text
本地事务：
  更新业务表
  插入 Outbox 事件
  一起 Commit

异步投递：
  扫描 Outbox 或订阅 CDC
  -> 发布 Kafka
  -> 记录发送结果
  -> 失败重试

消费端：
  幂等处理
  -> 成功后提交 Offset
  -> 失败重试/死信
```

**工程取舍：**

- Outbox 通用、边界清晰，但要治理表增长、扫描竞争和发送状态。
- CDC 减少业务侵入，但增加变更捕获平台、Schema 和顺序治理。
- 两者通常仍是至少一次投递，消费者幂等不能省略。

### 30. 延迟、重试和死信如何设计？

**结论先行：**

Kafka 原生更适合日志流。延迟和失败重试通常通过分级重试 Topic、时间轮/调度器或外部调度能力实现，并在最终失败后进入死信闭环。

```text
主 Topic
  -> 临时失败
  -> retry-1m -> retry-10m -> retry-1h
  -> 超过最大次数
  -> dead-letter Topic
  -> 告警 / 人工修复 / 隔离重放
```

**生产实践：**

- 重试消息保留原 Topic、Partition、Offset、`msg_id`、业务 ID、错误类型和重试次数。
- 使用指数退避和随机抖动，避免故障恢复瞬间形成重试风暴。
- 永久参数错误不要反复重试。
- 延迟触发后重新查询业务状态，例如关闭订单前确认仍未支付。
- 死信必须有负责人、处理时限和可审计操作入口。

## 八、性能优化与容量规划

### 31. Partition 数量如何估算？

**结论先行：**

先通过接近生产环境的压测得到单 Partition 写入和消费能力，再分别按生产峰值和目标消费速度计算，取较大值并预留故障与增长余量。

```text
partition count >= max(
  producer peak throughput / measured partition write throughput,
  required consumer throughput / measured partition consume throughput
)
```

**示例：**

生产峰值为 200 MB/s，压测单 Partition 稳定写入 20 MB/s，需要至少 10 个；消费目标为 300 MB/s，单 Partition 消费 25 MB/s，需要至少 12 个。基础值取 12，再结合 Broker 数量、故障降容和未来增长预留。

**必须修正的变量：**

- 消息大小和 Key 倾斜。
- 压缩率与 CPU。
- 副本复制和跨机架网络。
- 单消费者业务耗时及下游容量。
- Broker 故障后剩余节点承载能力。
- 分区过多带来的元数据和恢复成本。

### 32. 磁盘和 Broker 容量如何估算？

**结论先行：**

先算保留窗口内原始数据，再乘副本和安全余量；实际规划还要考虑压缩、索引、重放、磁盘水位和节点故障后的迁移空间。

```text
raw storage = peak bytes per second * retention seconds
cluster storage = raw storage * replica factor * headroom factor
```

**示例：**

若压缩后持续写入 50 MB/s，保留 3 天，原始保留量约为 `50 MB/s × 259200s`；再乘副本因子 3 和不低于业务要求的安全余量。不能让正常状态就逼近磁盘高水位，否则副本恢复和流量波动没有空间。

**Broker 数量还要考虑：**

- 单节点磁盘吞吐、容量和磁盘数量。
- 单节点网卡的生产、消费和复制总流量。
- Leader 分布和热点流量。
- 一个 Broker 故障后剩余 Broker 是否仍能承载峰值。
- 机架、可用区和维护窗口。

### 33. 消息积压多久可以恢复？

**结论先行：**

只有恢复消费速率大于持续流入速率时，Lag 才会下降；恢复时间取决于两者的净差值，而不是单看消费者数量。

```text
recovery time = accumulated messages /
                (recovery consume rate - incoming rate)
```

如果流入 8 万条/秒，扩容后可消费 12 万条/秒，净恢复速度只有 4 万条/秒。积压 1.44 亿条理论上需要约 1 小时，实际还要加入失败重试、下游限流和流量波动余量。

**面试追问：扩消费者为什么可能无效？**

有效并行度受 Partition 数限制；瓶颈也可能在数据库、外部 API、单 Key 顺序、锁竞争或热点 Partition。盲目扩容可能把下游压垮。

### 34. Producer、Consumer 和 Broker 如何系统调优？

| 层次 | 调优方向 | 必须同时观察 |
|---|---|---|
| Producer | 批量、等待、压缩、并发请求、缓冲 | 端到端延迟、错误、缓冲等待、CPU、网络 |
| Consumer | Fetch 批量、Poll 数量、业务并发、批量写 | Lag、处理分位数、失败率、下游压力、Rebalance |
| Broker | 磁盘、网络、请求线程、副本和 Leader 均衡 | 请求队列、磁盘延迟、ISR、Throttle、Page Cache |
| 数据模型 | 消息大小、Key 分布、Topic/Partition 数量 | 热点、大消息、文件句柄、元数据规模 |

**压测原则：**

- 使用接近真实的消息大小、Key 分布、压缩方式和副本配置。
- 同时压生产、消费和副本复制，不只测空消费者下的写入峰值。
- 测稳定吞吐和 P99 延迟，而不是短时间峰值。
- 注入 Broker 故障、消费者重启和下游变慢，验证降容状态。
- 输出容量基线、告警水位、扩容触发条件和恢复时间。

## 九、生产治理、安全与故障排查

### 35. Kafka 监控体系如何设计？

**结论先行：**

监控要覆盖 Producer、Broker、Consumer 和业务结果四层，并把指标放在同一时间线上，不能只看 Lag 或发送成功率。

| 层次 | 核心指标 |
|---|---|
| Producer | 成功率、错误类型、重试、发送延迟、批次、压缩率、缓冲等待、Throttle |
| Broker | 入出流量、请求延迟和队列、磁盘、网络、ISR、离线分区、Leader 分布 |
| Consumer | Lag 总量和增长率、消费速率、失败率、处理分位数、Rebalance、死信 |
| 业务 | 状态推进成功率、不一致数量、补偿数量、事件端到端延迟、对账差异 |

**告警原则：**

Lag 总量必须结合增长速度和业务时效。报表消息积压十分钟可能可接受，支付履约事件积压一分钟就可能需要高优先级响应。

### 36. 消息积压如何排查和止血？

**结论先行：**

先判断是生产突增、消费变慢、局部分区热点还是 Broker 问题，再在下游承受范围内扩容和恢复，不能看到 Lag 就直接加机器。

```text
所有 Group 同时变慢
  -> Broker / 网络 / 集群级问题

单个 Group 变慢
  -> 消费应用 / 下游依赖问题

单个 Partition 变慢
  -> 热点 Key / 毒消息 / Partition Worker 卡住

生产速率突增
  -> 业务流量 / 重放任务 / 异常重复生产
```

**处理步骤：**

1. 确认影响 Topic、Group、Partition、业务时效和增长速度。
2. 对齐生产速率、消费速率、错误率、处理耗时和下游指标。
3. 止血：隔离低优先级、限重放、转移毒消息、降低下游并发。
4. 确认 Partition 足够且下游有余量后再扩消费者。
5. 恢复时观察净消化速度、错误率、死信和下游水位。

### 37. 频繁 Rebalance 如何排查？

**证据顺序：**

1. 对齐 Rebalance 与发布、重启、OOM、GC、CPU 和网络事件。
2. 检查 `max.poll.interval.ms`、Session Timeout 和 Coordinator 错误。
3. 查看每批消息数、最慢消息、DB/HTTP 调用和 Worker 队列。
4. 检查实例是否频繁扩缩容、健康检查是否过于敏感。
5. 对比分配策略和 Rebalance 前后的 Partition 移动量。

**修复方向：**

- 缩小单批任务或把长任务改为有状态异步任务。
- 使用协作式分配和稳定成员降低迁移。
- 发布时错峰优雅退出。
- 修复慢依赖和无界 Worker，不用无限增大超时掩盖问题。

### 38. ISR 缩小、离线分区和 Broker 故障如何排查？

**结论先行：**

先保护剩余健康副本和核心流量，再判断是磁盘、网络、资源争抢、数据倾斜还是节点故障；不要在证据不足时删除日志或强制重置状态。

**排查顺序：**

- 查看 Partition Leader、Replica、ISR 和 Offline Partition。
- 对齐 Broker 磁盘容量、I/O 延迟、网络吞吐、请求队列、CPU 和进程停顿。
- 检查 Leader 是否集中、单 Broker 是否承载热点、大规模重放或副本迁移。
- 查看 Controller/KRaft 元数据、Leader Election 和 Broker 注册事件。
- 核对 `acks` 与最小同步副本是否导致拒绝写入。

**恢复原则：**

暂停非核心重放和迁移，先让副本追平并恢复 ISR；确认 Leader 分布和错误率稳定后，再分批恢复流量。

### 39. Offset 重置和消息重放有哪些风险？

**结论先行：**

Offset 重置会造成重复或跳过数据，必须先保存原位点、明确范围、验证消费者幂等，并使用隔离 Group 控制重放速度。

**安全流程：**

```text
确认修复目标和时间范围
  -> 保存当前 Group 各 Partition Offset
  -> 评估重复执行和外部副作用
  -> 使用隔离 Group / 隔离环境小范围验证
  -> 限速重放并监控下游
  -> 对账结果
  -> 决定是否切换或回滚
```

### 40. Kafka 安全和多租户如何治理？

**结论先行：**

安全治理至少包括传输加密、身份认证、资源授权和资源配额；多租户还要通过命名、所有权、容量和故障域避免相互影响。

| 机制 | 作用 |
|---|---|
| TLS | 加密客户端、Broker 和跨集群传输 |
| SASL | 对客户端和 Broker 身份认证 |
| ACL | 限制 Topic、Group、Cluster 等资源操作 |
| Quota | 限制租户生产、消费或请求速率，避免抢占集群 |

**治理清单：**

- Topic 命名包含环境、业务域、事件语义和版本。
- 每个 Topic 有 Owner、数据等级、保留时间、峰值和处置预案。
- 禁止大对象直接进入 Kafka，文件放对象存储，消息传引用和校验信息。
- 重要与非重要业务通过 Topic、配额或集群隔离。
- 凭证定期轮换，最小权限授权，审计高风险操作。

### 41. 灾备、跨集群和滚动升级如何考虑？

**结论先行：**

Kafka 副本主要解决单集群节点故障，不自动等同于跨地域灾备。灾备要明确复制方向、RPO/RTO、切换步骤、消费者位点和业务去重。

**生产实践：**

- 跨集群复制需要规划 Topic 配置、ACL、消息身份和消费位点映射。
- 双向复制要防止消息环路和重复。
- 灾备演练必须包含生产切换、消费恢复、重复消息和业务对账。
- 滚动升级前检查版本兼容、协议和元数据模式，先小范围节点验证。
- 升级期间监控 ISR、Leader Election、请求错误、Rebalance 和端到端业务指标。
- Kafka 不是业务数据库备份；权威数据和审计账本仍需自己的备份策略。

### 42. Kafka 故障如何复盘？

**复盘结构：**

```text
影响：哪些业务、持续多久、数据是否丢失或重复
时间线：告警、人工动作、恢复节点
根因：直接原因、触发条件、为何未被提前发现
止血：哪些动作有效，哪些动作放大了风险
恢复：如何验证 Lag、业务成功率、对账和下游压力
改进：监控、容量、发布、隔离、预案和演练
```

高级岗位表达重点是自己如何建立证据链、做风险判断和推进业务恢复，而不是强调人员管理。

## 十、业务架构与项目实战

### 43. Topic、Partition、Key 和 Consumer Group 如何设计？

**结论先行：**

先定义稳定事件语义和业务顺序维度，再按吞吐与恢复目标确定 Partition，按逻辑下游划分 Consumer Group，不能从组件默认值倒推业务模型。

**设计问题：**

1. 这是事实事件、任务命令还是状态快照？
2. 谁是权威数据源，是否需要历史重放？
3. 同一什么实体必须有序？
4. 峰值 TPS、字节流量、保留时间和消息时效是多少？
5. 有哪些独立下游，各自允许多大 Lag？
6. 失败后如何重试、补偿、对账和人工修复？

**生产实践：**

- 高低优先级拆 Topic，避免报表拖慢交易或投放主链路。
- 每个逻辑下游使用独立 Group。
- Key 选择订单、广告任务、设备等需要局部顺序的实体 ID。
- Topic 配置、Schema、Owner、SLA 和告警一起纳入发布评审。

### 44. 消息信封和 Schema 如何演进？

**推荐消息信封：**

```json
{
  "msg_id": "uuid",
  "msg_type": "campaign_created",
  "schema_version": 2,
  "occurred_at": 1730000000,
  "producer": "ad-api",
  "trace_id": "trace-xxx",
  "business_key": "campaign-123",
  "data": {},
  "extensions": {}
}
```

**演进原则：**

- 优先新增可选字段并提供默认语义，避免直接删除或改名。
- Producer 先兼容旧 Consumer，Consumer 先支持新旧版本，再切换生产。
- 明确字段缺失、空值和删除事件的含义。
- 重要事件使用 Schema Registry 或团队统一的 JSON/Protobuf/Avro 兼容检查流程。
- 灰度 Consumer 验证后再扩大生产流量。
- Schema 变化要能关联发布版本、Owner 和回滚方案。

### 45. 如何设计可靠的支付成功事件？

**业务目标：**

支付成功后推进订单、履约、积分和通知，同时避免回调重复、消息丢失和重复履约。

**设计方案：**

```text
支付回调
  -> 以支付流水号幂等
  -> 本地事务更新支付单 + 写 Outbox
  -> Publisher 发布 payment-succeeded
  -> order_id 作为 Key
  -> 履约、积分、通知使用独立 Group
```

**正确性：**

- 支付流水号唯一约束，状态只允许合法前进。
- 消费者按订单号和事件类型幂等。
- 履约成功后再提交安全 Offset。
- Outbox 重试、消费重试、支付渠道对账和业务补偿共同闭环。
- 监控支付成功到履约完成的端到端延迟，而非只看发送成功。

**降级：**

消息系统短暂不可用时，支付本地事务仍可提交 Outbox；通知可延后，核心履约通过恢复投递和对账补齐。

### 46. 广告投放异步任务如何设计？

**业务目标：**

用户提交广告创建任务后快速返回，后台调用多个媒体 API，并处理限流、超时、不确定状态和重复创建风险。

**方案：**

- Topic 表达“广告创建任务已就绪”，以 `campaign_id` 或 `task_id` 为 Key。
- 任务表或 Mongo 文档保存 `pending -> publishing -> success/failed/unknown` 状态机。
- 消费者通过条件更新抢占任务，重复消息看到终态直接确认。
- 媒体支持幂等 Key 时透传业务唯一 ID；否则保存请求身份和平台返回 ID。
- API 成功但本地更新失败时进入 `unknown`，调用媒体查询接口反查，不能盲目重试创建。
- 按媒体或优先级隔离 Topic/Group，设置并发上限、退避、熔断和配额。
- 监控任务成功率、端到端耗时、平台错误码、重试和未知状态数量。

**项目表达：**

强调自己负责业务边界、状态机、幂等和故障恢复设计，以及如何与产品和外部平台联调；管理经历只体现风险识别和协作能力，不把职业方向描述为管理岗位。

### 47. MySQL/MongoDB 到 Elasticsearch 如何保证最终一致？

**结论先行：**

数据库是权威数据源，Kafka 传递可追踪变更，索引消费者幂等写入 Elasticsearch，并通过版本校验、补偿扫描和全量重建保证最终一致。

```text
业务事务 / Change Stream / CDC
  -> Outbox 或变更事件
  -> Kafka
  -> ES Indexer Group
  -> 回查权威数据
  -> Bulk Upsert/Delete
  -> 记录同步版本和结果
```

**关键设计：**

- Key 使用文档或业务实体 ID，保证同一实体尽量局部有序。
- 消息带操作类型、数据版本、`msg_id` 和 `trace_id`。
- Upsert 前比较版本，旧事件不覆盖新索引。
- 消费者优先回查权威源构建最新文档，降低乱序影响。
- Bulk 部分失败必须逐项识别并重试，不能整批视为成功。
- 删除 404 可按幂等成功处理，但要保留审计。
- 定时扫描版本落后数据；大规模修复使用隔离 Group 或新索引重建后切别名。

### 48. 家庭防火墙设备事件如何削峰？

**业务目标：**

设备集中上线或网络恢复会产生突发事件，系统需要保护设备状态、App 推送和统计链路，核心状态优先于非实时分析。

**方案：**

- 按 `device_id` 分区，保证同一设备状态局部有序。
- 设备状态、通知、统计使用独立 Group；高低优先级必要时拆 Topic。
- 状态消费者采用版本或设备序列号，防止旧事件覆盖新状态。
- 批量写数据库和缓存，推送链路设置独立限流与重试。
- 容量按设备集中恢复峰值而不是日均流量规划。
- 积压时优先恢复状态更新，统计和非紧急通知可降级或延后。

**面试追问：为什么不能只扩消费者？**

可能受 Partition 数、单设备热点、数据库写入、推送接口限流和顺序要求限制。扩容前必须验证下游余量和净恢复速度。

### 49. Kafka、RocketMQ、RabbitMQ 如何做业务选型？

**结论先行：**

不按组件名做结论，先看吞吐、延迟、重放、顺序范围、延迟消息、事务边界、路由复杂度、团队运维能力和已有基础设施。

| 维度 | Kafka | RocketMQ | RabbitMQ |
|---|---|---|---|
| 核心定位 | 高吞吐、可重放的分布式日志流 | 交易和业务消息能力较完整 | 传统消息队列和灵活路由 |
| 消费模型 | Partition + Group + Offset | Topic/Queue + 消费组 | Exchange + Queue + ACK |
| 重放 | 强，适合索引和读模型重建 | 支持，按其消费和存储模型治理 | 通常不作为长期日志重放首选 |
| 局部顺序 | 单 Partition | 单队列/分片 | 单 Queue，可用并发受限 |
| 延迟消息 | 通常需要额外调度设计 | 业务能力较成熟 | 可通过 TTL/DLX 等实现，需评估精度和治理 |
| 事务边界 | Kafka 内部事务 | 事务消息和回查机制 | 依赖确认与业务补偿设计 |
| 路由 | Topic/Partition/Group，较简单 | Topic、Tag 等 | Exchange、Binding、Routing Key 灵活 |
| 典型优势 | 大事件流、多订阅、重放、CDC | 电商交易、延迟、事务和重试 | 中等吞吐、任务队列、复杂路由 |

**结合项目选择：**

- 广告行为流、设备事件、CDC、Elasticsearch 索引构建：优先考虑 Kafka。
- 强依赖事务消息回查、延迟和交易消息治理：评估 RocketMQ。
- 消息量中等但路由规则复杂、临时队列或任务分发突出：评估 RabbitMQ。
- 团队已有成熟平台时，迁移成本、SDK、监控、值班和故障经验也是重要约束。

### 50. 如何用两分钟回答 Kafka 项目题？

**推荐模板：**

> 这个场景的目标是把非主链路异步化并承接峰值流量。我先定义稳定事件和权威数据源，再按业务实体选择 Partition Key，按峰值吞吐和恢复目标规划 Partition。生产侧用本地事务加 Outbox 保证事件可恢复，Broker 侧使用多副本和同步副本约束，消费侧在业务成功后提交 Offset，并通过唯一约束、状态机或版本号保证幂等和防乱序。失败进入分级重试和死信，同时有补偿扫描和对账。线上监控端到端延迟、Lag 增长率、失败率、ISR 和业务完成率；积压时先保护下游，再按净恢复速度扩容。选型时还会评估延迟、路由、重放、团队运维能力和事务边界，而不是只看吞吐。

**资深 IC 加分点：**

- 能量化峰值、Partition、磁盘和恢复时间。
- 能说清异常状态、补偿入口和上线验证。
- 能用一次真实故障说明证据链和改进结果。
- 能推动接口、Schema、SLA 和监控标准落地。
- 管理经历只作为拆解问题、识别风险和跨团队协作的补充优势。
