# 技术｜Go 并发与运行时面试题

> 面向 9 年以上 Golang Web 后端业务研发岗位。本篇沿“代码语义 -> Runtime 执行 -> 资源成本 -> 生产模式”展开，重点深入 Channel、GMP、内存模型、逃逸、分配器、GC 和有界并发。

---

## 一、Go 并发与 Runtime 学习地图

### 1. Go 并发与 Runtime 的知识主线是什么？

```text
Goroutine / Channel / Select
  -> Mutex / Atomic / Memory Model
  -> G / M / P / Run Queue / Netpoller
  -> Stack / Escape Analysis
  -> mcache / mcentral / mheap
  -> GC / Write Barrier / Pacer / Scavenger
  -> Context / Worker Pool / Backpressure
  -> Race / Deadlock / Leak / Production Practice
```

### 2. 资深后端如何回答并发题？

采用“并发目标 -> 共享状态 -> 同步关系 -> 生命周期 -> 资源上限 -> 故障与观测”。不要只说“开 Goroutine 加 Channel”，而要说明谁创建、谁取消、谁关闭、错误如何传播、并发多少、下游能否承受。

### 3. 规范语义和 Runtime 源码如何结合？

先回答语言保证，例如 Channel 收发、锁和 Atomic 建立的 happens-before；再用 `hchan`、`sudog`、GMP 等源码解释成本。源码字段可变，规范语义和工程约束才是业务依赖。

### 4. 并发、并行和异步有什么区别？

- 并发：多个任务在时间上重叠推进。
- 并行：多个任务同一时刻在不同 CPU 上执行。
- 异步：发起后不阻塞等待最终结果的交互方式。

Goroutine 提供并发抽象，是否并行取决于可用 P、CPU 和阻塞状态；异步任务还需要持久化状态和可靠队列。

## 二、Goroutine、Channel、Select 与通信语义

### 5. Goroutine 为什么比线程轻？

Goroutine 初始栈较小且可增长，由 Runtime M:N 调度到 OS Thread，创建和切换通常低于线程。但它仍占栈、调度元数据和所引用对象；百万泄漏 Goroutine 一样会拖垮服务。

### 6. Channel 解决什么问题？

Channel 同时传递数据和建立同步关系，适合所有权转移、任务队列、结果汇聚和事件通知。它不是“比锁高级”；共享状态的短临界区通常用 Mutex 更直接。

### 7. `hchan` 的核心结构是什么？

概念字段包括：

```text
qcount/dataqsiz -> 当前数量与容量
buf             -> 环形缓冲区
sendx/recvx     -> 写入与读取位置
sendq/recvq     -> 等待发送/接收的 sudog 队列
closed          -> 关闭状态
elemtype/size   -> 元素类型和尺寸
lock            -> 保护 Channel 状态
```

因此 Channel 不是无锁队列；锁保证内部一致性，Runtime 再优化直接传递和快速路径。

### 8. 无缓冲 Channel 的发送链路是什么？

1. `chansend` 检查是否有等待 Receiver。
2. 有 Receiver 时直接从 Sender 栈复制到 Receiver 目标并唤醒。
3. 没有 Receiver 时构造 `sudog`，记录当前 G 和元素地址。
4. 将 `sudog` 放入 `sendq`，调用 `gopark` 挂起当前 G。
5. Receiver 到来后配对复制，通过 `goready` 使 Sender 可运行。

它表达同步交接，不提供排队容量。

### 9. 有缓冲 Channel 的发送和接收如何执行？

- 有等待对端时优先直接配对。
- 无等待 Receiver 且缓冲未满时，复制到 `buf[sendx]` 并推进环形索引。
- 缓冲满时 Sender 进入 `sendq`。
- Receiver 取缓冲头元素；若有等待 Sender，可把其元素补入缓冲并唤醒。

元素按值复制，大 Struct 会增加复制成本；传指针减少复制但引入共享和 GC 扫描。

### 10. `sudog`、`gopark` 和 `goready` 分别做什么？

`sudog` 表示 Goroutine 在某个同步对象上的等待关系；`gopark` 将当前 G 置为等待状态并让出 M；`goready` 将目标 G 变为 Runnable，放入运行队列等待调度，不代表立即执行。

### 11. 关闭 Channel 时 Runtime 做什么？

`closechan` 标记关闭，唤醒所有等待 Receiver：它们得到零值和 `ok=false`；唤醒所有等待 Sender：恢复后发生 send-on-closed Panic。重复关闭和关闭 nil Channel 都会 Panic。

关闭是广播完成信号，不负责释放 Channel 内存；对象不可达后才由 GC 回收。

### 12. Channel 的关闭原则是什么？

通常由唯一 Sender 或明确 Owner 关闭，Receiver 不关闭不拥有的 Channel。多 Sender 场景由协调者等待所有 Sender 结束后关闭，或只通过 Context 通知退出而不关闭共享输入。

不要用 Recover 掩盖重复关闭，应该修正所有权。

### 13. nil、open、closed Channel 的行为是什么？

| 操作 | nil | open | closed |
|---|---|---|---|
| send | 永久阻塞 | 正常/可能阻塞 | Panic |
| receive | 永久阻塞 | 正常/可能阻塞 | 立即零值、ok=false |
| close | Panic | 成功 | Panic |

将 Channel 设 nil 可在 Select 中动态禁用 Case，但误用会造成永久阻塞。

### 14. Select 的底层和选择语义是什么？

Select 先随机化轮询顺序，再按稳定锁顺序锁定涉及 Channel，检查就绪 Case；无就绪且无 Default 时把当前 G 的 `sudog` 注册到多个等待队列并挂起。某个 Case 唤醒后清理其他队列登记。

多 Case 就绪时伪随机选择，不保证业务公平；带 Default 的空转循环会烧 CPU。

### 15. Channel Buffer 应该设多大？

容量代表允许的突发和在途任务，不是越大越好。依据生产速率、消费速率、可接受等待、单元素内存和故障恢复设计。Buffer 只能吸收短期波动，长期生产大于消费最终仍会满。

必须监控长度、等待时间、拒绝数和最老任务年龄，并提供背压。

### 16. Channel 与 Mutex 如何选择？

- 所有权转移、任务流水线和事件通知：Channel。
- 保护内存中小块共享状态：Mutex。
- 单值计数/状态：Atomic。
- 复杂状态机可由单 Owner Goroutine 串行处理，但要评估队列和故障。

以语义清晰和压测为准，不奉行“不要通过共享内存通信”的字面教条。

## 三、同步原语、Atomic 与 Go Memory Model

### 17. Mutex 的工作状态和饥饿模式如何理解？

Mutex 有快速 CAS 路径，竞争时进入自旋或信号量等待；长期等待会切换更公平的交接倾向，降低尾部饥饿。具体位字段属于实现细节。

临界区应短，不在锁内做网络 IO；用 Mutex Profile 证明竞争。

### 18. Mutex 和 RWMutex 如何选择？

读多不等于 RWMutex 一定更快。读临界区足够长、并发高、写少时才可能受益；写到来会阻止新读进入。普通短临界区优先 Mutex，再 Benchmark。

### 19. 复制 Mutex 为什么危险？

Mutex、RWMutex、WaitGroup、Once 等使用后不可复制，复制会产生两个表面独立但状态错误的同步对象。含锁 Struct 使用指针接收者，通过 `go vet copylocks` 检查。

### 20. WaitGroup 常见错误有哪些？

- `Add` 与 `Wait` 生命周期竞争。
- Goroutine 内才 Add，主协程可能先 Wait 返回。
- 忘记 Done 或 Panic 导致永不结束。
- 计数变负 Panic。
- 复用时上一轮 Wait 尚未结束。

任务创建前 Add，Goroutine 开头 `defer Done`；需要错误传播和取消优先 `errgroup`。

### 21. Once、Cond 和 Pool 分别适合什么？

- Once：并发只执行一次初始化；函数 Panic 也视为执行过。
- Cond：在锁保护条件变化时唤醒等待者，必须循环检查条件。
- Pool：跨 GC 周期可被清空的临时对象复用，不是缓存和资源池。

### 22. Atomic 适合什么场景？

独立计数、只读配置指针、简单状态位。多个字段构成不变量时用锁或不可变对象 + `atomic.Pointer/Value` 整体替换。

Atomic 正确不代表业务原子，CAS 循环还可能争用和 ABA。

### 23. 什么是 Data Race？

两个 Goroutine 并发访问同一内存，至少一个写，且没有同步关系。结果可能受编译器优化和 CPU 重排影响，不能用“机器上一直正常”证明安全。

`go test -race` 只能发现执行覆盖路径上的 Race，不能证明不存在。

### 24. Go Memory Model 中重要的 happens-before 有哪些？

- Channel send happens-before 对应 receive 完成。
- close happens-before Receiver 观察到关闭。
- Mutex Unlock happens-before 后续成功 Lock。
- Once 函数完成 happens-before 任意 Do 返回。
- Atomic 操作提供文档定义的同步语义。
- Goroutine 创建本身不保证其完成对外可见，必须同步等待。

### 25. Double-Checked Locking 为什么容易错？

无同步读取初始化标志和对象会产生 Race，可能看到标志但对象尚未完整可见。使用 `sync.Once` 或 `atomic.Pointer` 的完整发布模式，不手写不受证明的双检锁。

## 四、GMP、调度、Netpoller、系统调用与 CGO

### 26. GMP 模型是什么？

- G：Goroutine，保存栈、状态和调度信息。
- M：OS Thread，真正执行机器指令。
- P：执行 Go 代码所需的调度资源，持有本地 Run Queue 和 mcache。

M 必须绑定 P 才能执行 Go 代码；阻塞系统调用时 P 可转交其他 M。

### 27. 新 Goroutine 放到哪里？

通常进入当前 P 的本地队列，部分情况下进入全局队列或 `runnext`。调度器综合本地、全局、Netpoller 和 Work Stealing 寻找 Runnable G。

具体频率会变化，理解局部性和负载均衡即可。

### 28. Work Stealing 如何工作？

空闲 P 从其他 P 的本地队列窃取一部分 G，避免某个 P 堆积而其他 CPU 空闲。它提高利用率，但大量短任务、频繁唤醒和锁竞争仍会产生调度成本。

### 29. Goroutine 在哪些时机发生调度？

Channel/Lock 阻塞、系统调用、网络等待、主动 `Gosched`、函数调用安全点的抢占、GC 协作等。现代 Go 支持更强的异步抢占，降低计算循环长期独占 P 的风险，但业务仍应避免无界 CPU 循环。

### 30. Sysmon 做什么？

后台监控长时间运行 G、系统调用中的 P、Netpoller、定时器和 GC 相关任务，触发抢占与资源回收。它是 Runtime 维持调度和时间进展的重要组件。

### 31. Netpoller 如何让网络 IO 不阻塞线程？

`net` 包使用非阻塞 FD 和操作系统多路复用。G 等待 IO 时由 Netpoller 登记并 Park，事件就绪后变为 Runnable；M 可执行其他 G。

这解释了 Go 适合高并发 IO，但每个连接仍消耗 FD、Buffer、G 和业务内存。

### 32. 阻塞系统调用发生什么？

Runtime 可将 P 从阻塞 M 分离，让新 M 接管 P 执行其他 G；调用返回后原 M 重新获取 P 或把 G 放回队列。大量不可感知的阻塞 CGO 调用仍可能创建许多线程。

### 33. `GOMAXPROCS` 应如何设置？

它限制同时执行 Go 代码的 P 数，不等于 Goroutine 或线程数。CPU 密集任务通常接近可用 CPU；容器中应与 CPU 配额协调。调高超过实际 CPU 可能增加调度，调低会限制并行。

用负载测试观察吞吐、P99、CPU throttling 和 Run Queue。

### 34. CGO 对调度有什么影响？

调用 C 需要跨边界，可能阻塞 OS Thread；C 持有 Go Pointer 和回调还受严格规则。高频小调用要批量化，监控线程数和 CGO 调用耗时，部署上处理动态库和交叉编译。

## 五、Goroutine 栈、逃逸分析与编译器决策

### 35. Goroutine 栈如何增长？

G 使用可增长连续栈。函数入口检查空间，不足时分配更大栈、复制内容并修正栈内指针，随后继续执行；栈也可收缩。

Go Pointer 在合法代码中不会因栈移动失效，Runtime 和编译器维护指针元数据。

### 36. 什么是逃逸分析？

编译器判断对象生命周期是否超出当前栈帧、地址是否可能被外部保留，从而决定栈或堆。返回局部变量地址在 Go 中安全，编译器会让对象活得足够久。

### 37. 常见逃逸场景有哪些？

- 返回指针或被长生命周期对象保存。
- Closure 捕获变量。
- Interface/反射使具体类型和生命周期难确定。
- 发送指针到可能更长生命周期的 Goroutine。
- 对象过大或编译器无法安全放栈。

这些是倾向，不是永久规则，应查看当前编译结果。

### 38. 如何查看逃逸分析？

```bash
go build -gcflags='all=-m=2' ./...
go test -gcflags='all=-m=2' ./path
```

输出很嘈杂，关注业务 Package 和热点函数；再用 Benchmark 的 B/op、allocs/op 与 Heap Profile 验证真实影响。

### 39. 逃逸一定需要优化吗？

不一定。对象数量小或不在热路径时，清晰代码更重要。优化优先减少算法工作量和对象生命周期，不要为“零逃逸”改成全局池或 Unsafe，反而增加共享和泄漏。

### 40. Stack Overflow 和深递归如何处理？

栈可增长但不是无限，深递归会持续扩栈并最终崩溃。图遍历和不可信深度数据应考虑迭代写法、深度限制和输入校验。

## 六、mcache、mcentral、mheap 与内存分配

### 41. Go 堆分配器的整体层次是什么？

```text
当前 P 的 mcache
  -> 对应 Size Class 的 mspan
  -> mcentral 获取可用 span
  -> mheap 管理 Page、Arena 和大对象
  -> OS 虚拟/物理内存
```

P-local mcache 让小对象常见分配避免全局锁。

### 42. Span、Page 和 Size Class 是什么？

Heap 按 Page 管理，连续 Page 组成 Span；小对象按 Size Class 放入固定槽位。相同 Class 对象共享 Span，分配是找到空槽。

对象实际占用可能向上取整，尺寸分布不佳会产生内部碎片。

### 43. Tiny、小对象和大对象如何分配？

无指针的极小对象可经 Tiny Allocator 合并；普通小对象从 mcache 对应 Class 分配；超过小对象范围的大对象直接向 mheap 申请合适 Page。

阈值属于实现细节，工程关注大对象和高频小对象对 GC、碎片与延迟的影响。

### 44. 为什么每个 P 有 mcache？

执行 Go 代码必须持有 P，因此分配可从 P-local Cache 快速完成，减少多线程竞争。G 切换 P 后使用新 P 的 mcache，不拥有固定 Allocator。

### 45. `sync.Pool` 如何降低分配？

Pool 保存临时可复用对象并利用 P-local 结构降低竞争，GC 时对象可能被丢弃。适合 `bytes.Buffer`、编码临时对象；取出必须 Reset，放回前清除大引用并限制异常大 Buffer。

不能放数据库连接、业务缓存和必须存在的对象。

### 46. 为什么释放对象后 RSS 不立即下降？

GC 证明对象不可达后，内存先成为 Runtime 可复用空间；Scavenger 再按策略将空闲 Page 归还 OS。HeapAlloc 下降、HeapIdle 上升而 RSS 暂时不降可能正常。

判断泄漏要看 GC 后存活对象趋势和 Profile，而非一次 RSS。

### 47. 常见内存指标如何区分？

- HeapAlloc：当前已分配存活/未回收堆字节。
- HeapInuse：标记为堆使用的 Span 字节。
- HeapIdle：保留但当前空闲的堆内存。
- HeapReleased：已归还 OS 的空闲内存。
- Sys：Runtime 从 OS 获取的各类内存总量。
- RSS：操作系统观察的驻留物理内存，包含 Go Heap 之外内容。

## 七、GC、写屏障、Pacer、GOGC 与 GOMEMLIMIT

### 48. Go GC 的主要目标是什么？

**结论先行：Go GC 的目标是在较短暂停、可接受 CPU 开销和受控内存占用之间取得平衡。**当前实现以并发、非移动的标记清扫为核心，大部分标记工作与业务 Goroutine 同时进行。

GC 的成本不只有 STW：并发 Mark Worker 会占用 CPU，分配过快时业务 Goroutine 还会执行 Mark Assist；活对象越多、指针越密，扫描成本通常越高。另一方面，允许更大的 Heap 能降低 GC 频率，但会提高 RSS 和容器 OOM 风险。

因此评价 GC 不能只看 Pause，还要一起观察 GC CPU、分配速率、GC 后存活堆、Heap Goal、Assist、P99 和内存余量。调 `GOGC` 或 `GOMEMLIMIT` 是容量取舍，不是替代对象生命周期优化。

**面试一句话：**Go GC 用并发标记清扫降低暂停，但成本会转移到 GC CPU、Mark Assist 和内存占用；优化时要同时看暂停、吞吐、活对象、分配率和容器余量。

### 49. 三色标记如何理解？

**结论先行：三色标记是一种描述对象扫描状态的逻辑模型。**白色表示尚未发现，灰色表示已经可达但其引用尚未扫描，黑色表示对象及其引用已经扫描完成。

```text
Root -> 灰色对象队列
       -> 取出灰色对象扫描其指针
       -> 新发现的白色对象染灰
       -> 当前对象染黑
队列清空后，仍为白色的对象不可达，可进入回收
```

并发标记期间业务仍会修改指针。如果黑色对象新指向白色对象而 GC 未记录这次变化，白色对象可能被误回收，因此 Runtime 需要写屏障维护可达性不变量。颜色不是对象里真实保存的枚举，当前实现会用标记位图和工作队列表达这些状态。

**面试一句话：**三色模型把对象分成未发现、待扫描和已扫描三类，从 Root 推进到灰色队列清空；并发修改指针会破坏可达性，所以必须配合写屏障。

### 50. 为什么需要写屏障？

并发标记期间业务仍修改指针，可能破坏“可达对象不会漏标”的不变量。写屏障在指针写入时记录必要关系，使 GC 正确追踪；它增加写操作成本。

### 51. GC 哪些阶段需要 STW？

开始和终止标记等阶段需要短暂 STW 完成状态切换和 Root 准备，大部分标记与业务并发。暂停短不代表 GC 无成本，Mark Worker 和 Assist 会消耗 CPU 并影响 P99。

### 52. Mark Assist 为什么会让接口变慢？

当分配速度超过 GC 标记进度，分配 Goroutine 被要求协助标记，业务请求承担 GC 工作。表现为高分配率、GC CPU 上升和尾延迟抖动。

根治是减少分配、缩小活对象和合理容量，不只是调参数。

### 53. `GOGC` 控制什么？

**结论先行：`GOGC` 控制 GC 触发目标相对“上一轮 GC 后存活堆”的增长比例，本质上调节 CPU 与内存的交换。**可近似理解为：

```text
下一轮堆目标 ≈ GC 后存活堆 + GC 后存活堆 × GOGC / 100
```

实际目标还会受 Root 扫描成本、`GOMEMLIMIT` 和 Runtime Pacer 调整影响，不能把公式当精确实现契约。值调高通常减少 GC 频率、增加内存；值调低通常降低堆目标，却增加 GC CPU 和 Mark Assist。若活对象本身很大，调低 `GOGC` 的收益有限，反而可能让接口更慢。

设置为 `off` 只是不按堆增长目标主动触发；显式 GC 或内存限制压力仍可能启动 GC。生产调整必须用相同负载比较 GC CPU、P99、Heap Goal、RSS 和 OOM 余量。

**面试一句话：**`GOGC` 决定堆相对上一轮存活量能增长多少；调高用内存换 CPU，调低用 CPU 换内存，但活对象过大时不能靠调参解决。

### 54. `GOMEMLIMIT` 控制什么？

**结论先行：`GOMEMLIMIT` 是 Go Runtime 管理内存的软上限，不是进程 RSS 的硬上限，也不等于容器 Memory Limit。**其目标范围可近似理解为 Runtime 指标 `/memory/classes/total:bytes` 减去已归还给 OS 的 `/memory/classes/heap/released:bytes`。

接近限制时 Pacer 会更积极触发 GC，即使 `GOGC` 原本允许更大的 Heap。若限制低于真实活对象及运行所需空间，Runtime 只能频繁 GC，却回收不出足够内存，形成 GC Thrashing、CPU 飙升和 P99 恶化；它也无法约束 CGO、直接 `mmap` 等全部进程外部内存。

容器预算应从 Cgroup Limit 中扣除线程栈、CGO、文件映射、网络/内核开销、Sidecar 和安全余量，再设置 `GOMEMLIMIT`。上线后同时监控 RSS、Runtime 管理内存、GC CPU、Heap Goal 和 OOM 事件。

**面试一句话：**`GOMEMLIMIT` 给 Runtime 一个软预算，接近时通过更积极 GC 控内存；它不限制全部 RSS，因此必须低于容器上限并给非 Go 内存留余量，否则可能 Thrash 后仍被 OOMKill。

### 55. Scavenger 做什么？

后台识别长期空闲 Page，并通过 OS 接口降低其物理驻留。它影响 RSS 回落但不决定对象可达性；GC 负责对象，Scavenger 负责 Page 归还。

### 56. 指针密度为什么影响 GC？

含指针对象需要扫描其指针图；大量 `map[string]*T`、链表和 Interface 会增加扫描。连续无指针数据更友好，但优化应由 GC CPU 和 Profile 驱动，不能为了少指针破坏模型。

### 57. 如何设置容器内存预算？

从 Limit 扣除非 Go 内存、线程栈、CGO、文件映射、内核和安全余量，再设置 GOMEMLIMIT。通过压测观察峰值 RSS、Heap Goal、GC CPU、P99 和 OOM；Limit 不能等于 GOMEMLIMIT。

## 八、Context、Worker Pool、Pipeline 与有界并发

### 58. Context 解决什么问题？

传播取消、Deadline 和请求级元数据。它是协作信号，不会强制终止代码；数据库、HTTP、循环和 Worker 必须主动监听。

Context 作为首参，不存长期 Struct，不传 nil，Value 只放跨进程请求元信息而非业务参数。

### 59. `WithCancel`、`WithTimeout`、`WithDeadline` 和 Cause 如何选择？

父流程主动停止用 Cancel；相对预算用 Timeout；已有绝对结束时间用 Deadline；需要保留取消原因用 Cause API。创建后尽快安排 Cancel，释放 Timer 和子树。

### 60. Worker Pool 怎样设计才完整？

- 有界 Worker 和 Queue。
- 提交支持 Context、等待或明确拒绝。
- 每个任务 Panic 隔离和错误分类。
- Owner 控制停止提交、关闭队列、等待 Worker。
- 指标覆盖活跃数、队列长度、等待、执行、拒绝和失败。
- 下游并发上限按承载能力，不按 CPU 随意设置。

### 61. `errgroup` 适合什么场景？

同一请求内有限并发子任务，需要首错取消和等待收敛。使用 `WithContext` 和 `SetLimit`；循环变量、共享结果和错误语义仍需正确处理。

长生命周期 Job 系统不应只靠内存 errgroup，应持久化状态。

### 62. Pipeline 如何避免泄漏？

每一级监听 Context；下游提前退出时上游 Send 能取消；Owner 负责关闭输出；错误有独立收敛路径；所有 Stage 最终可 Wait。

没有取消路径的 Fan-out/Fan-in 很容易因无人接收而泄漏。

### 63. Semaphore 和 Worker Pool 如何选择？

Semaphore 限制同时在途数量，任务仍由调用 Goroutine 执行；Worker Pool 同时限定执行者和排队。请求内批量调用适合 Semaphore/errgroup，长期任务入口适合有界 Queue + Worker。

### 64. 背压和丢弃策略怎样设计？

队列满时可阻塞等待、快速失败、丢最新、丢最旧或降级，必须根据业务语义选择。支付不能静默丢；指标采样可以丢。所有等待受 Deadline 限制，防止无限堆积。

### 65. 定时器和 Ticker 有哪些坑？

长期 Ticker 要 Stop；循环中反复 `time.After` 会创建计时对象，应复用 Timer；Reset 前按 API 契约处理状态；回调不能假设准点执行。定时任务跨实例还要做幂等和租约。

### 66. 如何优雅停机？

摘除 Readiness -> 停止接收新请求/拉消息 -> 取消根 Context -> 等待在途请求和 Worker -> Flush/提交 Offset -> 超时强制退出。宽限时间必须覆盖内部等待并小于平台终止期限。

## 九、Race、死锁、活锁、泄漏与运行时诊断

### 67. Goroutine 泄漏的常见原因是什么？

无人接收的 Send、无人发送的 Receive、无 Deadline 网络调用、Ticker 未停止、Pipeline 下游退出、锁永不释放、后台循环无取消。现象是 Goroutine 和 Heap 随时间上涨、P99 抖动。

### 68. 如何排查 Goroutine 泄漏？

对比两个时间点的 Goroutine Profile，按 Stack 聚合增长；结合创建位置、Trace、队列和连接池指标；用测试前后 `runtime.NumGoroutine` 或专用工具做回归，但允许 Runtime 背景波动。

### 69. 死锁、活锁和饥饿有什么区别？

- 死锁：参与者永久等待，系统无进展。
- 活锁：参与者持续动作但相互避让，无有效进展。
- 饥饿：某个任务长期得不到资源，其他任务仍推进。

锁顺序、Try/Backoff、公平队列和缩短临界区分别治理不同问题。

### 70. 如何使用 Race Detector？

```bash
go test -race ./...
go test -race -run TestName -count=20 ./path
```

它有明显 CPU/内存开销，适合 CI、测试和受控压测，不默认生产全量开启。报告应定位两条冲突访问和 Goroutine 创建栈。

### 71. Block、Mutex 和 Goroutine Profile 分别看什么？

- Goroutine：当前堆栈和等待状态。
- Block：Channel、Select、锁等发生的阻塞累计。
- Mutex：锁竞争造成的等待热点。

采样率会影响开销，故障前没有开启可能缺历史；生产按需短时启用并保护端点。

### 72. Runtime Trace 适合解决什么？

观察 Goroutine 调度、阻塞/唤醒、网络、Syscall、GC 和 Processor 利用率，适合“CPU 不高但延迟大”“并行度不足”“G 长时间等待”。热点函数优先 CPU Profile，内存对象优先 Heap Profile。

## 十、业务架构与项目实战

### 73. 广告素材同步如何设计有界并发？

分页读取任务，按媒体账号设置独立并发 Semaphore，外层 Worker Pool 有界；每个调用继承批次 Deadline，错误分可重试/永久；结果逐条落库，队列和媒体限流可观测。取消后不再取新任务，在途任务按 Deadline 收敛。

### 74. Kafka Consumer 中 Goroutine 如何组织？

消费组控制 Partition 并行度，Partition 内若要求顺序不能无约束并发。可按业务 Key 分片到固定 Worker；完成处理和幂等落库后提交 Offset。Rebalance 时停止取数、等待在途或安全取消，防止旧实例继续提交。

### 75. 外部 SDK 并发不安全如何治理？

先用 Race 和压测证明共享状态冲突；适配层给每账户实例独立 Owner 或加锁，禁止业务绕过；评估锁内是否包含 IO；升级 SDK 后保留并发回归。用 P99、Race 结果和锁 Profile 验证。

### 76. Go 服务内存问题如何串联回答？

> 编译器先通过逃逸分析决定对象是否进入堆；堆对象由 P-local mcache、mcentral 和 mheap 分配；GC 根据可达性标记，Pacer 用 GOGC 和 GOMEMLIMIT控制目标，Scavenger再尝试归还空闲页。线上我区分分配速率、GC 后活对象、Runtime 保留内存和 RSS；用 allocs/op、Heap Profile、runtime/metrics 与容器指标定位。优化优先缩短对象生命周期、限制缓存和批次、修复 Goroutine/子 Slice 持有，再考虑 Pool 和参数。

### 77. Channel 设计原理如何用两分钟回答？

> Channel 是带同步语义的类型化队列，底层 hchan 维护环形 Buffer、收发索引、sendq/recvq、关闭状态和锁。发送时若已有等待 Receiver 就直接复制并唤醒；有 Buffer 空间则入队；否则当前 G 通过 sudog 登记到 sendq 并 gopark，接收方到来后 goready。无缓冲 Channel 是同步交接，有缓冲 Channel 允许有限解耦。close 会广播唤醒 Receiver，并让等待 Sender 恢复后 Panic。Select 会在多个 Channel 注册等待并清理未选分支。工程上重点不是源码字段，而是谁拥有关闭权、Buffer 表示多少突发、元素复制与 GC 成本、取消和背压是否完整。

### 78. 如何用两分钟回答 Go Runtime？

> Go Runtime 用 G、M、P 将大量 Goroutine 调度到 OS Thread，P 持有本地运行队列和 mcache；阻塞 IO 通过 Netpoller 挂起 G，系统调用时 P 可转交其他 M，Sysmon 负责抢占和进展。G 使用可增长栈，编译器逃逸分析决定栈或堆；堆由 mcache、mcentral、mheap 分层管理，GC 并发标记并通过写屏障保持正确，GOGC 和 GOMEMLIMIT平衡 CPU 与内存。生产中我关注 Goroutine、队列、锁、分配率、GC CPU、Run Queue 和 P99，用 Race、pprof 和 Trace 定位，并用 Context、有界并发、背压和优雅停机控制生命周期。
