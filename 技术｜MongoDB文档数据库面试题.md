## 一、MongoDB 当前能力全景：文档模型、索引、事务、副本集与分片

> 先掌握 MongoDB 的文档建模、索引、事务和高可用能力。

### 1. MongoDB 适合哪些业务场景？

**结论先行：**
MongoDB 适合结构灵活、文档聚合、读写量大、字段变化快的业务，例如内容、配置、画像、日志元数据、商品扩展属性。

**分层展开：**

**适合场景**
- 内容系统：文章、笔记、评论扩展字段。
- 商品扩展属性：不同类目属性差异大。
- 用户画像：标签、偏好、行为摘要。
- 配置中心或表单系统：字段灵活变化。
- 地理位置和简单空间查询。

**不适合场景**
- 强事务账务。
- 复杂 join 报表。
- 高度规范化关系模型。
- 需要大量跨集合强一致更新的场景。

**工程判断**
- 如果一次查询通常需要完整对象，MongoDB 很合适。
- 如果频繁跨表 join 和强事务，优先关系型数据库。

**面试追问：内容业务为什么可能用 MongoDB？**
内容字段变化快、扩展属性多、单篇内容通常作为整体读写，文档模型能减少表结构变更和多表 join。

### 2. BSON 和 JSON 有什么区别？

**结论先行：**
BSON 是 MongoDB 的二进制文档格式，类似 JSON 但支持更多数据类型，并更适合存储和遍历。

**分层展开：**

**JSON**
- 文本格式。
- 易读易传输。
- 类型较少。

**BSON**
- 二进制格式。
- 支持 ObjectId、Date、Decimal、Binary 等类型。
- 字段带长度信息，便于解析。
- 比纯 JSON 更适合数据库内部存储。

**工程注意**
- 文档有大小限制，单文档不能无限膨胀。
- Go 中要注意 `bson` tag、零值、时间类型和 ObjectID 转换。

**面试追问：MongoDB 单文档大小限制有什么意义？**
它提醒我们不要把无限增长的数据都塞进一个文档，例如评论、流水、行为明细应拆集合或分页存储。

### 3. 嵌入文档和引用怎么选？

**结论先行：**
强关联、一起读写、生命周期一致的数据适合嵌入；独立增长、独立查询、多对多关系适合引用。

**分层展开：**

**适合嵌入**
- 用户地址列表规模可控。
- 订单快照中的商品信息。
- 内容的部分扩展属性。
- 读时总是一起返回的数据。

**适合引用**
- 评论列表无限增长。
- 用户和角色多对多。
- 商品和类目复杂关系。
- 被多个文档共享的数据。

**风险**
- 过度嵌入会导致大文档和大数组。
- 过度引用会退化成应用层 join。
- 文档模型要围绕核心查询设计。

**面试追问：订单里商品信息应该嵌入还是引用？**
订单明细通常要保存商品快照，适合嵌入订单或订单明细文档中，避免商品后续改名改价影响历史订单。

### 4. MongoDB 建模和 MySQL 建模有什么不同？

**结论先行：**
MySQL 建模偏关系规范化，MongoDB 建模偏查询驱动和聚合对象设计。

**分层展开：**

**MySQL**
- 强调表结构、主外键、范式。
- 通过 join 表达关系。
- 事务和约束能力强。

**MongoDB**
- 强调文档边界和访问模式。
- 通过嵌入减少查询次数。
- 通过冗余换读取性能。

**工程取舍**
- MongoDB 可以接受适度冗余。
- 冗余字段要有同步和修复机制。
- 高价值核心数据仍要明确 owner 和一致性边界。

**面试追问：MongoDB 冗余字段怎么保证一致？**
通过事件驱动更新、版本号、异步补偿和定期校验。核心字段不要无限冗余到不可治理。

### 5. MongoDB 索引有哪些类型？

**结论先行：**
MongoDB 常见索引包括单字段索引、复合索引、多键索引、唯一索引、TTL 索引、文本索引、地理空间索引和哈希索引。

**分层展开：**

**基础索引**
- 单字段索引：适合简单条件。
- 复合索引：适合多条件查询和排序。
- 唯一索引：保证字段唯一。

**特殊索引**
- 多键索引：数组字段索引。
- TTL 索引：自动过期删除。
- 文本索引：简单全文搜索。
- 地理索引：附近查询。
- 哈希索引：常用于哈希分片。

**工程建议**
- 索引必须围绕查询模式设计。
- 索引不是越多越好，写入会维护索引。
- 大集合建索引要评估资源和影响。

**面试追问：数组字段建索引有什么风险？**
数组会生成多键索引，一个文档可能产生多个索引项；数组过大时索引膨胀明显。

### P0 补充：MongoDB 高级索引怎么掌握？

**结论先行：**
MongoDB 索引不是只会单字段和复合索引就够了。面试要能说清楚不同索引适合什么场景、有什么限制、会带来什么写入和存储成本。

**高频索引类型：**

- multikey index：数组字段索引，一个文档可能产生多个索引项。
- unique index：保证字段唯一，适合业务唯一键。
- TTL index：按时间自动过期删除，适合临时任务、会话、日志保留。
- sparse index：只索引存在该字段的文档。
- partial index：只索引满足过滤条件的文档，比 sparse 更可控。
- text index：简单文本检索，不适合替代 Elasticsearch。
- geo index：地理位置查询。
- hashed index：哈希分布，常用于 hashed shard key。

**覆盖查询：**
如果查询字段和返回字段都在索引里，可以减少读取完整文档：

```javascript
db.tasks.find(
  { account_id: 1001, status: "pending" },
  { _id: 0, account_id: 1, status: 1, created_at: 1 }
)
```

配合索引：

```javascript
db.tasks.createIndex({ account_id: 1, status: 1, created_at: -1 })
```

**多键索引限制：**
数组字段会让索引项膨胀。复合索引里如果多个字段都是数组，容易产生组合爆炸，设计时要特别小心。

**面试追问：partial index 和 sparse index 有什么区别？**
sparse 只索引存在字段的文档；partial 可以指定更复杂的过滤条件，例如只索引 `status: "active"` 的文档。partial 更适合围绕业务热数据建小索引。

### 6. 副本集原理是什么？

**结论先行：**
副本集通过主节点写入、从节点复制 oplog、自动选主来实现高可用和读扩展基础。

**分层展开：**

**角色**
- primary：处理写请求。
- secondary：复制数据，可承担读。
- arbiter：只投票不存数据，不推荐滥用。

**复制**
- 主节点记录 oplog。
- 从节点拉取并重放 oplog。
- oplog 是有大小窗口的，落后太多可能需要重新同步。

**故障切换**
- 主节点不可用时触发选举。
- 多数派选出新主。
- 客户端需要感知拓扑变化。

**面试追问：副本集切主期间会发生什么？**
短时间写不可用，客户端可能收到重试错误；应用侧要配置合理超时和可重试写。

### 7. MongoDB oplog 有什么作用？

**结论先行：**
oplog 是副本集复制日志，从节点通过重放 oplog 追赶主节点，也常用于变更订阅和数据同步。

**分层展开：**

**作用**
- 主从复制。
- 故障恢复追赶。
- Change Stream 变更流基础。
- 外部同步系统可基于变更流消费。

**风险**
- oplog 太小，延迟从库可能追不上。
- 大批量写入会快速滚动 oplog。
- 复制延迟过大影响读从一致性。

**治理**
- 监控复制延迟。
- 评估 oplog 窗口。
- 大批量任务限速。

**面试追问：从库落后超过 oplog 窗口怎么办？**
无法增量追赶，通常需要重新全量同步或人工修复。

### 8. MongoDB 分片集群怎么工作？

**结论先行：**
MongoDB 分片通过 shard key 把集合数据分布到多个 shard，mongos 负责路由，config server 保存元数据。

**分层展开：**

**核心组件**
- shard：存储实际数据。
- mongos：查询路由。
- config server：存储集群元数据。

**数据分布**
- 基于 shard key 划分 chunk。
- chunk 在 shard 之间迁移保持均衡。
- 查询带 shard key 可以精准路由。

**工程风险**
- shard key 选错很难修。
- 广播查询会打所有分片。
- chunk 迁移会消耗资源。

**面试追问：分片后查询一定更快吗？**
不一定。带分片键的查询可能更快；不带分片键会广播到多个 shard，反而更慢。

### 9. 分片键怎么选择？

**结论先行：**
好的分片键要高基数、分布均匀、查询常带、写入不形成单点热点，并且尽量稳定不可变。

**分层展开：**

**好分片键特征**
- 高基数。
- 分布均匀。
- 常出现在查询条件中。
- 不频繁更新。
- 能支撑主要访问模式。

**坏分片键**
- 低基数字段，如 status。
- 单调递增时间，容易写热点。
- 查询很少带的字段。
- 业务未来可能变化的字段。

**常见方案**
- user_id 适合用户维度查询。
- tenant_id + hash 适合多租户。
- 时间 + 业务维度适合时序归档，但要防热点。

**面试追问：按时间做分片键有什么问题？**
新写入会集中在最新时间范围，容易形成写热点；历史查询和归档方便，但要配合 hash 或业务维度打散。

### P0 补充：WiredTiger 存储引擎要掌握到什么程度？

**结论先行：**
WiredTiger 是 MongoDB 主流存储引擎，核心能力包括文档级并发控制、缓存、checkpoint、journal 和压缩。面试不用背源码，但要能解释它如何影响性能和稳定性。

**核心能力：**

- 文档级锁：相比集合级锁，并发写入不同文档的能力更好。
- cache：热点数据和索引会缓存在 WiredTiger cache 中。
- checkpoint：周期性把内存脏数据刷到磁盘，形成一致性检查点。
- journal：崩溃恢复日志，保证异常宕机后能恢复已确认写入。
- 压缩：降低磁盘占用，但会带来 CPU 成本。

**工程影响：**

- 工作集超过内存时，查询会频繁读磁盘，延迟明显升高。
- 大文档和大数组会增加读写放大。
- 索引过多会增加写入维护成本和 cache 压力。
- checkpoint、批量写入、压缩都可能影响延迟毛刺。

**面试追问：MongoDB 为什么不是字段越灵活越随便用？**
因为灵活只是建模能力，不代表没有成本。文档过大、数组无限增长、索引过多、查询模式混乱，都会让 WiredTiger cache、磁盘 IO 和复制压力变大。

### P0 补充：分片治理里的 chunk、balancer、jumbo chunk 怎么讲？

**结论先行：**
分片不是简单把数据分到多台机器。真正的难点是 shard key 选择、chunk 分布、balancer 迁移、热点治理和扩容后的重平衡。

**核心概念：**

- chunk：按 shard key 范围切分的数据块。
- balancer：负责在 shard 之间迁移 chunk，保持数据相对均衡。
- jumbo chunk：过大且无法正常拆分或迁移的 chunk。
- zone sharding：把特定范围的数据固定到指定 shard，常用于地域或冷热隔离。

**常见问题：**

- shard key 基数低，chunk 不容易均匀拆分。
- shard key 单调递增，新写入集中到一个 shard。
- 查询不带 shard key，mongos 需要广播到所有 shard。
- jumbo chunk 迁移困难，扩容后也不容易均衡。

**面试追问：hashed shard key 能解决所有热点吗？**
不能。hashed 可以打散写入，但会牺牲范围查询能力。如果业务大量按时间范围查，纯 hash 可能让查询广播更多 shard。分片键要结合写入分布和查询模式一起设计。

## 二、核心原理分析：复合索引、explain、事务一致性与读写 concern

> 这一章回答“MongoDB 为什么这样建模和查询”。

### 10. 复合索引怎么设计？

**结论先行：**
复合索引设计遵循 ESR 思路：等值字段优先、排序字段其次、范围字段最后，同时结合选择性和查询频率。

**分层展开：**

**E：Equality**
- 等值过滤字段靠前。
- 例如 tenant_id、status、user_id。

**S：Sort**
- 排序字段紧随其后。
- 避免内存排序。

**R：Range**
- 范围字段放后面。
- 例如 create_time、price。

**工程注意**
- 不同查询模式可能需要不同索引。
- 低选择性字段单独建索引价值有限。
- 覆盖查询可以减少文档读取。

**面试追问：`{user_id:1, create_time:-1}` 适合什么查询？**
适合查询某个用户的记录并按时间倒序分页，例如用户订单、用户行为、用户消息列表。

### 11. explain 怎么看？

**结论先行：**
MongoDB explain 重点看是否走索引、扫描文档数、返回文档数、排序方式和执行耗时。

**分层展开：**

**核心字段**
- winningPlan：最终执行计划。
- stage：IXSCAN、COLLSCAN、FETCH、SORT 等。
- totalKeysExamined：扫描索引项数。
- totalDocsExamined：扫描文档数。
- nReturned：返回文档数。

**判断标准**
- 避免 COLLSCAN。
- 扫描数和返回数差距不要过大。
- 避免大规模内存排序。
- FETCH 过多说明可能回表读取文档多。

**治理方案**
- 调整复合索引顺序。
- 增加覆盖字段。
- 改写查询条件。
- 分页避免 skip 深分页。

**面试追问：keysExamined 很大但 nReturned 很小说明什么？**
说明索引过滤效率差，可能索引字段顺序不合理、条件选择性低或查询范围太大。

### 12. MongoDB 深分页怎么优化？

**结论先行：**
MongoDB 深分页不要依赖大 offset 的 skip，应使用基于排序字段和唯一键的游标分页。

**分层展开：**

**skip 问题**
- skip 需要跳过前面大量记录。
- 页码越深越慢。
- 对大集合非常不友好。

**游标分页**
- 按 `create_time + _id` 排序。
- 下一页带上上一页最后一条的时间和 id。
- 查询小于或大于游标的下一批。

**工程注意**
- 排序字段要有索引。
- 单排序字段不唯一时加 `_id` 兜底。
- 前端需要接受 cursor 模式，而不是任意页跳转。

**面试追问：为什么要加 `_id` 做第二排序？**
因为 create_time 可能重复，加 `_id` 能保证排序稳定，避免翻页重复或漏数据。

### 13. 聚合管道怎么优化？

**结论先行：**
聚合管道优化原则是尽早过滤、尽早投影、减少数据量、利用索引、控制 `$lookup` 和 `$group` 成本。

**分层展开：**

**优化方式**
- `$match` 尽量放前面。
- `$project` 去掉不需要字段。
- `$sort` 尽量利用索引。
- `$limit` 提前减少数据量。
- 避免大规模 `$unwind`。

**高风险操作**
- 大集合 `$group`。
- 跨集合 `$lookup`。
- 无索引 `$sort`。
- 大数组 `$unwind` 后爆炸。

**工程建议**
- 复杂报表尽量异步预计算。
- OLTP 主库避免重聚合。
- 使用 explain 验证执行计划。

**面试追问：MongoDB 能不能做复杂报表？**
能做，但不代表适合在主业务库实时做。复杂报表更适合离线数仓、预聚合表或搜索分析引擎。

### P0 补充：Aggregation 聚合管道的性能边界是什么？

**结论先行：**
聚合管道适合在 MongoDB 内做一定的数据加工，但不适合把主业务库当实时数仓。大 `$group`、大 `$sort`、大 `$lookup`、大 `$unwind` 都可能造成内存、CPU 和 IO 压力。

**重点规则：**

- `$match` 尽量前置，先减少数据量。
- `$project` 尽早裁剪大字段。
- `$sort` 尽量利用索引，否则可能内存排序。
- `$limit` 可以提前减少后续阶段数据量。
- `$lookup` 要控制关联集合规模和索引。
- `$unwind` 会放大数据行数，大数组尤其危险。
- `$group` 可能需要大量内存，必要时考虑预聚合。

**工程方案：**

- 用户在线请求避免跑重聚合。
- 报表类任务异步化，写入汇总集合。
- 大时间范围查询拆分分片、分批跑。
- 对聚合任务设置超时和资源隔离。
- 用 explain 看是否走索引、是否扫描过多文档。

**面试追问：allowDiskUse 是不是万能解法？**
不是。它允许部分聚合阶段溢写磁盘，避免内存限制失败，但会显著增加 IO 和延迟。它是兜底手段，不是性能优化手段。

### 14. MongoDB 事务怎么理解？

**结论先行：**
MongoDB 支持多文档事务，但最佳实践仍是优先通过单文档原子性和合理建模减少事务使用。

**分层展开：**

**单文档原子性**
- 单个文档更新天然原子。
- 嵌入式建模可把强一致对象放在一个文档内。

**多文档事务**
- 支持跨文档、跨集合事务。
- 可用于少量强一致变更。
- 成本高于单文档写入。

**工程建议**
- 高频核心链路少用大事务。
- 事务内操作要短。
- 失败要可重试。
- 仍要考虑幂等和补偿。

**面试追问：MongoDB 有事务后是否就能像 MySQL 一样用？**
不建议。MongoDB 事务是补充能力，不应把 MongoDB 当成关系型数据库来大量跨集合事务化使用。

### 15. readConcern 和 writeConcern 是什么？

**结论先行：**
readConcern 控制读到什么一致性级别的数据，writeConcern 控制写入被多少节点确认后返回成功。

**分层展开：**

**writeConcern**
- `w:1` 主节点确认。
- `majority` 多数节点确认。
- journal 控制是否等待日志持久化。

**readConcern**
- local：读本节点最新数据。
- majority：读多数提交的数据。
- snapshot：事务中一致性快照。

**工程取舍**
- 更强一致性通常意味着更高延迟。
- 核心写入可用 majority。
- 非核心日志类写入可降低确认要求。

**面试追问：writeConcern majority 能保证绝对不丢吗？**
它能显著降低故障切换丢数据概率，但仍要结合 journal、部署拓扑和业务补偿理解，不能替代完整业务审计。

### 16. readPreference 是什么？

**结论先行：**
readPreference 决定读请求路由到主节点还是从节点，影响读扩展、延迟和一致性。

**分层展开：**

**常见策略**
- primary：只读主。
- primaryPreferred：优先主。
- secondary：只读从。
- secondaryPreferred：优先从。
- nearest：读网络最近节点。

**风险**
- 读从可能读到旧数据。
- nearest 可能跨一致性边界。
- 核心写后读场景应读主或保证因果一致性。

**工程建议**
- 强一致读主。
- 报表、非核心展示可读从。
- 监控复制延迟。

**面试追问：读从库能提升写性能吗？**
不能。读从只能分担读压力，写仍由主节点处理。

### 17. MongoDB 乐观并发怎么做？

**结论先行：**
MongoDB 常用版本号、更新时间或条件更新实现乐观并发，更新时把预期状态放进 filter。

**分层展开：**

**版本号方式**
- 文档带 `version` 字段。
- 更新条件包含旧 version。
- 更新成功后 version + 1。

**状态机方式**
- 更新条件包含当前状态。
- 例如只允许 `pending -> paid`。
- 重复消息影响行数为 0 时按幂等处理。

**原子操作**
- `$inc`、`$set`、`$addToSet` 等单文档原子。
- 避免读改写竞争。

**面试追问：库存扣减能不能用 MongoDB？**
可以用条件更新和 `$inc` 做原子扣减，但高并发热点库存仍要关注单文档热点、重试和容量，交易核心更常见用 Redis 预扣 + DB 兜底。

### P0 补充：Change Streams 怎么用于数据同步？

**结论先行：**
Change Streams 基于 oplog 提供变更订阅能力，适合把 MongoDB 变更同步到 ES、缓存、数仓或异步任务系统。但它通常只能做到至少一次投递，业务侧必须幂等。

**核心概念：**

- oplog：复制日志，是变更流的基础。
- resume token：断点续传位置。
- fullDocument：需要时可返回变更后的完整文档。
- watch pipeline：可过滤关心的集合、操作类型和字段。

**工程风险：**

- 消费延迟超过 oplog 窗口，可能无法继续增量追赶。
- 网络抖动、进程重启会导致重复消费。
- ES/缓存更新失败需要重试和补偿。
- 变更顺序通常只能在一定范围内保证，跨分片要更谨慎。

**幂等方案：**

- 下游写入带业务唯一键。
- 记录最后处理的 resume token。
- 同步任务可重放。
- 下游更新使用版本号、更新时间或幂等 upsert。

**面试追问：Change Streams 能保证 exactly once 吗？**
不能简单认为能。工程上按至少一次处理，依靠幂等、断点、重试和校验保证最终一致。

## 三、实战设计与线上治理：慢查询、大文档、归档、Go Driver 和业务建模

### 18. MongoDB 慢查询怎么排查？

**结论先行：**
慢查询排查从慢日志、explain、索引、扫描量、排序、锁等待、数据量和集群状态逐层定位。

**分层展开：**

**查询层**
- 是否 COLLSCAN。
- 是否大范围扫描。
- 是否内存排序。
- 是否返回大文档。

**索引层**
- 索引是否匹配查询和排序。
- 是否存在低选择性索引。
- 是否多键索引膨胀。

**集群层**
- 主从延迟。
- 分片是否均衡。
- 是否广播查询。
- 磁盘 IO 和内存命中率。

**工程治理**
- 建合适复合索引。
- 改游标分页。
- 拆大文档。
- 复杂聚合异步化。

**面试追问：MongoDB 查询慢但有索引，可能是什么原因？**
索引顺序不匹配、过滤性差、返回文档过大、排序没用上索引、数组多键索引膨胀或分片广播查询。

### 19. 大文档和大数组有什么危害？

**结论先行：**
大文档和大数组会导致读写放大、内存压力、网络传输变大、索引膨胀和单文档更新冲突。

**分层展开：**

**大文档危害**
- 每次读取传输大量无用字段。
- 更新可能移动文档。
- 占用缓存。
- 接近单文档大小限制。

**大数组危害**
- `$push` 持续增长。
- 多键索引膨胀。
- 修改数组局部成本高。
- 分页困难。

**治理方案**
- 拆集合。
- 子文档分页。
- 冷热拆分。
- 只保留摘要字段。

**面试追问：评论列表能不能嵌入文章文档？**
少量评论预览可以嵌入，完整评论列表不适合，因为评论会无限增长，应该单独集合存储并分页查询。

### 20. MongoDB 如何做数据归档？

**结论先行：**
数据归档要按访问热度和业务生命周期拆分，常用冷热集合、历史库、TTL 索引和离线存储。

**分层展开：**

**归档方式**
- 热集合保近期数据。
- 历史集合保冷数据。
- TTL 自动清理临时数据。
- 同步到数仓或对象存储。

**注意事项**
- 归档任务限速。
- 归档前后校验数量和关键字段。
- 查询层能识别冷热数据。
- 保留回滚窗口。

**面试追问：TTL 索引适合订单数据吗？**
不适合核心订单。TTL 适合验证码、临时会话、短期日志等可自动过期数据，核心业务数据需要审计和归档策略。

### 21. Go 使用 MongoDB 要注意什么？

**结论先行：**
Go 使用 MongoDB 要关注 context 超时、连接池、BSON tag、ObjectID、错误分类、索引初始化和慢查询监控。

**分层展开：**

**连接治理**
- 配置连接池大小。
- 设置 server selection timeout。
- 每次操作传 context。
- 监控连接等待。

**编码规范**
- struct 字段写 `bson` tag。
- 区分零值和字段缺失。
- ObjectID 和字符串转换要统一。
- update 使用 `$set`，避免误替换整个文档。

**错误处理**
- duplicate key。
- no documents。
- write concern error。
- transient transaction error。

**面试追问：Go 里 update 忘记 `$set` 会怎样？**
可能把整个文档替换成传入对象，导致字段丢失。更新操作要明确使用 `$set/$inc/$push` 等操作符。

### 22. 如何用 MongoDB 设计用户画像？

**结论先行：**
用户画像适合用 MongoDB 存储半结构化标签和特征摘要，但明细行为不应无限塞进用户文档。

**分层展开：**

**文档设计**
- user_id 作为主查询键。
- tags 保存标签摘要。
- features 保存特征值。
- version/update_time 做版本治理。

**数据来源**
- 行为日志异步计算。
- MQ 或离线任务更新画像。
- 画像变更可发事件给推荐和营销。

**注意事项**
- 标签数组控制规模。
- 高频明细单独存储。
- 重要字段建索引。

**面试追问：画像更新如何保证不覆盖？**
使用 `$set` 更新局部字段，必要时带 version 条件；多来源写入要区分字段 owner 或通过统一画像服务写入。

## 四、项目实战表达：把 MongoDB 讲成复杂业务建模能力

> 这一章用于项目深挖，重点讲广告任务、状态流转、幂等和 MySQL 分工。

### 24. 枫叶互动广告任务为什么适合用 MongoDB？

**结论先行：**
广告任务适合 MongoDB，不是因为 MongoDB 比 MySQL “高级”，而是因为广告投放配置天然字段多、层级深、平台差异大、变化频繁，文档模型更适合承载这种半结构化业务。

**业务背景：**
Facebook、TikTok、Snapchat 的广告模型都包含 Campaign、AdGroup/AdSet/Squad、Ad 等层级，但字段并不完全一致。不同平台在预算、出价、定向、素材、像素、落地页、状态、错误码、审核信息等字段上差异很大，而且媒体 API 经常变化。

**MongoDB 建模方式：**
- 按媒体平台拆分任务集合或模型，保留平台字段差异。
- 一个广告任务文档保存任务配置、平台返回、状态流转、错误日志和 trace 信息。
- 使用 `task_id`、`parent_task_id`、`account_id`、`status`、`created_at` 等字段建索引。
- 对列表页只投影必要字段，避免大文档拖慢查询。
- 复杂检索逐步迁移到 ES，MongoDB 保留为权威存储。

**面试亮点：**
我会强调“MongoDB 承载灵活配置，MySQL 承载强事务结构化数据，ES 承载复杂检索”，而不是所有数据都放到一个存储里。

### 25. 广告任务状态流转怎么设计？

**核心状态：**
- draft：草稿。
- validating：校验中。
- pending：待发布。
- publishing：发布中。
- success：发布成功。
- failed：发布失败。
- partial_success：部分成功。
- canceled：已取消。

**关键原则：**
- 状态流转要单向推进，避免随意回退。
- 每次状态变化记录操作人、时间、原因、平台返回和 trace_id。
- 失败状态保留上下文，支持重试、复制任务或重新发布。
- 并发更新时使用条件更新，例如只允许 `pending -> publishing`，避免重复发布。

**追问：如果广告组成功，广告失败怎么办？**
这类任务不能简单整体失败，应该允许部分成功。任务树上每一层都有独立状态，最终总任务聚合子任务状态。失败的子任务保留平台错误码和请求上下文，支持按失败节点重试。

### 26. 素材同步任务如何用 MongoDB 做幂等？

**业务背景：**
一个短剧素材可能需要同步到多个媒体账号。运营可能重复点击同步，定时任务也可能重复扫描，如果没有幂等，会重复上传素材或产生多条任务。

**幂等键设计：**
可以使用以下字段组成唯一条件：

```text
channel + resource_id + resource_type + account_id + account_type
```

**处理流程：**
1. 查询素材和目标账号列表。
2. 过滤不符合平台要求的素材，例如视频尺寸、时长、格式。
3. 对每个素材账号组合执行 batch upsert。
4. 已存在且失败的任务更新为待上传。
5. 新增或重新激活的任务推送 Kafka，由消费者异步上传。
6. 每个任务记录 trace_id，便于排查。

**追问：Kafka 发送失败怎么办？**
严格方案是本地消息表/outbox；工程上也可以记录任务状态为 pending，由补偿任务扫描 pending 但长时间未消费的任务重新投递。核心是 MongoDB 里的任务状态不能依赖“消息一定发送成功”。

### 27. 自动投放模板为什么用聚合管道？

**业务背景：**
自动投放模板列表需要支持名称搜索、媒体平台、项目、状态、创建人、角色权限等筛选，还要展示模板绑定了多少策略。这个需求不是简单主键查询。

**Aggregation Pipeline 设计：**
- `$match`：组合基础筛选、权限筛选和搜索条件。
- `$lookup`：关联模板和策略关系。
- `$addFields`：计算策略数量。
- `$sort`：按更新时间或创建时间排序。
- `$facet`：一次查询返回分页列表和总数。

**风险：**
- `$lookup` 和 `$facet` 在数据量大时成本高。
- `$match` 要尽量前置，先缩小数据集。
- 搜索条件要结合索引设计，避免全表扫描。
- 如果列表查询继续膨胀，应把检索能力迁移到 ES。

### 28. MongoDB 和 MySQL 在你项目里怎么分工？

**推荐回答：**

在枫叶互动广告投放平台里，我会按数据形态和一致性要求选存储：

- MongoDB：广告任务、媒体平台返回、素材同步任务、自动投放模板等半结构化、字段变化快、嵌套层级深的数据。
- MySQL：投放工单、分配规则、绩效统计、轮次状态、权限审计等结构化强、事务要求高的数据。
- Redis：锁、限流、缓存版本、热点配置。
- Kafka：异步任务和最终一致。
- ES：素材检索、广告任务列表复杂查询。

**总结句：**
我的选型原则不是偏好某个数据库，而是让每类存储承担最适合自己的职责，减少后续演进成本。

**结论先行：**
商品扩展属性适合 MongoDB，因为不同类目字段差异大，但查询频繁字段需要标准化和建索引。

**分层展开：**

**模型设计**
- 固定字段独立存储，如商品 id、类目、状态。
- 扩展属性放 attributes 子文档。
- 类目定义属性 schema。

**查询设计**
- 高频筛选字段冗余到顶层或单独索引。
- 低频属性可只做展示。
- 复杂搜索筛选更适合同步到 ES。

**一致性**
- 商品主数据以商品服务为准。
- MongoDB 保存扩展属性。
- ES 做搜索索引。

**面试追问：商品筛选直接查 MongoDB 可以吗？**
简单条件可以，复杂多条件搜索、分词、排序和聚合更适合 ES，MongoDB 不要承担所有搜索职责。

## 五、面试方法论、速记与回答模板

> 本模块统一收纳“怎么答、怎么组织、怎么临场展开”的内容；前面章节负责知识本体，这里负责面试表达。

### 方法 1：资深后端如何回答 MongoDB 问题？

**结论先行：**
MongoDB 面试不能只说“JSON 存储灵活”，要围绕“文档建模、查询模式、索引设计、一致性边界、分片扩展和线上治理”回答。

**分层展开：**

**模型层**
- MongoDB 面向文档，数据以 BSON 存储。
- 文档天然适合聚合型对象和半结构化数据。
- 建模要从查询模式出发，而不是照搬关系型三范式。

**性能层**
- 索引决定查询效率。
- 聚合管道适合数据加工，但要控制内存和扫描量。
- 大文档、大数组、低选择性索引都会拖慢性能。

**一致性层**
- 单文档写入天然原子。
- 多文档事务支持存在，但成本高于单文档设计。
- readConcern/writeConcern/readPreference 决定读写一致性和可用性取舍。

**架构层**
- 副本集解决高可用。
- 分片集群解决容量和吞吐扩展。
- 分片键选择决定后续扩展质量。

**面试追问：MongoDB 适合替代 MySQL 吗？**
不适合简单替代。MongoDB 适合文档型、半结构化、查询模式相对明确且需要横向扩展的场景；强事务、复杂关联、强约束场景仍更适合 MySQL。
