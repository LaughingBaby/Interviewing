# 技术｜微服务与 RPC 治理面试题

> 面向 9 年以上 Golang Web 后端业务研发岗位。重点是服务为什么拆、边界如何定、RPC 失败如何治理、数据如何协作、变更如何安全发布，以及怎样用工程指标证明微服务没有变成分布式单体。

---

## 一、微服务学习地图与资深回答框架

### 1. 微服务与 RPC 的知识图谱主线是什么？

**一句话结论：** 微服务不是把代码拆成更多进程，而是围绕业务能力建立可独立演进、交付和治理的边界，并承担由网络调用和数据分治带来的复杂度。

```text
业务复杂度与组织边界
  -> 领域拆分、服务职责和数据所有权
  -> API / HTTP / gRPC / Protobuf 契约
  -> 注册发现、配置、负载均衡和健康检查
  -> Deadline、重试、幂等、限流、熔断、降级
  -> 本地事务、事件驱动和工作流
  -> 可观测性、发布、灰度、回滚和兼容
  -> 安全、容量、容灾和成本
  -> 线上排障与项目表达
```

### 2. 资深后端如何回答微服务问题？

采用“业务问题 -> 边界原则 -> 调用与数据 -> 失败治理 -> 发布运维 -> 指标收益”的结构。

**示例：为什么拆订单服务？**

- **业务问题：** 订单生命周期、交易 SLA 和迭代节奏与商品、营销不同。
- **边界：** 订单服务拥有订单状态与流水，其他服务不能直接改表。
- **调用与数据：** 同步 API 完成必须即时确认的操作，领域事件传播状态变化。
- **失败治理：** 幂等、状态机、Outbox、deadline、限流和补偿。
- **发布运维：** 接口兼容、灰度、回滚、数据迁移和独立容量。
- **收益：** 发布频率、故障域、P99、扩容效率改善；同时承认链路和一致性复杂度增加。

### 3. 微服务架构题的回答步骤是什么？

1. 量化业务规模、团队协作、变化频率和 SLA。
2. 识别领域边界、核心用例和数据所有权。
3. 设计同步 API 与异步事件，限制调用链深度。
4. 定义超时、重试、幂等、熔断、降级和过载保护。
5. 设计一致性、状态机、补偿、对账和重放。
6. 设计可观测性、灰度发布、兼容、回滚和故障演练。
7. 用交付、稳定性和成本指标验证拆分价值。

### 4. 微服务高频题如何串联？

| 层次 | 核心问题 | 关键词 |
|---|---|---|
| 拆分 | 为什么拆、拆多细 | 领域能力、变化率、数据所有权 |
| 契约 | HTTP 还是 gRPC | 语义、兼容、流式、调试、生态 |
| 路由 | 如何找到实例 | 注册发现、健康、LB、连接复用 |
| 韧性 | 下游变慢怎么办 | deadline、重试预算、熔断、隔离 |
| 数据 | 跨服务如何一致 | 本地事务、Outbox、Saga、对账 |
| 观测 | 如何定位跨服务问题 | RED、Trace、结构化日志、业务 ID |
| 发布 | 如何避免变更事故 | 向后兼容、灰度、回滚、Feature Flag |
| 安全 | 服务间如何信任 | mTLS、身份、授权、密钥、审计 |
| 实战 | 怎么证明设计有效 | SLA、故障域、交付效率、成本 |

## 二、服务拆分、领域边界与架构演进

### 5. 单体、模块化单体和微服务如何选择？

- 单体部署简单、调用本地、事务自然，适合早期和复杂度有限的系统。
- 模块化单体通过清晰包边界和数据访问约束控制耦合，是很多团队的优先起点。
- 微服务适合业务边界较稳定、团队需要独立交付、容量和 SLA 差异明显的场景。

如果没有独立演进和治理需求，拆成微服务只会增加网络、数据一致性、测试和运维成本。

### 6. 服务边界如何划分？

优先依据业务能力、聚合和变化原因，而不是按 Controller/DAO 技术层拆分。一个服务应有：

- 清晰业务职责和语言。
- 独立数据所有权。
- 可独立发布和扩缩容。
- 相对内聚的变更集合。
- 可接受的同步调用数量和团队认知负担。

领域边界不是一次设计永久不变，需要用事件风暴、调用数据和变更历史持续校准。

### 7. 什么是限界上下文？

同一业务词在不同上下文可有不同模型。例如“用户”在账户域是认证主体，在广告域是投放客户，在结算域是合同与账务主体。限界上下文明确模型语义、规则和责任，服务通常围绕它划分，但不必机械一一对应。

### 8. 服务粒度过细有什么问题？

- 一次请求跨多个服务，尾延迟和故障概率叠加。
- 本地事务变成分布式一致性。
- 发布、测试、版本和运维对象暴增。
- 团队频繁跨服务修改，边界并未降低耦合。
- 小服务各自占连接池和基础资源，成本上升。

若两个服务总是一起改、一起发布、共享数据库且频繁同步调用，通常应重新评估合并。

### 9. 什么是分布式单体？

系统虽然拆成多个进程，但服务共享数据库、调用链强耦合、必须一起发布、任何一个失败都影响全链路。这既失去单体简单性，又没有微服务独立性。

治理方向是明确数据所有权、减少同步依赖、建立兼容契约、允许独立发布，并合并没有独立业务价值的小服务。

### 10. 服务能否共享数据库？

长期目标应是服务拥有自己的数据，其他服务通过 API 或事件访问。共享数据库会绕过业务规则、导致 Schema 变更互相影响、难以独立扩容和审计。

迁移期可以共享实例但分 Schema/账号，并禁止跨域直接写；通过 CDC、事件和渐进迁移逐步收回所有权。

### 11. 如何从单体渐进迁移到微服务？

使用 Strangler Pattern：

1. 先模块化和补可观测性，找到高收益边界。
2. 在网关或应用路由层切出一个低耦合能力。
3. 通过 Anti-Corruption Layer 隔离旧模型。
4. 先同步历史数据，再双写/CDC 增量并对账。
5. 灰度流量，保留回切。
6. 稳定后删除旧路径，而不是永久双写。

### 12. 如何判断拆分是否成功？

- 服务可独立开发、测试、发布和回滚。
- 跨团队变更和发布协调次数下降。
- 故障影响范围和恢复时间下降。
- 可按服务容量独立扩缩容。
- 业务 P99、可用性和交付周期改善。
- 基础设施、人力和调用复杂度没有失控。

## 三、HTTP、gRPC、Protobuf 与接口契约

### 13. HTTP REST 与 gRPC 如何选择？

| 维度 | HTTP/JSON | gRPC/Protobuf |
|---|---|---|
| 可读与调试 | 直观、生态广 | 需工具和生成代码 |
| 契约 | OpenAPI 可描述 | IDL 强约束、代码生成 |
| 性能 | 文本编码开销较高 | 二进制、HTTP/2、多路复用 |
| 流式 | 支持方式较分散 | 原生四种 RPC 模式 |
| 浏览器/外部 API | 更自然 | 常需网关或 gRPC-Web |
| 内部服务 | 通用 | 强类型、高吞吐场景常见 |

外部开放 API 常优先 HTTP/JSON，内部稳定服务可优先 gRPC；也可通过网关提供双协议。

### 14. gRPC 的四种调用模式是什么？

- Unary：一请求一响应，最常见。
- Server Streaming：一个请求持续返回流。
- Client Streaming：客户端持续发送，服务端最终响应。
- Bidirectional Streaming：双方独立流式发送。

流式并不自动更快，要处理背压、生命周期、断线恢复、消息顺序、内存和负载均衡黏性。

### 15. Protobuf 为什么高效？

它使用字段编号和紧凑二进制编码，省去重复字段名；生成代码提供类型约束，未知字段可支持兼容转发。代价是可读性较差、需要 IDL 与生成流程、某些动态结构表达不如 JSON 直接。

性能收益仍应端到端压测，数据库和业务处理可能才是主要瓶颈。

### 16. Protobuf 如何保证兼容演进？

- 已发布字段号永不复用，删除字段用 `reserved` 保留编号和名称。
- 新增可选字段并提供明确默认语义。
- 不随意改变字段类型和语义。
- Enum 新增值时旧客户端必须能处理未知值。
- 先发布能兼容新旧数据的服务端/消费者，再启用新字段。
- 在 CI 中检查 Breaking Change。

### 17. API 如何做向后兼容？

优先增加字段和能力，不改变旧字段含义；服务端对缺失字段提供兼容；删除采用“废弃 -> 观测使用量 -> 通知 -> 双版本 -> 下线”。

版本号只能解决识别问题，不能替代兼容设计。内部 API 也必须有消费者清单和契约测试。

### 18. 接口应该返回领域对象还是专用 DTO？

使用专用 API DTO 隔离内部存储与领域实现，防止数据库字段直接成为永久外部契约。DTO 只暴露调用方需要的信息，明确可空性、单位、时区、分页和错误语义。

### 19. 错误码如何设计？

分为：参数/认证/权限、资源不存在、业务冲突、限流过载、依赖失败和内部错误。错误响应包含稳定机器码、可读信息、trace_id 和可选重试提示，不暴露堆栈和内部敏感信息。

gRPC 状态码要映射业务语义；`UNAVAILABLE` 可能重试，`INVALID_ARGUMENT` 不应重试，`ALREADY_EXISTS/ABORTED` 要结合幂等和并发语义。

### 20. 分页接口如何设计？

大数据集使用基于稳定排序键的 Cursor，令牌包含排序值、查询条件版本和必要签名；限制 page_size 和返回字段。Offset 适合浅分页和小集合，不适合深分页。

### 21. API Gateway 和 BFF 分别解决什么？

- Gateway 处理统一入口、认证、路由、限流、协议转换、观测和灰度。
- BFF 为特定客户端聚合和裁剪数据，减少移动端/网页多次调用。

不能把所有业务编排塞进网关；核心业务规则仍属于领域服务，避免网关成为新单体。

## 四、服务发现、配置、负载均衡与连接治理

### 22. 服务发现的基本流程是什么？

```text
实例启动 -> 注册地址和元数据 -> 健康/租约维护
客户端或网关查询实例列表 -> 负载均衡选择 -> 发起调用
实例下线 -> 先摘流量 -> 完成在途请求 -> 注销
```

注册中心保存控制面信息，不应进入每次请求的数据面关键路径；客户端需要本地缓存和异常兜底。

### 23. 客户端发现和服务端发现有什么区别？

- 客户端发现由调用方获取实例列表并负载均衡，少一跳但客户端复杂、语言 SDK 多。
- 服务端发现由 LB/Proxy 统一路由，客户端简单但增加代理层和运维成本。
- Service Mesh 属于代理化方案，将流量治理下沉到 Sidecar/节点代理。

### 24. 健康检查应该检查什么？

- Liveness：进程是否需要重启，不能因为临时下游失败就重启。
- Readiness：是否可以接收新流量，包括初始化、预热和关键资源状态。
- Startup：慢启动期间避免被误杀。
- gRPC 可暴露标准 Health Service，由服务更新 SERVING 状态。

健康检查要轻量、有超时，不能每次深查所有依赖造成探测风暴。

### 25. 常见负载均衡算法如何选择？

- Round Robin：实例同质、请求相近。
- Weighted Round Robin：实例容量不同。
- Least Request/Connection：请求耗时差异明显。
- Consistent Hash：需要会话或缓存局部性，但可能热点。
- EWMA/自适应：依据历史延迟和排队选择更快实例。

无论算法如何，都需配合异常实例摘除、慢启动和连接池治理。

### 26. gRPC 长连接如何影响负载均衡？

HTTP/2 多路复用让大量 RPC 复用少量连接。如果客户端只建立一个连接并在连接建立时选定实例，流量可能长期落在单实例。

需要使用合适的 Name Resolver、LB Policy、连接池或代理；扩容后还要让客户端及时发现并建立到新实例的 Subchannel。

### 27. 配置中心应如何治理？

- 配置有 Schema、默认值、环境和作用域。
- 变更经过权限、审计、评审和灰度。
- 客户端使用本地缓存，配置中心故障不应让服务立即不可用。
- 动态配置要验证范围和边界，支持快速回滚。
- 关键参数如超时、限流、开关要有变更事件和指标关联。

配置中心不是保存密码的普通 KV；密钥应进入专用 Secret 系统。

### 28. 优雅启动和优雅停机怎么做？

启动时先完成配置、连接池、缓存预热和依赖检查，再置 Ready。停机时先置 Not Ready/摘流量，停止接收新请求，等待在途 RPC 和消费者处理，提交 Offset/Flush 数据，超时后退出。

Golang 要处理 SIGTERM、Context Cancellation、HTTP Server Shutdown、gRPC GracefulStop 的超时兜底和后台 Goroutine 回收。

### 29. DNS、注册中心和本地缓存不一致怎么办？

客户端要容忍实例列表短暂陈旧：连接失败快速摘除并后台刷新，列表为空时短时使用最后成功快照，但不能无限信任。监控解析错误、实例收敛时间和过期实例调用率。

## 五、Deadline、重试、幂等、限流、熔断与隔离

### 30. 为什么每个 RPC 都必须设置 Deadline？

没有 Deadline 的调用可能无限等待，占用 Goroutine、连接、内存和上游请求，最终级联过载。Deadline 应根据端到端 SLO、历史 P99 和下游预算设置，并沿调用链传播。

服务端收到取消后必须停止数据库、HTTP 和后台工作，避免调用方放弃后仍浪费资源。

### 31. 超时预算如何分配？

若入口剩余 800ms，不能给三个串行下游各 800ms。需预留网关、排队、业务处理和响应时间，再按关键路径分配；并行调用预算取最大分支而非简单相加。

预算应动态使用剩余 Deadline，避免中间服务重新创建更长超时。

### 32. 哪些 RPC 可以重试？

只重试瞬时错误、幂等操作、未超过总 Deadline 且未触发过载保护的调用。使用有限次数、指数退避、抖动和 Retry Budget。

创建/扣款等写操作必须有幂等键；如果服务端已收到请求但响应丢失，透明重试也可能重复执行。

### 33. 如何实现接口幂等？

- 调用方生成业务唯一 request_id。
- 服务端用唯一索引或幂等表持久化请求状态和结果。
- 同一 key 参数摘要不同则拒绝，防止误复用。
- 并发请求看到处理中状态时等待、查询或返回明确语义。
- 幂等记录和业务写入处于同一本地事务。

### 34. 限流应该放在哪些层？

- CDN/WAF：恶意和大流量初筛。
- Gateway：租户、用户、API 全局配额。
- 服务实例：本地 QPS 与并发保护。
- 依赖侧：数据库、第三方 API 和热点资源配额。

令牌桶允许突发，并发限制更适合保护慢调用；全局精确和本地高可用常组合使用。

### 35. 熔断器的状态机是什么？

Closed 正常统计；错误率/慢调用达到阈值后进入 Open 快速失败；等待后进入 Half-Open，只允许少量探测，成功则关闭，失败再次打开。

熔断按下游方法或实例池隔离，设置最小请求量，避免低流量误判；恢复要渐进，不能瞬间放回全部流量。

### 36. 舱壁隔离解决什么问题？

将不同依赖、租户或业务等级放入独立连接池、并发池、队列或实例组，避免一个慢依赖占满全部资源。Golang 中可用有界 Worker Pool、Semaphore 和独立 HTTP/gRPC Client 实现。

### 37. 降级如何保持业务语义清晰？

- 推荐不可用返回热门内容，而不是伪造个性化结果。
- 搜索不可用返回基础筛选或缓存，并标识数据时间。
- 非核心统计缺失可省略字段。
- 核心写失败必须明确失败或处理中，不能返回假成功。

降级要定义触发、恢复、缓存陈旧上限和数据补偿。

### 38. 如何防止调用链放大？

- 限制同步调用深度和扇出数量。
- 批量 API 替代循环 N+1 RPC。
- 并行请求设置总并发和共享 Deadline。
- 请求级缓存/合并相同下游调用。
- 不在多层重复重试。
- 将非关键动作异步事件化。

### 39. 第三方 API 如何治理？

使用独立 Client、连接池、deadline、限流、熔断和并发隔离；请求带外部幂等号；超时后通过查询接口确认结果；错误分类重试，长期失败进入任务与人工处理；记录供应商成功率、P99、配额和成本。

## 六、数据所有权、事件驱动、事务与工作流

### 40. 微服务之间为什么不能直接改对方数据库？

直接改表绕过业务规则，使 Schema 和发布强耦合，责任与审计模糊，也让服务无法独立迁移。应通过命令 API 请求对方执行规则，或订阅其公开事件构建本地读模型。

### 41. 同步调用和异步事件如何选择？

- 调用方必须立即知道结果、操作短且交互式：同步 RPC。
- 可延迟、需削峰、一对多传播、希望解耦：异步事件。
- 核心命令常同步确认受理，后续工作异步执行。

异步不是自动可靠；需要事件契约、幂等、顺序、重试、死信、对账和可观测性。

### 42. Command 和 Event 有什么区别？

- Command 表达“请某个服务做某事”，有明确目标和可能失败。
- Event 表达“某个事实已经发生”，发布方不应要求某个消费者必须采取特定内部动作。

把事件命名为 `OrderCreated` 比 `NotifyInventoryToDeduct` 更解耦；但关键流程仍需确认消费者结果和超时治理。

### 43. 跨服务一致性如何设计？

单服务内使用本地事务保存业务事实和 Outbox；事件发布到 Kafka；消费者用 Inbox/唯一键幂等处理，并产生下一事件。长流程通过显式状态机或 Saga 编排，失败执行补偿，最终由对账发现遗漏。

强一致核心应尽量收敛到同一服务边界，而不是在所有服务间追求分布式锁。

### 44. Saga 编排和协同如何选择？

- Choreography：服务订阅事件自行反应，耦合低，但流程分散、排障困难。
- Orchestration：编排器显式下发步骤和记录状态，流程清晰，编排器复杂且需高可用。

短流程可协同，跨多个服务、超时和人工节点的关键流程常用编排。无论哪种都要幂等、补偿和审计。

### 45. Outbox Pattern 解决什么问题？

在同一本地事务中更新业务数据并插入 Outbox，后台 Relay/CDC 发布事件，消除“数据库提交成功但消息未发送”的窗口。

它仍可能重复发布，消费者必须幂等；还要治理 Outbox 积压、分区、清理、事件版本和发布延迟。

### 46. 事件 Schema 如何演进？

- 使用稳定 event_type、event_id、aggregate_id、version、occurred_at 和 trace context。
- 新增字段保持旧消费者可忽略，删除字段经过废弃周期。
- 不随意改变字段语义和单位。
- Schema Registry/CI 做兼容检查。
- 消费者先兼容新旧版本，再升级生产者。

### 47. 如何处理事件重复、乱序和迟到？

重复用 event_id/业务键去重；同一实体按 Key 分区；事件带 version，旧版本拒绝覆盖；缺版本可回查事实源或触发重建；迟到事件按业务窗口处理。

删除也要带版本或 Tombstone，防止迟到更新复活数据。

### 48. CQRS 适合什么场景？

写模型承载业务规则和事务，读模型为搜索、列表、报表按查询需求冗余。适合读写模型差异大、查询复杂、可接受最终一致的系统。

代价是模型同步、重复数据、延迟、对账和运维。普通 CRUD 不应为了模式完整而引入 CQRS。

### 49. 如何治理长时间业务流程？

使用持久化工作流状态机记录步骤、输入、尝试次数、Deadline 和补偿；每步幂等，定时扫描超时任务，支持人工暂停/重试/跳过；外部副作用保存请求号和响应证据。

不要用内存 Goroutine 或一次长 RPC 承载跨分钟、小时的流程。

## 七、可观测性、发布治理与变更安全

### 50. 微服务可观测性应该覆盖什么？

- 接口 RED：Rate、Errors、Duration。
- 实例 USE：Utilization、Saturation、Errors。
- Trace：入口到 DB、缓存、MQ、第三方的关键路径。
- 结构化日志：service、version、instance、trace_id、business_id、error_code。
- 业务不变量：订单状态、同步延迟、差异数和积压。
- 变更事件：发布、配置、扩容、路由和依赖切换。

### 51. Trace Context 如何传播？

在 HTTP Header、gRPC Metadata 和消息 Header 中传播 trace/span、request_id 和必要 baggage。异步消息创建 Consumer Span 并链接 Producer 上下文。

Baggage 只放小且必要的信息，敏感数据和高基数字段不能无界传播。

### 52. 日志如何避免“很多但没用”？

记录关键状态变化、依赖结果、错误分类和关联 ID；使用稳定字段而非拼接文本；控制级别、采样和敏感信息；一次错误由最理解语义的层记录，避免每层重复堆栈。

### 53. SLI、SLO 和 Error Budget 是什么？

- SLI 是实际测量值，如成功率和 P99。
- SLO 是目标，如月度 99.95% 成功率。
- Error Budget 是目标允许的失败额度，用于权衡发布速度和稳定性。

SLO 应面向用户体验，不只看容器存活；预算消耗过快时暂停高风险变更并治理可靠性。

### 54. 灰度发布如何设计？

按内部用户、租户、地域、设备或流量比例逐步放量；灰度版本与基线对比错误率、P99、资源和业务指标；设置自动暂停/回滚阈值；确保同一会话和状态流程路由一致。

数据库和消息 Schema 必须先兼容，再灰度应用，避免回滚后旧版本读不了新数据。

### 55. 蓝绿、滚动和金丝雀发布如何选择？

- 滚动成本低，版本短期混跑，要求兼容。
- 蓝绿切换和回滚快，但需要双份容量和数据变更策略。
- 金丝雀小流量验证风险，适合高频发布，但监控必须能区分版本。

选择取决于容量、兼容、风险和回滚速度，不应只由平台默认决定。

### 56. Feature Flag 如何治理？

Flag 要有负责人、范围、默认值、到期时间和回滚语义；变更需审计和灰度；代码同时支持开关两侧并做测试；功能稳定后及时删除 Flag，避免组合爆炸。

### 57. 数据库变更如何做到向后兼容？

采用 Expand-Contract：先新增字段/表并让旧代码可运行；新代码双读/双写或后台回填；验证后切读；最后停止旧写并删除旧结构。DDL、应用发布和数据迁移分阶段，每阶段可观察可回滚。

### 58. 消费者发布如何避免重复和丢处理？

停机前停止拉取、完成在途处理、提交 Offset 并关闭资源；新旧版本混跑时事件 Schema 兼容；每个消费者幂等；Rebalance 和超时期间不让同一批任务失控并发。大变更可新建 Consumer Group 影子消费验证。

## 八、安全、性能、容量与 Golang 工程实践

### 59. 微服务身份认证和授权如何分层？

- 入口验证用户身份和 Token，传递可信主体信息。
- 服务间使用工作负载身份和 mTLS，不能只信内网 IP。
- 每个服务在资源边界做授权，防止越权横向访问。
- 权限遵循最小化，敏感操作审计。

用户身份与服务身份是两层概念，不能用一个共享密钥替代全部授权。

### 60. mTLS 解决什么，没解决什么？

mTLS 提供双向身份认证和传输加密，可确认连接双方工作负载身份。它不自动决定“该服务能否访问某个订单”，仍需授权策略；也不解决应用层注入、越权和敏感日志。

### 61. Secret 如何管理？

密钥存专用 Secret Manager/KMS，通过短期凭证或工作负载身份获取；禁止写入代码、镜像、普通配置和日志；定期轮换并支持双密钥过渡；限制读取范围并审计。

### 62. 微服务容量如何估算？

```text
实例数 = 峰值 QPS / 单实例安全 QPS x 冗余系数
并发 ≈ QPS x 平均延迟
下游调用 QPS = 入口 QPS x 单请求调用次数 x 重试放大
```

还要计算连接池、响应大小、CPU/内存、消息积压恢复、N-1 容量和发布期间双版本容量。单实例安全 QPS 来自真实依赖和请求混合压测。

### 63. 如何降低 RPC 尾延迟？

- 减少串行调用和扇出。
- 治理慢实例、连接建立、DNS 和连接池等待。
- 缩小请求/响应，使用批量接口。
- 为热点数据缓存或预计算。
- 控制 GC、Goroutine、锁和下游查询。
- Hedging 只用于幂等读且有额外流量预算，不能默认开启。

### 64. Golang HTTP/gRPC 服务如何做资源治理？

- 入口设置读写、Header、Idle 和业务 Deadline。
- 限制请求体、并发、Goroutine 和有界队列。
- 复用 `http.Transport`/`grpc.ClientConn`，不要每请求建连接。
- 数据库和下游调用传播 Context。
- Interceptor/Middleware 统一认证、Trace、指标、恢复和错误映射。
- 优雅停机并等待在途任务，后台 Goroutine 可取消可回收。

### 65. gRPC Interceptor 适合做什么？

适合统一认证、授权、Trace、Metrics、日志、Panic Recovery、限流和错误规范化。业务规则、复杂重试和不同方法的降级不应全部塞进一个全局 Interceptor。

顺序很重要：恢复通常最外层，Trace 要覆盖后续处理，日志和指标应得到最终状态码。

### 66. Service Mesh 的收益和代价是什么？

收益是统一 mTLS、流量路由、观测和部分重试熔断，减少多语言 SDK 重复。代价是代理资源、网络跳数、控制面复杂度、排障层次和配置风险。

Mesh 解决通用流量治理，不理解业务幂等、补偿和降级语义；团队规模和异构程度不足时不必急于引入。

## 九、生产故障排查与稳定性治理

### 67. 微服务接口 P99 突然升高如何排查？

1. 确认影响接口、租户、地域、版本和时间线。
2. Trace 判断耗时在排队、应用、RPC、DB、缓存还是第三方。
3. 检查流量、错误、实例负载、GC、Goroutine、连接池和队列。
4. 检查下游 P99、超时、重试和熔断状态。
5. 对比发布、配置、扩容、证书和路由变化。
6. 先限流、降级、回滚或隔离慢依赖，再做代码/索引根治。

### 68. 大量 RPC `DeadlineExceeded` 如何定位？

区分 Deadline 本身过短、下游处理慢、客户端排队、连接建立/DNS、网络、服务端未响应取消和上游剩余预算不足。查看 Client/Server Span、剩余 Deadline、状态码、连接池等待和下游资源。

不要第一反应只把超时调大，否则会增加在途请求并放大故障。

### 69. 服务发现异常如何处理？

判断注册中心不可用、实例心跳失效、DNS 缓存、客户端 Resolver 未更新还是实例 Ready 状态错误。短期使用最后成功实例快照、快速摘除连接失败节点、停止扩散；恢复后验证实例列表收敛和连接重建。

### 70. 发布后错误率上升如何止血？

立即冻结放量，比较新旧版本的错误码、P99、资源和业务指标；若满足阈值直接回滚或关 Feature Flag。检查 Schema/配置兼容和下游依赖，回滚后继续观察积压与数据副作用，不能只看 Pod 恢复。

### 71. 消息积压导致业务延迟如何处理？

拆分生产速率、消费速率、失败重试和单条耗时；识别毒消息、下游 DB/ES 变慢和分区热点。止血时暂停低优先级生产/重放、隔离毒消息、控制消费者并发保护下游；计算净恢复速率和预计恢复时间。

### 72. 级联故障如何切断？

关闭多层重试，限制入口和并发，熔断慢依赖，启用缓存/静态降级，释放卡住连接和 Goroutine。按最底层依赖到入口逐层恢复，并渐进放量，避免刚恢复就被积压打回故障。

### 73. 微服务稳定性治理清单是什么？

- 每个外部调用有 Deadline、错误分类和幂等策略。
- 每个队列、连接池和并发池有界。
- 每个核心接口有 SLO、降级和容量基线。
- 每次发布可灰度、可回滚、可区分版本指标。
- 每条关键事件链可追踪、可重放、可对账。
- 定期做实例故障、依赖变慢、注册中心异常和备份恢复演练。

### 74. 故障复盘如何避免只归因于个人？

分析直接错误之外的系统条件：为什么评审没发现、测试为何没覆盖、权限为何允许、监控为何晚报、降级为何无效、回滚为何慢。改进要进入自动检查、平台能力、容量模型、演练和责任流程，而不只是“以后注意”。

## 十、业务架构、项目实战与高级 IC 表达

### 75. 广告投放平台如何划分服务？

可按业务能力划分账户/授权、广告资产、投放任务、调度执行、媒体适配、报表等边界。投放任务拥有状态机；媒体适配封装外部平台差异；报表消费事件构建读模型。

同步链路只做校验和受理，耗时发布通过工作流/Kafka 异步执行。服务间不能直接改对方表，状态通过 API 与领域事件协作。

### 76. 第三方媒体发布 RPC 如何设计？

- 请求包含 task_id、platform、account_id、request_id 和业务版本。
- 调用前做账号级限流和并发隔离。
- 使用明确 Deadline、错误分类和退避。
- 超时后用 request_id 查询媒体侧结果，避免重复创建。
- 结果持久化并产生状态事件，永久失败进入人工处理。
- 指标按媒体、账号、错误码统计成功率、P99、限流和重试。

### 77. 如何把一个同步长接口改造成异步任务？

1. API 校验并幂等创建任务，快速返回 task_id 和 accepted 状态。
2. 本地事务写任务与 Outbox，事件进入 Kafka。
3. Worker 领取并按状态机执行，每一步保存进度。
4. 客户端轮询、SSE/WebSocket 或回调获取状态。
5. 超时、重试、取消和人工介入都有明确语义。
6. 监控排队时间、执行时间、成功率和积压恢复时间。

### 78. 微服务拆分项目如何证明不是过度设计？

展示拆分前的具体痛点：发布互相阻塞、故障域过大、某模块容量独特、代码变更冲突或数据职责混乱。再给出拆分后的量化变化：独立发布比例、发布周期、故障影响、实例成本、P99 和跨团队协作次数。

同时坦诚新增的 RPC、最终一致性和运维成本，以及如何用平台和规范控制。

### 79. 如何表达跨团队推动能力但保持业务研发定位？

> 我有一年多带团队和协作经验，所以在做核心业务研发时，会主动推动接口契约、风险清单、容量评审、灰度方案和故障复盘落地。我能理解排期和协作成本，也能和产品、前端、测试及基础设施团队对齐。但我的职业方向仍是高级业务研发/架构职责，希望继续亲自承担核心设计、编码和线上问题，而不是转为纯管理角色。

项目回答要具体说明自己写过什么、做过什么取舍和解决过什么故障，管理经验只作为推动复杂交付的加分项。

### 80. 如何回答“是否所有服务都要上 gRPC、Mesh 和 Kafka”？

不需要。协议和基础设施由需求决定：外部开放 API 可能更适合 HTTP；低频简单调用不必为微小性能收益上 gRPC；团队语言单一且治理简单时 SDK 已足够，不必上 Mesh；必须立即返回且无异步价值的操作不必上 Kafka。

高级判断是以业务收益、故障模型、团队能力和总成本做选择，而不是追求组件齐全。

### 81. 如何用两分钟回答微服务架构题？

> 我不会把微服务理解为把单体拆成很多进程，而是围绕业务能力、数据所有权和独立变化率建立边界。先确认团队规模、发布冲突、容量差异和 SLA，边界不清时优先模块化单体。服务间即时确认走有明确契约的 HTTP/gRPC，非关键传播走事件；限制同步链路深度和 N+1 调用。每个 RPC 都传播 Deadline，重试只用于幂等瞬时错误并受 Retry Budget 控制，同时配置并发上限、熔断、隔离和业务降级。数据由服务自己拥有，单服务内用本地事务，跨服务通过 Outbox、Kafka、幂等消费者、状态机、补偿和对账保证最终一致。发布遵循 Schema 向后兼容、灰度和快速回滚；运行时用 RED、Trace、结构化日志和业务不变量定位问题。最后用独立发布率、P99、可用性、故障域、恢复时间和资源成本证明拆分是否真正带来收益。
