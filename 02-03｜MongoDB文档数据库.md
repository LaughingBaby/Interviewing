# 技术｜MongoDB 文档数据库面试题

> 面向 9 年以上 Golang Web 后端业务研发岗位。重点不是记命令，而是围绕聚合文档建模、索引和查询计划、读写一致性、副本集、分片、容量治理与业务边界形成完整判断。

---

## 一、MongoDB 学习地图与资深回答框架

### 1. MongoDB 的知识图谱主线是什么？

**一句话结论：** 从“为什么选择文档模型”出发，沿着聚合根建模、BSON 存储、索引查询、并发事务、副本集、分片、生产治理和业务实战逐层深入。

```text
业务访问模式与能力边界
  -> Database / Collection / BSON Document
  -> 聚合根、嵌入与引用
  -> WiredTiger、Cache、Checkpoint、Journal
  -> 索引、Query Planner、Aggregation
  -> 原子更新、事务、Read/Write Concern
  -> Replica Set、Oplog、选举和回滚
  -> Sharding、Chunk、Balancer、Shard Key
  -> 容量、治理、排障和项目表达
```

### 2. 资深后端如何回答 MongoDB 问题？

采用“结论 -> 访问模式 -> 原理 -> 方案 -> 风险 -> 落地指标”的结构。

**示例：嵌入还是引用？**

- **结论：** 一起读取、一起更新且规模有界的数据优先嵌入；生命周期独立、关系多对多或会无限增长的数据使用引用。
- **访问模式：** 先说明核心查询和更新路径，而不是先谈范式。
- **原理：** 嵌入可用单文档原子性减少查询，引用可控制文档大小和更新放大。
- **方案：** 主信息嵌入，明细流水独立集合，必要时冗余少量展示字段。
- **风险：** 大数组、热点文档、重复数据同步和跨集合一致性。
- **指标：** 文档大小、P99、扫描文档数、更新冲突、工作集和增长速度。

### 3. MongoDB 场景设计题如何展开？

1. 明确实体边界、读写比例、查询组合、排序、分页和增长模型。
2. 确定聚合根，选择嵌入、引用和必要冗余。
3. 由高频查询反推联合索引和字段顺序。
4. 明确单文档原子性是否足够，事务是否真的必要。
5. 选择读偏好、读关注和写关注，说明一致性与延迟权衡。
6. 评估副本集、分片键、容量、备份、归档和故障恢复。
7. 给出 Go 连接池、超时、幂等、监控、降级与迁移方案。

### 4. MongoDB 高频题如何串联？

| 层次 | 核心问题 | 必须讲出的关键词 |
|---|---|---|
| 定位 | 为什么用 MongoDB | 聚合文档、灵活结构、访问模式 |
| 建模 | 嵌入还是引用 | 一起读写、有界性、原子边界 |
| 存储 | WiredTiger 做什么 | Cache、MVCC、Checkpoint、Journal |
| 索引 | 为什么走/不走索引 | ESR、覆盖、选择性、Explain |
| 一致性 | 如何保证正确 | 单文档原子、事务、Concern、版本 |
| 高可用 | 主节点挂了怎么办 | Oplog、选举、回滚、Majority |
| 扩展 | Shard Key 怎么选 | 基数、均匀、定向、单调热点 |
| 治理 | 慢和抖动怎么查 | 扫描比、Cache、Ticket、磁盘、复制延迟 |
| 实战 | 项目如何表达 | 边界、权衡、指标、迁移和复盘 |

## 二、使用场景、文档建模与能力边界

### 5. MongoDB 适合哪些业务场景？

- 字段变化较快、不同类型记录结构存在差异。
- 数据天然以聚合文档整体读写，如内容、素材、配置、用户画像。
- 需要嵌套对象、数组和丰富的单文档更新表达式。
- 查询主要围绕聚合根，跨实体 Join 和强约束较少。
- 希望通过副本集和分片水平扩展读写与容量。

适合不等于“没有 Schema”；生产系统仍需应用层校验、Schema Validation、索引和演进策略。

### 6. 什么场景不适合 MongoDB？

- 大量跨实体事务、外键约束和复杂关系查询。
- 财务账务等要求严格关系模型和审计约束的核心事实。
- 数据访问高度随机，无法形成稳定聚合边界或索引组合。
- 数组或嵌套对象会无限增长，导致巨型热点文档。
- 团队只是为了“字段随便加”而引入，缺少容量和治理能力。

### 7. MongoDB、MySQL、ES、Redis 和 Kafka 如何分工？

| 组件 | 最擅长 | 典型边界 |
|---|---|---|
| MySQL | 事务、关系、约束、权威结构化数据 | 灵活嵌套和全文检索成本较高 |
| MongoDB | 聚合文档、灵活模型、单文档原子更新 | 复杂 Join 与全局强约束较弱 |
| Elasticsearch | 全文召回、相关性、组合过滤和聚合 | 不作为交易事实源 |
| Redis | 热点缓存、计数、限流和短期状态 | 不保存大规模主数据 |
| Kafka | 变更事件、异步解耦、削峰和重放 | 不提供随机业务查询 |

选择 MongoDB 应回答“文档聚合带来了什么业务收益”，而不是只说 JSON 写起来方便。

### 8. MongoDB 与 MySQL 建模思想有什么不同？

- MySQL 通常从实体关系、范式、约束和事务边界出发。
- MongoDB 通常从访问模式和聚合根出发，允许为读性能做反范式冗余。
- MySQL 通过 Join 在查询时组合；MongoDB 倾向在写入时把一起读取的数据组织到同一文档。
- MongoDB 仍可引用和使用 `$lookup`，但不应把关系模型原样搬过去。

### 9. 什么是聚合根，为什么它是 MongoDB 建模核心？

聚合根是业务上具有一致生命周期和事务边界的一组数据的入口。例如广告任务包含基本配置、定向摘要和当前状态，可作为一个聚合；执行流水则增长无界，应独立集合。

好的聚合根同时满足：高频一起读取、更新边界清晰、大小可控、热点可接受、生命周期相近。

### 10. 嵌入文档和引用如何选择？

**优先嵌入：** 一对一或有界一对少、一起查询、一起更新、需要单文档原子性。

**优先引用：** 一对多无界、多对多、子实体独立访问/更新、共享实体、文档可能接近大小上限。

折中方式是引用主实体，同时冗余名称、状态等展示快照；用事件同步冗余字段，并接受短暂最终一致。

### 11. 如何处理不断增长的数组？

不要将日志、评论、执行记录、成员历史无限 push 到一个文档。可采用：

- 子项独立集合，使用 parent_id + 时间联合索引。
- Bucket Pattern 按数量或时间分桶。
- 主文档只保留最新 N 条摘要和计数。
- 冷数据归档到历史集合或对象存储。

这同时避免文档搬迁、更新冲突、单文档大小限制和热点写。

### 12. Schema 灵活是否意味着不需要版本管理？

恰恰相反。需要：

- 用 `schema_version` 标记文档版本。
- 新代码同时兼容旧字段一段时间。
- 写新读旧、后台迁移、校验、灰度和回滚。
- 对核心字段启用 Schema Validation。
- 禁止相同字段在不同文档中出现混乱类型。

## 三、BSON、WiredTiger、存储与请求执行链路

### 13. BSON 与 JSON 有什么区别？

BSON 是二进制文档格式，除字符串、数字、布尔和数组外，还支持 ObjectId、Date、Decimal128、Binary 等类型，便于数据库编码、遍历和驱动映射。

代价是字段名会随文档重复存储，某些类型比紧凑二进制格式更占空间。Golang 中要明确 `int32/int64`、时间、Decimal 和空值语义，避免跨语言类型漂移。

### 14. ObjectId 的组成和使用边界是什么？

ObjectId 具备全局低冲突生成和大致时间有序特性，适合默认 `_id`。但它不等于严格连续 ID，也不应依赖其时间部分代替业务时间。

若业务已有稳定唯一 ID，可直接用作 `_id`，减少一次唯一索引并方便幂等；分片场景仍要评估 `_id` 与 Shard Key 的关系。

### 15. WiredTiger 的核心职责是什么？

- 管理 B-Tree、页面、压缩和磁盘文件。
- 使用 Cache 保存内部页、数据页和脏页。
- 通过 MVCC 提供快照读和并发控制。
- 使用 Journal 与 Checkpoint 实现持久化和崩溃恢复。
- 写操作主要采用文档级并发控制和乐观冲突检测。

数据库慢不只看 CPU，还要看工作集是否装入 Cache、脏页回写、Checkpoint、磁盘延迟和读写排队。

### 16. WiredTiger Cache 与操作系统页缓存是什么关系？

WiredTiger Cache 保存解压后的页面和脏数据，操作系统页缓存还可能缓存压缩后的数据文件。把机器内存全部分给 WiredTiger 会挤压 OS 和其他进程。

应观察 Cache used、dirty bytes、eviction、pages read into cache、pages written、应用延迟和磁盘，而不是只盯一个使用率。

### 17. Checkpoint 和 Journal 分别保证什么？

- Journal 记录尚未进入稳定 Checkpoint 的写入意图，用于崩溃恢复。
- Checkpoint 将某个一致时间点的脏页状态持久化，并推进可恢复基线。

写关注决定客户端等待哪些确认；Journal/Checkpoint 解决节点内部持久化，副本集 Majority 解决跨节点确认，它们不是同一个层次。

### 18. 一条查询请求如何执行？

1. Driver 依据拓扑、Read Preference 和 Server Selection 选择节点。
2. `mongod` 解析过滤、排序、投影和会话语义。
3. Query Planner 生成候选计划，选择索引扫描或集合扫描。
4. 执行器从索引定位 RecordId，必要时 FETCH 文档并过滤。
5. WiredTiger 从 Cache 或磁盘读取页面。
6. 结果按 Batch 返回，Driver Cursor 继续 `getMore`。

慢查询可能出在选节点、排队、计划、扫描、FETCH、排序、磁盘或客户端消费任一阶段。

### 19. 一条写请求如何执行？

1. Driver 选择 Primary 并携带 Write Concern、Session 和超时。
2. Primary 校验文档、索引和唯一约束，执行更新并处理写冲突。
3. 变更写入本地存储和 Oplog。
4. Secondary 拉取 Oplog 并按顺序应用。
5. 达到 Write Concern 后向客户端确认。

网络超时时结果可能未知，应用必须依赖稳定业务 ID、条件更新或事务编号保证重试幂等。

### 20. MongoDB 为什么也会有锁和写冲突？

WiredTiger 并非无锁。MongoDB 在全局、数据库、集合层使用意向锁，在文档层做并发控制；两个事务修改同一文档可能发生 write conflict 并重试。

长事务、大批更新、DDL、热点单文档和索引构建仍会影响并发。业务设计不能因为“文档级锁”就忽略热点。

## 四、索引、Query Planner、Explain 与聚合查询

### 21. MongoDB 有哪些常见索引？

- 单字段和联合索引。
- 唯一索引、稀疏索引和 Partial Index。
- Multikey 数组索引。
- TTL 索引。
- Text、Geospatial、Hashed、Wildcard 索引。
- Hidden Index，用于评估隐藏后的影响。

索引不是越多越好：每个索引都消耗磁盘、Cache 和写入维护成本。

### 22. 联合索引如何按照 ESR 原则设计？

ESR 是常用启发式：Equality、Sort、Range。

```text
{ tenant_id: 1, status: 1, created_at: -1 }
```

若查询固定等值过滤租户和状态，再按时间倒序，此顺序通常合理。但 ESR 不是定律：字段选择性、查询比例、范围大小、排序是否可避免内存排序都要用 Explain 和压测验证。

### 23. 什么是最左前缀和索引边界？

联合索引按字段顺序组织。缺少前导字段，后续字段通常无法形成高效连续边界；前导范围条件之后的字段也可能无法继续缩小扫描范围。

面试中应结合具体查询说明 `indexBounds`、keysExamined 和 docsExamined，而不是只背“最左匹配”。

### 24. Multikey 索引有什么特殊限制和风险？

数组字段会为多个元素生成索引键，数组越大写放大越明显。联合索引中多个数组字段会受到限制，数组元素组合查询还需 `$elemMatch` 保证条件落在同一元素。

大数组不应靠加索引硬撑，应重新评估文档边界和子集合建模。

### 25. Partial、Sparse 和 TTL 索引如何选择？

- Partial Index 只索引满足表达式的文档，控制更精确，适合只查询有效状态的数据。
- Sparse Index 跳过缺失字段，语义较粗，唯一约束要特别小心。
- TTL Index 后台异步删除过期文档，删除不是到点即时发生，不适合精确业务定时器。

### 26. 覆盖索引为什么快？

当过滤、排序和返回字段都可从索引获得时，可以避免回表 FETCH 完整 BSON 文档，减少 Cache 和磁盘访问。

代价是索引更宽、写入更慢、内存占用更大。仅对高频且字段稳定的查询设计覆盖索引。

### 27. Explain 必须看哪些指标？

- `winningPlan`：COLLSCAN、IXSCAN、FETCH、SORT 等阶段。
- `totalKeysExamined` 与 `totalDocsExamined`。
- `nReturned`，用于计算扫描/返回比。
- 是否出现内存排序、重复键消除和大量 FETCH 后过滤。
- 执行时间、索引边界和各 Shard 表现。

一个查询“用了索引”不等于高效；若扫百万 key 只返回十条，仍需优化。

### 28. Query Planner 为什么可能选错索引？

数据分布变化、参数差异、相似索引过多、统计信息和计划缓存都可能导致某个计划在部分条件下退化。

应先保存 Explain 和真实参数，比较候选索引与扫描量；必要时调整索引、查询形态或清理/分析计划缓存。`hint` 适合作为受控手段，不应成为长期掩盖建模问题的默认方案。

### 29. 深分页为什么慢，如何优化？

`skip N` 仍需扫描并丢弃前 N 条，页数越深成本越高。推荐基于稳定排序键做 Seek Pagination：

```javascript
db.tasks.find({
  tenant_id: "t1",
  created_at: { $lt: lastCreatedAt }
}).sort({ created_at: -1, _id: -1 }).limit(50)
```

游标应同时携带 `created_at` 和 `_id`，解决相同时间的稳定翻页。随机跳页需产品或预计算方案配合。

### 30. Aggregation Pipeline 如何优化？

- 尽早 `$match` 和 `$project`，缩小文档和字段。
- 让过滤与排序命中联合索引。
- 谨慎使用 `$lookup`、`$unwind`、`$group` 和大范围 `$sort`。
- 限制结果集和内存，观察是否落盘。
- 复杂报表可预聚合、异步计算或交给分析系统。
- 在分片集群中关注各 Shard 执行和 mongos 归并成本。

## 五、原子更新、事务、并发控制与一致性语义

### 31. MongoDB 的原子性边界是什么？

单文档写入天然原子，包括更新文档内多个字段和有界数组。合理聚合建模可以把很多业务一致性收敛到单文档，无需事务。

跨文档、跨集合或跨 Shard 操作才考虑多文档事务；事务不是弥补错误文档边界的默认工具。

### 32. 如何实现乐观并发控制？

使用版本号或当前状态进入更新条件：

```javascript
db.tasks.updateOne(
  { _id: taskId, version: expectedVersion, status: "pending" },
  { $set: { status: "running" }, $inc: { version: 1 } }
)
```

检查 `matchedCount/modifiedCount` 判断是否竞争失败。它适合状态机、库存配额、任务领取等场景，并天然支持幂等重试。

### 33. 多文档事务如何工作，代价是什么？

事务基于 Session、快照视图、WiredTiger MVCC 和副本集/分片协调提交。它能提供 ACID，但会延长快照和锁生命周期、增加缓存压力、冲突、网络往返和分片协调成本。

原则：事务要短、小、有界；禁止在事务中调用慢外部服务；为 UnknownTransactionCommitResult 和 TransientTransactionError 设计正确重试。

### 34. Read Concern、Write Concern 和 Read Preference 分别控制什么？

| 机制 | 控制的问题 |
|---|---|
| Read Concern | 能读到什么一致性/持久性级别的数据 |
| Write Concern | 写入需等待多少副本及持久化确认 |
| Read Preference | 请求发往 Primary 还是 Secondary |

三者必须组合讨论。读 Secondary 并不自动提高一致性；写 Majority 后立即读延迟副本，仍可能读不到最新数据。

### 35. `w: majority` 保证了什么，没保证什么？

它要求写入达到多数确认边界，提高节点故障后保留写入的概率。它不保证业务跨系统事务、不消除网络超时的结果未知，也不意味着任意 Secondary 立即可见。

应用仍需幂等键、合理超时、读关注和补偿机制。

### 36. 如何实现“写后读到自己刚写的数据”？

可选方案：

- 写后读 Primary。
- 使用因果一致 Session，结合合适的 Read/Write Concern。
- 将写入结果直接返回并在短时间内由调用方使用。
- 对允许最终一致的页面展示“处理中”状态。

不要为了所有查询都强制最强语义；按订单确认、后台列表、统计报表分别定义一致性。

### 37. MongoDB 事务与 MySQL 事务如何比较？

两者都可提供 ACID，但 MongoDB 的优势仍是聚合文档和单文档原子性，MySQL 的优势是成熟关系约束、Join 和事务建模。若业务长期依赖大量跨文档事务和复杂关联，通常说明 MySQL 更贴合主模型，而不是证明 MongoDB 也能勉强实现。

### 38. 幂等写如何设计？

- 用业务唯一键或事件 ID 建唯一索引。
- Upsert 时把唯一条件放入过滤器。
- 状态迁移使用 `current_status + version` 条件更新。
- 消费事件记录 `last_event_version`，拒绝旧版本。
- 网络超时后先按业务键查询结果，再决定是否重试。

## 六、副本集、Oplog、选举与 Change Streams

### 39. 副本集的核心对象和链路是什么？

副本集通常包含一个 Primary 和多个 Secondary。Primary 接收写入并追加 Oplog，Secondary 持续拉取和应用 Oplog，成员通过心跳感知状态并在 Primary 故障时选举。

它解决高可用和数据冗余，但应用仍需处理短暂不可写、连接切换、重试和读延迟。

### 40. Oplog 是什么，为什么容量重要？

Oplog 是有界的复制操作日志，Secondary 用它追赶 Primary，Change Streams 也依赖可用的变更历史。

若 Secondary 离线时间超过 Oplog 覆盖窗口，就无法增量追赶，需要 Initial Sync。容量应按峰值写入、停机维护窗口和异常恢复时间估算，并监控时间窗口而非只看字节。

### 41. Primary 故障后会发生什么？

1. 成员通过心跳发现 Primary 不可达。
2. 满足条件的 Secondary 发起选举，获得多数票后成为新 Primary。
3. Driver 发现拓扑变化并选择新 Primary。
4. 客户端在选举窗口内可能收到不可写或超时错误。
5. 旧 Primary 恢复后回到 Secondary，未被多数确认的分叉写可能回滚。

业务要设置合理 Server Selection Timeout，并仅对幂等操作进行有界重试。

### 42. 什么是 Rollback，如何降低风险？

旧 Primary 上尚未进入新多数派历史的写入，在重新加入时可能被回滚。降低风险的方法包括使用 Majority Write Concern、合理副本拓扑、稳定网络、监控复制延迟和避免在失去多数时强行重配。

关键业务还需通过对账和业务幂等处理异常，不能只依赖数据库副本。

### 43. 为什么通常推荐奇数个投票成员？

多数派需要超过半数投票。三个投票成员可容忍一个故障，五个可容忍两个；增加偶数投票成员通常不增加同等故障容忍，却增加部署和协调成本。

投票成员、数据副本和只读扩展不是同一概念，拓扑需结合故障域和成本设计。

### 44. Secondary 读有什么收益和风险？

收益是分担可容忍陈旧的读取、就近读取和报表负载。风险是复制延迟、读到可回滚数据、读写不一致和 Secondary 被重查询拖慢复制。

订单确认等强写后读走 Primary；报表和推荐可按最大陈旧度使用 Secondary，并隔离分析节点。

### 45. Change Streams 如何工作，适合什么场景？

Change Streams 在复制日志之上提供集合、数据库或集群级变更订阅，事件包含 resume token，可用于缓存失效、搜索索引同步和审计通知。

它不是业务消息队列的完全替代：需要处理重复、断线恢复、Oplog 窗口、权限、事件顺序边界和下游背压。重要业务通常再进入 Kafka，获得更长保留、消费组和重放能力。

### 46. 副本集如何备份和恢复？

- 小规模可用逻辑备份，但恢复时间随数据量增长。
- 大规模通常使用一致性快照、托管备份或存储快照。
- 需要支持时间点恢复时保留 Oplog/增量日志。
- 备份必须异地、加密、校验并定期恢复演练。
- 明确 RPO、RTO、恢复顺序、索引重建时间和应用回切步骤。

## 七、Sharding、Shard Key、Chunk 与水平扩展

### 47. MongoDB 分片集群由哪些组件组成？

- `mongos`：无状态查询路由与结果归并入口。
- Config Server Replica Set：保存集群元数据和 Chunk 分布。
- Shard：真正保存数据，每个 Shard 通常是副本集。
- Balancer：迁移 Chunk 以改善分布。

分片解决容量和吞吐扩展，但引入路由、迁移、跨 Shard 查询和事务复杂度。

### 48. Shard Key 为什么是最重要的设计？

Shard Key 决定文档分布、查询能否定向、写入是否热点、Chunk 能否均衡以及未来扩容上限。一个差的 Shard Key 很难低成本修改。

评估四维：

1. **基数：** 能否形成足够多分区。
2. **频率：** 值分布是否均匀，是否有超级大租户。
3. **单调性：** 是否把新增写集中到尾部 Chunk。
4. **查询相关性：** 高频查询是否携带 Shard Key，从而定向路由。

### 49. Range Sharding 和 Hashed Sharding 如何取舍？

- Range 保留相邻值局部性，适合范围查询，但单调递增键易产生写热点。
- Hashed 分布更均匀，适合点查和写扩散，但范围查询会广播。
- 复合 Shard Key 可用租户、哈希或时间组合平衡路由与写入。

选择必须基于真实查询矩阵和最大租户模型，不能只说哈希更均匀。

### 50. 什么是 Scatter-Gather 查询？

查询不包含可定向的 Shard Key 条件时，mongos 将请求发往多个甚至全部 Shard，再归并结果。它会随 Shard 数放大 CPU、网络、尾延迟和排序聚合成本。

治理方法是调整 API 约束、索引和 Shard Key，使核心查询可定向；无法定向的运营报表应隔离或预计算。

### 51. Chunk、Balancer 和迁移如何工作？

数据按 Shard Key 范围组织为 Chunk。Balancer 根据分布和 Zone 约束触发迁移，将数据从源 Shard 复制到目标 Shard，提交元数据后清理旧数据。

迁移消耗网络、磁盘和 Cache，业务高峰期要限速或设置窗口。监控不能只看 Chunk 数，还要看数据大小、迁移队列、热点和查询分布。

### 52. Jumbo Chunk 是什么，如何治理？

当 Chunk 因低基数、不可分割键分布或超大相同键值而无法正常拆分迁移时，可能形成 Jumbo。根因往往是 Shard Key 数据分布不合理。

短期可评估手动拆分、迁移或归档；长期需重新设计 Shard Key、Resharding 或重建集合。操作前必须验证版本能力和回滚路径。

### 53. 什么时候才应该分片？

- 单副本集存储、IO、CPU 或写吞吐经过优化后仍接近上限。
- 工作集远超单机内存且业务无法归档拆分。
- 需要跨机器扩展容量，并有稳定 Shard Key。

不要把分片作为慢查询的第一解决方案。无索引查询分片后只会变成并行扫描更多机器。

## 八、性能优化、容量规划与 Golang 工程实践

### 54. MongoDB 容量如何估算？

```text
文档平均 BSON 大小 x 文档数量
  + 全部索引大小
  + Oplog
  + Journal/临时空间
  + 副本倍数
  + 增长与安全余量
```

同时估算工作集：热点文档页、热点索引页和写入脏页能否进入 Cache。容量方案必须在单节点故障、Initial Sync、备份和索引构建时仍有余量。

### 55. 如何系统优化慢查询？

1. 保存真实过滤、排序、投影、耗时和业务上下文。
2. Explain 检查 winningPlan、扫描/返回比和排序。
3. 调整查询和联合索引，减少 FETCH 与返回字段。
4. 治理深分页、无界数组、`$lookup` 和大范围聚合。
5. 检查 Cache、eviction、读写排队、磁盘、CPU 和复制延迟。
6. 最后再考虑读扩展、分片或数据归档。

### 56. 如何处理热点文档和热点租户？

- 把高频计数拆成分桶计数或 Redis 聚合后异步落库。
- 将无界子项拆成独立集合。
- 对超大租户使用更细分区键或独立集合/集群。
- 条件更新失败采用有界退避，避免并发自旋。
- 通过 tenant_id、document_id、operation 指标识别倾斜。

### 57. 大批量写入如何设计？

- 使用 `bulkWrite`，根据业务选择 ordered/unordered。
- 控制批次字节数、并发和连接池，建立背压。
- 预先创建正确索引，避免导入中途反复建索引。
- 逐项解析错误，区分重复键、校验失败和临时错误。
- 大迁移要限速，监控复制延迟、Oplog 窗口、Cache 和磁盘。

### 58. Golang MongoDB Driver 如何治理？

- 复用全局 `mongo.Client`，不要每请求 Connect/Disconnect。
- 所有操作使用 `context` deadline，并区分服务选择、连接、Socket 和业务超时。
- 按压测配置 `maxPoolSize/minPoolSize/maxConnecting` 等参数，避免连接风暴。
- Cursor 必须关闭并检查 `cursor.Err()`，限制 Batch 和返回字段。
- 事务回调只包含数据库操作，正确重试临时事务错误和提交结果未知。
- 日志记录 collection、operation、duration、matched/modified、error labels 和 trace，脱敏查询参数。

### 59. MongoDB 压测要覆盖什么？

必须使用真实文档大小、数组分布、索引和读写混合，分别测热点与均匀流量、Primary 切换、Secondary 延迟、Checkpoint、索引构建和迁移。

输出 QPS、P50/P95/P99、扫描比、Cache/eviction、读写排队、磁盘延迟、复制延迟、Oplog 窗口、连接池等待和故障恢复时间。

## 九、生产治理、安全与故障排查

### 60. MongoDB 监控体系包括什么？

| 层次 | 核心指标 |
|---|---|
| 业务 | 成功率、P99、数据新鲜度、状态冲突、积压 |
| 查询 | 慢查询、扫描/返回比、COLLSCAN、内存排序 |
| 存储 | Cache、dirty、eviction、Checkpoint、磁盘延迟 |
| 并发 | 连接、活跃操作、读写排队、锁和 write conflict |
| 复制 | 成员状态、复制延迟、Oplog 窗口、选举和 rollback |
| 分片 | Shard 分布、Chunk、迁移、热点、Scatter-Gather |
| 容量 | 数据/索引增长、磁盘水位、备份和恢复耗时 |

### 61. MongoDB 线上变慢如何排查？

1. 确认时间线、影响租户、接口、集合、读写类型和 P99。
2. 查 profiler/慢日志和 currentOp，保存真实查询与计划。
3. 判断是扫描、排序、锁/冲突、连接池等待还是磁盘 IO。
4. 检查 Cache eviction、dirty、Checkpoint、Ticket/排队和 CPU。
5. 检查复制延迟、选举、Balancer、索引构建和批量任务。
6. 先限流重查询、暂停迁移/批任务或切换降级，再做索引和模型根治。

### 62. 连接数打满或请求排队如何处理？

先区分应用连接池等待、服务端连接过多和存储引擎读写排队。排查发布是否创建了多 Client、请求是否未关闭 Cursor、慢查询是否长期占连接，以及实例扩容是否同时造成连接风暴。

止血是限制入口和批任务、回收异常实例、降低每实例连接上限；根治是单例 Client、合理池大小、deadline、慢查询治理和连接指标。

### 63. 复制延迟如何排查？

- Primary 峰值写入和大事务是否超过 Secondary 应用速度。
- Secondary 磁盘、CPU、Cache 是否不足。
- Secondary 是否承载重查询、备份或索引构建。
- 网络延迟和丢包是否异常。
- Oplog 窗口是否足以覆盖延迟。

延迟期间避免让强一致读继续走 Secondary；优先限制重任务，保护 Oplog 窗口，必要时扩容或重新同步落后节点。

### 64. 磁盘水位高和数据增长失控如何处理？

按数据、索引、Oplog、日志和临时文件拆分占用。短期停止非核心写入和大索引构建，清理可确认的过期数据并扩磁盘；长期通过 TTL、归档、索引精简、压缩评估、分片和容量告警治理。

禁止在未确认备份和业务范围时直接删除集合或文件。

### 65. MongoDB 安全如何治理？

- 开启认证、TLS 和最小权限，区分应用、运维、备份和审计账号。
- 数据库不得直接暴露公网，限制网络来源和管理端口。
- 凭证进入密钥系统并定期轮换。
- 敏感字段按合规要求加密、脱敏或使用字段级保护能力。
- 审计 DDL、用户权限、批量导出、删除和恢复操作。
- 应用层强制租户过滤，防止越权查询。

### 66. MongoDB 故障复盘如何组织？

给出“现象 -> 直接原因 -> 触发条件 -> 放大因素 -> 止血 -> 恢复 -> 防复发”。例如慢查询只是直接原因，缺少查询预算、索引评审、压测覆盖和扫描比告警才是治理缺口。

复盘必须落到负责人和可验证动作：索引变更、容量阈值、故障演练、备份恢复、连接池基线和发布检查。

## 十、业务架构与项目实战

### 67. 广告任务或素材文档如何建模？

**主文档：** 任务 ID、租户、项目、状态、当前配置、素材摘要、创建与更新时间；这些字段经常一起读取且更新边界清晰。

**独立集合：** 执行流水、审核历史、投放结果和大规模明细，它们增长无界并需要独立分页。

**索引：** 由后台列表查询反推，例如 `{tenant_id, status, updated_at}`；名称全文搜索复杂时同步到 ES。

**一致性：** 状态迁移使用状态 + version 条件更新，事件通过 Kafka 幂等传播。

### 68. 如何设计任务状态流转？

```text
pending -> running -> succeeded
                  -> failed
pending/running -> cancelled（按业务规则）
```

更新条件必须包含当前状态和版本，检查 matchedCount；同时写入独立状态流水，事件带 task_id、version 和 event_id。重复消费同一版本直接返回成功，旧版本不允许覆盖新状态。

### 69. MongoDB 到 ES 的同步链路如何设计？

- MongoDB 是完整业务文档，ES 是搜索读模型。
- 变更通过 Outbox/Change Stream 进入 Kafka。
- 消费者按 ID 回查最新文档，构建精简 ES Document。
- 使用业务版本防止乱序，Bulk item 失败分类重试。
- 对账任务校验数量、版本、状态和删除标记。
- Mapping 变化时新建索引、全量导入、追增量、Alias 灰度切换。

### 70. 用户画像为什么可能适合 MongoDB？

画像字段按用户类型和业务阶段变化，标签、偏好和设备摘要天然形成单用户聚合文档。读取常以 user_id 为入口，适合点查和部分字段更新。

但事件明细不应无限嵌入；高频实时计数可先在 Redis/流处理聚合，离线分析进入数仓，MongoDB 只保存服务在线查询的画像快照。

### 71. 如何进行 MongoDB 灰度迁移？

1. 定义旧库与 MongoDB 的字段映射和一致性口径。
2. 全量迁移使用稳定游标、限速和断点续传。
3. 增量使用事件或 CDC，写入带版本并幂等。
4. 双读影子对比关键字段和耗时，允许明确的延迟窗口。
5. 按租户/流量切读，再切写；保留回滚链路。
6. 完成后对账、观察、归档旧数据，最后下线双写。

### 72. 如何用两分钟回答 MongoDB 架构题？

> 我先把 MongoDB 定位为围绕聚合根的业务文档存储，而不是无 Schema 的万能库。建模从访问模式出发：一起读写且规模有界的数据嵌入，生命周期独立或增长无界的数据引用并独立分页；高频查询按照过滤、排序和范围设计联合索引，用 Explain 验证扫描/返回比。正确性优先利用单文档原子更新和版本条件，跨文档事务保持短小；再按业务选择 Read Concern、Write Concern 和 Read Preference。高可用由副本集、Oplog、选举和备份承担，容量超过单副本集后才依据基数、均匀性、单调性和查询路由选择 Shard Key。线上重点监控 P99、慢查询、Cache、磁盘、连接、复制延迟、Oplog 窗口和分片热点。需要全文检索时同步到 ES，需要事件解耦时进入 Kafka，热点状态交给 Redis，每个组件只承担自己最擅长的职责。
