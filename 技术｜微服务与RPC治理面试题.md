# 技术｜微服务与RPC治理面试题

> 面向 9 年+ Golang Web 后端业务岗。
> 覆盖 HTTP/gRPC、服务治理、网关、限流熔断、灰度、可观测性和稳定性。

## 一、微服务总框架

### 1. 微服务解决什么问题，又带来什么问题？

**结论先行：**
微服务解决复杂系统的团队协作、独立部署和局部扩展问题，但会带来分布式调用、数据一致性、可观测性和治理复杂度。

**分层展开：**

**解决的问题**
- 服务按业务域拆分。
- 独立发布和扩容。
- 技术边界清晰。
- 故障隔离更容易。

**带来的问题**
- RPC 延迟和失败。
- 分布式事务复杂。
- 链路追踪和排障困难。
- 版本兼容和灰度治理。

**适用边界**
- 团队和业务复杂到需要拆分时再拆。
- 小团队小系统不应过度微服务化。

**面试追问：微服务拆分原则是什么？**
围绕业务能力和数据所有权拆分，高内聚、低耦合，避免按技术层横切拆分。

## 二、HTTP、gRPC 与接口设计

### 2. HTTP 和 gRPC 怎么选？

**结论先行：**
对外开放和通用 Web API 优先 HTTP/REST；内部高性能强契约服务调用优先 gRPC。

**分层展开：**

**HTTP/REST**
- 通用性好。
- 易调试。
- 生态成熟。
- 适合开放 API 和前后端交互。

**gRPC**
- 基于 HTTP/2。
- Protobuf 强契约。
- 性能好。
- 支持流式调用。
- 适合内部微服务。

**工程取舍**
- gRPC 对浏览器和网关支持要额外处理。
- REST 更灵活但契约约束弱。
- 核心是接口稳定性和治理能力。

**面试追问：gRPC 一定比 HTTP 快吗？**
通常内部调用效率更高，但性能还取决于 payload、连接复用、序列化、网络、服务实现和压测场景。

### 3. 接口如何设计才能长期演进？

**结论先行：**
接口设计要做到语义清晰、向后兼容、错误码稳定、幂等可控、版本可演进。

**分层展开：**

**请求设计**
- 字段命名稳定。
- 必填和可选明确。
- 避免暴露内部模型。

**响应设计**
- 明确错误码。
- 支持 trace id。
- 分页和排序规则清晰。

**兼容策略**
- 新增字段优先。
- 不随意删除或改变字段语义。
- 重大变化通过版本号或新接口。

**面试追问：为什么不能直接复用数据库 model 做 API？**
数据库模型是内部存储结构，API 是外部契约。直接暴露会导致内部结构难以演进，也可能泄露敏感字段。

### 4. 超时应该怎么设置？

**结论先行：**
超时要分层设置，整体请求 deadline 要向下游传递，子调用超时要小于上游剩余时间。

**分层展开：**

**设置原则**
- 客户端、网关、服务端、DB/RPC 都要有超时。
- 下游超时不能大于上游超时。
- 根据 P99 和业务 SLA 设置。

**工程实践**
- Go 中用 context 传递 deadline。
- 连接超时、读写超时分开。
- 超时后及时释放资源。

**风险**
- 超时太短误杀正常请求。
- 超时太长导致资源堆积。
- 多层重试叠加导致雪崩。

**面试追问：超时后服务端一定停止处理了吗？**
不一定。客户端超时只是放弃等待，服务端要主动监听 context 或连接关闭信号才能停止。

### 5. 重试怎么设计？

**结论先行：**
重试只适合可重试错误和幂等操作，要限制次数、设置退避、避免多层重试和重试风暴。

**分层展开：**

**适合重试**
- 网络抖动。
- 短暂超时。
- 限流后的短退避。
- 幂等查询或幂等写。

**不适合重试**
- 参数错误。
- 业务规则失败。
- 非幂等扣款、下单。
- 下游已经明确处理中的请求。

**治理**
- 指数退避。
- jitter 随机抖动。
- 最大次数。
- 重试预算。
- 熔断配合。

**面试追问：为什么重试会放大故障？**
下游变慢时，上游重试会让请求量成倍增加，进一步压垮下游和线程池，形成雪崩。

## 三、服务治理

### 6. 限流、熔断、降级分别是什么？

**结论先行：**
限流控制入口流量，熔断阻断异常依赖，降级牺牲非核心能力保护主链路。

**分层展开：**

**限流**
- 防止请求超过系统承载。
- 常用令牌桶、漏桶、滑动窗口。
- 可按用户、接口、服务、租户维度。

**熔断**
- 下游错误率或超时过高时快速失败。
- 半开探测恢复。
- 防止故障扩散。

**降级**
- 返回默认值、缓存旧值、关闭非核心功能。
- 保核心链路可用。

**面试追问：限流返回什么更合适？**
对用户请求返回明确的稍后重试或排队状态；对内部调用返回可识别错误码，便于上游降级。

### 7. 服务注册发现原理是什么？

**结论先行：**
服务注册发现通过注册中心维护服务实例列表，客户端或网关根据列表进行负载均衡和健康检查。

**分层展开：**

**注册**
- 服务启动时注册实例信息。
- 定期心跳续约。
- 下线时注销。

**发现**
- 客户端拉取或订阅实例列表。
- 本地缓存服务地址。
- 根据负载均衡策略选择实例。

**健康检查**
- 心跳超时剔除。
- 主动探测。
- 优雅下线避免流量打到退出实例。

**面试追问：注册中心挂了服务还能调用吗？**
如果客户端有本地缓存，短时间可以继续调用已有实例，但新实例发现、故障剔除和配置变更会受影响。

### 8. 负载均衡有哪些策略？

**结论先行：**
常见负载均衡策略有轮询、随机、加权、最少连接、一致性哈希，选择取决于实例能力、请求特性和状态依赖。

**分层展开：**

**轮询/随机**
- 简单通用。
- 适合实例能力接近的无状态服务。

**加权**
- 实例配置不同或灰度流量不同。
- 可逐步放量。

**最少连接**
- 适合请求耗时差异较大的场景。

**一致性哈希**
- 适合有本地缓存或状态亲和的场景。
- 节点变化时迁移较少。

**面试追问：为什么微服务尽量无状态？**
无状态服务更容易扩容、故障迁移和负载均衡；状态放到 DB、Redis、对象存储等专门组件中治理。

### 9. 灰度发布怎么设计？

**结论先行：**
灰度发布要做到小流量、可观测、可回滚、按维度放量，避免一次性全量发布。

**分层展开：**

**灰度维度**
- 用户 ID。
- 地区。
- 租户。
- 设备版本。
- 请求 Header。

**发布流程**
- 部署新版本。
- 小流量验证。
- 观察核心指标。
- 逐步放量。
- 异常自动或手动回滚。

**监控指标**
- 错误率。
- P99。
- 业务转化。
- 下游依赖耗时。
- 日志异常。

**面试追问：数据库变更如何配合灰度？**
遵循向前兼容：先加字段和兼容代码，再灰度新逻辑，确认后清理旧字段或旧逻辑。

## 四、可观测性与排障

### 10. 日志、指标、Trace 分别解决什么问题？

**结论先行：**
日志回答发生了什么，指标回答系统是否异常，Trace 回答请求经过哪里和慢在哪里。

**分层展开：**

**日志**
- 记录关键事件和错误上下文。
- 适合排查单个请求问题。

**指标**
- QPS、延迟、错误率、资源使用。
- 适合告警和趋势分析。

**Trace**
- 展示跨服务调用链。
- 定位慢调用和错误传播。

**工程建议**
- 三者都要带 trace id。
- 日志避免敏感信息。
- 指标要控制基数。

**面试追问：高基数指标有什么问题？**
高基数会让监控系统存储和查询压力暴涨，例如把 user_id 作为 label 通常是错误做法。

### 11. 微服务接口 P99 变高怎么排查？

**结论先行：**
P99 变高要沿调用链从入口、服务内部、下游依赖、资源瓶颈和最近变更逐层定位。

**分层展开：**

**入口层**
- QPS 是否上涨。
- 是否有异常用户或流量来源。
- 网关是否限流或排队。

**服务层**
- CPU、内存、GC。
- goroutine、线程池、连接池。
- 锁竞争和慢逻辑。

**依赖层**
- DB 慢 SQL。
- Redis 慢查询。
- MQ 积压。
- RPC 下游超时。

**变更层**
- 最近发布。
- 配置调整。
- 数据量变化。

**面试追问：平均耗时正常但 P99 高说明什么？**
说明长尾请求存在，可能是锁竞争、连接池排队、GC、下游抖动、大 key 或部分热点流量导致。

### 12. 如何做服务降级预案？

**结论先行：**
降级预案要提前定义核心链路、非核心功能、降级开关、兜底数据和恢复流程。

**分层展开：**

**核心链路**
- 下单、支付、登录、核心查询。
- 优先保障资源。

**可降级能力**
- 推荐。
- 排行榜。
- 个性化展示。
- 非核心通知。

**降级方式**
- 返回默认值。
- 使用缓存旧值。
- 关闭部分功能。
- 限制低优先级用户或接口。

**恢复**
- 灰度恢复。
- 观察指标。
- 避免流量瞬间回灌。

**面试追问：降级是不是失败？**
不是。降级是稳定性设计的一部分，是在异常情况下保护核心业务的主动策略。

## 六、你的项目版服务治理追问

### 14. 广告平台调用外部媒体 API 如何做治理？

**业务特点：**
Facebook/TikTok/Snapchat 这类外部媒体 API 不受我们控制，存在限流、超时、错误码变更、授权失效、接口抖动等问题。服务治理重点不是传统 RPC 注册发现，而是外部依赖治理。

**治理方案：**
- 超时：每类接口设置独立超时时间，写接口和读接口区分。
- 重试：只对可重试错误重试，使用指数退避和最大次数。
- 限流：Redis 全局滑动窗口，按平台/账号/接口维度控制。
- 熔断：平台连续异常时短时间熔断，避免错误风暴。
- 降级：任务进入可重试状态，前台不长时间阻塞。
- 观测：记录平台、账号、接口、错误码、耗时、trace_id。

**面试总结：**
外部 API 治理的目标不是保证每次都成功，而是失败可控、可重试、可追踪、不拖垮主链路。

### 15. 微服务 P99 变高如何结合项目回答？

**排查路径：**
- 入口：网关/HTTP P99 是否变高。
- 服务：业务代码耗时、goroutine、GC、CPU。
- 存储：MySQL 慢 SQL、Mongo 慢查询、Redis 慢命令、ES 慢 DSL。
- MQ：是否同步等待异步结果，是否消费积压反压。
- 外部依赖：媒体 API、BI、通知服务是否抖动。

**广告平台例子：**
如果广告创建接口 P99 变高，我会优先看是否同步等待媒体 API。如果是，就把创建改成“提交任务 + 后台异步执行”；如果已经异步，则看任务创建阶段是否卡在 Mongo 写入、Redis 限流、Kafka 发送或权限校验。

### 16. 如何设计灰度发布和快速回滚？

**项目场景：**
广告任务列表从 MongoDB 查询切到 ES 查询，不能一次性全量发布。

**灰度方案：**
- 按用户白名单灰度。
- 按接口功能灰度，例如主列表和 keyword 搜索分开。
- 开关配置可实时调整。
- 响应中带查询路径，便于排障。
- 指标对比 Mongo/ES 的耗时、错误率、结果数量。
- 发现异常快速回切 Mongo。

**可迁移到红果：**
短剧内容发布、播放页配置、推荐策略、运营活动，都适合通过灰度、实验和开关降低发布风险。

### 17. 如何表达技术负责但不走管理？

**推荐表达：**
我带过 11 人后端团队，所以对需求拆解、方案评审、风险识别和跨团队协作比较熟悉。但我下一阶段更希望做资深后端 IC，不是纯管理岗。我希望把这段经历转化成技术影响力：能独立负责复杂链路，也能在方案评审、稳定性治理和关键项目推进上帮助团队。

**面试官真正关心：**
- 你会不会排斥协作？
- 你是不是只想写代码不管结果？
- 你能不能承担复杂项目？

**回答重点：**
不是“不负责”，而是“主要产出方式选择技术建设”。

**结论先行：**
可以说管理经验帮助自己更好地做业务 Owner，包括拆解目标、识别风险、推动协作和保障交付，但当前更希望在一线负责业务和核心技术方案。

**分层展开：**

**表达重点**
- 我更希望保持代码和业务深度。
- 管理经验让我更懂项目推进和风险控制。
- 可以承担技术 Owner 和 mentoring，但不追求纯管理路径。

**避免表达**
- 不要说“我不想管人”。
- 不要否定管理工作。
- 不要显得只接受个人任务。

**推荐说法**
- “我更适合也更希望做业务后端 Owner，管理经历让我在跨团队协同、方案拆解和交付风险识别上更成熟。”

**面试追问：能不能带新人？**
可以带新人、做 code review、方案评审和技术辅导，但职业目标仍是一线业务后端和技术 Owner。
