# 技术｜计算机网络与 HTTP 工程面试题

> 面向 9 年以上 Golang Web 后端业务研发岗位。本篇从报文如何跨主机到 HTTP/RPC 如何稳定调用，串联 IP、TCP、UDP/QUIC、Socket、DNS、TLS、代理、负载均衡、连接池与生产故障。

---

## 一、网络学习地图与资深回答框架

### 1. 后端网络知识图谱是什么？

```text
Link / ARP
  -> IP / Route / ICMP / MTU
  -> TCP / UDP / QUIC
  -> Socket / Nonblocking / epoll
  -> DNS / TLS
  -> HTTP/1.1 / HTTP/2 / HTTP/3 / gRPC
  -> Proxy / LB / NAT / Connection Pool
  -> Security / Diagnostics / Incidents
```

### 2. 资深后端如何回答网络题？

采用“所在层 -> 报文/状态流程 -> 可靠性机制 -> 性能代价 -> 应用配置 -> 线上证据”。例如 TIME_WAIT 不只讲定义，还要讲为何产生、谁主动关闭、连接复用和端口耗尽如何验证。

### 3. 网络故障为什么要分层？

DNS、路由、握手、TLS、协议、应用和下游都可能表现为 timeout。分层能选择正确证据：解析耗时、SYN 重传、TLS Alert、HTTP Status、应用 Trace。

### 4. 延迟由哪些部分组成？

排队、传播、传输、处理，以及 DNS、TCP/TLS 握手、代理和应用依赖。小请求常由 RTT/排队主导，大响应受带宽和拥塞主导。

### 5. 网络排障第一现场看什么？

错误类型、目标 Host/IP/Port、协议、DNS、连接建立、TLS、TTFB、重传、丢包、连接状态和代理日志；保留客户端与服务端同一 Trace/时间线。

## 二、链路层、IP、ARP、路由、ICMP 与 MTU

### 6. OSI/TCP-IP 分层的工程价值是什么？

分层提供职责和故障定位：链路传 Frame，IP 尽力交付 Packet，TCP 提供可靠字节流，HTTP 定义应用消息。真实实现可跨层优化，但排障仍需明确问题在哪层。

### 7. 同网段通信如何经过 ARP？

主机通过路由判断目标在本地链路，查询 ARP/邻居缓存得到目标 MAC；未知时广播请求，收到应答后封装 Ethernet Frame。跨网段则解析下一跳网关 MAC，而非远端 MAC。

### 8. IP 协议提供什么保证？

无连接、尽力交付，不保证到达、顺序和不重复。TTL 防路由环，Checksum/上层校验检测错误，可靠性由 TCP或应用承担。

### 9. 路由表如何选择路径？

按最长前缀匹配选目标网段，再结合优先级/Metric 和策略路由确定接口与下一跳。多网卡、容器网段和 VPN 问题常来自路由/Source IP 不一致。

### 10. ICMP 有什么作用？

传递不可达、超时、Fragmentation Needed 等控制信息；Ping 使用 Echo。防火墙完全丢弃 ICMP 可能破坏 Path MTU Discovery。Ping 通不代表 TCP/HTTP 一定通，反之亦然。

### 11. MTU 和 MSS 有什么区别？

MTU 是链路层最大 IP Packet 尺寸，MSS 是 TCP Payload 上限，通常从 MTU 减 IP/TCP Header。隧道增加头部会降低有效 MTU。

### 12. IP 分片和 PMTUD 如何工作？

IPv4 可能由路由器分片，任一 Fragment 丢失导致整个包失败；IPv6 路由器不分片。PMTUD 通过 ICMP 发现路径 MTU。ICMP 被丢可能形成 MTU Black Hole：小包通、大包卡。

### 13. NAT 解决什么并带来什么问题？

转换地址/端口以共享公网或连接网络，维护状态表。高并发可能耗尽 SNAT Port，Idle Timeout 关闭长连接，源 IP 在代理后丢失。需监控 NAT 表、端口和代理 Header。

### 14. Anycast、ECMP 和多路径有什么影响？

Anycast 将同一 IP 从多地宣告，ECMP 在等价路径散流；路径可能随哈希/路由变化，导致延迟和连接迁移差异。长连接通常保持五元组路径，QUIC Connection ID 更利于迁移。

## 三、TCP 连接、可靠传输、流控与拥塞控制

### 15. TCP 三次握手为什么是三次？

双方交换并确认初始序列号与收发能力：SYN、SYN-ACK、ACK。两次无法让服务端确认客户端已收到其序列信息，也难抵抗旧重复连接请求。

### 16. 四次挥手为什么通常是四次？

TCP 全双工，两方向独立关闭。收到 FIN 只表示对端不再发送，本端可能仍有数据，先 ACK，处理完再 FIN；某些情况下 ACK 与 FIN 可合并。

### 17. TIME_WAIT 为什么存在？

主动关闭方等待足够时间，确保最后 ACK 可重传，并让旧连接延迟 Segment 消失。大量 TIME_WAIT 常见于短连接；优先连接复用和合理架构，不粗暴修改内核语义。

### 18. CLOSE_WAIT 多说明什么？

对端已发 FIN，本端 Socket 尚未 Close，通常应用未关闭 Response Body/连接或处理卡住。定位持有 FD 的进程和调用栈，调内核超时不能根治应用泄漏。

### 19. TCP 如何保证可靠有序？

Sequence、ACK、Checksum、重传、接收缓存和乱序重组。可靠不等于业务请求只执行一次：连接断开和应用重试仍会重复副作用。

### 20. 超时重传和快速重传有什么区别？

RTO 到期重传，依据 RTT 动态估计；收到多个 Duplicate ACK 可触发快速重传，较早判断中间 Segment 丢失。现代算法还有 SACK 等改进，具体由系统实现。

### 21. 滑动窗口和流量控制是什么？

Receiver 通告可用窗口，Sender 控制未确认数据，避免压垮接收缓存。Zero Window 时发送方探测；应用不读取会让窗口变小并反向产生背压。

### 22. 拥塞控制解决什么？

通过 Congestion Window 控制网络在途数据，基于丢包/延迟等信号调整，保护共享网络。实际发送上限受 `min(cwnd, rwnd)` 等约束。算法如 CUBIC/BBR 取舍不同。

### 23. TCP 粘包是什么？

TCP 是字节流，不保留消息边界。一次 Send 不对应一次 Receive。应用用长度前缀、分隔符、固定长度或 HTTP/Protobuf Framing，解析器处理半包、多包和恶意长度。

### 24. Keepalive 和 HTTP Keep-Alive 是一回事吗？

TCP Keepalive 是内核探测长期空闲连接存活；HTTP Keep-Alive 是复用连接发送多个请求。应用还需要请求 Deadline 和 LB Idle Timeout，不能只靠 TCP Keepalive。

### 25. SYN Flood 如何防护？

攻击制造大量半连接。内核 SYN Cookie/队列治理、LB/WAF、源限速和容量共同防护。调大 Backlog 只能缓解，需监控 SYN_RECV、Drop 和攻击流量。

### 26. `listen backlog` 涉及哪些队列？

实现概念上有半连接和已完成等待 Accept 的队列，应用 Accept 慢或参数/内核上限不足会丢连接。排查同时看 SYN、Accept Queue、应用 Event Loop 和 CPU。

## 四、UDP、QUIC 与实时/低延迟通信

### 27. UDP 与 TCP 的核心区别是什么？

UDP 保留 Datagram 边界、无连接状态和内建可靠/有序/拥塞控制，Header 小、可一对多；应用按需实现重传、顺序、分片和安全。不是“UDP一定快”，可靠功能仍有成本。

### 28. UDP 适合什么场景？

DNS、实时音视频、游戏、遥测和基于 UDP 的 QUIC。允许部分丢失或应用有专门协议时合适；交易请求通常不直接手写 UDP 可靠层。

### 29. UDP 丢包在哪里发生？

发送/接收 Socket Buffer、网卡队列、内核处理、网络拥塞、MTU/分片、防火墙和应用读取不及时。需要端到端序号和各层 Drop 指标。

### 30. QUIC 解决什么问题？

在 UDP 上实现加密、可靠 Stream、拥塞控制和连接迁移；整合 TLS，减少握手；各 Stream 独立传输，避免 TCP 层一个丢包阻塞所有 Stream。代价是用户态 CPU、生态和网络设备兼容。

### 31. QUIC 为什么仍然会拥塞？

它仍共享网络并执行拥塞控制，只是多 Stream 的应用层队头阻塞改善。单个 Stream 内仍需有序，带宽、丢包和服务器 CPU 仍是限制。

### 32. 连接迁移如何实现？

QUIC 使用 Connection ID 而非仅依赖五元组识别连接，客户端 IP/网络变化后可验证新路径并继续。服务端负载均衡需能按 Connection ID 路由或共享状态。

## 五、Socket、非阻塞 IO、epoll 与服务器模型

### 33. Socket 是什么？

应用访问网络协议栈的内核对象，通过 FD 操作；TCP Socket 有状态、收发 Buffer 和地址元组。创建、Bind、Listen、Accept、Connect、Read/Write、Close 构成基本生命周期。

### 34. 阻塞和非阻塞 Socket 有什么区别？

阻塞调用无数据/空间时让线程睡眠；非阻塞立即返回 EAGAIN，由 Event Loop 在就绪后重试。非阻塞不等于无等待，只是等待由调度器管理。

### 35. IO 多路复用解决什么？

一个/少量线程等待大量 FD 的就绪事件，避免每连接一线程。Select/Poll 线性扫描，epoll 使用注册的 Interest/Ready 结构提高大量 FD 场景效率。

### 36. epoll LT 和 ET 如何选择？

LT 简单稳健，只要仍就绪就通知；ET 事件更少但必须非阻塞并读写到 EAGAIN。框架已封装时遵循框架，不为理论性能自行改底层。

### 37. Reactor 和 Proactor 有什么区别？

Reactor 通知“可以做 IO”，应用执行 Read/Write；Proactor 通知异步 IO 已完成。Linux 常见高性能网络库以 Reactor 为主，io_uring 提供更广异步提交/完成模型。

### 38. Go Netpoller 如何映射这些机制？

Go `net` 使用非阻塞 FD 和平台 Poller；G 等待网络时 Park，事件就绪后 Ready，不长期占用 OS Thread。业务仍需 Deadline、连接/并发上限和慢客户端治理。

### 39. Socket Buffer 越大越好吗？

大 Buffer 可吸收 RTT/带宽和突发，但增加单连接内存、排队和延迟；小 Buffer 限吞吐。结合 Bandwidth-Delay Product、连接数和业务消息压测，系统 Max 也限制应用设置。

### 40. Zero Copy 是什么？

减少用户态/内核态之间复制和上下文切换，例如 sendfile、splice、mmap 等。适合文件传输；TLS、内容变换和跨平台会限制。端到端 Profile 后再用。

### 41. Backpressure 如何沿网络传播？

应用消费慢 -> Receive Buffer 满 -> TCP Window 缩小 -> Sender Write/Buffer 排队 -> 上游请求积压。每层都应有有界 Buffer 和 Deadline，不能无限内存队列切断信号。

## 六、DNS、TLS、证书与安全连接

### 42. DNS 查询链路是什么？

应用/系统 Resolver 查本地缓存和 Hosts，递归 Resolver 依次获取 Root、TLD、权威结果并缓存。实际可有搜索域、Stub、Sidecar 和 Service Discovery。

### 43. TTL 如何影响变更？

TTL 控制缓存时长，低 TTL 收敛快但增加查询，高 TTL 稳定但迁移慢。客户端长连接即使 DNS 更新也可能继续旧 IP，因此发布/容灾要同时治理连接生命周期。

### 44. DNS 为什么会成为性能瓶颈？

Resolver 慢、缓存 Miss、搜索域多次尝试、并发限制、网络和权威故障。记录 DNS Duration/Error，使用缓存和冗余 Resolver，但避免永久缓存失效地址。

### 45. TLS 握手大致发生什么？

协商版本/Cipher，服务器发送证书，客户端验证域名、链、有效期和信任根，双方完成密钥交换并建立会话密钥。现代 TLS减少往返并支持会话恢复。

### 46. 对称和非对称加密如何配合？

非对称用于身份与密钥协商，成本高；建立会话后用对称加密传数据。证书把公钥与身份由 CA 签名关联。

### 47. 证书验证为什么会失败？

过期/未生效、域名不匹配、链不完整、根不信任、SNI 错、时钟偏差和撤销/策略。不能生产跳过验证，应修信任、证书和时间。

### 48. mTLS 与普通 TLS 有什么区别？

普通 TLS 主要验证服务端，mTLS 还要求服务端验证客户端证书，适合服务身份。它提供认证和加密，不自动提供资源级授权。

### 49. TLS 如何影响性能？

握手 RTT、非对称计算、记录加解密和证书验证。连接复用、会话恢复、硬件/库优化可降低；大规模短连接最明显。不能为性能关闭验证。

### 50. 0-RTT 有什么安全边界？

早期数据可能被重放，只适合幂等读或有重放防护的操作。交易写不能因降低一次 RTT 忽视业务幂等。

## 七、HTTP/1.1、HTTP/2、HTTP/3、gRPC 与长连接

### 51. HTTP/1.1 的核心特性和限制是什么？

文本 Header、Persistent Connection、Chunked 等；同连接请求响应顺序约束导致应用层队头问题，浏览器常开多连接。连接复用减少握手，但 Idle 与池大小需治理。

### 52. HTTP/2 如何多路复用？

将连接上的消息拆为二进制 Frame，按 Stream ID 交错传输，并有 Header Compression 和流控。一个 TCP丢包仍会阻塞连接上所有 Stream 的字节到达。

### 53. HTTP/3 为什么改善队头阻塞？

基于 QUIC，各 Stream 独立可靠，某 Stream 丢包不阻塞其他 Stream；同时整合 TLS 和连接迁移。服务器、客户端和中间网络支持度及 CPU 需要评估。

### 54. HTTP Keep-Alive 和连接池如何配置？

按单实例并发和下游总预算配置 Max Idle/Per Host/Total，设置 Idle Timeout 和连接年龄；确保 Body 读取/关闭；监控池等待、建连和复用。池越大不等于吞吐越高。

### 55. Chunked、Content-Length 和流式响应有何差异？

已知长度可发 Content-Length；未知长度 HTTP/1.1 可 Chunked；HTTP/2/3 用 Frame 结束语义。流式响应要 Flush、Backpressure、取消和代理 Buffer 配置。

### 56. HTTP 状态码 502、503、504 如何理解？

502 代理收到无效/失败上游响应；503 服务不可用/过载；504 代理等待上游超时。实际由网关定义，必须结合代理日志、Upstream Timing 和应用 Trace。

### 57. gRPC 为什么依赖 HTTP/2？

利用二进制 Frame、多路复用、双向 Stream、Header/Trailer 和流控。gRPC Status 在 Trailer，代理和观测工具需正确支持。

### 58. WebSocket 和 SSE 如何选择？

WebSocket 双向、适合交互；SSE 基于 HTTP 单向服务端推送、浏览器重连简单。两者都是长连接，需心跳、慢客户端、连接上限、重连游标和发布治理。

### 59. HTTP 幂等和网络重试有什么关系？

GET/PUT 等定义上的幂等仍需正确实现；POST 可用 Idempotency-Key。网络超时只表示未知，自动重试前必须知道服务端是否可能已执行。

### 60. Header 为什么也需要限制？

超大 Header/Cookie 会耗内存、带宽和解析 CPU，并可能触发代理差异。服务端、网关统一限制；Trace/Baggage 不传播无界业务数据。

## 八、代理、负载均衡、NAT、服务发现与连接治理

### 61. 正向代理和反向代理有什么区别？

正向代理代表客户端访问外部，反向代理代表服务端接收用户并转发后端。API Gateway/LB 常是反向代理，承担 TLS、路由、限流和观测。

### 62. L4 和 L7 负载均衡如何选择？

L4 按连接/五元组，性能高且协议透明；L7理解 HTTP/gRPC，可按路径/Header路由和重试，但解析成本与语义复杂。架构可组合。

### 63. 常见负载算法有哪些？

Round Robin、Weighted、Least Request/Connection、Consistent Hash、EWMA/自适应。长连接下连接均匀不等于请求均匀；一致性哈希可能热点。

### 64. 健康检查为什么可能误判？

阈值太短把慢当死，深度探测依赖异常导致全部摘除，检查路径与业务路径不同产生假健康。区分 Liveness、Readiness 和被动错误，设置抖动和慢启动。

### 65. 代理重试为什么危险？

代理不知道完整业务幂等，重试写请求可能重复；多层重试放大。仅对安全方法/明确条件、有限次数和总 Deadline 重试，并输出 attempt 指标。

### 66. X-Forwarded-For 和 Forwarded 如何信任？

客户端可伪造，只有来自可信代理的 Header 可接受；网关清洗并追加，应用按可信跳数解析。安全审计保留直接 Peer 与逻辑 Client IP。

### 67. 服务发现更新为何不立即生效？

DNS/注册缓存和现有长连接仍指向旧实例。客户端 Resolver 更新地址后还需连接轮换和 LB 创建新 Subchannel；下线先摘流量、等待连接收敛再终止。

### 68. SNAT Port 耗尽如何发生？

大量到同一目标短连接需要唯一源端口/五元组，NAT 表与端口范围耗尽导致 Connect 失败。连接复用、增加出口 IP、扩大资源和降低短连接共同治理。

## 九、网络安全、观测与故障诊断

### 69. 网络问题常用哪些命令？

```bash
ip addr
ip route get <ip>
ss -s
ss -tanp
dig +trace <domain>
curl -v --trace-time <url>
openssl s_client -connect host:443 -servername host
tcpdump -i any -nn host <ip>
```

生产抓包要限定 Host/Port/时长和权限，注意敏感数据。

### 70. 如何用 tcpdump 判断 TCP 问题？

看 SYN/SYN-ACK/ACK、Retransmission、Duplicate ACK、RST、Zero Window 和 RTT；同时抓客户端/服务端更可靠。抓包未看到应用内部排队，必须与 Trace 结合。

### 71. `connection refused`、timeout、reset 有何差异？

Refused 常是目标可达但端口无监听/主动拒绝；Connect Timeout 常是丢包、路由、防火墙或积压；Reset 是连接被一方/中间设备异常关闭。错误仍需结合阶段和抓包。

### 72. 丢包和重传如何定位？

按客户端、主机网卡、交换/云网络、服务端逐段；看 TCP Retrans、NIC Drop/Error、Softnet、队列和带宽。重传也可能接收端应用不读导致，不全是物理网络。

### 73. 如何防 DDoS 和应用层攻击？

边缘清洗/CDN/WAF、L4 SYN防护、L7认证限流、请求大小/复杂度限制、连接与并发上限、缓存和降级。源站隐藏和容量预案同样重要。

### 74. 网络故障复盘需要哪些证据？

目标域名/IP/端口、协议版本、DNS答案、路由、握手/TLS/TTFB分段、连接状态、重传、代理 Upstream 日志、Trace、发布和网络变更。只写“网络抖动”不是根因。

## 十、业务架构与生产案例

### 75. 广告创建接口大量超时如何排查？

按入口网关、Go服务、HTTP连接池、DNS/TLS、媒体 API和账号限流拆 Trace；确认是池等待、平台429还是网络。止血按平台熔断、任务异步排队并保护其他媒体；恢复限速回放。

### 76. 大量 TIME_WAIT 如何处理？

确认主动关闭方和目标分布，检查每请求建 Client、Keep-Alive 被禁、Body 未读、LB Idle 不匹配；修复连接复用和池参数。只有端口确实瓶颈时再评估系统参数和出口扩展。

### 77. HTTP/2 单连接故障为什么影响大？

大量 Stream 复用一个 TCP，丢包/连接关闭影响多个请求；Server Max Concurrent Streams和客户端连接数也会排队。可使用合理连接池、Keepalive和重连，但连接过多又增加资源。

### 78. 小包正常、大响应超时如何处理？

检查 MTU Black Hole、代理 Body/Buffer/Timeout、带宽和客户端读取；用不同 Payload、Ping DF/Tracepath和抓包验证。不要只调应用 WriteTimeout。

### 79. DNS 切流为何仍有请求到旧集群？

递归/本地缓存未过 TTL，客户端长连接未重建，代理有独立缓存。迁移前降低 TTL，旧集群保留过渡，连接设置合理年龄，监控新旧流量并渐进下线。

### 80. 如何用两分钟回答网络架构题？

> 我按链路、网络、传输、加密和应用层定位。域名先经 DNS，路由与 ARP找到下一跳；TCP通过三次握手、序列/ACK、流控和拥塞控制提供可靠字节流，应用必须定义消息边界；TLS建立身份和会话密钥；HTTP/1.1复用连接，HTTP/2在单TCP多路复用，HTTP/3用QUIC改善跨Stream队头阻塞。生产上代理/LB、NAT、连接池和服务发现都会影响路径，所以我监控 DNS、Connect、TLS、TTFB、池等待、重传和状态码。故障时用Trace先找慢阶段，再用ss、dig、抓包和代理日志取证；重试只用于幂等请求并受总Deadline控制，恢复时渐进放流和轮换旧连接。
