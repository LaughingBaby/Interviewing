**基础能力**
- 数组、链表、字符串。
- 哈希表、栈、队列、堆。

**核心思想**
- 双指针。
- 滑动窗口。
- 二分。
- 递归和回溯。
- 动态规划。
- 贪心。

**表达方式**
- 先说思路。
- 再说复杂度。
- 再处理边界。
- 最后写代码和自测。

**工程映射**

## 二、数组、哈希与双指针

### 2. 两数之和怎么做？

**结论先行：**
使用哈希表记录已遍历数字到下标的映射，遍历时查找目标差值，时间复杂度 O(n)。

**分层展开：**

**思路**
- 遍历数组。
- 对当前值 x，查 `target - x` 是否出现过。
- 如果出现，返回两个下标。
- 否则把 x 放入 map。

**复杂度**
- 时间 O(n)。
- 空间 O(n)。

**边界**
- 重复数字。
- 负数。
- 无解时返回空或约定值。

**面试追问：为什么不能先排序？**
排序会丢失原始下标，需要额外保存；如果题目要求原下标，哈希表更直接。

### 3. 三数之和怎么做？

**结论先行：**
先排序，再固定一个数，用双指针寻找另外两个数，并通过跳过重复值去重。

**分层展开：**

**思路**
- 排序。
- 枚举第一个数 i。
- left 指向 i+1，right 指向末尾。
- 三数和小了左移，大了右移，等于目标则记录。

**去重**
- i 和前一个相同跳过。
- 找到答案后跳过相同 left 和 right。

**复杂度**
- 时间 O(n^2)。
- 空间取决于排序实现。

**面试追问：为什么排序后可以双指针？**
因为数组有序，和小了只能增大 left，和大了只能减小 right，方向明确。

### 4. 滑动窗口适合什么题？

**结论先行：**
滑动窗口适合连续子数组或子串问题，尤其是求最长、最短、计数且窗口左右边界可单调移动的场景。

**分层展开：**

**典型题**
- 最长无重复子串。
- 最小覆盖子串。
- 长度最小的子数组。
- 固定窗口最大值。

**核心模板**
- 右指针扩张窗口。
- 更新窗口状态。
- 当窗口不满足条件时左指针收缩。
- 更新答案。

**关键点**
- 窗口内状态怎么维护。
- 什么时候收缩。
- 答案在扩张时更新还是收缩后更新。

**面试追问：滑动窗口为什么通常是 O(n)？**
左右指针都只单调向前，每个元素最多进窗口一次、出窗口一次。

### 5. 最长无重复子串怎么做？

**结论先行：**
用滑动窗口和哈希表记录字符最近位置，发现重复时移动左边界。

**分层展开：**

**思路**
- right 遍历字符串。
- map 记录字符最近下标。
- 如果字符出现且位置不小于 left，left 移到重复字符后一位。
- 每步更新最大长度。

**复杂度**
- 时间 O(n)。
- 空间 O(字符集大小)。

**边界**
- 空字符串。
- 全重复。
- 全不重复。
- Unicode 场景注意 rune 和 byte。

**面试追问：Go 里字符串遍历要注意什么？**
`for i := range s` 的 i 是 byte 下标；中文等多字节字符要按 rune 处理，否则长度和切片可能错误。

## 三、链表、栈和队列

### 6. 反转链表怎么做？

**结论先行：**
用三个指针迭代反转，每次保存 next，再把当前节点指向前驱。

**分层展开：**

**思路**
- prev 初始 nil。
- cur 初始 head。
- 保存 next。
- cur.Next = prev。
- prev = cur，cur = next。

**复杂度**
- 时间 O(n)。
- 空间 O(1)。

**边界**
- 空链表。
- 单节点。
- 多节点。

**面试追问：递归能做吗？**
能，但递归有栈空间 O(n)，链表很长时可能有栈风险，迭代更稳。

### 7. 如何判断链表有环？

**结论先行：**
使用快慢指针，快指针每次走两步，慢指针每次走一步，如果相遇说明有环。

**分层展开：**

**思路**
- slow=head，fast=head。
- slow 每次一步，fast 每次两步。
- fast 或 fast.Next 为空则无环。
- slow==fast 则有环。

**复杂度**
- 时间 O(n)。
- 空间 O(1)。

**扩展**
- 找入环点：相遇后一个指针回 head，两个指针每次一步，再次相遇就是入口。

**面试追问：为什么找入口这样可行？**
由快慢指针路程差可推导，head 到入口距离等于相遇点绕环到入口的距离。

### 8. LRU 缓存怎么实现？

**结论先行：**
LRU 用哈希表 + 双向链表实现，哈希表 O(1) 定位节点，双向链表 O(1) 调整访问顺序。

**分层展开：**

**数据结构**
- map：key -> 链表节点。
- 双向链表：头部最近使用，尾部最久未使用。

**get**
- map 查节点。
- 找到后移动到头部。
- 返回 value。

**put**
- 已存在则更新并移动到头部。
- 不存在则插入头部。
- 超容量删除尾部节点。

**面试追问：为什么不用单链表？**
删除尾部或移动任意节点需要前驱节点，单链表无法 O(1) 完成，双向链表更合适。

### 9. 单调栈适合什么题？

**结论先行：**
单调栈适合寻找下一个更大/更小元素、柱状图最大矩形、每日温度等问题。

**分层展开：**

**核心思想**
- 栈内元素保持单调。
- 当前元素破坏单调性时，弹出并结算答案。

**典型题**
- 每日温度。
- 下一个更大元素。
- 接雨水。
- 柱状图最大矩形。

**复杂度**
- 每个元素最多入栈一次、出栈一次。
- 时间 O(n)。

**面试追问：单调栈和堆有什么区别？**
单调栈解决相邻顺序关系中的最近更大/更小；堆解决动态集合中的全局最大/最小。

## 四、二分、堆和 TopK

### 10. 二分查找的本质是什么？

**结论先行：**
二分的本质是在单调性上排除一半搜索空间，不只用于有序数组，也用于答案空间。

**分层展开：**

**数组二分**
- 查找目标值。
- 查找第一个大于等于。
- 查找最后一个小于等于。

**答案二分**
- 最小可行值。
- 最大可行值。
- 通过 check 函数判断可行性。

**关键点**
- 明确区间开闭。
- 明确 mid 是否保留。
- 避免死循环。

**面试追问：二分最容易错在哪里？**
边界更新和循环条件。建议固定一种模板，比如左闭右闭或左闭右开，不要混用。

### 11. TopK 问题怎么做？

**结论先行：**
TopK 常用小顶堆维护 K 个最大元素，时间 O(nlogK)，空间 O(K)。

**分层展开：**

**思路**
- 建一个大小为 K 的小顶堆。
- 遍历元素。
- 堆未满直接加入。
- 当前元素大于堆顶，则替换堆顶。

**复杂度**
- 时间 O(nlogK)。
- 空间 O(K)。

**其他方案**
- 快速选择平均 O(n)。
- 排序 O(nlogn)。
- 数据范围小可计数排序。

**面试追问：海量数据 TopK 怎么做？**
分片统计局部 TopK，再合并全局 TopK；如果数据无法一次进内存，使用外部排序或流式堆。

## 五、树、图和动态规划

### 12. 二叉树遍历怎么掌握？

**结论先行：**
二叉树遍历要掌握前中后序和层序，递归写法表达结构，迭代写法体现栈和队列理解。

**分层展开：**

**DFS**
- 前序：根左右。
- 中序：左根右。
- 后序：左右根。

**BFS**
- 队列层序遍历。
- 适合最短层数、层级聚合。

**工程类比**
- 递归目录扫描。
- 组织树。
- 评论树。
- 权限菜单。

**面试追问：递归有什么风险？**
树特别深时可能栈溢出，生产系统处理深层结构要考虑迭代或深度限制。

### 13. 动态规划题怎么识别？

**结论先行：**
动态规划适合有重叠子问题和最优子结构的问题，关键是定义状态、转移方程、初始化和遍历顺序。

**分层展开：**

**四步法**
- 定义 dp 含义。
- 写状态转移。
- 设置初始值。
- 确定遍历顺序。

**典型题**
- 爬楼梯。
- 最长递增子序列。
- 背包问题。
- 编辑距离。
- 最大子数组和。

**优化**
- 一维滚动数组。
- 状态压缩。
- 剪枝。

**面试追问：动态规划和递归记忆化有什么区别？**
本质相同。记忆化是自顶向下，DP 表通常是自底向上。

### 14. 图的 BFS 和 DFS 怎么选？

**结论先行：**
BFS 适合最短路径和层级扩散，DFS 适合连通性、回溯、拓扑和路径搜索。

**分层展开：**

**BFS**
- 队列。
- 无权图最短路径。
- 层级遍历。

**DFS**
- 栈或递归。
- 连通分量。
- 拓扑排序。
- 回溯枚举。

**工程场景**
- 依赖任务编排。
- 组织关系。
- 风控关系网络。
- 推荐关系扩散。

**面试追问：如何判断有向图有环？**
可以用 DFS 三色标记，或拓扑排序看最终处理节点数是否等于总节点数。

### 15. 并查集适合什么问题？

**结论先行：**
并查集适合动态连通性问题，例如朋友圈、账号合并、岛屿数量、图连通分量。

**分层展开：**

**核心操作**
- find：查找根节点。
- union：合并集合。
- connected：判断是否同集合。

**优化**
- 路径压缩。
- 按秩或按大小合并。

**复杂度**
- 接近 O(1) 的均摊复杂度。

**面试追问：并查集不能解决什么？**
不擅长处理删除边后的连通性变化，动态删除通常需要更复杂的数据结构或离线处理。

## 六、后端高频算法工程题

### 16. 如何实现令牌桶限流？

**结论先行：**
令牌桶按固定速率生成令牌，请求到来时消耗令牌，有令牌放行，无令牌拒绝或排队。

**分层展开：**

**状态**
- 当前令牌数。
- 最大桶容量。
- 令牌生成速率。
- 上次刷新时间。

**流程**
- 请求到来先根据时间差补充令牌。
- 令牌数不超过桶容量。
- 有令牌则扣减并放行。
- 无令牌则拒绝。

**工程注意**
- 单机限流可本地实现。
- 分布式限流可用 Redis + Lua。
- 高 QPS 全局限流要避免 Redis 成瓶颈。

**面试追问：令牌桶和漏桶区别？**
令牌桶允许一定突发，漏桶以固定速率流出，更平滑但对突发不友好。

### 17. 如何做一致性哈希？

**结论先行：**
一致性哈希把节点和 key 映射到哈希环上，key 顺时针找到第一个节点，节点增删时只影响局部 key。

**分层展开：**

**基本流程**
- 对节点 hash 放到环上。
- 对 key hash 后顺时针找节点。
- 节点变化只迁移相邻区间。

**虚拟节点**
- 每个真实节点映射多个虚拟节点。
- 缓解数据倾斜。
- 支持权重。

**应用场景**
- 分布式缓存。
- 分片路由。
- 负载均衡。

**面试追问：一致性哈希能完全避免倾斜吗？**
不能，只能降低倾斜概率。热点 key 仍需要单独治理。

## 七、字节后端定向：手写代码与项目化算法

### 19. 字节一面手写代码应该重点准备什么？

**结论先行：**
以字节后端为目标，算法不要只刷“能 AC”，要能在 20-30 分钟内用 Go 写出可运行、边界清晰、复杂度能解释的代码。优先准备高频基础题、工程型手写题和并发型手写题。

**P0 高频基础题：**
- 哈希：两数之和、最长连续序列、字母异位词分组。
- 双指针：三数之和、接雨水、盛最多水的容器。
- 滑动窗口：最长无重复子串、最小覆盖子串、找到字符串中所有字母异位词。
- 二分：搜索旋转数组、寻找峰值、查找第一个/最后一个位置。
- 链表：反转链表、合并 K 个有序链表、环形链表、相交链表。
- 栈和队列：有效括号、最小栈、单调栈、每日温度。
- 堆：TopK、高频词、数据流中位数。
- 树：层序遍历、最近公共祖先、路径和、二叉树直径。
- 图：岛屿数量、课程表、最短路径基础。
- DP：爬楼梯、打家劫舍、最长递增子序列、最长公共子序列。

**P0 工程型手写题：**
- LRU Cache。
- 令牌桶/滑动窗口限流器。
- Worker Pool。
- 并发安全 Map。
- 生产者消费者。
- 带超时控制的批量任务执行。
- TopK 热点统计。
- 一致性哈希。

**回答方式：**
1. 先说思路和复杂度。
2. 再写核心代码。
3. 主动补边界：空输入、重复元素、并发安全、溢出、超时。
4. 写完后用 1-2 个 case 口头跑一遍。

### 20. 如何手写 Worker Pool？

**适用场景：**
枫叶互动广告素材同步、ES fallback 补偿、广告任务批量发布、媒体 API 批量拉取这类任务，都不能无限开 goroutine，否则会打满外部 API、DB、Redis 或本机资源。Worker Pool 的核心是控制并发、收敛错误、支持取消。

```go
package main

import (
	"context"
	"fmt"
	"sync"
)

type Job struct {
	ID int
}

func RunWorkerPool(ctx context.Context, jobs []Job, workerNum int, fn func(context.Context, Job) error) error {
	if workerNum <= 0 {
		workerNum = 1
	}

	ctx, cancel := context.WithCancel(ctx)
	defer cancel()

	jobCh := make(chan Job)
	errCh := make(chan error, 1)
	var wg sync.WaitGroup

	for i := 0; i < workerNum; i++ {
		wg.Add(1)
		go func() {
			defer wg.Done()
			for {
				select {
				case <-ctx.Done():
					return
				case job, ok := <-jobCh:
					if !ok {
						return
					}
					if err := fn(ctx, job); err != nil {
						select {
						case errCh <- err:
							cancel()
						default:
						}
						return
					}
				}
			}
		}()
	}

	go func() {
		defer close(jobCh)
		for _, job := range jobs {
			select {
			case <-ctx.Done():
				return
			case jobCh <- job:
			}
		}
	}()

	done := make(chan struct{})
	go func() {
		wg.Wait()
		close(done)
	}()

	select {
	case <-done:
		return nil
	case err := <-errCh:
		return err
	case <-ctx.Done():
		return ctx.Err()
	}
}

func main() {
	jobs := []Job{{1}, {2}, {3}}
	err := RunWorkerPool(context.Background(), jobs, 2, func(ctx context.Context, job Job) error {
		fmt.Println(job.ID)
		return nil
	})
	fmt.Println(err)
}
```

**面试追问：**
- 如果允许部分失败怎么办？把 error 收集成 slice，不要第一个错误就 cancel。
- 如果要限速怎么办？在 worker 内部加入 `time.Ticker` 或 Redis/本地限流器。
- 如果任务很多怎么办？不要一次性加载全部任务，可以分页扫描 DB/Mongo 后分批投递。

### 21. 如何手写 LRU Cache？

**核心思路：**
HashMap 负责 O(1) 定位节点，双向链表负责 O(1) 移动和淘汰。最近访问的节点放到链表头部，容量满时淘汰尾部。

```go
type Node struct {
	key, val int
	prev    *Node
	next    *Node
}

type LRUCache struct {
	cap  int
	data map[int]*Node
	head *Node
	tail *Node
}

func Constructor(capacity int) LRUCache {
	head := &Node{}
	tail := &Node{}
	head.next = tail
	tail.prev = head
	return LRUCache{
		cap:  capacity,
		data: make(map[int]*Node),
		head: head,
		tail: tail,
	}
}

func (c *LRUCache) Get(key int) int {
	node, ok := c.data[key]
	if !ok {
		return -1
	}
	c.moveToHead(node)
	return node.val
}

func (c *LRUCache) Put(key int, value int) {
	if node, ok := c.data[key]; ok {
		node.val = value
		c.moveToHead(node)
		return
	}

	node := &Node{key: key, val: value}
	c.data[key] = node
	c.addToHead(node)

	if len(c.data) > c.cap {
		removed := c.removeTail()
		delete(c.data, removed.key)
	}
}

func (c *LRUCache) moveToHead(node *Node) {
	c.remove(node)
	c.addToHead(node)
}

func (c *LRUCache) addToHead(node *Node) {
	node.prev = c.head
	node.next = c.head.next
	c.head.next.prev = node
	c.head.next = node
}

func (c *LRUCache) remove(node *Node) {
	node.prev.next = node.next
	node.next.prev = node.prev
}

func (c *LRUCache) removeTail() *Node {
	node := c.tail.prev
	c.remove(node)
	return node
}
```

### 22. 如何手写滑动窗口限流器？

**简化版本地实现：**

```go
type SlidingWindowLimiter struct {
	mu      sync.Mutex
	limit   int
	window  time.Duration
	records []time.Time
}

func NewSlidingWindowLimiter(limit int, window time.Duration) *SlidingWindowLimiter {
	return &SlidingWindowLimiter{limit: limit, window: window}
}

func (l *SlidingWindowLimiter) Allow() bool {
	l.mu.Lock()
	defer l.mu.Unlock()

	now := time.Now()
	start := now.Add(-l.window)
	idx := 0
	for idx < len(l.records) && l.records[idx].Before(start) {
		idx++
	}
	l.records = l.records[idx:]

	if len(l.records) >= l.limit {
		return false
	}
	l.records = append(l.records, now)
	return true
}
```

**项目升级版回答：**
本地限流只能限制单实例，枫叶互动媒体 API 调用是多实例并发，所以更适合 Redis ZSET + Lua 做全局滑动窗口。Lua 保证清理窗口、计数、写入、设置过期的原子性；ZSET member 不能只用秒级时间戳，否则同一秒多个请求会被去重，导致限流失真。
