# 技术｜架构设计与项目表达面试题

> 面向 9 年+ Golang Web 后端业务岗。
> 目标：把技术能力表达成“能负责复杂业务”的业务 Owner 能力，同时把 1 年多管理经验表达成协作和交付优势，而不是管理岗诉求。

## 一、个人定位与面试表达

### 1. 9 年+ Go 后端应该如何自我介绍？

**结论先行：**
自我介绍要突出“业务后端 Owner、复杂系统落地、高并发稳定性、Go 技术栈”，管理经验只作为补充能力。

**分层展开：**

**推荐结构**
- 工作年限和主技术栈。
- 负责过的核心业务域。
- 擅长的技术方向。
- 代表项目和量化结果。
- 当前岗位偏好。

**表达重点**
- 我希望继续深耕业务后端。
- 能独立负责复杂模块。
- 能做方案设计、编码实现、上线治理和复盘。
- 管理经验帮助我更好协作和推进，但不是求职主方向。

**示例表达**
- “我主要做 Go Web 后端，过去负责过交易/内容/用户增长等业务模块，比较擅长高并发读写、缓存一致性、MQ 最终一致和线上稳定性治理。之前有一年多带小团队和项目推进经验，这段经历让我在方案拆解、风险识别和协作推进上更成熟，但我当前更希望承担业务后端 Owner 的角色，保持一线技术和业务深度。”

**面试追问：为什么不继续做管理？**
可以回答：“我认可管理价值，但我现阶段更希望把优势放在复杂业务和核心系统建设上。管理经历会帮助我更好地协同和推动落地，但我的职业主线还是业务后端技术 Owner。”

### 2. 项目介绍怎么讲才有层级？

**结论先行：**
项目介绍要讲业务目标、技术挑战、你的职责、关键方案、结果数据和复盘，不要只罗列技术栈。

**分层展开：**

**背景**
- 业务是什么。
- 用户量、QPS、数据量。
- 原系统痛点。

**职责**
- 你负责哪些模块。
- 哪些方案由你主导。
- 如何协调上下游。

**方案**
- 架构设计。
- 数据模型。
- 缓存/MQ/DB 设计。
- 一致性和稳定性。

**结果**
- 延迟降低。
- QPS 提升。
- 成本下降。
- 故障减少。
- 业务指标提升。

**复盘**
- 方案边界。
- 后续优化。
- 如果重做会怎么改。

**面试追问：没有准确数据怎么办？**
可以给区间或相对变化，例如“DB QPS 降低约 70%”、“P99 从秒级降到 300ms 内”，但不要编造无法自洽的数据。

## 二、业务建模与方案设计

### 3. 如何做复杂业务需求分析？

**结论先行：**
复杂需求要先识别业务目标、核心实体、状态流转、边界条件和异常补偿，再进入技术设计。

**分层展开：**

**业务目标**
- 解决什么问题。
- 面向哪些用户。
- 成功指标是什么。

**领域对象**
- 用户、订单、商品、库存、支付单、内容、评论等。
- 对象之间关系。

**状态流转**
- 初始状态。
- 中间状态。
- 终态。
- 非法流转。

**异常场景**
- 重复请求。
- 下游失败。
- 超时。
- 并发冲突。
- 数据补偿。

**面试追问：为什么要先画状态机？**
状态机能把业务规则显性化，避免并发、重复消息和异常补偿时状态错乱。

### 4. 技术方案评审应该关注什么？

**结论先行：**
方案评审关注目标是否明确、边界是否清晰、风险是否识别、数据是否正确、稳定性是否可控、上线是否可回滚。

**分层展开：**

**业务正确性**
- 流程是否闭环。
- 状态是否完整。
- 异常是否补偿。

**技术合理性**
- 架构是否过度。
- 组件选型是否匹配。
- 数据模型是否可扩展。

**稳定性**
- 限流降级。
- 幂等重试。
- 监控告警。
- 容量评估。

**交付风险**
- 依赖方。
- 排期。
- 灰度。
- 回滚。

**面试追问：你评审时最常拦什么问题？**
常见是没有幂等、没有补偿、没有监控、没有灰度回滚、数据库变更不兼容。

### 5. 如何做技术选型？

**结论先行：**
技术选型要围绕业务目标、团队能力、生态成熟度、成本、风险和演进空间，而不是追新。

**分层展开：**

**评估维度**
- 是否解决核心问题。
- 团队是否能维护。
- 社区和生态是否成熟。
- 性能和成本是否匹配。
- 故障时是否可控。

**常见取舍**
- MySQL vs NoSQL。
- Kafka vs RocketMQ。
- Redis 缓存 vs 本地缓存。
- 同步 RPC vs MQ 异步。

**落地方式**
- 小范围验证。
- 压测。
- 灰度。
- 预案。

**面试追问：你会如何说服团队不要过度设计？**
回到业务目标和当前瓶颈，用数据说明简单方案已能满足，并保留未来演进接口。

## 三、系统设计表达模板

### 6. 如何设计一个高并发商品详情页？

**结论先行：**
商品详情页要用多级缓存、读模型拆分、热点预热、价格库存独立治理和降级兜底。

**分层展开：**

**读链路**
- CDN 静态资源。
- 本地缓存热点详情。
- Redis 缓存商品主数据。
- DB 或商品服务兜底。

**数据拆分**
- 商品基础信息低频变化。
- 价格、库存高频变化，独立缓存或服务。
- 推荐、评论、营销信息异步加载。

**一致性**
- 商品变更写 DB 后删缓存。
- MQ 广播删除本地缓存。
- TTL 兜底。

**稳定性**
- 热点商品预热。
- 逻辑过期。
- 回源互斥。
- Redis 故障返回本地旧值或降级字段。

**面试追问：为什么价格库存不和详情放一起？**
变化频率和一致性要求不同，混在一起会导致详情缓存频繁失效，也增加不一致风险。

### 7. 如何设计订单支付链路？

**结论先行：**
订单支付链路核心是状态机、幂等、事务边界、支付回调可靠处理和对账补偿。

**分层展开：**

**下单**
- 创建订单。
- 幂等号防重复。
- 预占库存。
- 生成支付单。

**支付**
- 调用支付渠道。
- 支付回调验签。
- 回调幂等。
- 更新支付单和订单状态。

**异步**
- 支付成功事件。
- 履约、积分、通知。
- MQ 消费幂等。

**补偿**
- 支付渠道对账。
- 超时关单。
- 库存释放。
- 异常订单扫描。

**面试追问：支付回调成功但订单更新失败怎么办？**
支付单状态要先可靠落库，后续订单状态通过本地消息表、重试任务或对账补偿推进。

### 8. 如何设计内容发布系统？

**结论先行：**
内容发布系统要区分写入、审核、分发、搜索索引、推荐索引和用户可见状态，核心是状态流转和异步一致。

**分层展开：**

**写入链路**
- 内容草稿。
- 发布请求。
- 内容存储。
- 审核状态。

**异步链路**
- 审核通过后发事件。
- 更新搜索索引。
- 更新推荐索引。
- 推送 Feed。

**查询链路**
- 作者看自己内容。
- 用户看公开内容。
- 审核状态控制可见性。

**稳定性**
- 内容事件幂等。
- 索引失败补偿。
- 敏感内容回撤。
- 缓存失效。

**面试追问：搜索索引和 DB 不一致怎么办？**
以 DB 为准，通过消息重试、定时校验和索引重建修复。关键操作查询 DB 状态二次校验。

## 四、管理经验转化为业务技术优势

### 9. 如何表达带人经验但不变成管理岗候选人？

**结论先行：**
把带人经验表达为“技术影响力和交付能力”，不要表达为“组织管理诉求”。

**分层展开：**

**可以强调**
- 做过任务拆解。
- 做过 code review。
- 帮新人理解业务。
- 推动跨团队接口对齐。
- 识别项目风险。

**避免强调**
- 汇报线人数。
- 管理制度。
- 绩效管理。
- 只想做协调。

**推荐表达**
- “我不排斥在项目中做技术牵引和辅导，但我更希望评价重心放在业务交付、系统设计和核心代码质量上。”

**面试追问：如果入职后让你带几个人可以吗？**
可以承担项目维度的技术协作和新人辅导，但希望岗位定位仍是业务后端开发，而不是纯管理。

### 10. 如何体现 9 年+ 的成熟度？

**结论先行：**
9 年+ 成熟度体现在能提前识别风险、设计可回滚方案、关注长期维护、用数据证明收益，而不是只会写功能。

**分层展开：**

**技术判断**
- 知道什么时候不用复杂技术。
- 能解释方案取舍。
- 能识别一致性和稳定性风险。

**工程习惯**
- 单测和压测。
- 监控告警。
- 灰度发布。
- 故障预案。

**业务意识**
- 关注用户体验。
- 关注业务指标。
- 能在成本和收益之间平衡。

**协作能力**
- 推动接口对齐。
- 写清楚方案文档。
- review 关键代码。

**面试追问：资深和中级最大的区别是什么？**
资深不仅解决当前问题，还能识别问题背后的系统风险，并设计可持续演进的方案。

## 五、项目深挖高频问题

### 11. 面试官问“你项目里最大的难点是什么”怎么答？

**结论先行：**
不要说“时间紧”这种泛泛困难，要讲技术难点、业务约束和你做出的取舍。

**分层展开：**

**可选难点**
- 高并发下的数据一致性。
- 热点流量治理。
- 跨系统最终一致。
- 数据迁移不停机。
- 复杂状态机。
- 线上性能瓶颈。

**回答结构**
- 难点背景。
- 为什么难。
- 你尝试了哪些方案。
- 最终选择和原因。
- 结果和复盘。

**面试追问：失败过吗？**
可以讲一次受控失败或线上问题，重点放在你如何止血、复盘和改进，而不是回避问题。

### 12. 如何回答“你负责到什么程度”？

**结论先行：**
要明确自己在需求分析、方案设计、核心编码、上线验证、故障处理中的职责边界。

**分层展开：**

**高质量表达**
- 我主导了技术方案。
- 我负责核心模块编码。
- 我推动上下游接口对齐。
- 我设计了监控和灰度。
- 我参与上线和故障复盘。

**避免表达**
- “我们做了一个系统”但说不清自己做了什么。
- 只讲团队成果不讲个人贡献。
- 只讲代码不讲方案和结果。

**面试追问：如果团队其他人也参与了，怎么说？**
客观说明协作边界，突出自己负责的关键决策和模块，不夸大但要讲清贡献。

### 13. 如何回答“如果重做这个项目你会怎么改”？

**结论先行：**
这个问题考察复盘能力。回答要体现你理解方案边界，并能提出更成熟的演进方向。

**分层展开：**

**可回答方向**
- 更早引入监控和压测。
- 数据模型预留扩展。
- 缓存一致性补偿更完善。
- 灰度和回滚更自动化。
- 拆分过重模块。

**注意事项**
- 不要全盘否定原方案。
- 要结合当时约束。
- 说明为什么当时没这么做。

**面试追问：为什么当时没一步到位？**
因为项目有业务窗口、团队资源和风险边界。资深工程师要在当前约束下做合理方案，并保留演进空间。

### 18. 如何回答“你最有含金量的项目难点是什么”？

**推荐选题一：媒体 API 限流与广告创建稳定性**

难点不是简单调用 API，而是多账号、多渠道、多任务并发调用外部 Marketing API，外部平台有动态限流、错误码复杂、状态最终一致、失败后还要能追踪和重试。如果控制不好，会造成广告创建失败、任务状态不一致、运营重复操作。

我的处理方式是：
- 广告创建链路异步化，前台只提交任务，实际创建由任务流推进。
- 基于 Redis ZSET + Lua 做全局滑动窗口限流。
- 对平台破频做动态 QPS 衰减和恢复。
- 任务状态和平台错误日志落 MongoDB，便于重试和排查。
- Kafka 消息带 trace_id、msg_id、msg_type，消费侧做幂等。
- 对关键任务建设监控：成功率、失败原因、重试次数、队列积压、平台错误码分布。

**推荐选题二：ES 查询灰度和同步补偿**

难点是素材/广告任务数据原来在 MongoDB，随着查询条件复杂和数据量增长，需要引入 ES，但不能一次性切换，否则 ES 数据不一致或 DSL 问题会影响运营使用。

我的处理方式是：
- 写链路通过 Kafka 异步同步 ES。
- 查询链路支持 Mongo/ES 双路径灰度。
- 按用户白名单、主列表、keyword 反查维度控制切流。
- 响应中返回查询路径，便于排障。
- fallback 定时扫描未同步数据，Bulk upsert/delete，删除 404 视为成功保证幂等。
- 保留 Mongo 兜底，出现 ES 异常可以快速回切。

### 19. 红果方向常见系统设计题怎么准备？

**题目一：设计一个短剧内容发布系统**

回答骨架：
- 业务对象：剧集、章节、素材、字幕、封面、标签、审核记录、发布记录。
- 写链路：创作者/运营提交内容，进入草稿、审核中、待发布、已发布、下线状态机。
- 读链路：C 端内容详情页、播放页、推荐流读取发布态数据，不能读草稿态。
- 存储：MySQL 存强结构化主数据，MongoDB 存扩展配置，Redis 缓存热点剧集和章节，ES 做内容检索。
- 异步：转码、审核、封面处理、搜索索引、推荐特征更新都通过 MQ 解耦。
- 一致性：发布主状态以 DB 为准，下游通过可靠消息最终一致；失败任务进入补偿表。
- 灰度：按 App 版本、地区、语言、实验分桶控制内容和配置可见性。

**题目二：设计一个短剧素材/广告投放平台**

回答骨架：
- 核心对象：素材、媒体账号、广告任务、Campaign、AdGroup、Ad、像素、落地页、投放工单、效果数据。
- 前台链路：运营创建任务、校验素材和账号、预览、提交。
- 后台链路：任务拆分、限流调用媒体 API、状态同步、失败重试。
- 检索链路：素材和任务列表走 ES，DB/Mongo 作为权威数据源。
- 稳定性：Redis 限流、分布式锁、Kafka 幂等消费、ES fallback、监控报警。
- 数据回流：媒体报表和 BI 数据异步拉取，按任务/素材/剧目聚合，反哺投放决策。

**题目三：设计一个高并发短剧播放页**

回答骨架：
- 读多写少，核心是剧集详情、章节列表、播放地址、推荐结果、用户播放进度。
- 热点剧集使用 Redis 缓存，多级缓存和短 TTL 防止击穿。
- 播放进度、曝光、点击、完播等事件走 MQ/日志链路异步写入。
- 推荐结果可按用户和场景做短时缓存，保证降级时有兜底内容。
- 风险点是热点内容、推荐服务超时、埋点洪峰、缓存失效和下游依赖抖动。

### 20. 如何回答“你带过 11 人，但为什么不继续做组长”？

**推荐回答：**

我在枫叶互动确实承担过后端组长职责，包括需求拆解、排期协调、技术方案评审、核心模块设计和线上质量把控。这段经历让我对项目推进、风险识别和跨团队协作更成熟。  

但我下一阶段更希望做资深后端 IC / 技术骨干，而不是纯管理岗。原因是我更擅长也更有热情的部分，是复杂系统设计、核心链路研发、稳定性治理和技术问题攻坚。带团队的经验会帮助我更好地做方案评审和协作，但我希望主要产出仍然来自技术建设本身。

**重点：**
不要说“管理太累”“容易内耗”。面试里要表达成“阶段性职业选择”和“希望深耕技术影响力”。

**结论先行：**
反问要围绕岗位真实工作、业务挑战、技术栈和团队期望，避免只问福利和晋升。

**可问问题**
- 这个岗位主要负责哪条业务线？
- 当前团队最需要解决的技术或业务挑战是什么？
- 对 9 年+ 后端的期待更偏业务 Owner、架构治理还是团队管理？
- 当前 Go 技术栈和微服务治理体系是怎样的？
- 入职后前三个月希望我解决什么问题？

**面试追问：如果对方强调管理怎么办？**
可以温和澄清：“我可以承担项目推进和技术协作，但我更希望岗位核心仍是一线业务后端和技术方案落地。”
