# 技术｜核心数据结构与存储引擎索引对比

> 这份文档用于承接 Redis、MySQL、MongoDB、Elasticsearch、Kafka 等组件背后的通用数据结构问题。
> 面试时不要孤立背“某组件用了什么结构”，而要从访问模式、存储介质、读写比例、范围查询、空间成本和工程复杂度解释为什么。

## 一、先建立全局视角

### 1. 数据结构选型看什么？

**结论先行：**
数据结构不是越高级越好，而是要匹配访问模式。内存结构更关注 CPU、指针、实现复杂度；磁盘结构更关注 IO 次数、页大小、顺序读写；搜索引擎更关注 term 到文档集合的倒排关系；日志系统更关注顺序追加和快速定位。

| 问题 | 典型结构 |
| --- | --- |
| 快速点查 | Hash、B+Tree、LSM + Bloom |
| 有序范围查询 | B+Tree、SkipList、红黑树 |
| 排名/排行榜 | SkipList、堆、排序数组 |
| 全文检索 | 倒排索引、FST、Posting List |
| 高写入吞吐 | LSM Tree |
| 顺序日志定位 | 稀疏索引 |
| 布尔状态压缩 | Bitmap |
| 近似去重统计 | HyperLogLog、Bloom Filter |

## 二、二叉搜索树和平衡树

### 2. 二叉搜索树 BST 是什么？

**定义：**
二叉搜索树满足左子树小于当前节点，右子树大于当前节点。

```text
        8
      /   \
     3     10
    / \      \
   1   6      14
      / \     /
     4   7   13
```

**特点：**
- 平衡时查找、插入、删除平均 `O(logN)`。
- 如果按有序数据插入，可能退化成链表，最坏 `O(N)`。

```text
1
 \
  2
   \
    3
     \
      4
```

所以工程里通常不用普通 BST，而用平衡树。

### 3. 平衡树、AVL 树、红黑树是一回事吗？

**结论先行：**
不是一回事。平衡树是大类，AVL 树和红黑树都是平衡二叉搜索树的具体实现。

```text
平衡二叉搜索树
├── AVL 树
├── 红黑树
├── Treap
├── Splay Tree
└── 其他变种
```

### 4. AVL 树是什么？

**定义：**
AVL 树要求任意节点左右子树高度差不超过 1，是一种严格平衡的二叉搜索树。

```text
        30
      /    \
    20      40
   /  \       \
 10   25      50
```

插入 `10, 20, 30` 时，普通 BST 会倾斜：

```text
10
  \
   20
     \
      30
```

AVL 会通过旋转变成：

```text
     20
    /  \
   10   30
```

**特点：**
- 查询稳定，树高更低。
- 插入删除可能旋转频繁。
- 适合读多写少、对查询延迟敏感的场景。

### 5. 红黑树是什么？

**定义：**
红黑树也是二叉搜索树，但它通过红/黑颜色规则维持近似平衡，不像 AVL 那么严格。

```text
          20(B)
        /       \
     10(R)      30(R)
    /   \       /   \
  5(B) 15(B) 25(B) 40(B)
```

核心规则可以简化理解：
- 根节点是黑色。
- 红色节点不能连续。
- 任意节点到叶子路径上的黑色节点数量一致。
- 通过变色和旋转保持近似平衡。

| 对比 | AVL 树 | 红黑树 |
| --- | --- | --- |
| 平衡程度 | 更严格 | 相对宽松 |
| 查询性能 | 略好 | 稍弱 |
| 插入删除 | 旋转更多 | 调整更少 |
| 常见场景 | 读多写少 | 读写都多 |
| 工程使用 | 相对少 | Java TreeMap、C++ map/set 等 |

**一句话：**
AVL 更偏查询性能，红黑树更偏综合读写性能。

## 三、跳表 SkipList

### 6. 跳表是什么？

**定义：**
跳表是多层有序链表。底层保存完整数据，上层是稀疏索引。

```text
Level 3: 1 ------------------------------ 30
Level 2: 1 -------------- 15 ------------ 30
Level 1: 1 ----- 8 ------ 15 ----- 22 --- 30
Level 0: 1 -> 3 -> 5 -> 8 -> 10 -> 15 -> 18 -> 22 -> 25 -> 30
```

查找 `22`：

```text
从最高层开始
能往右就往右
右边太大就往下
最后在底层定位到目标
```

**特点：**
- 查找、插入、删除平均 `O(logN)`。
- 范围遍历很自然，找到起点后沿底层链表扫。
- 实现比红黑树简单，不需要旋转和染色。
- Redis ZSet、LevelDB MemTable 都能看到跳表思想。

### 7. 跳表和红黑树怎么对比？

| 对比 | 跳表 | 红黑树 |
| --- | --- | --- |
| 查找 | 平均 `O(logN)` | `O(logN)` |
| 插入删除 | 调整多层指针 | 旋转 + 变色 |
| 范围查询 | 找到起点后顺序扫 | 中序遍历或额外迭代器 |
| 排名 rank | 可维护 span | 要维护 subtree size |
| 实现复杂度 | 较低 | 较高 |
| 最坏情况 | 理论可能退化，概率极低 | 严格平衡 |

Redis ZSet 用 `dict + skiplist`：

```text
dict: member -> score
skiplist: 按 score/member 有序
```

这是空间换时间：dict 负责按 member 点查，skiplist 负责排序、范围和排名。

## 四、B 树、B-树、B+ 树

### 8. B 树和 B-树是一回事吗？

**结论先行：**
是。`B-树` 不是“B 减树”，它就是 `B Tree` 的中文写法，也叫 `B树`。

```text
B 树 = B-树 = B Tree
B+ 树 = B+ Tree
```

### 9. B 树是什么？

**定义：**
B 树是一种多路平衡搜索树。一个节点可以存多个 key，并拥有多个子节点。

```text
             [20 | 40]
           /     |      \
 [5 | 10 | 15] [25 | 30] [45 | 50 | 60]
```

为什么要这样设计？

```text
二叉树一个节点只有一个 key，树高较高。
B 树一个节点放多个 key，树更矮，磁盘 IO 更少。
```

**特点：**
- 内部节点和叶子节点都可以存数据。
- 查询可能在中间节点命中。
- 树高低，适合磁盘页。
- 范围扫描不如 B+ 树自然。

### 10. B+ 树是什么？

**定义：**
B+ 树是 B 树的变种。非叶子节点只存索引，真实数据只存叶子节点；叶子节点之间通过链表相连。

```text
              [20 | 40]
            /     |      \
         [10]   [20|30]  [40|50]
          |       |        |
          v       v        v
叶子: [1,5,10] -> [20,25,30] -> [40,45,50]
```

**特点：**
- 非叶子节点只做索引，一个页能放更多 key。
- 所有查询最终都到叶子节点，路径更稳定。
- 叶子节点有序链表，范围查询非常强。
- MySQL InnoDB 索引典型使用 B+ 树。

### 11. B 树和 B+ 树区别

| 对比 | B 树 / B-树 | B+ 树 |
| --- | --- | --- |
| 数据存储 | 内部节点和叶子节点都可存数据 | 数据只在叶子节点 |
| 非叶子节点 | 索引 + 数据 | 只存索引 |
| 查询路径 | 可能中途命中 | 一定到叶子 |
| 范围查询 | 相对麻烦 | 叶子链表顺序扫描 |
| 单节点索引数 | 较少，因为可能存数据 | 更多，因为只存 key 和指针 |
| 树高 | 相对高一点 | 更矮 |
| 数据库索引 | 有使用，但较少 | MySQL InnoDB 典型使用 |

### 12. 为什么 MySQL InnoDB 用 B+ 树？

**结论先行：**
MySQL 索引主要面向磁盘页和范围查询，B+ 树能减少 IO、支持排序和范围扫描，所以非常适合 OLTP 场景。

```sql
where id = 100
where age between 20 and 30
order by created_at
limit 100
```

B+ 树适合这些查询：
- 点查：从根节点一路定位到叶子。
- 范围：找到起点后沿叶子链表扫描。
- 排序：索引天然有序。
- 分页：顺序读取叶子节点。

为什么不用红黑树？
- 红黑树节点通常只有一个 key，树高更高。
- 指针跳转多，不适合磁盘页。
- 磁盘 IO 成本远大于内存指针调整成本。

## 五、LSM Tree

### 13. LSM 是什么？

**定义：**
LSM 全称 `Log-Structured Merge Tree`，核心思想是把随机写尽量变成顺序写，以提升写入吞吐。

```text
写入
  |
  v
MemTable 内存有序表
  |
  | 满了刷盘
  v
Level 0: SSTable_1  SSTable_2  SSTable_3
  |
  | Compaction 合并
  v
Level 1: SSTable_A
  |
  v
Level 2: SSTable_B
```

LSM 通常由几部分组成：

| 结构 | 作用 |
| --- | --- |
| WAL | 先写日志，防止宕机丢数据 |
| MemTable | 内存有序表，承接写入 |
| Immutable MemTable | 等待刷盘的只读 MemTable |
| SSTable | 磁盘有序不可变文件 |
| Bloom Filter | 判断 key 一定不存在，减少无效读 |
| Sparse Index | 快速定位 SSTable 内部数据块 |
| Compaction | 后台合并、清理旧版本和删除标记 |

### 14. LSM 和 B+ 树怎么对比？

| 对比 | B+ 树 | LSM Tree |
| --- | --- | --- |
| 写入 | 可能随机写、页分裂 | 顺序写为主 |
| 读取 | 路径稳定 | 可能查多层 SSTable |
| 范围查询 | 强 | 也支持，但受多层文件影响 |
| 空间 | 较紧凑 | Compaction 前可能有重复数据 |
| 删除 | 直接改索引/记录 | 写 tombstone，后续合并清理 |
| 适合 | 读多、范围查询、事务型 OLTP | 写多、大吞吐、KV/日志型存储 |

**一句话：**
B+ 树偏读和范围查询稳定，LSM 偏写入吞吐。

## 六、倒排索引、FST、BKD Tree

### 15. 倒排索引是什么？

**定义：**
倒排索引是从词到文档列表的映射。

```text
term -> docID list

"redis" -> [1, 5, 9]
"mysql" -> [2, 5, 8]
"广告"  -> [3, 9, 10]
```

搜索 `redis AND mysql`：

```text
[1,5,9] ∩ [2,5,8] = [5]
```

它适合全文检索，因为全文搜索的核心问题是：

```text
某个词出现在哪些文档里？
多个词怎么求交集？
相关性怎么排序？
```

### 16. ES/Lucene 常见结构

| 结构 | 用途 |
| --- | --- |
| 倒排索引 | term 到 docID 列表 |
| Term Dictionary | 词典，记录有哪些 term |
| FST | 压缩 term dictionary，支持快速查词 |
| Posting List | 某个 term 对应的文档列表 |
| BKD Tree | 数值、日期、地理位置范围查询 |
| Doc Values | 排序、聚合、脚本访问 |
| Segment | 不可变索引段，便于顺序写和合并 |

**为什么 ES 不用 B+ 树做全文检索？**
B+ 树适合按字段值比较和范围扫描；全文检索需要从 term 快速找到文档集合，并做交集、并集、相关性排序，所以倒排索引更合适。

## 七、Kafka 稀疏索引

### 17. Kafka 为什么用稀疏索引？

Kafka 是顺序日志系统，不是通用数据库。每个 partition 被切成多个 segment：

```text
00000000000000000000.log
00000000000000000000.index
00000000000000000000.timeindex
```

`.index` 保存 offset 到物理位置的稀疏映射：

```text
offset -> position

1000 -> 0
1500 -> 4096
2000 -> 8192
```

查找 offset 1800：

```text
1. 在 index 中找到 <= 1800 的最近 offset，例如 1500
2. 跳到 position 4096
3. 顺序扫描到 offset 1800
```

为什么不用 B+ 树？
- Kafka 主要顺序追加写。
- segment 文件不可变或近似追加。
- 稀疏索引空间小，写入成本低。
- 定位到附近后顺序扫描一点点很快。

## 八、组件索引结构对比

### 18. MySQL、MongoDB、ES、Kafka、Redis 都用什么？

| 组件 | 核心索引/类似索引 | 为什么 |
| --- | --- | --- |
| MySQL InnoDB | B+ 树 | 磁盘页友好、树高低、范围查询强、排序分页友好 |
| MongoDB WiredTiger | B-Tree 类结构 | 文档字段索引、复合索引、范围查询和排序 |
| Elasticsearch / Lucene | 倒排索引、FST、BKD Tree、Doc Values | 全文检索、term 查询、数值/地理范围、聚合排序 |
| Kafka | 稀疏 offset index、time index | 顺序日志，低成本定位 segment 位置 |
| Redis ZSet | Hash + SkipList | 内存有序集合，点查、排名、范围遍历综合均衡 |
| RocksDB / LevelDB | LSM Tree | 高写入吞吐、顺序刷盘、后台合并 |
| PostgreSQL | B-Tree、Hash、GIN、GiST、BRIN | 根据查询类型提供多种索引 |

## 九、限流算法结构：固定窗口、滑动窗口、令牌桶与漏桶

### 19. 限流算法为什么也属于数据结构选型？

**结论先行：**
限流本质是在“时间维度上管理请求状态”。不同限流算法背后依赖不同数据结构：固定窗口是计数器，滑动窗口是时间有序集合，令牌桶是状态机，漏桶是队列或调度时间。面试时不要只背算法名，要说明它解决什么问题、用什么状态表示、适合什么工程场景。

| 算法 | 核心数据结构/状态 | 典型实现 | 优点 | 缺点 | 适合场景 |
| --- | --- | --- | --- | --- | --- |
| 固定窗口 | 计数器 | Redis `INCR + EXPIRE`、内存 map | 简单、快、内存少 | 窗口边界可能突刺 | 粗粒度接口保护 |
| 滑动窗口 | 时间有序集合 / 时间队列 | Redis `ZSet + Lua`、内存 deque | 精确统计最近 N 秒 | 高 QPS 下内存成本较高 | 外部 API 精确配额 |
| 令牌桶 | `tokens + lastRefillTime` 状态机 | Go `x/time/rate`、Guava、Redis Lua | 允许突发，长期平均受控 | 需要理解突发容量 | 网关、RPC、第三方 API |
| 漏桶 | 队列 / 下一次可执行时间 | Nginx `limit_req`、任务队列、Redis Lua | 匀速流出，削峰明显 | 增加排队延迟 | 慢下游、短信、邮件、批任务 |
| 并发数限制 | semaphore / 计数器 | Go channel、`errgroup.SetLimit`、Redis running key | 控制同时执行数量 | 不控制 QPS | DB 连接、RPC in-flight 保护 |

### 20. 令牌桶和漏桶怎么对比？

**令牌桶：允许突发，控制长期平均速率**

```text
固定速率生成 token
        │
        ▼
Token Bucket，最多存 capacity 个
        │
        ▼
请求到来先拿 token
        │
        ├── 拿到：放行
        └── 拿不到：拒绝 / 等待 / 降级
```

核心参数：
- `rate`：每秒生成多少 token，决定长期平均速率。
- `capacity`：桶容量，决定最大突发能力。
- `requested`：每次请求消耗多少 token。

例子：`rate=100/s，capacity=500`，表示长期平均 100 QPS，但低峰积攒 token 后，瞬间最多可以承接约 500 个突发请求。

**漏桶：稳定流出，保护慢下游**

```text
请求进入桶 / 队列
      │
      ▼
桶容量有限，满了拒绝
      │
      ▼
按固定速率流出
      │
      ▼
真正调用下游
```

核心参数：
- `capacity`：最大排队数量。
- `leakRate` / `interval`：固定流出速度。
- `maxDelay`：最大可接受等待时间。

例子：`capacity=1000，leakRate=100/s`，表示最多排队 1000 个请求，下游每秒稳定处理 100 个。它不追求瞬间放行，而是把突发流量抹平。

### 21. 主流框架和组件一般用什么限流思路？

| 框架/组件 | 常见限流思路 | 适合解决的问题 |
| --- | --- | --- |
| Go `golang.org/x/time/rate` | 令牌桶 | 单实例 QPS 控制、客户端 SDK 限速 |
| Go channel / semaphore | 并发数限制 | DB 连接、RPC in-flight、批任务并发 |
| Nginx `limit_req` | 漏桶思想，支持 burst | 接入层 IP/URI 粗粒度限流 |
| Envoy / API Gateway | 本地 token bucket + 全局限流服务 | 微服务入口统一治理 |
| Redis | 固定窗口、滑动窗口、令牌桶、漏桶都可实现 | 多实例共享额度、账号/API/租户限流 |
| Sentinel | QPS、并发、热点参数、熔断降级 | Java 服务治理和热点保护 |
| Guava RateLimiter | 令牌桶 / 平滑预热 | Java 客户端本地限速 |
| Resilience4j | RateLimiter、Bulkhead | 限流、隔离、熔断组合治理 |
| MQ / 任务队列 | 漏桶 / 队列削峰 | 把突发写入变成稳定消费 |
| 数据库条件更新 | 原子扣减额度 | 强一致配额、库存、资金类扣减 |

**工程选型口诀：**
- 单实例保护：Go 本地令牌桶 + semaphore。
- 多实例共享额度：Redis 滑动窗口 / 令牌桶 Lua。
- 入口粗粒度防刷：Nginx、Envoy、API Gateway。
- 慢下游削峰：漏桶、队列、MQ。
- 强一致额度：数据库条件更新或事务，不要用普通缓存计数替代。

## 十、面试总结模板

### 22. 如何回答“为什么不同组件索引结构不一样？”

可以这样答：

> 数据结构选型要看访问模式和存储介质。MySQL 是磁盘型 OLTP，核心瓶颈是 IO，所以用 B+ 树降低树高，并通过叶子链表支持范围查询和排序。Redis ZSet 是内存有序集合，需要排名、范围查询和插入删除，跳表实现简单、范围遍历自然。Elasticsearch 是全文检索，核心是 term 到 docID 的倒排索引，数值范围再用 BKD Tree。Kafka 是顺序日志，只需要稀疏 offset/time index 快速定位 segment。LSM 则适合高写入，把随机写转成顺序写。

### 23. 速记口诀

```text
BST 左小右大，但会退化；
AVL 严格平衡，查找强；
红黑树 近似平衡，工程常用；
跳表 多层链表，范围和排名友好；
B树/B-树 多 key 多子节点；
B+树 数据在叶子，叶子有链表；
LSM 内存写入，磁盘有序文件，后台合并；
倒排索引 词到文档；
Kafka 稀疏索引 offset 到 position；
```
