# 技术｜分布式与高并发架构面试题

> 面向 9 年以上 Golang Web 后端业务研发岗位。重点是从故障模型和业务目标推导一致性、幂等、事务、协调、流量治理与容量方案，而不是背 CAP、锁和中间件名词。

---

## 一、分布式系统学习地图与资深回答框架

### 1. 分布式与高并发的知识图谱主线是什么？

**一句话结论：** 分布式系统是在网络、节点和时钟都不完全可靠的条件下，通过冗余、协议和业务约束实现可接受的一致性、可用性、性能与成本。

```text
业务目标：正确性 / 可用性 / 延迟 / 吞吐 / 成本
  -> 故障模型：超时、重复、乱序、分区、崩溃、慢节点
  -> 一致性模型：线性一致、因果、最终一致
  -> 身份与时间：ID、版本、幂等键、逻辑时钟
  -> 协调：Quorum、共识、Leader、Lease、Lock、Fencing
  -> 事务：本地事务、Outbox、TCC、Saga
  -> 流量：缓存、限流、熔断、降级、背压
  -> 高并发设计：读扩展、写扩展、削峰和热点治理
  -> 容量、容灾、可观测性和项目表达
```

### 2. 资深后端如何回答分布式问题？

采用“目标 -> 约束 -> 失败场景 -> 不变量 -> 方案 -> 权衡 -> 验证”的结构。

**示例：如何防止库存超卖？**

- **目标：** 已确认订单的扣减总量不能超过可售库存。
- **约束：** 多实例并发、请求重试、MQ 重复、下游超时。
- **不变量：** `available >= 0`，同一订单只扣一次。
- **方案：** 数据库条件更新或原子脚本完成预扣，订单号唯一约束保证幂等，超时后按订单号查结果；取消通过状态机补偿。
- **权衡：** Redis 预扣吞吐高但需要与数据库对账；DB 条件更新简单但热点行上限较低。
- **验证：** 并发压测、重复/乱序/超时注入、库存账实对账和故障恢复演练。

### 3. 架构题为什么要先定义不变量？

不变量是任何故障下都必须成立的业务规则，例如：

- 钱不能凭空增加或消失。
- 同一订单只能成功支付一次。
- 已售库存不能超过总库存。
- 已发布版本不能被旧事件覆盖。

先定义不变量，才能判断哪些操作必须强一致，哪些状态允许最终一致，哪些重试需要幂等，哪些失败必须人工介入。

### 4. 如何区分功能需求和非功能需求？

- 功能需求：下单、扣库存、发布、查询、取消。
- 非功能需求：峰值 QPS、P99、可用性、RPO/RTO、数据规模、一致性窗口、安全和成本。

高级架构答案必须量化非功能需求。没有 QPS、数据量和 SLA 的“分库分表、加缓存、上 MQ”只是组件罗列。

### 5. 分布式题的通用失败清单是什么？

```text
请求没到
请求到了但响应没回来
请求被执行多次
请求乱序到达
节点崩溃或暂停
网络分区或高延迟
主从切换与旧主恢复
时钟漂移
容量耗尽和级联过载
人工误操作与错误发布
```

每个方案至少回答：检测方式、业务影响、自动恢复、人工恢复和数据校验。

## 二、CAP、BASE、一致性模型、时间与故障语义

### 6. CAP 定理应该如何准确理解？

CAP 讨论的是发生网络分区时，系统无法同时保证线性一致性和所有请求可用：

- C：对外表现得像单副本，读到符合线性顺序的值。
- A：每个到达非故障节点的请求最终得到非错误响应。
- P：即使节点间消息丢失或延迟，系统仍继续处理。

现实分布式系统必须面对 P，因此真正的选择是**分区期间对具体操作偏 C 还是偏 A**，而不是永久给系统贴 CP/AP 标签。

### 7. BASE 与 ACID 是什么关系？

BASE 强调基本可用、软状态和最终一致，是跨服务大规模系统常见工程思想；ACID 描述事务原子性、一致性、隔离性和持久性。

它们不是非此即彼：单个服务内部可用 ACID 保证核心事实，服务间通过事件和补偿实现最终一致。

### 8. 强一致、线性一致、顺序一致和最终一致有什么区别？

- **线性一致：** 每个操作像在调用与返回之间某一时刻原子发生，且尊重真实时间先后。
- **顺序一致：** 所有进程看到相同全序，但不要求尊重真实时间。
- **因果一致：** 有因果关系的操作顺序一致，无关操作可不同序。
- **最终一致：** 停止新写后，各副本最终收敛，不规定收敛前读到什么。

面试中“强一致”必须具体化，不要把所有较强语义都混称强一致。

### 9. 读己之写、单调读和因果一致分别解决什么？

- 读己之写：用户提交后刷新页面能看到自己的修改。
- 单调读：同一会话不会先看到新值再看到旧值。
- 因果一致：回复不会出现在原帖之前，状态 B 不会早于导致 B 的状态 A。

很多业务不需要全局线性一致，只需会话粘滞、版本令牌或因果 Session 即可降低成本。

### 10. 网络超时为什么不等于执行失败？

超时只说明调用方在期限内没有收到响应，可能是：请求未到、服务执行中、已成功但响应丢失、下游提交后进程崩溃。

因此重试前必须判断操作是否幂等；关键写操作要用业务请求号查询最终状态，不能看到 timeout 就直接再扣一次。

### 11. 系统时钟为什么不可靠？

机器时钟会漂移，NTP 校时可能跳变，跨节点时间戳不能天然表示因果顺序。墙上时钟适合展示和粗粒度过期，不适合作为唯一并发正确性依据。

版本控制可使用数据库序列、日志 Offset、Term/Index、Lamport Clock 或业务版本；Lease 还要考虑时钟漂移和暂停。

### 12. Lamport Clock 和 Vector Clock 解决什么？

- Lamport Clock 用逻辑计数建立“若 A 因果先于 B，则 L(A) < L(B)”的顺序，但不能判断并发。
- Vector Clock 记录各参与方版本，可识别先后或并发冲突，但元数据随参与方增加。

业务系统常用更简单的实体版本号、事件序号和单分区顺序来实现同类目标。

### 13. 什么是故障检测器的局限？

在异步网络中，无法仅凭一段时间没收到心跳区分“节点已死”和“节点很慢/网络分区”。超时阈值越短切换越快但误判越多，越长则故障恢复越慢。

因此高可用不是只调心跳，还需多数派、Term、Fencing、幂等和恢复校验防止误判造成双写。

## 三、身份、幂等、去重、顺序、超时与重试

### 14. 什么是幂等，为什么分布式系统必须做幂等？

幂等是同一业务操作执行一次或多次，最终业务效果相同。网络超时、客户端重试、MQ 至少一次投递、进程重启都会制造重复。

幂等不是“接口返回一样”，而是余额、库存、订单状态和外部副作用不被重复改变。

### 15. 幂等键如何设计？

优先使用调用方生成且在业务语义内唯一的 request_id/order_id/event_id：

- 数据库唯一索引阻止重复创建。
- 幂等表记录 key、请求摘要、状态和响应。
- 状态机条件更新限制合法迁移。
- 消费端 Inbox 记录已处理事件。

需要定义作用域、有效期、请求参数不一致时的处理和并发中的“处理中”语义。

### 16. 去重表和唯一索引如何选择？

- 唯一索引最简单可靠，适合结果直接落同一数据库。
- 去重表可记录处理状态和响应，适合复杂操作，但需和业务写在同一本地事务中。
- Redis `SET NX` 可做快速挡板，但过期、故障和双写使其不适合单独承担资金类最终正确性。

### 17. 如何处理请求结果未知？

为写请求提供查询状态接口，以业务 ID 查询：不存在才重试，成功则返回原结果，处理中则稍后查询，失败且可重试才重新执行。

服务端要持久化请求状态，并使“记录请求 + 修改业务数据”处于同一本地事务或原子边界。

### 18. 重试的正确条件是什么？

同时满足：错误是瞬时的、操作幂等、仍在总 deadline 内、下游有恢复可能、系统未过载。

策略使用有限次数、指数退避、随机抖动和 Retry Budget。参数错误、权限错误、业务冲突和资源长期不足不应重试。

### 19. 为什么重试会引发重试风暴？

下游变慢后上游超时重试，流量倍增使下游更慢；多层服务各自重试又产生乘法放大。

治理：只在一个合适层重试、限制尝试次数、共享 deadline、设置 Retry Budget、配合限流熔断，并让下游通过背压/`Retry-After` 明确拒绝。

### 20. Timeout 和 Deadline 有什么区别？

Timeout 是允许本次操作执行多久，Deadline 是整条调用最晚结束时间。跨服务应传播 Deadline，并给网络、业务处理和返回预留预算。

下游发现剩余时间不足应尽早失败，服务端也必须响应 Context Cancellation，停止数据库查询和后台工作。

### 21. 消息顺序如何保证？

先定义顺序范围，通常只要求同一订单、设备或用户内有序，而非全局有序。将相同业务键路由到同一分区，由单消费序列处理；事件携带实体版本，消费端拒绝旧版本。

并发 Worker 必须按 key 串行或分片，不要在取到消息后无约束并行破坏顺序。

### 22. 全局唯一 ID 有哪些方案？

| 方案 | 优点 | 代价 |
|---|---|---|
| 数据库自增/号段 | 简单、趋势递增 | 中心依赖或号段管理 |
| UUID | 本地生成、低冲突 | 较大、索引局部性较差 |
| Snowflake 类 | 趋势递增、吞吐高 | 时钟回拨、节点号管理 |
| Redis/协调服务 | 全局递增 | 额外中心依赖 |

ID 方案要考虑唯一性、趋势、信息泄露、分库路由、长度、时钟回拨和故障降级。

## 四、Quorum、共识、Leader、Lease、锁与 Fencing

### 23. Quorum 读写的基本思想是什么？

在 N 个副本中，写等待 W 个、读查询 R 个。若 `W + R > N`，读写集合至少相交，但是否得到最新值还取决于版本选择、失败和实现细节。

Quorum 是副本交集思想，不自动等于线性一致；还需要处理并发写、旧主、读修复和成员变更。

### 24. 共识算法解决什么问题？

共识使多个节点即使发生故障和消息延迟，也对一系列值或日志顺序达成一致。它常用于选主、元数据、配置、锁服务和复制状态机。

它不直接解决业务跨系统事务，也不能在失去多数时同时保持写可用和一致。

### 25. Raft 的核心流程是什么？

- 节点处于 Follower、Candidate、Leader 状态。
- 选举通过递增 Term 和多数票产生 Leader。
- 客户端写入先进入 Leader Log，再复制到多数节点后提交。
- 已提交日志按顺序应用到状态机。
- 日志匹配和 Term 规则避免旧 Leader 覆盖新历史。

真正工程难点还包括快照、日志压缩、成员变更、磁盘持久化和慢节点。

### 26. 为什么多数派系统通常使用奇数节点？

三个节点可容忍一个故障，五个可容忍两个；四个节点仍需三个多数，只能容忍一个故障。增加偶数节点通常不增加故障容忍，却增加复制成本。

节点应跨独立故障域部署，否则物理上多个节点可能同时失效。

### 27. Leader Election 与业务主节点有什么风险？

网络分区或长 GC 可能使旧 Leader 不知道自己已失去领导权，继续操作外部资源，形成双主副作用。因此仅有“选主成功”不够，外部写操作需要 Term/Epoch 或 Fencing Token 拒绝旧 Leader。

### 28. Lease 与 Lock 有什么区别？

- Lock 表示互斥所有权。
- Lease 是带有效期、需要续约的临时所有权，持有方失联后能自动释放。

Lease 适合服务注册、Leader 和任务租约，但进程暂停、网络延迟和时钟漂移可能让客户端误以为仍持有。执行外部副作用仍应配合 Fencing。

### 29. 分布式锁的正确性要求是什么？

1. 互斥：同一资源同一时刻只有合法持有者。
2. 所有权：只有持有者能释放，value 使用唯一 token。
3. 有界存活：持有者崩溃后锁最终释放。
4. 续约：长任务在确认仍持有时延长 Lease。
5. Fencing：下游拒绝旧持有者。
6. 可观测：记录等待、持有、续约失败和超时。

### 30. Fencing Token 为什么重要？

锁服务每次授予锁时返回单调递增 token。资源服务只接受 token 大于已见 token 的请求。即使旧持有者因暂停后恢复并继续执行，也会被资源层拒绝。

没有资源端校验的 Fencing，锁过期后旧客户端仍可能与新客户端同时写。

### 31. Redis 锁、etcd/ZooKeeper 锁和数据库锁如何选择？

- Redis 锁性能高，适合可补偿、短临界区，需正确 token、过期和续约。
- etcd/ZooKeeper 基于共识与 Lease/临时节点，语义较强但吞吐和运维成本更高。
- 数据库条件更新/行锁最接近业务事实，适合低到中等并发核心数据。
- 能用唯一约束、原子更新、队列串行化解决时，优先避免分布式锁。

### 32. 一致性哈希解决什么问题？

普通取模在节点数变化时会重映射大量 Key。一致性哈希将节点和 Key 放在环上，节点变化只迁移相邻区间；虚拟节点改善分布和异构权重。

它常用于缓存、路由和分片，但仍需处理热点 Key、节点容量差异、复制和迁移期间双读双写。

## 五、分布式事务、最终一致性与补偿

### 33. 分布式事务有哪些主流方案？

| 方案 | 一致性/适用 | 主要代价 |
|---|---|---|
| 2PC/XA | 强原子提交、资源支持 XA | 阻塞、协调者和资源占用 |
| TCC | 高价值、动作可 Try/Confirm/Cancel | 业务侵入、空回滚/悬挂/幂等 |
| Saga | 长事务、多步骤业务 | 中间状态可见、补偿不等于回滚 |
| Outbox + MQ | 跨服务事件最终一致 | 延迟、重复、乱序、对账 |
| 本地消息表 | 基础设施简单 | 扫描、清理和吞吐治理 |

选择依据是不变量、一致性窗口、吞吐、参与方能力、补偿可行性和运维复杂度。

### 34. 2PC 的流程和问题是什么？

Prepare 阶段协调者询问参与者是否能提交，参与者预留资源并记录日志；Commit 阶段根据所有投票决定提交或回滚。

问题是锁和资源持有时间长、协调者故障、网络分区下阻塞、参与者吞吐下降。它适合同构资源和低延迟强原子场景，不适合任意微服务调用链。

### 35. TCC 要解决哪些异常？

- Try 预留资源，Confirm 真正确认，Cancel 释放资源。
- 三个动作都必须幂等。
- 空回滚：Try 未执行但 Cancel 到达，需识别并安全返回。
- 悬挂：Cancel 先到，迟到 Try 不得再次预留。
- Confirm/Cancel 失败要持续重试和人工兜底。

### 36. Saga 的补偿为什么不等于数据库回滚？

每一步本地事务提交后对外可见，失败时通过语义相反的补偿动作恢复业务，例如取消订单、退还额度。期间其他业务可能已经观察并依赖中间状态，补偿也可能失败。

因此需要状态机、幂等、可重试、人工介入和审计，而不能把 Saga 描述成自动 ACID。

### 37. Transactional Outbox 如何工作？

```text
同一本地事务：
  更新业务表
  + 插入 outbox(event_id, aggregate_id, version, payload)

异步 Relay：
  扫描/CDC Outbox -> 发布 Kafka -> 标记或归档
```

它消除“DB 已提交但消息未产生”的窗口。消息仍可能重复，因此 Relay 和 Consumer 都要幂等，Outbox 还需分片、归档和积压监控。

### 38. 消费 MQ 更新数据库如何保证一致？

消费端在同一本地事务中完成：插入 Inbox/幂等记录、修改业务数据、记录待发下游事件。事务成功后才提交 Offset。

如果提交 Offset 失败会重复投递，但 Inbox 阻止重复副作用。业务 Exactly-once 来自幂等和状态约束，不来自口号。

### 39. 如何处理乱序事件？

- 按实体 ID 路由同一分区，尽量保证源端顺序。
- 事件携带实体 version/sequence。
- Consumer 只接受大于当前版本的事件。
- 有缺口时暂存、回查事实源或触发重建。
- 删除使用 Tombstone 或状态版本，防止旧更新复活已删除数据。

### 40. 对账和补偿系统为什么是最终一致性的组成部分？

重试只能处理已知失败，无法发现漏事件、人工改数、程序 Bug 和长期数据漂移。对账应比较业务主键、状态、金额/数量、版本和时间窗口，输出可追踪差异并按风险自动或人工修复。

没有可重建、可对账和可人工处置的最终一致性方案是不完整的。

## 六、缓存、热点、限流、熔断、降级与背压

### 41. Cache Aside 如何工作，主要不一致窗口是什么？

读：先查缓存，未命中查 DB 并回填。写：先更新 DB，再删除缓存。

不一致来自并发回填、删除失败、主从延迟和多级缓存。常用版本、短 TTL、MQ/CDC 失效、延迟双删和对账降低概率，但不能宣称绝对强一致。

### 42. 缓存穿透、击穿和雪崩如何区分？

- 穿透：查询不存在数据，持续打到后端；用参数校验、空值缓存、Bloom Filter。
- 击穿：单个热点 Key 失效，大量请求同时回源；用互斥重建、逻辑过期、请求合并。
- 雪崩：大量 Key 同时失效或缓存整体故障；用 TTL 抖动、多级缓存、限流降级和数据库保护。

### 43. 热点 Key 和热点资源如何治理？

- 识别 Top Key、租户、接口和分区。
- 读热点可本地缓存、复制、拆 Key 或预计算。
- 写热点可分桶、合并写、队列串行化或按业务维度拆分。
- 超级租户可独立资源池或隔离集群。
- 方案必须提供聚合与一致性机制，不能只“随机打散”。

### 44. 常见限流算法如何选择？

- 固定窗口简单但边界突刺。
- 滑动窗口更精确但状态成本更高。
- 漏桶输出平滑，适合保护固定速率下游。
- 令牌桶允许可控突发，最常用于 API。
- 并发限制直接控制在途请求，适合慢依赖。

限流维度包括全局、租户、用户、接口和资源；返回明确错误并支持降级，而非静默丢弃核心请求。

### 45. 本地限流和分布式限流如何选择？

本地限流延迟低、无中心依赖，但实例间只能近似公平；分布式限流可全局精确，但增加网络依赖和热点。

常用组合是入口网关做租户全局配额，服务实例做本地并发与突发保护，核心资源再做细粒度保护。

### 46. 熔断解决什么问题？

当下游错误率、慢调用或并发超过阈值时，熔断器快速失败，避免资源被持续占用和级联故障；半开阶段用少量探测判断恢复。

熔断必须按依赖/方法隔离，配合 deadline、限流、重试预算和降级结果。阈值要基于调用量窗口，低流量偶发错误不应误触发。

### 47. 降级应该如何设计？

- 核心写链路优先，关闭推荐、聚合和装饰信息。
- 搜索失败返回热门或基础列表。
- 非核心同步转异步或延迟处理。
- 只读缓存允许短暂陈旧并标识时间。
- 明确触发条件、恢复条件、数据补偿和产品体验。

降级不是返回空成功；调用方必须知道结果语义是否完整。

### 48. 背压为什么重要？

当下游处理速度低于上游产生速度时，背压让压力在可控边界显现，而不是无限堆积内存、连接和队列。实现包括有界队列、并发信号量、消费速率调节、429/503、暂停拉取和生产端限速。

每个队列都要有容量、等待时间、拒绝策略和监控。

## 七、高并发读写、秒杀、订单、Feed 与计数设计

### 49. 高并发系统的通用拆解方法是什么？

```text
入口：CDN / WAF / 网关 / 限流
  -> 无状态服务：水平扩展 / 连接池 / deadline
  -> 缓存：热点 / 请求合并 / 降级
  -> MQ：削峰 / 异步 / 重放
  -> 存储：索引 / 分片 / 批量 / 幂等
  -> 治理：容量 / 监控 / 演练 / 预案
```

每层都要说明容量上限和失败后压力会流向哪里。

### 50. 秒杀系统如何设计？

1. CDN/本地缓存承载静态信息。
2. 网关按活动、用户、设备限流，验证码和资格校验挡流量。
3. Redis/Lua 原子校验资格、去重和预扣，快速返回排队中。
4. Kafka/RocketMQ 削峰，订单消费者幂等创建订单并落库。
5. 数据库条件更新或预留记录保证最终库存不超卖。
6. 失败释放预扣，超时订单关闭，对账修正 Redis 与 DB。

还要防刷、热点 Key、消息积压、重复下单和活动结束后的容量回收。

### 51. 库存扣减有哪些实现方案？

- DB 条件更新：`UPDATE ... SET stock=stock-n WHERE stock>=n`，正确简单，热点吞吐受限。
- Redis Lua 预扣：吞吐高，需要持久化订单、补偿和对账。
- 分桶库存：降低单点热点，需处理余量碎片和汇总。
- 队列串行化：顺序清晰，延迟和单分区吞吐受限。

最终选择由峰值、正确性、实现成本和可补偿性决定。

### 52. 订单系统如何保证状态正确？

- 用明确状态机约束合法迁移。
- 更新条件包含当前状态和 version。
- 创建使用 order_id 唯一索引。
- 支付、库存、履约通过 Outbox/Saga 最终一致。
- 每次状态变化记录不可变流水。
- 超时关单与支付回调竞争通过条件更新裁决。

### 53. 高并发读系统如何扩展？

依次考虑：浏览器/CDN 缓存、服务本地缓存、Redis、数据库读副本、搜索读模型、预计算和分片。每增加一层缓存都要定义 Key、TTL、失效、回源、穿透和降级。

不要一开始就分库；先优化数据模型、索引、请求合并和热点。

### 54. 高并发写系统如何扩展？

- 缩短同步主链路，只保留必须立即确认的事实。
- 批量、合并写和异步事件降低随机 IO。
- 按业务键分区，保证局部顺序和水平扩展。
- 通过 Shard Key/分库键分散写入。
- 热点计数分桶并周期聚合。
- 所有写入具备幂等、重放和对账能力。

### 55. Feed 流如何设计？

- 写扩散：发布时把内容 ID 推入粉丝 Inbox，读快但大 V 写放大严重。
- 读扩散：读取时聚合关注者 Outbox，写轻但读延迟和计算高。
- 混合：普通用户写扩散，大 V 读扩散，读取时合并、去重和排序。

需要处理游标分页、删除/屏蔽、热点、时间线一致性和缓存。

### 56. 点赞、播放量等计数如何设计？

强准确且低并发可数据库原子更新；高频计数可 Redis 分桶或流式聚合，周期批量落库。用户是否点赞用唯一关系记录，展示总数允许短暂最终一致。

必须定义去重口径、迟到事件、回放、跨天边界和账实对账。

### 57. 短链接服务如何设计？

- 发号器生成唯一短码，映射持久化并缓存热点。
- 跳转服务无状态，先查本地/Redis，再查数据库。
- 访问日志异步进入 Kafka，不阻塞跳转。
- 防止恶意 URL、枚举扫描和热点链接攻击。
- 过期与删除要同步缓存，分析数据单独存储。

## 八、性能、容量、弹性、容灾与成本

### 58. 容量评估的基本公式是什么？

```text
峰值 QPS = 日均请求量 x 峰谷系数 / 有效秒数
服务实例数 = 峰值 QPS / 单实例安全 QPS x 冗余系数
存储 = 日增量 x 保留天数 x 副本/索引系数 + 临时空间
带宽 = 峰值 QPS x 平均请求/响应字节
积压恢复时间 = backlog /（恢复消费速率 - 新增速率）
```

所有单机能力必须来自接近生产的压测，并在 N-1、发布和依赖变慢场景下验证。

### 59. 为什么只看平均值会误判容量？

平均 QPS 掩盖峰值，平均延迟掩盖长尾，平均文档大小掩盖大对象，平均租户掩盖超级租户。容量至少看 P95/P99、最大值、时间分布、Key 分布和故障期间容量。

### 60. Little's Law 如何用于并发估算？

稳态下：

```text
在途请求数 L = 到达率 λ x 平均响应时间 W
```

若 1000 QPS、平均耗时 0.1 秒，平均约 100 个在途请求。下游变慢到 1 秒，在途变成约 1000，连接、Goroutine 和内存随之膨胀，因此 deadline、并发限制和背压是容量设计的一部分。

### 61. 弹性扩容为什么不是瞬时解法？

实例启动、镜像拉取、预热、服务发现、连接池建立、JIT/缓存填充都需要时间。若告警触发时系统已雪崩，扩容可能来不及，还可能给数据库制造连接风暴。

需要基础冗余、提前预测、预热、最大扩容速率和下游容量联动。

### 62. 多可用区和异地多活如何选择？

- 同城多 AZ 通常延迟低，适合多数派和自动故障切换。
- 异地灾备重点是备份、异步复制和受控切换。
- 异地多活要解决流量路由、数据归属、冲突、一致性和回切，成本最高。

先定义 RPO/RTO 和业务是否允许降级，再决定架构，不要为“多活”而多活。

### 63. RPO 和 RTO 分别是什么？

- RPO：最多允许丢失多长时间的数据。
- RTO：故障后允许多长时间恢复服务。

它们直接决定同步/异步复制、备份频率、跨地域方案、演练投入和成本。口头说“高可用”却没有 RPO/RTO 不完整。

### 64. 如何控制架构成本？

按业务等级分配可用性、缓存、副本和灾备；冷热分层、数据归档、自动缩容和容量归属；评估中间件的机器、网络、存储和人力成本。

成熟架构不是组件最多，而是在 SLA 内以可理解、可运维的方式实现目标。

## 九、可观测性、稳定性治理与故障排查

### 65. 分布式系统如何建立可观测性？

- Metrics 看趋势和告警：RED（Rate、Errors、Duration）与 USE（Utilization、Saturation、Errors）。
- Logs 提供结构化事件和业务上下文。
- Traces 串联跨服务关键路径。
- Events 记录发布、扩容、配置和故障切换。
- Business invariants 监控库存、金额、状态和数据差异。

三大信号必须通过 trace_id、request_id、order_id 等关联。

### 66. 线上故障如何快速分层？

```text
是单接口、单租户、单机房还是全局？
  -> 流量是否变化？
  -> 应用 CPU/内存/Goroutine/GC 是否异常？
  -> 连接池、线程池、队列是否饱和？
  -> DB/Redis/MQ/ES 是否变慢？
  -> 网络、DNS、证书和服务发现是否异常？
  -> 最近是否发布、改配置、迁移或重放？
```

先保存证据和止血，再追根因；不要在影响范围未知时直接重启全部实例。

### 67. 如何识别级联故障？

典型信号是下游延迟上升、上游连接和 Goroutine 增长、超时重试放大、线程池/队列饱和，最终多个服务同时不可用。

止血要切断放大链：关闭重试、限流、熔断慢依赖、降级非核心功能、释放卡住资源，并按依赖顺序恢复。

### 68. 如何设计稳定性防线？

1. 设计阶段：SLO、容量、故障模型和降级语义。
2. 开发阶段：deadline、幂等、有界队列和资源隔离。
3. 发布阶段：灰度、自动观测、回滚和配置审计。
4. 运行阶段：监控、值班、预案、容量巡检。
5. 演练阶段：故障注入、备份恢复和数据对账。
6. 复盘阶段：修系统机制，不只要求个人小心。

### 69. 混沌工程应该如何开展？

从明确稳态指标和小故障域开始，注入延迟、超时、实例退出、网络丢包、只读、磁盘不足和消息重复。必须有停止条件、回滚、观察和业务批准。

目的不是制造事故，而是验证假设：限流是否生效、重试是否放大、主从是否切换、数据是否可校验。

### 70. 故障复盘最重要的输出是什么？

除时间线和根因外，还要说明：为什么故障被允许发生、为什么影响被放大、为什么监控没提前发现、为什么恢复这么慢。改进项必须可验证，有负责人、期限和验收指标。

## 十、业务架构与项目实战

### 71. 广告投放任务如何设计分布式执行？

- MySQL/MongoDB 保存任务事实和状态机。
- 调度器使用租约领取分片，任务更新用 version 条件防双执行。
- Kafka 按账户或任务 Key 分区，Worker 幂等执行并记录 event_id。
- 外部媒体 API 使用 deadline、限流、退避和请求号查询结果。
- 状态变化通过 Outbox 传播，失败进入重试/人工处理。
- 指标包括任务延迟、重复率、外部 API 成功率、积压和状态差异。

### 72. 短剧发布链路如何保证最终一致？

发布涉及元数据、媒资、审核、搜索索引和 CDN。核心状态在主库按状态机推进；每一步本地提交后发事件，消费者按内容 ID 和版本幂等处理。搜索和 CDN 允许短暂延迟，失败持续补偿；只有必需步骤完成才将版本标记为可见。

下线事件使用更高版本，防止迟到发布事件让内容重新上线。

### 73. 设备事件高并发接入如何设计？

1. 网关鉴权、限流和请求大小限制。
2. 事件带 device_id、sequence、event_id 和发生时间。
3. Kafka 按 device_id 分区，保证设备内顺序。
4. 消费端批量写存储，event_id 唯一去重。
5. 迟到和缺序事件按业务窗口处理，设备最新状态用版本条件更新。
6. 冷数据归档，查询和统计使用不同读模型。

### 74. 如何设计一个可靠的计费/额度系统？

账本使用不可变流水和唯一业务单号，余额是流水聚合或事务内维护的快照。扣减使用数据库事务、行锁/条件更新保证不为负；外部请求超时按业务单号查询，不直接重扣。跨服务通过 Outbox 传播，定期做流水、余额和外部账单三方对账。

资金类系统优先正确和可审计，不能把 Redis 分布式锁作为唯一真相。

### 75. 如何表达你的 1 年多管理经验但不应聘管理岗？

将其表达为高级 IC 的工程优势：

- 能推动接口契约、容量评审、故障复盘和跨团队协作。
- 能把复杂项目拆成可交付阶段，识别风险并推动闭环。
- 理解排期、人员和业务优先级，但职业选择是继续深耕业务架构与核心研发。
- 项目回答以自己承担的设计、编码、排障和结果为主，管理经验只作为协同补充。

### 76. 如何用两分钟回答分布式架构题？

> 我会先量化业务 QPS、P99、可用性、RPO/RTO 和一致性要求，再定义库存、金额、状态等不可破坏的不变量。因为网络超时不代表失败、消息会重复乱序、节点和时钟会异常，所以写链路首先用业务 ID、唯一约束、版本和状态机保证幂等；单服务内用本地事务，跨服务根据业务选择 Outbox、Saga 或 TCC，并提供重试、对账和人工补偿。高并发上通过缓存、请求合并、MQ 削峰、批量写、分区和热点拆分扩展，但每层都配置 deadline、并发上限、背压、熔断和降级。涉及 Leader、Lease 或锁时还要用 Term/Fencing 防旧持有者。最后用真实压测规划 N-1 容量，通过指标、日志、Trace、业务对账、灰度发布和故障演练验证，而不是只描述理想流程。
