# 技术｜系统架构设计方法论面试题

> 面向 9 年以上 Golang Web 后端业务研发岗位。本篇不从“堆组件”开始，而是从业务目标、不变量和故障模型推导数据、流量、一致性、高可用、容量、安全与演进方案。

---

## 一、系统设计地图与资深回答框架

### 1. 系统架构知识图谱是什么？

```text
Business Goal / User Journey
  -> Scale / SLO / RPO / RTO / Cost
  -> Domain / State Machine / Invariant
  -> API / Data / Read-Write Path
  -> Consistency / Idempotency / Transaction
  -> Cache / MQ / Partition / Search
  -> Availability / Degrade / Disaster Recovery
  -> Security / Observability / Evolution
  -> Capacity / Trade-off / Verification
```

### 2. 系统设计题的标准步骤是什么？

1. 澄清用户、功能范围和成功标准。
2. 量化 QPS、数据量、P99、可用性、RPO/RTO。
3. 定义核心实体、状态机和业务不变量。
4. 画 API、读写链路和数据模型。
5. 找瓶颈并按需要引入缓存、MQ、分片和读模型。
6. 覆盖超时、重复、乱序、故障和降级。
7. 做容量、监控、迁移、成本和验证闭环。

### 3. 为什么先定义不变量？

不变量决定一致性边界，例如库存不为负、同一订单只支付一次、已下线内容不被旧事件重新发布。没有不变量就无法判断事务、锁、幂等和补偿应放在哪里。

### 4. 架构图应该画什么？

先画用户和核心请求，再画服务边界、事实存储、缓存/MQ/搜索等派生组件及数据方向；标出同步/异步、所有者和故障边界。不要一开始画满网关、注册中心等无关盒子。

### 5. 如何表达 Trade-off？

说明选择优化了什么、牺牲了什么、为何在当前约束下值得、触发什么条件会重选。例如异步化提升吞吐和可用性，但引入延迟、重复、对账和运维成本。

## 二、需求澄清、SLO、规模与约束

### 6. 功能需求和非功能需求如何拆？

功能是发布、查询、下单、支付；非功能是峰值、P99、可用性、数据正确性、保留周期、安全和成本。资深答案必须量化后者。

### 7. QPS 如何估算？

从 DAU/业务次数/时段得到平均，再乘峰谷、活动和重试系数；拆读写和不同 API。给区间并说明假设，随后通过压测校正。

### 8. 数据量如何估算？

日新增 = 用户/设备 × 每日事件 × 平均大小；总存储再乘保留、索引、副本、压缩和安全余量。冷热、删除、重建和备份临时空间也要考虑。

### 9. SLI、SLO、SLA 如何区分？

SLI 是测量，SLO 是内部目标，SLA 是对外承诺和责任。设计要明确成功口径、统计窗口和排除项；健康检查成功不等于用户请求可用。

### 10. P99 为什么影响架构？

调用链串行和扇出会累积尾延迟；平均值无法决定 Deadline和容量。需减少同步层级、隔离慢依赖、缓存/预计算并设置端到端预算。

### 11. RPO 和 RTO 如何决定灾备？

RPO决定可丢数据窗口，RTO决定恢复时间；进而选择同步/异步复制、备份频率、跨地域和演练投入。高目标增加延迟、复杂度和成本。

### 12. 如何给业务分级？

按收入/正确性/用户影响分核心写、核心读、非核心体验和离线任务；分别设置 SLO、资源、降级和恢复顺序。不能所有功能都按最高等级建设。

### 13. 如何识别硬约束？

合规、预算、上线时间、现有团队、中间件能力、地域和第三方配额。方案必须在现实约束下可交付，而不是理论最优。

## 三、领域建模、状态机、API 与数据所有权

### 14. 如何识别核心实体和聚合？

从用户旅程、业务语言、生命周期和一致性边界识别。聚合内规则强一致，跨聚合用 ID/事件协作。实体不是数据库表的一比一翻译。

### 15. 状态机为什么重要？

显式列出合法迁移、触发者、幂等、超时和补偿，防止任意更新状态。每次迁移用当前状态+版本条件裁决并记录流水。

### 16. API 设计从什么开始？

从业务命令和查询语义，不从表 CRUD。定义请求 ID、权限、幂等、状态码、分页、异步任务和兼容；限制批量和复杂度。

### 17. 服务边界如何决定？

按业务能力、数据所有权、变化率、SLA 和团队认知拆分。频繁一起改/发布、共享事务和深同步链说明边界可能过细。

### 18. 数据所有权意味着什么？

只有所属服务能直接写事实数据，其他服务通过 API/事件协作。读性能可构建本地冗余模型，但必须标明来源、版本和重建方式。

### 19. 关系模型和文档模型如何选择？

关系、约束、事务和灵活查询优先 MySQL；聚合文档、结构变化和整体访问优先 MongoDB。搜索用 ES、热点用 Redis、事件用 Kafka，各自不越界。

### 20. 范式与反范式如何取舍？

事实模型优先正确和单一来源；读模型可按查询冗余。反范式降低读成本但增加同步和修复，必须有版本、事件和对账。

### 21. 如何设计审计与流水？

关键状态变化写不可变流水，包含业务 ID、前后状态、操作者、请求/事件 ID、时间和原因。流水支持对账、追责和重建，不只打印日志。

## 四、读写链路、缓存、异步化、搜索与数据分发

### 22. 读链路如何逐层优化？

正确索引/模型 -> 只读必要字段 -> 本地/Redis缓存 -> 读副本 -> 搜索/CQRS读模型 -> 预计算/CDN。每层定义命中、失效、回源、降级和一致性。

### 23. Cache Aside 的边界是什么？

读缓存未命中回源，写 DB 后删缓存。并发回填和删除失败会短暂不一致；用 TTL、版本、CDC/MQ失效、热点互斥和对账治理，不能宣称绝对强一致。

### 24. 什么时候异步化？

非必须即时完成、可持久化重试、需削峰或一对多传播时。异步任务必须有状态、幂等、错误、取消、超时、补偿和可观测，不是随手 `go func`。

### 25. MQ 在架构中承担什么？

事件持久化、解耦、削峰和重放。不能替代事实存储；引入重复、乱序、积压和 Schema 演进，Consumer 要幂等并保护下游。

### 26. 什么时候引入 ES？

全文召回、复杂过滤、相关性排序和聚合超过主库能力时。ES 是可重建读模型，主库先提交，通过 Outbox/CDC/MQ同步并对账。

### 27. CQRS 适合什么？

写规则和查询模型差异大、读多且允许最终一致时。代价是双模型、同步和运维；普通 CRUD 不需为模式而使用。

### 28. 热点数据如何治理？

识别 Top Key/租户；读热点复制、本地缓存、请求合并；写热点分桶、批量、队列串行或拆分大租户。拆散后要有聚合和一致性。

### 29. 深分页和大结果如何处理？

Cursor/Seek、限制 Page/Fields、异步导出和对象存储。禁止在线接口无限 Offset 和一次返回全量；导出任务有资源配额。

### 30. 数据分发为什么需要版本？

事件会重复乱序，派生存储用实体版本拒绝旧更新；删除也带更高版本/Tombstone。版本由事实源生成，不能只依赖发生时间。

## 五、一致性、幂等、事务、锁与补偿

### 31. 如何确定一致性级别？

按不变量区分强一致核心事实、读己之写、因果或最终一致。订单金额与库存更强，搜索/报表可延迟。避免全系统统一最高级别。

### 32. 幂等如何设计？

业务 request_id/event_id、唯一索引、状态机条件更新、Inbox和结果查询。定义 Key作用域、参数摘要、处理中和过期；记录与业务写同一事务。

### 33. 网络超时为什么需要状态查询？

超时是结果未知。创建/支付等通过业务 ID查询不存在/处理中/成功/失败，再决定重试；不能见 Timeout 就再执行副作用。

### 34. 跨服务一致性有哪些方案？

2PC、TCC、Saga、Outbox+MQ。根据不变量、参与方、吞吐、补偿可行性和一致性窗口选。多数业务以本地事务+Outbox+幂等+对账为主。

### 35. Outbox 为什么重要？

业务更新和事件记录同一本地事务，Relay/CDC异步发布，消除 DB成功消息未产生窗口。发布和消费仍会重复，必须幂等和治理积压。

### 36. 分布式锁什么时候必要？

需要跨实例互斥且无法用唯一约束、条件更新、分区串行化时。锁需 token、Lease、续约、Owner释放和 Fencing；锁不替代业务幂等。

### 37. 乐观锁和悲观锁如何选择？

冲突少用版本条件，失败重试；冲突高且临界操作短可数据库锁。锁内不调用慢外部服务，监控冲突、等待和死锁。

### 38. 补偿为什么不等于回滚？

跨服务已提交状态可能被观察，补偿是新的业务动作，如退款/释放，不抹去历史。补偿也会失败，需要幂等、重试、人工和审计。

### 39. 对账系统检查什么？

按业务主键比较状态、金额/数量、版本和时间窗口，产生可追踪差异并自动/人工修复。它发现重试看不到的漏事件、Bug和人工改数。

## 六、高可用、过载保护、容灾与恢复

### 40. 高可用由哪些层构成？

多实例/故障域、健康与摘流量、依赖冗余、Deadline/限流/熔断/降级、数据副本/备份、灰度和演练。单纯多副本不是高可用。

### 41. 为什么所有调用都要 Deadline？

无界等待占连接、线程/Goroutine和内存，形成级联故障。端到端预算向下传播，下游响应取消；超时不是重试理由本身。

### 42. 过载保护的顺序是什么？

Admission Control、并发上限、有界队列、背压、限流、熔断和降级。拒绝要早、明确、有业务优先级；无限队列是延迟 OOM。

### 43. 降级如何设计？

核心写优先；读可用缓存/热门/简化结果并标识陈旧；非核心统计关闭；写未完成返回处理中而非假成功。定义触发、恢复和补偿。

### 44. 多可用区如何部署？

实例、副本和主控跨独立故障域；考虑多数派、网络延迟和容量 N-1。避免主副本在同机架，配置 Allocation Awareness。

### 45. 异地灾备和多活如何选择？

先按 RPO/RTO决定备份、异步复制、热备或多活。多活需流量归属、冲突、一致性和回切，成本最高；不是所有业务都需要。

### 46. Backup 和 Replica 有什么区别？

Replica抵抗节点故障，但误删/逻辑污染同步传播；Backup提供历史恢复点。备份必须异地、加密、校验并定期恢复演练。

### 47. 级联故障如何切断？

控制入口、关闭重试放大、熔断慢依赖、降级非核心、释放卡住资源，保护事实存储；从底层依赖向上渐进恢复并限制积压回放。

### 48. 如何设计故障演练？

定义稳态指标、最小故障域和停止条件，注入延迟、超时、实例退出、网络分区、磁盘满、消息重复；验证告警、降级、恢复和数据正确性。

## 七、容量、性能、分片、扩展与成本

### 49. 容量评估基本公式是什么？

```text
实例数 = 峰值 QPS / 单实例安全 QPS × 冗余
并发 ≈ QPS × 平均延迟
存储 = 日增 × 保留 × 索引/副本系数 + 临时余量
带宽 = 峰值 QPS × 平均报文大小
```

能力来自真实混合压测，并验证 N-1、发布和依赖变慢。

### 50. 如何找到系统瓶颈？

负载阶梯中观察哪个资源先饱和：CPU、锁、连接池、DB、Redis、MQ、网络或第三方配额；用 Trace和Queue Wait解释 P99。瓶颈随流量和数据分布移动。

### 51. 什么时候读写分离？

读压力大且可容忍复制延迟，查询已优化后。必须处理写后读、路由、延迟和故障切换；不用于解决慢 SQL 本身。

### 52. 什么时候分库分表？

单库容量/吞吐经模型、索引、归档和读扩展后仍不足，并有稳定分片键时。它引入跨分片查询、事务、全局ID和迁移成本。

### 53. 分片键如何选择？

基数、均匀性、单调热点、查询路由和扩容迁移。最大租户而非平均租户决定倾斜；核心查询最好携带分片键。

### 54. 如何处理超级租户？

资源配额、独立分片/集群、二级分区、按租户限流和成本归属。共享架构必须承认极端租户会破坏平均模型。

### 55. 弹性扩容为什么不是即时救命？

启动、预热、发现和连接池需时间，还会增加下游连接。要基础冗余、预测、慢启动、下游联动和提前限流。

### 56. 如何控制架构成本？

按业务等级设置副本/SLO，冷热分层和归档，优化实例利用率、网络和存储；计算中间件机器与人力。最成熟的方案不是组件最多。

## 八、安全、可观测性、运维与治理

### 57. 威胁建模从哪里开始？

识别资产、信任边界、入口、攻击者和影响；覆盖认证、授权、注入、越权、数据泄露、供应链和拒绝服务。安全要求进入设计而非上线前补。

### 58. 多租户隔离有哪些层次？

应用行级、Schema/库、实例/集群和账号/网络隔离。隔离越强成本越高；无论哪层，租户 ID从可信身份注入并做资源配额。

### 59. 可观测性三支柱如何落地？

Metrics看趋势/告警，Trace定位链路，Logs给事件上下文，再加业务不变量和变更事件。用 Trace/Business ID关联，控制高基数和敏感数据。

### 60. 架构评审检查什么？

目标/约束、不变量、边界、数据、失败场景、容量、安全、观测、灰度/回滚、成本和备选。评审输出决策与 Owner，不只开会。

### 61. 如何治理配置和 Feature Flag？

强类型、范围校验、权限/审计、灰度和回滚；Flag有 Owner和过期，稳定后删除。配置中心故障使用最后快照，不让全部服务启动失败。

### 62. 运行手册包含什么？

告警含义、确认步骤、止血、风险、恢复、数据校验和升级联系人。命令可执行并定期演练；危险重置/删除不是第一步。

### 63. 如何用 SLO 驱动治理？

按用户请求定义 SLI，Error Budget消耗过快时限制高风险发布并治理主要失败来源。避免只用CPU/存活作为服务目标。

## 九、架构演进、迁移、灰度、回滚与决策记录

### 64. 为什么架构演进优于一次重写？

大重写需求仍变化、验证周期长、双系统成本和切换风险高。通过边界、适配、双读/影子和 Strangler小步迁移，每步可验证回滚。

### 65. 数据迁移的完整流程是什么？

Schema兼容 -> 全量稳定游标/断点 -> 增量CDC/事件 -> 版本幂等 -> 对账 -> 双读影子 -> 灰度切流 -> 观察 -> 下线旧链路。

### 66. Expand-Contract 如何用于 Schema 变更？

先新增兼容结构，新旧代码都可运行；回填并切读写；确认旧消费者消失后删除。DDL、应用和数据迁移分发布。

### 67. 灰度发布怎样设计？

按租户、用户、地域或比例；对比新旧错误、P99、资源和业务；自动暂停阈值；状态流程保持版本一致；支持快速回滚和数据副作用处理。

### 68. ADR 记录什么？

背景、约束、备选、决策、后果和复审条件。用于重要选型和边界，帮助后人理解为何而非只看结果。

### 69. 如何识别过度设计？

没有真实规模/独立演进需求却引入分片、微服务、多活和复杂消息编排；团队无法运维；收益无法量化。优先最简单可演进方案，并写触发升级条件。

### 70. 技术债如何进入架构治理？

具体描述风险、业务影响、偿还成本和触发条件，分核心高风险与普通维护；结合需求渐进修复，重大债独立里程碑。不是把所有不喜欢的代码称债。

## 十、系统设计案例与两分钟表达

### 71. 高并发商品详情页如何设计？

CDN/本地/Redis多级读，Cache Aside和热点保护；商品事实在DB，库存价格按一致性独立查询；ES负责搜索不进入交易；缓存故障限流回源并降级；容量按热点、命中和DB保护评估。

### 72. 订单支付链路如何设计？

订单状态机和唯一业务号；本地事务写订单/Outbox；支付请求带幂等号，超时查询状态；回调幂等；库存/履约用Saga/事件；不可变流水和三方对账保证资金正确。

### 73. 内容发布系统如何设计？

元数据、媒资、审核、搜索和CDN分步骤工作流；主库版本是事实，Outbox事件驱动下游；下线高版本防旧事件复活；查询走ES/缓存，核心发布不依赖搜索；支持灰度和回滚。

### 74. 广告投放任务系统如何设计？

任务状态机、账号级限流、持久化队列和按账号/任务分区；Worker幂等，第三方超时按request_id查询；平台适配隔离；失败可重试/人工；指标含排队、平台成功率、429和状态差异。

### 75. 埋点事件平台如何设计？

入口鉴权/限流/大小，事件带device_id、event_id、sequence；Kafka按Key分区，批量落湖/存储，去重与迟到窗口；Schema Registry；冷热分层和重放；在线状态与离线分析拆模型。

### 76. 如何用两分钟回答系统设计题？

> 我先澄清用户旅程和功能边界，量化峰值QPS、数据量、P99、可用性、RPO/RTO和成本，再定义核心实体、状态机与不可破坏的不变量。主链路先用最简单事实存储和正确索引完成，按瓶颈引入缓存、MQ、搜索、读模型或分片，并说明一致性、重复乱序和运维代价。写请求用业务ID、唯一约束和版本幂等，单服务本地事务，跨服务用Outbox、事件、补偿和对账。所有依赖有Deadline、有界并发、限流熔断和业务降级，数据有副本、备份和演练。最后用压测规划N-1容量，以Metrics/Trace/业务不变量观测，通过兼容Schema、灰度和回滚演进，并明确方案何时需要升级。
