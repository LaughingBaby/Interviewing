## 二、Elasticsearch 当前能力全景：能力边界、核心模型与工程用法

> 本模块回答“它能做什么、怎么正确使用”。先掌握能力边界，再进入底层原理。

### 1. ES 的核心对象如何串起来？

**结论先行：**
Elasticsearch 是以 JSON Document 为数据模型、以 Index 组织数据、以 Mapping 定义字段、以 Shard 实现分布式存储、以 Query DSL 完成搜索和分析的近实时搜索引擎。

```text
业务主库 MongoDB/MySQL
          |
          | Kafka / CDC / Outbox
          v
      ES Index
          |
   +------+------+
   v             v
Primary Shard  Replica Shard
   |
   +-- Lucene Segment
   +-- Inverted Index
   +-- Doc Values

查询请求 -> Coordinating Node -> 多个 Shard 查询 -> 结果归并 -> 返回
```

| 对象 | 作用 | 工程关注点 |
|---|---|---|
| Document | 一条 JSON 文档 | 不等于权威业务记录，通常是派生读模型 |
| Index | 文档集合和索引边界 | mapping、生命周期、读写别名 |
| Mapping | 字段类型、分词、索引能力 | 类型一旦错误，常需重建索引 |
| Primary Shard | 数据分片和写入目标 | 分片数影响扩展、并行和查询放大 |
| Replica Shard | 高可用和读扩展 | 增加副本也增加磁盘、复制和恢复成本 |
| Alias | 指向一个或多个 Index | 灰度、重建、无感切换 |
| Query DSL | JSON 查询语言 | 查询类型决定召回、评分和资源消耗 |

### 2. ES 能力、适用场景和边界

| 能力 | 适合场景 | 主要边界 |
|---|---|---|
| 全文检索 | 标题、正文、素材名称、广告描述 | 分词和相关性需要持续调优 |
| 精确过滤 | 状态、账号、媒体、类目、时间 | 字段类型通常应为 keyword/date/numeric |
| 排序 | 相关性、时间、热度、价格 | 深分页和跨分片排序成本高 |
| 聚合 | 运营统计、筛选项、报表分析 | 高基数聚合可能打满 heap |
| 多字段查询 | 关键词 + 过滤 + 权重 | DSL 复杂后要通过 profile 定位 |
| 近实时搜索 | 写入后秒级可见 | 不是强实时、强事务数据库 |
| 重建和重放 | 修复索引、迁移 mapping、重建读模型 | 需要主数据、事件和幂等能力 |

### 3. ES 关键配置和 API 怎么理解？

| 配置/API | 作用 | 生产判断 |
|---|---|---|
| `number_of_shards` | Primary Shard 数量 | 创建前按数据量、并发和生命周期规划 |
| `number_of_replicas` | Replica 数量 | 高可用、读扩展和存储成本的取舍 |
| `refresh_interval` | 写入数据变为可搜索的周期 | 批量导入可临时调大，在线搜索关注可见延迟 |
| `index.translog.durability` | Translog 持久化策略 | 可靠性和写入性能取舍 |
| `index.max_result_window` | `from + size` 最大窗口 | 防止深分页把内存打爆 |
| `search_after` | 游标式分页 | 需要稳定排序，适合实时列表翻页 |
| `PIT` | 固定搜索视图 | 配合 `search_after` 做一致分页 |
| `Bulk API` | 批量写入 | 控制批大小、并发和逐条错误处理 |
| `profile` | 分析查询各阶段耗时 | 排查慢查询，不建议线上默认开启 |
| `reindex` | 重建或迁移文档 | 配合新 Index + Alias 做线上切换 |

### 4. ES 为什么搜索快？

**结论先行：**
ES 搜索快的核心是倒排索引，它把“文档包含哪些词”转成“词出现在哪些文档”，从而快速定位候选文档。

**分层展开：**

**倒排索引**
- 正排：文档 -> 词。
- 倒排：词 -> 文档列表。
- 搜索关键词时直接查词典和倒排列表。

**Lucene segment**
- 数据写入形成不可变 segment。
- 查询可并行搜索多个 segment。
- 后台 merge 合并 segment。

**缓存和并行**
- filter cache。
- filesystem cache。
- 多 shard 并行查询。

**面试追问：为什么 MySQL like 不适合复杂搜索？**
前导模糊 like 难以利用 B+Tree 索引，复杂分词、相关性排序、高亮和聚合也不是 MySQL 的强项。

### 5. text 和 keyword 有什么区别？

**结论先行：**
text 会分词，适合全文检索；keyword 不分词，适合精确匹配、过滤、排序和聚合。

**分层展开：**

**text**
- 经过 analyzer。
- 适合标题、正文、描述。
- 查询通常用 match。

**keyword**
- 整体作为一个 term。
- 适合 id、状态、枚举、标签、手机号。
- 查询通常用 term。

**工程建议**
- 同一字段可用 multi-fields 同时支持 text 和 keyword。
- 排序和聚合不要用 text 字段。
- 精确匹配不要误用 match。

**面试追问：为什么 term 查 text 可能查不到？**
text 字段索引时被分词和归一化，term 查询不会分析查询词，必须精确匹配索引里的 term。

### 6. mapping 设计要注意什么？

**结论先行：**
mapping 是 ES 的索引结构契约，字段类型、分词器、doc_values、nested 等设计会直接影响查询效果和性能。

**分层展开：**

**字段类型**
- text：全文。
- keyword：精确值。
- date：时间。
- numeric：数值。
- boolean：布尔。
- nested：嵌套数组对象。

**设计原则**
- 明确每个字段用于搜索、过滤、排序还是聚合。
- 控制 dynamic mapping，避免字段爆炸。
- 高频过滤字段用 keyword/numeric/date。
- 大文本字段避免无意义聚合排序。

**变更风险**
- 字段类型通常不能直接修改。
- mapping 错了常需要重建索引。
- 要用 alias 支持平滑切换。

**面试追问：dynamic mapping 有什么风险？**
字段自动推断可能类型不符合预期，还可能因大量动态字段造成 mapping explosion，影响集群稳定性。

### 7. object 和 nested 有什么区别？

**结论先行：**
object 会把数组对象拍平，可能导致字段之间匹配错位；nested 会把每个嵌套对象作为独立隐藏文档，保证对象内部关系正确。

**分层展开：**

**object**
- 默认对象类型。
- 数组对象字段会被扁平化。
- 查询多个子字段时可能跨对象误匹配。

**nested**
- 每个子对象独立索引。
- nested query 保证条件命中同一个子对象。
- 查询和存储成本更高。

**选型建议**
- 简单对象用 object。
- 数组对象且需要对象内多条件匹配时用 nested。
- nested 不宜过多，避免文档膨胀。

**面试追问：商品 SKU 属性适合 nested 吗？**
如果需要保证同一个 SKU 的颜色、尺码、库存等条件同时匹配，适合 nested；如果只是展示，不一定需要。

### 8. query 和 filter 有什么区别？

**结论先行：**
query 关注相关性打分，filter 只做是/否过滤，filter 可缓存且性能通常更好。

**分层展开：**

**query**
- 计算 score。
- 用于全文检索。
- match、multi_match 等。

**filter**
- 不计算 score。
- 用于精确过滤。
- term、range、exists 等。
- 更适合缓存。

**工程建议**
- 状态、类目、时间范围放 filter。
- 关键词搜索放 query。
- bool query 组合 must/filter/should/must_not。

**面试追问：为什么过滤条件不要都放 must？**
must 会参与查询上下文和打分，纯过滤条件放 filter 更清晰，也更利于缓存和性能。

### 9. match 和 term 有什么区别？

**结论先行：**
match 会对查询词进行分词分析，term 不分析查询词，直接精确匹配倒排索引中的 term。

**分层展开：**

**match**
- 适合 text 字段。
- 查询词经过 analyzer。
- 用于全文搜索。

**term**
- 适合 keyword、数字、枚举字段。
- 精确匹配。
- 不做分词。

**常见错误**
- 用 term 查 text 原文。
- 用 match 查 keyword 导致语义不清。
- 字段 mapping 不符合查询意图。

**面试追问：搜索订单号用 match 还是 term？**
订单号应建 keyword 字段，用 term 精确匹配。

### 10. ES 相关性评分怎么理解？

**结论先行：**
ES 默认相关性基于 BM25，综合词频、逆文档频率和字段长度等因素计算 score。

**分层展开：**

**影响因素**
- 关键词出现频率。
- 关键词稀有程度。
- 字段长度。
- boost 权重。

**业务调优**
- 标题字段权重大于正文。
- 新鲜度加权。
- 热度加权。
- 类目和用户偏好重排。

**工程实践**
- ES 负责召回和初排。
- 复杂业务排序可在应用层或推荐系统重排。
- 打分逻辑要可解释和可监控。

**面试追问：为什么搜索不是只按关键词匹配？**
因为用户希望更相关、更优质、更新或更符合业务目标的结果，搜索排序通常是文本相关性和业务特征共同作用。

### 11. 中文分词怎么处理？

**结论先行：**
中文没有天然空格分隔，ES 需要合适的中文分词器，并结合业务词典、同义词和搜索分析器优化召回。

**分层展开：**

**常见问题**
- 分词过细导致噪声大。
- 分词过粗导致召回低。
- 新词、品牌词、行业词识别差。

**优化方式**
- 使用中文分词器。
- 维护自定义词典。
- 同义词扩展。
- index analyzer 和 search analyzer 区分。

**工程注意**
- 词典更新要有发布流程。
- 同义词可能扩大召回并降低精度。
- 搜索效果要用样例集评估。

**面试追问：分词器换了为什么要重建索引？**
因为索引阶段已经按旧分词器生成倒排索引，换分词器后历史文档不会自动重新分词。

### 12. shard 和 replica 是什么？

**结论先行：**
shard 是索引的分片单元，用于水平扩展；replica 是主分片副本，用于高可用和读扩展。

**分层展开：**

**primary shard**
- 数据写入的主分片。
- 创建索引时数量通常固定。
- 决定数据水平分布。

**replica shard**
- primary 的副本。
- 提高查询并发。
- primary 故障时可提升。

**工程取舍**
- shard 太少扩展不足。
- shard 太多管理开销大。
- replica 增加读能力但占更多存储和写入成本。

**面试追问：分片越多越好吗？**
不是。过多小 shard 会增加集群状态、文件句柄、内存和调度开销，反而降低稳定性。

### 13. ES 分片数量怎么设计？

**结论先行：**
分片数量要根据数据量、增长速度、查询并发、节点数和生命周期规划，避免单 shard 过大或 shard 过多。

**分层展开：**

**评估维度**
- 单索引数据量。
- 日增量。
- 保留周期。
- 查询 QPS。
- 节点规模。

**常见策略**
- 按时间滚动索引。
- 使用 ILM 管理冷热生命周期。
- 单 shard 控制在合理大小区间。
- 利用 rollover 自动切换。

**工程建议**
- 日志类按天或按容量滚动。
- 商品搜索类按业务索引 + alias 切换。
- 重建索引用 alias 做无感迁移。

**面试追问：索引创建后 primary shard 数能改吗？**
不能直接修改，只能通过 split/shrink 或 reindex 到新索引解决，设计前要评估。

### 14. ES 脑裂是什么？怎么避免？

**结论先行：**
脑裂是集群出现多个 master 导致元数据分叉。现代 ES 通过选主机制降低风险，但仍要合理配置 master 节点和网络。

**分层展开：**

**产生原因**
- 网络分区。
- master 候选节点配置不当。
- 集群状态无法达成多数派。

**治理方式**
- 至少 3 个 dedicated master 候选节点。
- 保证多数派选主。
- 避免 master 和 data 混部导致资源争抢。
- 监控 master 切换。

**工程建议**
- 生产集群角色隔离。
- 网络稳定性重要。
- 不要随意重启多个 master 节点。

**面试追问：为什么推荐 3 个 master 候选节点？**
可以容忍一个节点故障，同时通过多数派避免两个分区各自选主。

### 15. ES heap 为什么重要？

**结论先行：**
ES 依赖 JVM heap 管理查询、聚合、集群状态等内存，同时也依赖操作系统 page cache 缓存 segment，因此 heap 不能盲目调太大。

**分层展开：**

**heap 用途**
- 查询和聚合。
- fielddata。
- 集群状态。
- indexing buffer。

**page cache**
- Lucene segment 依赖文件系统缓存。
- 给 OS 留足内存很重要。

**工程建议**
- heap 一般不超过物理内存一半。
- 避免超过压缩指针阈值。
- 监控 GC、heap 使用率、fielddata、breaker。

**面试追问：ES OOM 常见原因是什么？**
高基数聚合、fielddata、超大查询、mapping explosion、bulk 过大、shard 过多和集群状态膨胀。

## 三、核心能力原理分析：为什么这样设计、代价是什么

> 本模块回答“为什么能做到、哪里会失效”。复杂原理要结合案例和线上问题理解。

### 16. 倒排索引、Term Dictionary 和 Posting List

**结论先行：**
倒排索引是从词项到文档集合的映射，通常还保存词频、位置、偏移量等信息，用于全文检索和相关性计算。

**分层展开：**

**核心结构**
- term dictionary：词典。
- postings list：倒排列表。
- doc id：文档编号。
- term frequency：词频。
- position：词位置。

**查询过程**
- 查询语句分词。
- 查每个 term 的倒排列表。
- 求交集、并集或打分。
- 返回排序后的文档。

**工程影响**
- 分词影响召回。
- mapping 影响索引结构。
- 字段类型选错会导致搜索效果差或性能差。

**面试追问：倒排索引适合范围查询吗？**
ES 支持范围查询，但倒排索引的核心优势是词项检索。大量复杂范围过滤要结合字段类型、BKD tree、filter 和数据规模评估。

### 17. ES 写入流程是什么？

**结论先行：**
ES 写入先路由到 primary shard，写入内存 buffer 和 translog，再复制到 replica，refresh 后才可被搜索。

**分层展开：**

**写入步骤**
- 根据 routing 定位 shard。
- 写 primary shard。
- 写 translog。
- 写入内存 buffer。
- 同步到 replica。
- refresh 生成可搜索 segment。

**关键概念**
- refresh：让数据可搜索。
- flush：持久化并清理 translog。
- merge：合并小 segment。

**工程影响**
- ES 是近实时搜索，不是写入立即可搜。
- refresh interval 越短，搜索实时性越好但写入成本越高。

**面试追问：写入成功后为什么马上搜不到？**
因为写入成功不等于 refresh 完成。文档要等 refresh 后进入可搜索 segment。

### 18. refresh、flush、merge 区别是什么？

**结论先行：**
refresh 让数据可搜索，flush 保障持久化并清理 translog，merge 合并 segment 提升查询效率和回收删除空间。

**分层展开：**

**refresh**
- 内存 buffer 生成 segment。
- segment 进入搜索可见。
- 代价是产生更多小 segment。

**flush**
- 执行 Lucene commit。
- translog 可被裁剪。
- 偏持久化语义。

**merge**
- 多个小 segment 合并成大 segment。
- 清理被删除文档。
- 后台 IO 和 CPU 成本高。

**面试追问：为什么大批量导入时可以调大 refresh_interval？**
减少 refresh 次数和小 segment 生成，提升写入吞吐。导入完成后再恢复 refresh 并手动 refresh。

### 19. ES 如何保证写入可靠性？

**结论先行：**
ES 通过 primary + replica、translog、ack 策略和集群健康机制提高写入可靠性，但它不是强事务数据库。

**分层展开：**

**副本机制**
- 写入 primary。
- 同步到 replica。
- 副本可用于故障恢复和查询扩展。

**translog**
- 记录尚未 flush 的操作。
- 节点异常后可重放恢复。

**ack 策略**
- 等待多少活跃 shard 确认可配置。
- 副本不足时写入可失败或降级。

**工程建议**
- 主数据仍以 DB 为准。
- ES 写入失败要可重试。
- DB 到 ES 同步要有补偿。

**面试追问：ES 数据丢了怎么办？**
如果 ES 是索引库，应从主库重新构建索引；因此必须保留主数据和可重建能力。

### 20. DB 到 ES 数据同步怎么设计？

**结论先行：**
DB 到 ES 同步常用 MQ、binlog CDC 或 outbox，核心是保证可重试、幂等、可补偿和可重建。

**分层展开：**

**同步方案**
- 业务写 DB 后发 MQ 更新 ES。
- 订阅 binlog 同步 ES。
- 本地 outbox 事件投递。
- 定时全量或增量校验。

**一致性问题**
- ES 更新延迟。
- 消息重复。
- 消息乱序。
- ES 写入失败。

**治理方案**
- 文档 version 或 update_time 防乱序覆盖。
- 消费端幂等。
- 失败重试和死信。
- 定期对账和重建索引。

**面试追问：先写 ES 再写 DB 可以吗？**
通常不建议。主数据应先落 DB，ES 是派生索引；先写 ES 会导致搜索结果出现未真正落库的数据。

### 21. ES 分布式查询为什么会有结果归并成本？

**结论先行：**
一次 ES 搜索通常要经过协调节点、多个分片的 Query Phase、结果归并和 Fetch Phase；分片越多、排序窗口越大、聚合越复杂，协调节点和网络归并成本越高。

```text
Client
  |
  v
Coordinating Node
  |
  +-- Shard 0: 查询候选 TopN
  +-- Shard 1: 查询候选 TopN
  +-- Shard 2: 查询候选 TopN
  |
  +-- Query Phase：归并分数/排序/聚合桶
  +-- Fetch Phase：批量拉取完整文档
  v
最终响应
```

**工程影响：**

- `from + size` 会让每个分片准备更多候选结果。
- 跨分片排序需要协调节点保留和比较更多数据。
- 聚合结果需要分片先局部计算，再由协调节点合并。
- 返回完整 `_source` 过大时，Fetch Phase 和网络传输也会成为瓶颈。

### 22. ES 的 Refresh、Translog 和副本确认分别保证什么？

不要把三个概念混在一起：

| 机制 | 主要解决的问题 | 不负责什么 |
|---|---|---|
| Refresh | 文档何时进入可搜索 Segment | 不等于业务事务提交 |
| Translog | 节点异常后恢复尚未 flush 的操作 | 不替代主库和跨系统一致性 |
| Replica | 副本冗余、故障恢复和读扩展 | 不保证外部 DB/ES 同步一致 |
| Flush | 提交 Lucene 索引并推进持久化 | 不代表每条业务事件都已被下游消费 |

### 23. ES 为什么不是事务数据库？

ES 的强项是搜索和分析，不是跨文档、跨索引、跨 MySQL/Mongo 的事务一致性。生产上一般采用：

```text
权威数据落库
    -> Outbox / CDC / Kafka 可靠产生事件
    -> ES Consumer 幂等消费
    -> Bulk Upsert/Delete
    -> 失败重试、死信、对账、全量重建
```

因此，ES 里的文档应当具备可重建能力，不能把 ES 作为唯一业务事实来源。

## 四、Elasticsearch 实战设计：会用、用好、知道边界

> 本模块回答“怎么落到业务设计里”。重点是方案取舍、正确性、性能和扩展。

### 1. ES 深分页为什么慢？怎么优化？

**结论先行：**
ES 深分页慢是因为每个 shard 都要取 `from + size` 条候选结果再全局排序，页数越深成本越高。

**分层展开：**

**问题原因**
- 多 shard 分布式排序。
- 每个 shard 返回大量候选。
- coordinator 汇总排序。
- 内存和 CPU 成本高。

**优化方案**
- `search_after` 做游标分页。
- PIT 保持分页视图。
- scroll 用于离线遍历，不用于用户实时翻页。
- 限制最大翻页深度。

**工程建议**
- C 端搜索一般不允许翻到很深。
- 后台导出用 scroll 或异步任务。
- 排序字段要稳定，加唯一字段兜底。

**面试追问：search_after 为什么要稳定排序？**
如果排序不稳定，下一页可能重复或漏数据。通常用业务排序字段 + `_id` 或唯一字段。

### 2. ES 聚合查询怎么优化？

**结论先行：**
ES 聚合要控制字段基数、数据范围和 shard 数，优先在 keyword/numeric/date 字段上聚合，并避免高基数全量聚合。

**分层展开：**

**优化方式**
- 使用 filter 缩小范围。
- 聚合字段使用 keyword。
- 控制 size。
- 高基数字段慎用 terms 聚合。
- 使用预聚合或离线分析。

**风险**
- fielddata 打爆内存。
- 高基数聚合消耗 heap。
- 大范围日期聚合慢。
- 多层嵌套聚合成本高。

**面试追问：为什么 text 字段聚合危险？**
text 默认不适合聚合，开启 fielddata 会把大量词项加载到内存，可能导致 heap 压力暴涨。

### 3. 如何做 ES 索引重建？

**结论先行：**
索引重建要使用“新 Index + 新 Mapping + Alias 切换”，实现不停机迁移，并保留快速回滚能力。

**标准流程：**

1. 创建新索引和新 mapping。
2. 通过 reindex 或离线任务导入历史数据。
3. 继续消费增量事件，避免重建期间产生数据缺口。
4. 校验文档数量、关键字段、版本和抽样查询结果。
5. 先灰度读新索引，再原子切换 alias。
6. 保留旧索引，确认稳定后再删除。

**风险与治理：**

- 全量导入要限速，避免打满主库和 ES。
- 增量事件要带版本号，防止旧消息覆盖新数据。
- Bulk 必须逐条检查失败结果，不能只看 HTTP 200。
- alias 切换前后要核对读写 alias，避免写入仍落到旧索引。
- 旧索引保留一段时间，出现异常可以快速回滚 alias。

**面试追问：为什么不直接修改旧索引 mapping？**
很多字段类型、分词和索引结构不能安全地直接修改，强行修改会造成查询语义和历史数据不一致；新索引配合 alias 更可控。

### 4. 如何设计商品搜索？

**结论先行：**
商品搜索要把 DB 作为主存储，ES 作为搜索索引，围绕召回、过滤、排序、同步和降级设计。

**分层展开：**

**索引模型**
- 商品 id、标题、类目、品牌。
- 价格、销量、评分、状态。
- 扩展属性。
- 上下架和库存状态。

**查询链路**
- 关键词 match。
- 类目、品牌、价格 filter。
- 业务排序：相关性、销量、新品、价格。
- 聚合筛选项。

**同步链路**
- 商品变更写 DB。
- MQ/binlog 同步 ES。
- update_time/version 防乱序。
- 定期对账。

**降级**
- ES 故障时返回热门商品或类目列表。
- 核心交易不依赖 ES。

**面试追问：商品价格频繁变化怎么办？**
价格可作为独立字段局部更新，但高频变更要控制写入压力；极高频库存不一定实时同步 ES，可搜索召回后再查库存服务二次过滤。

### 5. 如何设计内容搜索？

**结论先行：**
内容搜索要重点处理中文分词、相关性排序、审核状态、时效性、作者权重和数据同步。

**分层展开：**

**索引字段**
- 标题、正文摘要、标签。
- 作者、类目、发布时间。
- 审核状态、可见范围。
- 互动数、质量分。

**搜索效果**
- 标题 boost。
- 标签和类目召回。
- 发布时间衰减。
- 热度和质量加权。

**安全和一致性**
- 只索引审核通过内容。
- 内容下架要及时更新 ES。
- 查询时必要字段做权限过滤。

**面试追问：内容删除后 ES 还搜得到怎么办？**
说明同步链路延迟或失败。要通过删除事件重试、对账任务修复；敏感内容可查询时回源 DB 二次校验。

## 五、线上治理与故障排查：发现问题、定位问题、解决问题

> 本模块回答“线上出问题怎么办”。按“确认影响面 -> 定向止血 -> 采集证据 -> 根因定位 -> 分批恢复 -> 复盘治理”执行。

### ES 生产故障总手册：从告警到恢复

#### 1. 告警后的前 3 分钟看什么？

**应用侧：**

- 接口 QPS、P95/P99、超时率、错误率和 ES span 耗时。
- 查询 DSL、Index、Alias、请求参数、返回文档数和响应大小。
- Bulk 成功率、逐条失败数、重试数、消费 lag 和同步延迟。
- MongoDB/MySQL、Kafka、Redis、对象存储等上下游依赖是否同步异常。

**ES 集群侧：**

- Cluster health、节点存活、Primary/Replica 分配状态。
- CPU、heap、GC、磁盘容量、磁盘 IO、网络和线程池队列。
- 查询/写入拒绝、Circuit Breaker、慢查询、hot threads。
- 分片大小、分片数量、节点热点、Segment Merge 和 refresh 压力。

**时间线：**

- 是否刚发布了 DSL、Mapping、分词词典或查询参数。
- 是否刚执行了 reindex、bulk 导入、索引迁移、扩容或副本调整。
- 是否发生活动流量、批量导出、Kafka 重放、主库同步积压。

#### 2. 根据现象选择止血动作

| 现象 | 先做什么 | 不要盲目做什么 |
|---|---|---|
| 新版本发布后查询变慢/报错 | 关闭新查询开关、回滚版本、切回旧 Index/主库 | 先调大 heap 或增加副本 |
| ES 查询过载导致接口超时 | 限制入口 QPS、关闭非核心聚合、降低返回字段和分页深度 | 让所有请求继续重试 |
| Bulk 写入压垮 ES | 限制 producer/consumer 并发和 batch，暂停低优先级同步 | 继续扩大 Bulk 批次和并发 |
| DB/Mongo 到 ES 积压 | 保证主库稳定，按优先级处理增量，暂停全量重建 | 盲目扩消费者打爆主库 |
| 集群 Yellow | 查看未分配 Replica，处理节点/磁盘/分配规则 | 直接删除索引或强制覆盖数据 |
| 集群 Red | 先确认受影响 Primary 和业务范围，恢复节点或从 Snapshot 恢复 | 直接执行 destructive 操作 |
| heap/GC 飙高 | 限制聚合、深分页、大响应和脚本查询，摘除异常流量 | 只把 JVM heap 无限调大 |
| 磁盘水位过高 | 暂停重放和低优先级写入，按 ILM/retention 清理或扩容 | 直接删除数据目录文件 |

止血原则：

- ES 是搜索读模型，查询异常时可以回源主库、返回降级结果或暂时关闭非核心搜索能力。
- 同步链路异常时优先保护 MongoDB/MySQL，不要为了追 lag 把主库和 ES 同时打垮。
- 重试必须有上限和退避；Bulk 失败要逐条记录，不能用无限重试放大故障。
- 任何 reindex、reset、delete、force merge 操作前都要保存状态、确认范围和回滚路径。

#### 3. 最小证据集和常用命令

```bash
# 集群健康、节点和分片状态
GET /_cluster/health
GET /_cat/nodes?v
GET /_cat/indices?v
GET /_cat/shards?v

# 查看未分配分片的具体原因
GET /_cluster/allocation/explain

# 查看线程池、节点资源和热点线程
GET /_nodes/stats/thread_pool,jvm,fs,indices
GET /_nodes/hot_threads

# 查看索引配置和 Mapping
GET /ad-campaign-*/_settings
GET /ad-campaign-*/_mapping

# 只在定位具体 DSL 时使用 Profile
GET /ad-campaign-*/_search
{
  "profile": true,
  "query": {}
}
```

每次故障至少保存：时间、Index/Alias、节点、Shard、请求 DSL、trace_id、业务参数、错误响应、慢查询、集群指标、发布版本和上下游指标。没有具体 Index、Shard 和请求证据的“ES 挂了”不能直接作为根因。

#### 4. 故障分支一：查询慢或接口超时

**判断路径：**

```text
只有某类 DSL 慢        -> 查询写法、字段类型、分页、聚合
所有查询都慢           -> 集群资源、磁盘、GC、线程池、分片状态
ES span 不慢但接口慢   -> 连接池、网络、序列化、业务层等待
查询结果越翻越慢       -> from + size 深分页
聚合请求单独慢         -> 高基数、范围过大、heap 或 fielddata
```

**操作步骤：**

1. 通过 APM 找到具体 Index、DSL、参数和耗时阶段。
2. 用 `profile` 判断是 query、fetch、排序、聚合还是脚本耗时。
3. 检查是否误用 `wildcard`/`regexp`、深分页、脚本排序和大 `_source`。
4. 检查 Mapping：过滤是否使用 keyword，排序/聚合字段是否支持 Doc Values。
5. 检查 shard 数量、分片大小、节点热点、heap、GC、磁盘 IO 和线程池。
6. 修复后先小流量验证 P99、错误率和节点资源，再扩大流量。

#### 5. 故障分支二：写入慢、Bulk 失败或同步积压

**常见根因：**

- Bulk 批次过大或并发过高。
- refresh 过于频繁。
- Replica 同步、Translog、磁盘 IO 或 Segment Merge 成为瓶颈。
- Mapping 动态字段爆炸。
- Kafka 消费者处理速度超过 ES 或主库承载能力。
- Bulk 返回 HTTP 200，但其中部分 item 实际失败。

**排查和处理：**

1. 区分 producer 发送慢、Kafka lag、consumer 处理慢和 ES Bulk 拒绝。
2. 记录每批文档数、字节数、耗时、成功数和逐条失败原因。
3. 降低并发和批量大小，观察 indexing latency、线程池队列和磁盘 IO。
4. 批量导入场景评估临时调整 refresh 和 replica，完成后恢复并校验。
5. 对 mapping、分词、动态字段和单条文档大小做限制。
6. 失败消息按临时错误/永久错误分流，进入重试或死信，不阻塞正常数据。

#### 6. 故障分支三：Yellow、Red 和分片未分配

**排查顺序：**

1. 查看受影响的 Index、Primary/Replica、节点和分片分配。
2. 调用 `_cluster/allocation/explain` 获取未分配原因。
3. 检查节点是否离线、磁盘是否超过水位、节点角色和分配规则是否冲突。
4. 判断是 Replica 未分配还是 Primary 未分配：Yellow 通常影响冗余，Red 可能影响数据可读性。
5. 恢复节点、释放磁盘或修复分配规则后，等待分片恢复。
6. 只有在确认数据不可恢复时，才按 Snapshot、主库重建或备份恢复方案处理。

#### 7. 故障分支四：heap、GC、Circuit Breaker 或 OOM

**重点怀疑：**

- 高基数 `terms` 聚合。
- text 字段开启 fielddata。
- 深分页、大范围排序和超大响应。
- 脚本查询、复杂 nested 查询。
- shard 过多、Mapping 字段爆炸。
- Bulk 或 reindex 并发过高。

**治理：**

- 限制聚合范围、桶数量、分页窗口和响应字段。
- 使用 keyword/numeric/date 字段进行聚合排序。
- 关闭或限制不可控脚本。
- 将大对象放对象存储，ES 只存检索字段和引用。
- 控制 Bulk/reindex 速率；heap 调整必须结合 page cache、GC 和压测。

#### 8. 故障分支五：ES 与 Mongo/MySQL 数据不一致

```text
主库是否已提交？
  -> 事件是否产生？
     -> Kafka 是否积压或丢事件？
        -> 消费者是否成功？
           -> Bulk 是否部分失败？
              -> 文档版本是否落后？
                 -> 删除事件是否正确处理？
```

修复手段：

- 事件携带业务 ID、操作类型、版本号和 trace_id。
- 消费者回查权威数据源，避免旧消息覆盖新状态。
- Upsert、Delete 和 404 处理保持幂等。
- 对账任务扫描主库与 ES 的数量、版本和关键字段差异。
- 严重不一致时使用新索引全量重建，Alias 灰度切换。

#### 9. 恢复和复盘

1. 先恢复核心搜索和写入，再恢复报表、推荐、重放和全量同步。
2. 逐步放开限流，观察 P99、heap、GC、磁盘、线程池和错误率。
3. 数据同步恢复后，分批消费 lag，防止回灌打满主库或 ES。
4. 确认分片、Replica、Alias、Mapping 和关键查询结果后再结束应急状态。
5. 复盘补齐慢 DSL、分片容量、Mapping 校验、Bulk 逐条失败、对账和故障演练。

**ES 生产排障面试模板：**

> 我会先确认影响范围和时间线，再区分是查询链路、写入同步、集群资源还是数据一致性问题。查询慢就从 DSL、Mapping、分页、聚合、分片和资源逐层定位；写入积压就区分 Kafka lag、消费者处理、Bulk 拒绝和磁盘/副本瓶颈；Yellow/Red 则先看未分配分片和节点水位。止血时保护主库、限流非核心查询、暂停重放或降低同步并发，恢复时分批放量并校验 Alias、分片、数据版本和业务结果，最后补监控、对账和容量治理。

### 1. ES 查询慢怎么排查？

**结论先行：**
查询慢要从 DSL、mapping、分片、segment、缓存、数据量、资源和热点查询逐层排查。

**分层展开：**

**查询层**
- 是否有深分页。
- 是否 wildcard 前导模糊。
- 是否高亮大字段。
- 是否大范围聚合。
- 是否脚本排序。

**索引层**
- mapping 是否合理。
- 字段类型是否正确。
- 是否存在过多 segment。
- refresh/merge 是否繁忙。

**集群层**
- CPU、heap、GC。
- 磁盘 IO。
- shard 是否过多或过大。
- 是否有热点节点。

**工具**
- profile API。
- slow log。
- hot threads。
- cat APIs。

**面试追问：ES CPU 高但 QPS 没涨可能是什么？**
可能是 merge、复杂聚合、脚本查询、GC、热点查询、分片迁移或异常查询导致。

### 2. ES 写入慢怎么排查？

**结论先行：**
写入慢要看 bulk 大小、refresh、replica、translog、merge、mapping、磁盘和线程池拒绝。

**分层展开：**

**写入方式**
- 使用 bulk。
- 控制单批大小。
- 并发写入不要过高。

**索引设置**
- 批量导入调大 refresh_interval。
- 必要时临时降低副本数。
- 避免动态字段爆炸。

**资源瓶颈**
- 磁盘 IO。
- CPU。
- heap。
- indexing thread pool。
- merge backlog。

**面试追问：bulk 越大越好吗？**
不是。bulk 太大会占内存、导致请求超时和线程池压力。要通过压测找到合适批大小。

## 六、组件对比与选型：ES、MySQL、MongoDB、Redis、Kafka

### 1. 这些组件如何分工？

| 组件 | 权威数据/核心能力 | 不适合承担的职责 |
|---|---|---|
| MySQL | 事务、约束、关系查询、权威业务数据 | 复杂全文搜索和大规模相关性排序 |
| MongoDB | 灵活文档模型、业务聚合文档、主数据 | 复杂全文相关性搜索和强事务跨系统同步 |
| Redis | 缓存、热点数据、计数、短期状态 | 大规模持久搜索和复杂条件查询 |
| Kafka | 事件流、异步解耦、重放和削峰 | 低延迟随机查询和权威业务存储 |
| Elasticsearch | 全文搜索、过滤、排序、聚合和搜索读模型 | 强事务主库、跨系统一致性事实来源 |

### 2. 广告投放系统的推荐链路

```text
MySQL/MongoDB：Campaign、AdGroup、Ad、素材权威数据
        |
        v
Kafka：变更事件、异步解耦、重试和重放
        |
        v
Elasticsearch：运营检索、复杂筛选、排序、聚合
        |
        v
Redis：热点列表、短期缓存、限流和任务状态
```

选型时不要说“ES 查询快所以全部数据放 ES”，而要说明：**ES 是可重建的派生读模型，主数据仍由 MySQL/MongoDB 负责，Kafka 负责可靠传播，Redis 负责热点和短期能力。**

## 七、项目实战表达：把 Elasticsearch 讲成项目能力

> 本模块用于项目深挖，把技术点落到业务复杂度、稳定性和工程收益上。

### 1. 枫叶互动素材库为什么要引入 ES？

**结论先行：**
素材库和广告任务列表之所以引入 ES，是因为运营查询条件越来越复杂，既有关键词搜索，又有项目、素材类型、媒体平台、状态、创建人、时间范围等组合筛选。MongoDB/MySQL 可以做基础查询，但复杂检索、排序和多条件过滤继续堆在主库上，会影响主链路稳定性。

**业务场景：**
- 素材名称、短剧项目、素材类型、媒体账号、同步状态检索。
- 广告任务列表按关键词、平台、账号、状态、创建人筛选。
- 运营需要快速定位素材和任务，用于投放复盘和问题排查。

**设计原则：**
- MongoDB/MySQL 是权威数据源。
- ES 是查询加速和检索视图。
- 写链路异步同步，读链路灰度切换。
- 数据不一致时允许回源和补偿。

### 2. Mongo/DB 到 ES 同步链路怎么设计？

**标准回答：**

业务数据变更后，先写权威存储，再通过 Kafka 发送 ES 同步消息。消费者收到消息后，回查权威数据源，构造 ES document，通过 Bulk API 批量 upsert 或 delete。同步失败时记录错误并重试，同时有 fallback 任务定时扫描未同步或版本落后的数据，保证最终一致。

**为什么要回查权威数据源？**
因为消息可能延迟、重复或乱序。如果消息体里带完整数据，旧消息可能覆盖新数据。消费者回查 DB/Mongo，可以拿到当前最新状态。

**删除如何幂等？**
ES delete 遇到 404 可以视为成功，因为目标状态就是“不存在”。这能避免重复删除消息导致任务失败。

### 3. ES 查询灰度迁移怎么讲？

**业务背景：**
广告任务列表原来走 MongoDB。为了提升复杂条件查询性能，需要切到 ES。但 ES 上线初期可能存在 mapping、同步延迟、DSL 逻辑和排序差异，不能一次性全量替换。

**灰度方案：**
- 支持 Mongo/ES 双路径查询。
- 按用户白名单控制是否走 ES。
- 主列表查询和 keyword 反查设置独立开关。
- 响应中返回查询路径，便于客服/研发排障。
- 保留 Mongo 兜底，ES 异常时可快速回切。
- 对比两边结果数量、关键字段和耗时，逐步扩大灰度。

**面试亮点：**
这个方案体现的是工程迁移能力：不是“上 ES 就完了”，而是考虑数据一致性、灰度、回滚、可观测性和用户影响面。

### 4. 素材搜索 mapping 怎么设计？

**示例字段：**
- `resource_id`：keyword，用于精确过滤。
- `resource_name`：text + keyword，支持全文搜索和精确排序。
- `project_id`：keyword。
- `channel`：keyword。
- `resource_type`：keyword。
- `status`：keyword/integer。
- `creator_id`：keyword。
- `created_at`、`updated_at`：date。
- `tags`：keyword 数组。

**设计要点：**
- 精确筛选字段用 keyword，不要误用 text。
- 需要模糊搜索的名称字段可以 text + keyword 多字段。
- 时间字段用 date，便于范围查询和排序。
- 枚举字段控制基数，避免 mapping 爆炸。
- 不要把大 JSON 全量塞进 ES，只放查询和展示需要的字段。

### 5. ES 慢查询如何结合项目排查？

**排查路径：**
1. 看是所有查询慢，还是某些关键词/筛选条件慢。
2. 看 DSL：是否用了 wildcard、regexp、script、深分页、大聚合。
3. 看索引：mapping 是否合理，filter 字段是否 keyword。
4. 看分页：大 offset 改 search_after 或业务游标。
5. 看集群：CPU、heap、GC、磁盘 IO、segment merge、线程池队列。
6. 看数据：是否分片过大、冷热数据未隔离、单 shard 热点。

**项目表达：**
素材和任务列表是运营高频页面，我会先从慢 DSL 和分页方式入手，再看集群资源。如果是业务组合查询越来越复杂，会考虑增加冗余字段、调整 mapping、拆索引或将报表类统计从 ES 中拆出去。
