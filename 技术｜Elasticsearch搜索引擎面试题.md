## 二、Elasticsearch 当前能力全景：能力边界、核心模型与工程用法

> 本模块回答“它能做什么、怎么正确使用”。先掌握能力边界，再进入底层原理。

### 1. ES 为什么搜索快？

**结论先行：**
ES 搜索快的核心是倒排索引，它把“文档包含哪些词”转成“词出现在哪些文档”，从而快速定位候选文档。

**分层展开：**

**倒排索引**
- 正排：文档 -> 词。
- 倒排：词 -> 文档列表。
- 搜索关键词时直接查词典和倒排列表。

**Lucene segment**
- 数据写入形成不可变 segment。
- 查询可并行搜索多个 segment。
- 后台 merge 合并 segment。

**缓存和并行**
- filter cache。
- filesystem cache。
- 多 shard 并行查询。

**面试追问：为什么 MySQL like 不适合复杂搜索？**
前导模糊 like 难以利用 B+Tree 索引，复杂分词、相关性排序、高亮和聚合也不是 MySQL 的强项。

### 2. text 和 keyword 有什么区别？

**结论先行：**
text 会分词，适合全文检索；keyword 不分词，适合精确匹配、过滤、排序和聚合。

**分层展开：**

**text**
- 经过 analyzer。
- 适合标题、正文、描述。
- 查询通常用 match。

**keyword**
- 整体作为一个 term。
- 适合 id、状态、枚举、标签、手机号。
- 查询通常用 term。

**工程建议**
- 同一字段可用 multi-fields 同时支持 text 和 keyword。
- 排序和聚合不要用 text 字段。
- 精确匹配不要误用 match。

**面试追问：为什么 term 查 text 可能查不到？**
text 字段索引时被分词和归一化，term 查询不会分析查询词，必须精确匹配索引里的 term。

### 3. mapping 设计要注意什么？

**结论先行：**
mapping 是 ES 的索引结构契约，字段类型、分词器、doc_values、nested 等设计会直接影响查询效果和性能。

**分层展开：**

**字段类型**
- text：全文。
- keyword：精确值。
- date：时间。
- numeric：数值。
- boolean：布尔。
- nested：嵌套数组对象。

**设计原则**
- 明确每个字段用于搜索、过滤、排序还是聚合。
- 控制 dynamic mapping，避免字段爆炸。
- 高频过滤字段用 keyword/numeric/date。
- 大文本字段避免无意义聚合排序。

**变更风险**
- 字段类型通常不能直接修改。
- mapping 错了常需要重建索引。
- 要用 alias 支持平滑切换。

**面试追问：dynamic mapping 有什么风险？**
字段自动推断可能类型不符合预期，还可能因大量动态字段造成 mapping explosion，影响集群稳定性。

### 4. object 和 nested 有什么区别？

**结论先行：**
object 会把数组对象拍平，可能导致字段之间匹配错位；nested 会把每个嵌套对象作为独立隐藏文档，保证对象内部关系正确。

**分层展开：**

**object**
- 默认对象类型。
- 数组对象字段会被扁平化。
- 查询多个子字段时可能跨对象误匹配。

**nested**
- 每个子对象独立索引。
- nested query 保证条件命中同一个子对象。
- 查询和存储成本更高。

**选型建议**
- 简单对象用 object。
- 数组对象且需要对象内多条件匹配时用 nested。
- nested 不宜过多，避免文档膨胀。

**面试追问：商品 SKU 属性适合 nested 吗？**
如果需要保证同一个 SKU 的颜色、尺码、库存等条件同时匹配，适合 nested；如果只是展示，不一定需要。

### 5. query 和 filter 有什么区别？

**结论先行：**
query 关注相关性打分，filter 只做是/否过滤，filter 可缓存且性能通常更好。

**分层展开：**

**query**
- 计算 score。
- 用于全文检索。
- match、multi_match 等。

**filter**
- 不计算 score。
- 用于精确过滤。
- term、range、exists 等。
- 更适合缓存。

**工程建议**
- 状态、类目、时间范围放 filter。
- 关键词搜索放 query。
- bool query 组合 must/filter/should/must_not。

**面试追问：为什么过滤条件不要都放 must？**
must 会参与查询上下文和打分，纯过滤条件放 filter 更清晰，也更利于缓存和性能。

### 6. match 和 term 有什么区别？

**结论先行：**
match 会对查询词进行分词分析，term 不分析查询词，直接精确匹配倒排索引中的 term。

**分层展开：**

**match**
- 适合 text 字段。
- 查询词经过 analyzer。
- 用于全文搜索。

**term**
- 适合 keyword、数字、枚举字段。
- 精确匹配。
- 不做分词。

**常见错误**
- 用 term 查 text 原文。
- 用 match 查 keyword 导致语义不清。
- 字段 mapping 不符合查询意图。

**面试追问：搜索订单号用 match 还是 term？**
订单号应建 keyword 字段，用 term 精确匹配。

### 7. ES 相关性评分怎么理解？

**结论先行：**
ES 默认相关性基于 BM25，综合词频、逆文档频率和字段长度等因素计算 score。

**分层展开：**

**影响因素**
- 关键词出现频率。
- 关键词稀有程度。
- 字段长度。
- boost 权重。

**业务调优**
- 标题字段权重大于正文。
- 新鲜度加权。
- 热度加权。
- 类目和用户偏好重排。

**工程实践**
- ES 负责召回和初排。
- 复杂业务排序可在应用层或推荐系统重排。
- 打分逻辑要可解释和可监控。

**面试追问：为什么搜索不是只按关键词匹配？**
因为用户希望更相关、更优质、更新或更符合业务目标的结果，搜索排序通常是文本相关性和业务特征共同作用。

### 8. 中文分词怎么处理？

**结论先行：**
中文没有天然空格分隔，ES 需要合适的中文分词器，并结合业务词典、同义词和搜索分析器优化召回。

**分层展开：**

**常见问题**
- 分词过细导致噪声大。
- 分词过粗导致召回低。
- 新词、品牌词、行业词识别差。

**优化方式**
- 使用中文分词器。
- 维护自定义词典。
- 同义词扩展。
- index analyzer 和 search analyzer 区分。

**工程注意**
- 词典更新要有发布流程。
- 同义词可能扩大召回并降低精度。
- 搜索效果要用样例集评估。

**面试追问：分词器换了为什么要重建索引？**
因为索引阶段已经按旧分词器生成倒排索引，换分词器后历史文档不会自动重新分词。

### 9. shard 和 replica 是什么？

**结论先行：**
shard 是索引的分片单元，用于水平扩展；replica 是主分片副本，用于高可用和读扩展。

**分层展开：**

**primary shard**
- 数据写入的主分片。
- 创建索引时数量通常固定。
- 决定数据水平分布。

**replica shard**
- primary 的副本。
- 提高查询并发。
- primary 故障时可提升。

**工程取舍**
- shard 太少扩展不足。
- shard 太多管理开销大。
- replica 增加读能力但占更多存储和写入成本。

**面试追问：分片越多越好吗？**
不是。过多小 shard 会增加集群状态、文件句柄、内存和调度开销，反而降低稳定性。

### 10. ES 分片数量怎么设计？

**结论先行：**
分片数量要根据数据量、增长速度、查询并发、节点数和生命周期规划，避免单 shard 过大或 shard 过多。

**分层展开：**

**评估维度**
- 单索引数据量。
- 日增量。
- 保留周期。
- 查询 QPS。
- 节点规模。

**常见策略**
- 按时间滚动索引。
- 使用 ILM 管理冷热生命周期。
- 单 shard 控制在合理大小区间。
- 利用 rollover 自动切换。

**工程建议**
- 日志类按天或按容量滚动。
- 商品搜索类按业务索引 + alias 切换。
- 重建索引用 alias 做无感迁移。

**面试追问：索引创建后 primary shard 数能改吗？**
不能直接修改，只能通过 split/shrink 或 reindex 到新索引解决，设计前要评估。

### 11. ES 脑裂是什么？怎么避免？

**结论先行：**
脑裂是集群出现多个 master 导致元数据分叉。现代 ES 通过选主机制降低风险，但仍要合理配置 master 节点和网络。

**分层展开：**

**产生原因**
- 网络分区。
- master 候选节点配置不当。
- 集群状态无法达成多数派。

**治理方式**
- 至少 3 个 dedicated master 候选节点。
- 保证多数派选主。
- 避免 master 和 data 混部导致资源争抢。
- 监控 master 切换。

**工程建议**
- 生产集群角色隔离。
- 网络稳定性重要。
- 不要随意重启多个 master 节点。

**面试追问：为什么推荐 3 个 master 候选节点？**
可以容忍一个节点故障，同时通过多数派避免两个分区各自选主。

### 12. ES heap 为什么重要？

**结论先行：**
ES 依赖 JVM heap 管理查询、聚合、集群状态等内存，同时也依赖操作系统 page cache 缓存 segment，因此 heap 不能盲目调太大。

**分层展开：**

**heap 用途**
- 查询和聚合。
- fielddata。
- 集群状态。
- indexing buffer。

**page cache**
- Lucene segment 依赖文件系统缓存。
- 给 OS 留足内存很重要。

**工程建议**
- heap 一般不超过物理内存一半。
- 避免超过压缩指针阈值。
- 监控 GC、heap 使用率、fielddata、breaker。

**面试追问：ES OOM 常见原因是什么？**
高基数聚合、fielddata、超大查询、mapping explosion、bulk 过大、shard 过多和集群状态膨胀。

## 三、核心能力原理分析：为什么这样设计、代价是什么

> 本模块回答“为什么能做到、哪里会失效”。复杂原理要结合案例和线上问题理解。

### 13. 倒排索引是什么？

**结论先行：**
倒排索引是从词项到文档集合的映射，通常还保存词频、位置、偏移量等信息，用于全文检索和相关性计算。

**分层展开：**

**核心结构**
- term dictionary：词典。
- postings list：倒排列表。
- doc id：文档编号。
- term frequency：词频。
- position：词位置。

**查询过程**
- 查询语句分词。
- 查每个 term 的倒排列表。
- 求交集、并集或打分。
- 返回排序后的文档。

**工程影响**
- 分词影响召回。
- mapping 影响索引结构。
- 字段类型选错会导致搜索效果差或性能差。

**面试追问：倒排索引适合范围查询吗？**
ES 支持范围查询，但倒排索引的核心优势是词项检索。大量复杂范围过滤要结合字段类型、BKD tree、filter 和数据规模评估。

### 14. ES 写入流程是什么？

**结论先行：**
ES 写入先路由到 primary shard，写入内存 buffer 和 translog，再复制到 replica，refresh 后才可被搜索。

**分层展开：**

**写入步骤**
- 根据 routing 定位 shard。
- 写 primary shard。
- 写 translog。
- 写入内存 buffer。
- 同步到 replica。
- refresh 生成可搜索 segment。

**关键概念**
- refresh：让数据可搜索。
- flush：持久化并清理 translog。
- merge：合并小 segment。

**工程影响**
- ES 是近实时搜索，不是写入立即可搜。
- refresh interval 越短，搜索实时性越好但写入成本越高。

**面试追问：写入成功后为什么马上搜不到？**
因为写入成功不等于 refresh 完成。文档要等 refresh 后进入可搜索 segment。

### 15. refresh、flush、merge 区别是什么？

**结论先行：**
refresh 让数据可搜索，flush 保障持久化并清理 translog，merge 合并 segment 提升查询效率和回收删除空间。

**分层展开：**

**refresh**
- 内存 buffer 生成 segment。
- segment 进入搜索可见。
- 代价是产生更多小 segment。

**flush**
- 执行 Lucene commit。
- translog 可被裁剪。
- 偏持久化语义。

**merge**
- 多个小 segment 合并成大 segment。
- 清理被删除文档。
- 后台 IO 和 CPU 成本高。

**面试追问：为什么大批量导入时可以调大 refresh_interval？**
减少 refresh 次数和小 segment 生成，提升写入吞吐。导入完成后再恢复 refresh 并手动 refresh。

### 16. ES 如何保证写入可靠性？

**结论先行：**
ES 通过 primary + replica、translog、ack 策略和集群健康机制提高写入可靠性，但它不是强事务数据库。

**分层展开：**

**副本机制**
- 写入 primary。
- 同步到 replica。
- 副本可用于故障恢复和查询扩展。

**translog**
- 记录尚未 flush 的操作。
- 节点异常后可重放恢复。

**ack 策略**
- 等待多少活跃 shard 确认可配置。
- 副本不足时写入可失败或降级。

**工程建议**
- 主数据仍以 DB 为准。
- ES 写入失败要可重试。
- DB 到 ES 同步要有补偿。

**面试追问：ES 数据丢了怎么办？**
如果 ES 是索引库，应从主库重新构建索引；因此必须保留主数据和可重建能力。

### 17. DB 到 ES 数据同步怎么设计？

**结论先行：**
DB 到 ES 同步常用 MQ、binlog CDC 或 outbox，核心是保证可重试、幂等、可补偿和可重建。

**分层展开：**

**同步方案**
- 业务写 DB 后发 MQ 更新 ES。
- 订阅 binlog 同步 ES。
- 本地 outbox 事件投递。
- 定时全量或增量校验。

**一致性问题**
- ES 更新延迟。
- 消息重复。
- 消息乱序。
- ES 写入失败。

**治理方案**
- 文档 version 或 update_time 防乱序覆盖。
- 消费端幂等。
- 失败重试和死信。
- 定期对账和重建索引。

**面试追问：先写 ES 再写 DB 可以吗？**
通常不建议。主数据应先落 DB，ES 是派生索引；先写 ES 会导致搜索结果出现未真正落库的数据。

## 四、Elasticsearch 实战设计：会用、用好、知道边界

> 本模块回答“怎么落到业务设计里”。重点是方案取舍、正确性、性能和扩展。

### 18. ES 深分页为什么慢？怎么优化？

**结论先行：**
ES 深分页慢是因为每个 shard 都要取 `from + size` 条候选结果再全局排序，页数越深成本越高。

**分层展开：**

**问题原因**
- 多 shard 分布式排序。
- 每个 shard 返回大量候选。
- coordinator 汇总排序。
- 内存和 CPU 成本高。

**优化方案**
- `search_after` 做游标分页。
- PIT 保持分页视图。
- scroll 用于离线遍历，不用于用户实时翻页。
- 限制最大翻页深度。

**工程建议**
- C 端搜索一般不允许翻到很深。
- 后台导出用 scroll 或异步任务。
- 排序字段要稳定，加唯一字段兜底。

**面试追问：search_after 为什么要稳定排序？**
如果排序不稳定，下一页可能重复或漏数据。通常用业务排序字段 + `_id` 或唯一字段。

### 19. ES 聚合查询怎么优化？

**结论先行：**
ES 聚合要控制字段基数、数据范围和 shard 数，优先在 keyword/numeric/date 字段上聚合，并避免高基数全量聚合。

**分层展开：**

**优化方式**
- 使用 filter 缩小范围。
- 聚合字段使用 keyword。
- 控制 size。
- 高基数字段慎用 terms 聚合。
- 使用预聚合或离线分析。

**风险**
- fielddata 打爆内存。
- 高基数聚合消耗 heap。
- 大范围日期聚合慢。
- 多层嵌套聚合成本高。

**面试追问：为什么 text 字段聚合危险？**

### 23. ES 查询慢怎么排查？

**结论先行：**
查询慢要从 DSL、mapping、分片、segment、缓存、数据量、资源和热点查询逐层排查。

**分层展开：**

**查询层**
- 是否有深分页。
- 是否 wildcard 前导模糊。
- 是否高亮大字段。
- 是否大范围聚合。
- 是否脚本排序。

**索引层**
- mapping 是否合理。
- 字段类型是否正确。
- 是否存在过多 segment。
- refresh/merge 是否繁忙。

**集群层**
- CPU、heap、GC。
- 磁盘 IO。
- shard 是否过多或过大。
- 是否有热点节点。

**工具**
- profile API。
- slow log。
- hot threads。
- cat APIs。

**面试追问：ES CPU 高但 QPS 没涨可能是什么？**
可能是 merge、复杂聚合、脚本查询、GC、热点查询、分片迁移或异常查询导致。

### 24. ES 写入慢怎么排查？

**结论先行：**
写入慢要看 bulk 大小、refresh、replica、translog、merge、mapping、磁盘和线程池拒绝。

**分层展开：**

**写入方式**
- 使用 bulk。
- 控制单批大小。
- 并发写入不要过高。

**索引设置**
- 批量导入调大 refresh_interval。
- 必要时临时降低副本数。
- 避免动态字段爆炸。

**资源瓶颈**
- 磁盘 IO。
- CPU。
- heap。
- indexing thread pool。
- merge backlog。

**面试追问：bulk 越大越好吗？**
不是。bulk 太大会占内存、导致请求超时和线程池压力。要通过压测找到合适批大小。

## 六、项目实战表达：把 Elasticsearch 讲成项目能力

> 本模块用于项目深挖，把技术点落到业务复杂度、稳定性和工程收益上。

### 25. 枫叶互动素材库为什么要引入 ES？

**结论先行：**
素材库和广告任务列表之所以引入 ES，是因为运营查询条件越来越复杂，既有关键词搜索，又有项目、素材类型、媒体平台、状态、创建人、时间范围等组合筛选。MongoDB/MySQL 可以做基础查询，但复杂检索、排序和多条件过滤继续堆在主库上，会影响主链路稳定性。

**业务场景：**
- 素材名称、短剧项目、素材类型、媒体账号、同步状态检索。
- 广告任务列表按关键词、平台、账号、状态、创建人筛选。
- 运营需要快速定位素材和任务，用于投放复盘和问题排查。

**设计原则：**
- MongoDB/MySQL 是权威数据源。
- ES 是查询加速和检索视图。
- 写链路异步同步，读链路灰度切换。
- 数据不一致时允许回源和补偿。

### 26. Mongo/DB 到 ES 同步链路怎么设计？

**标准回答：**

业务数据变更后，先写权威存储，再通过 Kafka 发送 ES 同步消息。消费者收到消息后，回查权威数据源，构造 ES document，通过 Bulk API 批量 upsert 或 delete。同步失败时记录错误并重试，同时有 fallback 任务定时扫描未同步或版本落后的数据，保证最终一致。

**为什么要回查权威数据源？**
因为消息可能延迟、重复或乱序。如果消息体里带完整数据，旧消息可能覆盖新数据。消费者回查 DB/Mongo，可以拿到当前最新状态。

**删除如何幂等？**
ES delete 遇到 404 可以视为成功，因为目标状态就是“不存在”。这能避免重复删除消息导致任务失败。

### 27. ES 查询灰度迁移怎么讲？

**业务背景：**
广告任务列表原来走 MongoDB。为了提升复杂条件查询性能，需要切到 ES。但 ES 上线初期可能存在 mapping、同步延迟、DSL 逻辑和排序差异，不能一次性全量替换。

**灰度方案：**
- 支持 Mongo/ES 双路径查询。
- 按用户白名单控制是否走 ES。
- 主列表查询和 keyword 反查设置独立开关。
- 响应中返回查询路径，便于客服/研发排障。
- 保留 Mongo 兜底，ES 异常时可快速回切。
- 对比两边结果数量、关键字段和耗时，逐步扩大灰度。

**面试亮点：**
这个方案体现的是工程迁移能力：不是“上 ES 就完了”，而是考虑数据一致性、灰度、回滚、可观测性和用户影响面。

### 28. 素材搜索 mapping 怎么设计？

**示例字段：**
- `resource_id`：keyword，用于精确过滤。
- `resource_name`：text + keyword，支持全文搜索和精确排序。
- `project_id`：keyword。
- `channel`：keyword。
- `resource_type`：keyword。
- `status`：keyword/integer。
- `creator_id`：keyword。
- `created_at`、`updated_at`：date。
- `tags`：keyword 数组。

**设计要点：**
- 精确筛选字段用 keyword，不要误用 text。
- 需要模糊搜索的名称字段可以 text + keyword 多字段。
- 时间字段用 date，便于范围查询和排序。
- 枚举字段控制基数，避免 mapping 爆炸。
- 不要把大 JSON 全量塞进 ES，只放查询和展示需要的字段。

### 29. ES 慢查询如何结合项目排查？

**排查路径：**
1. 看是所有查询慢，还是某些关键词/筛选条件慢。
2. 看 DSL：是否用了 wildcard、regexp、script、深分页、大聚合。
3. 看索引：mapping 是否合理，filter 字段是否 keyword。
4. 看分页：大 offset 改 search_after 或业务游标。
5. 看集群：CPU、heap、GC、磁盘 IO、segment merge、线程池队列。
6. 看数据：是否分片过大、冷热数据未隔离、单 shard 热点。

**项目表达：**
素材和任务列表是运营高频页面，我会先从慢 DSL 和分页方式入手，再看集群资源。如果是业务组合查询越来越复杂，会考虑增加冗余字段、调整 mapping、拆索引或将报表类统计从 ES 中拆出去。

**结论先行：**
索引重建要用新索引 + alias 切换，实现不停机迁移，并保留回滚能力。

**分层展开：**

**流程**
- 创建新索引和新 mapping。
- 全量导入历史数据。
- 增量同步变更。
- 校验文档数量和抽样数据。
- alias 从旧索引切到新索引。
- 保留旧索引一段时间。

**风险**
- 全量导入影响主库。
- 增量乱序。
- alias 切换时读写不一致。
- mapping 设计再次错误。

**治理**
- 导入限速。
- version 防旧数据覆盖新数据。
- 灰度查询。
- 快速回滚 alias。

**面试追问：为什么不用直接改旧索引？**
很多 mapping 变更无法直接修改，直接改风险大；新索引 + alias 是更安全的线上迁移方式。
