## 一、网络与排障总框架

### 1. 资深后端如何回答线上排障题？

**结论先行：**
线上排障要先止血，再定位，再修复，再复盘；定位时按流量、应用、依赖、系统资源和最近变更逐层收敛。

**分层展开：**

**止血**
- 限流。
- 降级。
- 回滚。
- 扩容。
- 隔离异常实例。

**定位**
- 错误率、P99、QPS。
- 日志、指标、Trace。
- DB/Redis/MQ 下游状态。
- CPU、内存、磁盘、网络。
- 最近发布和配置变更。

**修复**
- 临时修复和长期修复分开。
- 核心链路优先。
- 变更要灰度。

**复盘**
- 时间线。
- 根因。
- 影响面。
- 改进项。

**面试追问：排障最忌讳什么？**
最忌讳没有止血就深入分析，或者没有证据直接猜根因。

## 二、TCP 与 HTTP

### 2. TCP 三次握手和四次挥手怎么理解？

**结论先行：**
三次握手用于双方确认收发能力和初始序列号，四次挥手用于双方分别关闭发送方向。

**分层展开：**

**三次握手**
- 客户端 SYN。
- 服务端 SYN+ACK。
- 客户端 ACK。
- 建立可靠连接。

**四次挥手**
- 一方发送 FIN。
- 对方 ACK。
- 对方处理完剩余数据后 FIN。
- 发起方 ACK 并进入 TIME_WAIT。

**工程意义**
- 握手失败影响连接建立。
- 大量 TIME_WAIT 可能来自短连接。
- CLOSE_WAIT 多通常是应用没有关闭连接。

**面试追问：为什么需要 TIME_WAIT？**
为了确保最后 ACK 能被对方收到，并让旧连接的延迟报文在网络中自然消失，避免影响新连接。

### 3. TCP 粘包是什么？

**结论先行：**
TCP 是字节流协议，没有消息边界。所谓粘包是应用层没有正确定义和解析消息边界。

**分层展开：**

**原因**
- TCP 只保证字节有序可靠。
- 多个小包可能合并。
- 一个应用消息可能拆成多段。

**解决方案**
- 固定长度。
- 分隔符。
- 长度前缀。
- 应用层协议，如 HTTP、Protobuf framing。

**面试追问：UDP 有粘包问题吗？**
UDP 是数据报协议，保留消息边界，但不保证可靠、有序和不丢失。

### 4. HTTP keep-alive 有什么作用？

**结论先行：**
keep-alive 通过复用 TCP 连接减少握手成本和延迟，是高并发 HTTP 服务的基础优化。

**分层展开：**

**优点**
- 减少 TCP 握手。
- 降低延迟。
- 减少端口和连接创建成本。

**风险**
- 空闲连接占资源。
- 连接池配置不当会导致连接耗尽。
- 长连接故障要及时摘除。

**工程建议**
- 客户端连接池设置最大空闲连接。
- 服务端设置 idle timeout。
- 监控连接数和等待时间。

**面试追问：连接池越大越好吗？**
不是。连接池过大会压垮下游，过小会排队。要按 QPS、耗时、实例数和下游承载计算。

### 5. HTTP/1.1、HTTP/2、HTTP/3 有什么区别？

**结论先行：**
HTTP/1.1 以文本协议和连接复用为主，HTTP/2 引入多路复用和二进制帧，HTTP/3 基于 QUIC 改善队头阻塞和连接迁移。

**分层展开：**

**HTTP/1.1**
- keep-alive。
- 管线化实践较少。
- 同连接上仍有队头阻塞。

**HTTP/2**
- 二进制分帧。
- 多路复用。
- header 压缩。
- 单 TCP 连接上 TCP 层仍可能队头阻塞。

**HTTP/3**
- 基于 QUIC/UDP。
- 改善 TCP 队头阻塞。
- 支持连接迁移。

**面试追问：gRPC 为什么常用 HTTP/2？**
HTTP/2 支持多路复用、流式传输和二进制帧，适合高效 RPC 调用。

## 三、Linux 资源排查

### 6. CPU 飙高怎么排查？

**结论先行：**
CPU 飙高先确认是用户态、内核态、IO wait 还是 steal，再结合进程、线程和应用 profile 定位。

**分层展开：**

**系统层**
- `top` 看整体 CPU。
- `mpstat` 看各核。
- `pidstat` 看进程和线程。
- `perf` 看内核和用户态热点。

**应用层**
- Go 服务抓 pprof。
- 看 QPS、P99、错误率。
- 查最近发布。

**常见原因**
- 计算热点。
- GC 压力。
- 锁竞争。
- 日志过量。
- 重试风暴。

**面试追问：CPU user 高和 sys 高分别说明什么？**
user 高通常是应用计算多；sys 高可能是系统调用、网络、锁、文件 IO 或内核态开销高。

### 7. 内存高怎么排查？

**结论先行：**
内存高要区分进程堆内存、缓存、RSS、泄漏、mmap 和系统 page cache。

**分层展开：**

**系统层**
- `free` 看整体内存。
- `top` 看进程 RSS。
- `pmap` 看进程内存映射。
- `vmstat` 看 swap。

**应用层**
- Go heap profile。
- goroutine profile。
- 缓存大小。
- 大对象持有。

**常见原因**
- 本地缓存无上限。
- goroutine 泄漏。
- slice 持有大数组。
- 请求堆积。
- 内存碎片。

**面试追问：Linux free 里 available 少就一定危险吗？**
要结合 buffer/cache 和 available 看。Linux 会用空闲内存做 page cache，不代表都被应用占满。

### 8. 磁盘 IO 高怎么排查？

**结论先行：**
磁盘 IO 高要看读写吞吐、IOPS、await、util 和具体进程，再判断是日志、数据库、临时文件还是持久化任务。

**分层展开：**

**系统命令**
- `iostat` 看磁盘延迟和利用率。
- `iotop` 看进程 IO。
- `df` 看空间。
- `du` 定位目录。

**常见原因**
- 日志暴增。
- MySQL 慢查询和刷盘。
- Redis AOF/RDB。
- 大文件上传下载。
- 临时文件未清理。

**治理**
- 日志采样和异步。
- 磁盘隔离。
- 限制批任务。
- 告警和清理策略。

**面试追问：磁盘 util 100% 一定代表吞吐满了吗？**
不一定。可能是单盘处理小 IO 延迟高，吞吐不大但繁忙度满了。

### 9. 网络问题怎么排查？

**结论先行：**
网络问题要从 DNS、连通性、延迟、丢包、带宽、连接数和应用超时逐层排查。

**分层展开：**

**基础检查**
- DNS 解析。
- ping 延迟和丢包。
- telnet/nc 端口连通。
- traceroute 路径。

**连接状态**
- `ss -s` 看总览。
- `ss -antp` 看连接状态。
- TIME_WAIT、CLOSE_WAIT、SYN_SENT。

**抓包**
- tcpdump。
- Wireshark 分析。
- 看重传、RST、窗口大小。

**面试追问：CLOSE_WAIT 很多说明什么？**
通常说明对端已关闭连接，但本地应用没有及时 close，可能是代码连接泄漏。

## 四、线上故障场景

### 10. 接口突然大量超时怎么排查？

**结论先行：**
大量超时先确认影响范围和止血，再沿入口、应用、下游、资源和变更定位。

**分层展开：**

**影响范围**
- 单接口还是全站。
- 单机房还是全区域。
- 部分用户还是全部用户。

**入口层**
- QPS 是否异常。
- 网关是否排队。
- 限流是否触发。

**应用层**
- goroutine 是否暴涨。
- 连接池是否耗尽。
- CPU/GC 是否异常。

**依赖层**
- DB 慢 SQL。
- Redis 慢查询。
- MQ 积压。
- RPC 下游超时。

**面试追问：为什么要先看最近变更？**
线上故障高比例与发布、配置、流量切换、数据任务有关，最近变更能快速缩小范围。

### 11. 502、503、504 分别可能是什么？

**结论先行：**
502 通常是网关从上游收到无效响应，503 表示服务不可用，504 表示网关等待上游超时。

**分层展开：**

**502**
- 上游进程崩溃。
- 连接被重置。
- 协议错误。

**503**
- 服务实例不可用。
- 熔断或限流。
- 上游没有健康实例。

**504**
- 上游响应超时。
- 下游依赖慢。
- 应用线程或连接池阻塞。

**面试追问：504 一定是网关问题吗？**
不是。网关只是报超时的位置，根因常在应用或更下游依赖。

### 12. 如何做故障复盘？

**结论先行：**
复盘不是追责，而是还原事实、定位根因、沉淀机制，防止同类问题再次发生。

**分层展开：**

**时间线**
- 故障开始。
- 告警触发。
- 处理动作。
- 恢复时间。

**影响面**
- 用户范围。
- 接口和业务损失。
- 错误率和时长。

**根因**
- 直接原因。
- 深层机制缺陷。
- 为什么监控没提前发现。

**改进**
- 代码修复。
- 监控告警。
- 发布流程。
- 压测和预案。

**面试追问：复盘里最重要的产出是什么？**
可落地的改进项和 owner，而不是一份漂亮文档。

### 14. 广告创建接口突然大量超时怎么排查？

**先判断影响面：**
- 是所有广告创建都超时，还是某个平台/账号/接口超时？
- 是前台 HTTP 超时，还是后台异步任务超时？
- 是发布后出现，还是外部媒体 API 抖动？

**排查路径：**
1. 看接口 QPS、P95/P99、错误率、超时比例。
2. 看 trace，确认耗时卡在 DB、Redis、Kafka、ES，还是外部媒体 API。
3. 看媒体平台错误码和限流返回。
4. 看 Redis 限流命中数和 forbidden 标记。
5. 看 Kafka 是否积压，消费者是否异常。
6. 看数据库连接池、慢 SQL、锁等待。

**应急处理：**
- 对异常媒体平台临时降级或暂停自动投放。
- 降低账号 QPS，延迟重试。
- 前台改为提交任务成功，后台慢慢执行。
- 对失败任务保留状态，不让运营重复创建。

**复盘关注：**
是否缺少平台级熔断、是否限流阈值过高、是否监控没有按平台/账号维度拆分。

### 15. MQ 消费积压怎么排查？

**典型场景：**
广告创建、素材同步、ES 更新、设备事件消费都可能出现 MQ 积压。

**排查步骤：**
- 看 topic 和 consumer group 的 lag。
- 判断是所有 partition 积压，还是个别 partition 积压。
- 看消费者日志：是否大量失败重试。
- 看单条消息处理耗时：外部 API、DB、ES、Redis 哪个慢。
- 看消费者实例数和分区数是否匹配。
- 看消息 key 是否导致分区倾斜。

**应急方案：**
- 临时扩容消费者。
- 关闭非核心消费逻辑，例如通知、低优先级同步。
- 失败消息转入重试/死信，不阻塞主队列。
- 批量消费和批量写入。
- 如果下游被打满，不能盲目扩容消费者，要先保护下游。

**项目表达：**
360 设备事件消费里，突发流量导致积压时，可以通过多 topic 隔离和拆分 App 推送逻辑来保护主消费链路。枫叶互动素材同步和 ES 更新也可以用类似思路，区分主链路和补偿链路。

### 16. Redis 超时或 CPU 飙高怎么排查？

**排查路径：**
1. 看 Redis 慢查询，确认是否有大 key、复杂命令、Lua 脚本耗时。
2. 看 ops、CPU、连接数、内存、网络流量。
3. 看业务发布，是否新增高频 key 或大范围 scan。
4. 看 hot key，是否某个广告账号、热门素材、热门剧集被集中访问。
5. 看客户端连接池，是否连接泄漏或超时时间过短。

**项目场景：**
- 媒体 API 限流使用 ZSET + Lua，如果 key 维度过粗，会让某个 key 成为热点。
- 缓存版本指纹如果一次读取账号过多，会增加 Redis 压力。
- 热门短剧/广告配置如果 TTL 同时过期，可能导致缓存击穿。

**处理方案：**
- 拆 key 维度，避免单 key 过热。
- Lua 脚本控制复杂度和窗口长度。
- 热点缓存加随机 TTL、互斥重建、本地短 TTL 缓存。
- 对 Redis 异常设置降级策略，保护 DB。

### 17. ES 数据同步延迟怎么排查？

**现象：**
运营更新了素材或广告任务，但列表搜索不到，或者搜索结果还是旧数据。

**排查路径：**
1. 看业务主库/Mongo 是否已经更新。
2. 看 Kafka 同步消息是否发送成功。
3. 看 consumer lag 是否积压。
4. 看消费者是否 Bulk 写入失败。
5. 看 ES refresh 间隔和查询路径。
6. 看是否走了 Mongo/ES 双路径灰度中的错误路径。

**应急方案：**
- 单条数据手动触发重建。
- 开启 fallback 扫描补偿。
- 查询临时回切 Mongo。
- 对失败原因做告警：mapping 错误、字段过长、ES 拒绝写入、集群只读等。

### 18. Go 服务发布后 P99 抖动怎么排查？

**排查路径：**
- 先看发布前后 QPS 是否变化，确认是否和发布相关。
- 看 P99 是所有接口上涨，还是个别接口上涨。
- 看 CPU、内存、GC、goroutine、连接池。
- 用 pprof 看 CPU/heap/goroutine。
- 看下游依赖耗时：MySQL、Redis、Mongo、ES、外部 API。
- 看日志是否有重试放大、错误风暴。

**常见原因：**
- 新增同步调用外部 API。
- goroutine 无限制创建。
- JSON 大对象序列化。
- 慢 SQL 或 ES 深分页。
- Redis hot key。
- 错误重试没有退避。

**回答亮点：**
不要只说“看日志”。资深回答要能把指标、trace、pprof、下游依赖、最近发布和应急降级串起来。

### 19. 线上故障复盘怎么体现成熟度？

**复盘结构：**
- 故障时间线：发现、定位、止血、恢复。
- 影响范围：接口、用户、业务指标、数据影响。
- 直接原因：哪个模块出错。
- 根因：为什么测试、监控、灰度没拦住。
- 改进项：代码、架构、监控、流程、预案。

**示例改进项：**
- 增加平台/账号维度的媒体 API 错误率告警。
- Kafka lag 按 topic/group 分级报警。
- ES 同步失败记录补偿状态。
- Redis hot key 和慢 Lua 脚本监控。
- 发布前压测和灰度白名单。

**面试表达：**
故障复盘不是追责，而是把偶发问题变成系统能力：可发现、可定位、可止血、可恢复、可预防。

**结论先行：**
常用命令按 CPU、内存、磁盘、网络、进程、日志分类记忆，面试中要说明每个命令解决什么问题。

**分层展开：**

**CPU**
- `top`：整体进程资源。
- `pidstat`：进程和线程 CPU。
- `perf top`：热点函数。

**内存**
- `free`：系统内存。
- `vmstat`：内存和 swap。
- `pmap`：进程内存映射。

**磁盘**
- `df`：磁盘空间。
- `du`：目录大小。
- `iostat`：IO 延迟和利用率。
- `iotop`：进程 IO。

**网络**
- `ss`：连接状态。
- `tcpdump`：抓包。
- `curl`：接口验证。
- `dig`：DNS 解析。

**日志**
- `grep/rg`：搜索日志。
- `tail`：实时查看。
- `journalctl`：系统服务日志。

**面试追问：你排障时会先敲命令还是先看监控？**
先看监控和告警确认范围，再用命令深入到具体机器和进程，避免盲查。

## Redis 网络 IO、socket 可读可写与 Pipeline 深挖

**结论先行：**
Redis 快不是简单因为“单线程”，而是主线程用 IO 多路复用管理大量连接，socket 就绪后再读请求、执行命令、写响应；Pipeline 则通过批量发送命令、批量读取响应，减少 RTT 和 read/write 系统调用次数。

可以把 Redis 一次请求拆成三层：

| 层次 | 读是什么意思 | 写是什么意思 |
| --- | --- | --- |
| Redis 命令语义层 | `GET` 读取 Redis 内存中的 value | `SET` 修改 Redis 内存数据 |
| 网络事件层 | socket 可读，表示客户端请求已经到达服务端内核接收缓冲区 | socket 可写，表示服务端可以把响应写入内核发送缓冲区 |
| 系统调用层 | `read/recv` 从 socket 读字节到 Redis 用户态 | `write/send` 把 Redis 响应写入 socket |

一句话区分：

```text
GET/SET 的读写是数据语义；
socket 可读/可写是网络事件；
read/write 是系统调用。
```

**socket 是什么？**
客户端和 Redis 通信，本质是 TCP socket。应用层拿到的是 fd，fd 背后是内核里的 socket 对象，里面有接收缓冲区和发送缓冲区：

```text
应用程序
  |
  | fd
  v
内核 socket 对象
  ├── recv buffer  接收缓冲区
  └── send buffer  发送缓冲区
```

Redis 服务端常见两类 fd：

```text
Redis Server
├── listen_fd       // 监听新连接
├── client_fd_1     // 和客户端 1 通信
├── client_fd_2     // 和客户端 2 通信
└── client_fd_3     // 和客户端 3 通信
```

- `listen_fd` 可读：有新连接来了，Redis 可以 `accept()`，拿到新的 `client_fd`。
- `client_fd` 可读：某个客户端发来了 Redis 命令，Redis 可以 `read/recv`。
- `client_fd` 可写：Redis 可以把响应写回客户端，Redis 可以 `write/send`。

注意：

```text
客户端发 GET，对 Redis 服务端来说是 client_fd 可读。
客户端发 SET，对 Redis 服务端来说也是 client_fd 可读。
Redis 返回 GET 结果，对 Redis 服务端来说是 client_fd 可写。
Redis 返回 SET 的 +OK，对 Redis 服务端来说也是 client_fd 可写。
```

**GET 的完整链路：**

```text
Client                         Redis
  |                              |
  | 发送 GET name                |
  | ---------------------------> |
  |                              | client_fd 可读
  |                              | read() 读取命令
  |                              | 执行 GET：读内存数据
  |                              | 生成响应 "tom"
  |                              | client_fd 可写
  | <--------------------------- |
  | 返回 "tom"                   |
```

这里有三种“读”：

```text
1. Redis 网络层 read socket：读取客户端发来的 GET 命令
2. Redis 命令语义 GET：读取内存中的 value
3. 客户端 read socket：读取 Redis 返回结果
```

**SET 的完整链路：**

```text
Client                         Redis
  |                              |
  | 发送 SET name tom            |
  | ---------------------------> |
  |                              | client_fd 可读
  |                              | read() 读取命令
  |                              | 执行 SET：写内存数据
  |                              | 生成响应 +OK
  |                              | client_fd 可写
  | <--------------------------- |
  | 返回 +OK                     |
```

重点是：

```text
SET 是 Redis 数据层面的写；
但客户端发送 SET 命令，对 Redis 服务端来说仍然是 socket 可读事件。

GET 是 Redis 数据层面的读；
但 Redis 返回 GET 结果，对 Redis 服务端来说是 socket 可写事件。
```

**为什么需要 IO 多路复用？**

如果 Redis 对某个客户端使用阻塞 IO：

```c
read(client_fd, buffer, size)
```

一旦这个客户端没发数据，Redis 就可能卡在这个连接上，其他客户端即使已经发来请求，也无法被处理。

如果一个连接一个线程：

```text
client A -> thread A
client B -> thread B
client C -> thread C
```

连接数上来后，线程栈内存、调度成本、上下文切换、锁竞争都会变重。

IO 多路复用的思路是：

```text
应用程序不用挨个问每个 socket 有没有数据，
而是把 socket 注册给操作系统，
哪个 socket 准备好了，操作系统通知应用程序。
```

对应 Redis：

```text
哪个 socket 可读，Redis 就 read 哪个 socket；
哪个 socket 可写，Redis 就 write 哪个 socket；
没就绪的连接，不浪费主线程时间。
```

**select、poll、epoll 怎么区分？**

- `select`：能监听多个 fd，但 fd 数量通常受限制，返回后还要扫描。
- `poll`：用数组保存 fd，没有 select 固定 fd 集合大小限制，但连接多时仍要扫描。
- `epoll`：先把关心的 fd 注册给内核，内核维护 interest list 和 ready list；应用调用 `epoll_wait` 时，直接拿到已经就绪的 fd，更适合大量连接。

如果 Redis 有 1 万个连接，但只有 10 个连接有请求：

```text
select/poll：
    可能要关注和扫描很多连接。

epoll：
    内核直接告诉 Redis 哪 10 个连接就绪。
```

**Redis 事件循环可以怎么讲？**

```text
while true:
    1. 等待 socket 事件就绪
    2. 处理可读事件：读取客户端请求
    3. 解析 Redis 命令
    4. 执行命令
    5. 把响应写入客户端输出缓冲区
    6. 处理可写事件：发送响应
    7. 处理时间事件：serverCron、过期扫描、统计等
```

图示：

```text
                Redis 主线程
                     |
                     v
                  事件循环
                     |
       +-------------+-------------+
       |             |             |
       v             v             v
    可读事件       可写事件       时间事件
 client_fd 可读  client_fd 可写  serverCron
       |             |          过期处理等
       v             v
    read 请求     write 响应
       |
       v
    解析命令 -> 执行命令 -> 生成响应
```

**Redis 为什么不一直监听可写事件？**
因为大多数时候 socket 都是可写的，只要内核发送缓冲区没满，socket 通常就可写。如果对所有连接一直注册可写事件，事件循环会被大量无意义的可写事件唤醒。

更合理的做法是：

```text
没有响应要发：
    不关心可写事件

有响应没发完：
    注册可写事件

socket 可写：
    尽量把输出缓冲区里的响应写出去

写完了：
    取消可写事件
```

**Pipeline 为什么能提速？**

Redis 是请求/响应协议，不用 Pipeline 时：

```text
write cmd1
read resp1

write cmd2
read resp2

write cmd3
read resp3
```

使用 Pipeline 后：

```text
write cmd1+cmd2+cmd3
read resp1+resp2+resp3
```

对比图：

```text
不用 pipeline：

Client                         Redis
  | -- write cmd1 ------------> |
  | <- read resp1 ------------- |
  | -- write cmd2 ------------> |
  | <- read resp2 ------------- |
  | -- write cmd3 ------------> |
  | <- read resp3 ------------- |


使用 pipeline：

Client                         Redis
  | -- write cmd1+cmd2+cmd3 --> |
  |                             | 依次执行 cmd1/cmd2/cmd3
  | <- read resp1+resp2+resp3 - |
```

Pipeline 的收益有两个：

```text
1. 减少网络 RTT，不用每条命令都等一次响应。
2. 减少 read/write 系统调用次数，把多次读写合并成少数几次读写。
```

但要说严谨一点：Pipeline 不一定永远只有一次 `read/write`，还受 TCP 缓冲区、命令大小、客户端库缓冲策略、网络状况影响。更准确的说法是：Pipeline 把多条命令和响应批量化，把很多次 `read/write` 减少为少数几次 `read/write`。

**完整链路总结：以 `GET user:1` 为例**

```text
1. 客户端从连接池拿一个 socket 连接
2. 客户端 write GET user:1 到 socket
3. 数据经 TCP/IP 到达 Redis 服务端内核 socket 接收缓冲区
4. epoll 发现 client_fd 可读
5. Redis 主线程 read 请求到用户态输入缓冲区
6. Redis 解析 RESP 协议
7. Redis 执行 GET，在内存 dict 中查找 key
8. Redis 生成响应
9. 响应进入 Redis 输出缓冲区
10. client_fd 可写时，Redis write 响应
11. 客户端 read 到结果
```

**面试追问 1：GET 是读命令，为什么对 Redis 来说还是 socket 可读？**
GET 是 Redis 数据语义上的读，表示读取内存中的 value。但客户端发送 GET 命令这个动作，在网络层是客户端往 Redis 发数据。Redis 服务端要先从 socket 读取这个请求，所以对 Redis 来说，收到 GET 命令是 socket 可读事件。执行 GET 后返回 value，才是 socket 可写事件。

**面试追问 2：SET 是写命令，为什么也是 socket 可读？**
SET 是 Redis 数据语义上的写，表示修改 Redis 内存数据。但 SET 命令本身是客户端发给 Redis 的请求数据。Redis 服务端必须先通过 socket 可读事件读取 SET 命令，然后再执行内存写入。执行完 SET 后返回 `+OK`，这个响应才对应 socket 可写。

**面试追问 3：Pipeline 是否会让 Redis 并行执行命令？**
不会。Pipeline 只是客户端把多条命令批量发送给 Redis，Redis 服务端仍然会按照同一个连接上的命令顺序依次执行。Pipeline 提升的是网络效率和系统调用效率，不是把命令变成并行执行。

**面试追问 4：Pipeline 能无限大吗？**
不能。Pipeline 太大会导致客户端一次发送大量命令，服务端需要缓存大量响应，增加输出缓冲区压力，也可能造成延迟抖动。大批量命令要按合理批次发送。
