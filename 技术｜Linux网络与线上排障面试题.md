# 技术｜Linux网络与线上排障面试题

> 面向 9 年+ Golang Web 后端业务岗。
> 覆盖 TCP/HTTP、连接池、Linux 资源排查、线上故障定位和复盘。

## 一、网络与排障总框架

### 1. 资深后端如何回答线上排障题？

**结论先行：**
线上排障要先止血，再定位，再修复，再复盘；定位时按流量、应用、依赖、系统资源和最近变更逐层收敛。

**分层展开：**

**止血**
- 限流。
- 降级。
- 回滚。
- 扩容。
- 隔离异常实例。

**定位**
- 错误率、P99、QPS。
- 日志、指标、Trace。
- DB/Redis/MQ 下游状态。
- CPU、内存、磁盘、网络。
- 最近发布和配置变更。

**修复**
- 临时修复和长期修复分开。
- 核心链路优先。
- 变更要灰度。

**复盘**
- 时间线。
- 根因。
- 影响面。
- 改进项。

**面试追问：排障最忌讳什么？**
最忌讳没有止血就深入分析，或者没有证据直接猜根因。

## 二、TCP 与 HTTP

### 2. TCP 三次握手和四次挥手怎么理解？

**结论先行：**
三次握手用于双方确认收发能力和初始序列号，四次挥手用于双方分别关闭发送方向。

**分层展开：**

**三次握手**
- 客户端 SYN。
- 服务端 SYN+ACK。
- 客户端 ACK。
- 建立可靠连接。

**四次挥手**
- 一方发送 FIN。
- 对方 ACK。
- 对方处理完剩余数据后 FIN。
- 发起方 ACK 并进入 TIME_WAIT。

**工程意义**
- 握手失败影响连接建立。
- 大量 TIME_WAIT 可能来自短连接。
- CLOSE_WAIT 多通常是应用没有关闭连接。

**面试追问：为什么需要 TIME_WAIT？**
为了确保最后 ACK 能被对方收到，并让旧连接的延迟报文在网络中自然消失，避免影响新连接。

### 3. TCP 粘包是什么？

**结论先行：**
TCP 是字节流协议，没有消息边界。所谓粘包是应用层没有正确定义和解析消息边界。

**分层展开：**

**原因**
- TCP 只保证字节有序可靠。
- 多个小包可能合并。
- 一个应用消息可能拆成多段。

**解决方案**
- 固定长度。
- 分隔符。
- 长度前缀。
- 应用层协议，如 HTTP、Protobuf framing。

**面试追问：UDP 有粘包问题吗？**
UDP 是数据报协议，保留消息边界，但不保证可靠、有序和不丢失。

### 4. HTTP keep-alive 有什么作用？

**结论先行：**
keep-alive 通过复用 TCP 连接减少握手成本和延迟，是高并发 HTTP 服务的基础优化。

**分层展开：**

**优点**
- 减少 TCP 握手。
- 降低延迟。
- 减少端口和连接创建成本。

**风险**
- 空闲连接占资源。
- 连接池配置不当会导致连接耗尽。
- 长连接故障要及时摘除。

**工程建议**
- 客户端连接池设置最大空闲连接。
- 服务端设置 idle timeout。
- 监控连接数和等待时间。

**面试追问：连接池越大越好吗？**
不是。连接池过大会压垮下游，过小会排队。要按 QPS、耗时、实例数和下游承载计算。

### 5. HTTP/1.1、HTTP/2、HTTP/3 有什么区别？

**结论先行：**
HTTP/1.1 以文本协议和连接复用为主，HTTP/2 引入多路复用和二进制帧，HTTP/3 基于 QUIC 改善队头阻塞和连接迁移。

**分层展开：**

**HTTP/1.1**
- keep-alive。
- 管线化实践较少。
- 同连接上仍有队头阻塞。

**HTTP/2**
- 二进制分帧。
- 多路复用。
- header 压缩。
- 单 TCP 连接上 TCP 层仍可能队头阻塞。

**HTTP/3**
- 基于 QUIC/UDP。
- 改善 TCP 队头阻塞。
- 支持连接迁移。

**面试追问：gRPC 为什么常用 HTTP/2？**
HTTP/2 支持多路复用、流式传输和二进制帧，适合高效 RPC 调用。

## 三、Linux 资源排查

### 6. CPU 飙高怎么排查？

**结论先行：**
CPU 飙高先确认是用户态、内核态、IO wait 还是 steal，再结合进程、线程和应用 profile 定位。

**分层展开：**

**系统层**
- `top` 看整体 CPU。
- `mpstat` 看各核。
- `pidstat` 看进程和线程。
- `perf` 看内核和用户态热点。

**应用层**
- Go 服务抓 pprof。
- 看 QPS、P99、错误率。
- 查最近发布。

**常见原因**
- 计算热点。
- GC 压力。
- 锁竞争。
- 日志过量。
- 重试风暴。

**面试追问：CPU user 高和 sys 高分别说明什么？**
user 高通常是应用计算多；sys 高可能是系统调用、网络、锁、文件 IO 或内核态开销高。

### 7. 内存高怎么排查？

**结论先行：**
内存高要区分进程堆内存、缓存、RSS、泄漏、mmap 和系统 page cache。

**分层展开：**

**系统层**
- `free` 看整体内存。
- `top` 看进程 RSS。
- `pmap` 看进程内存映射。
- `vmstat` 看 swap。

**应用层**
- Go heap profile。
- goroutine profile。
- 缓存大小。
- 大对象持有。

**常见原因**
- 本地缓存无上限。
- goroutine 泄漏。
- slice 持有大数组。
- 请求堆积。
- 内存碎片。

**面试追问：Linux free 里 available 少就一定危险吗？**
要结合 buffer/cache 和 available 看。Linux 会用空闲内存做 page cache，不代表都被应用占满。

### 8. 磁盘 IO 高怎么排查？

**结论先行：**
磁盘 IO 高要看读写吞吐、IOPS、await、util 和具体进程，再判断是日志、数据库、临时文件还是持久化任务。

**分层展开：**

**系统命令**
- `iostat` 看磁盘延迟和利用率。
- `iotop` 看进程 IO。
- `df` 看空间。
- `du` 定位目录。

**常见原因**
- 日志暴增。
- MySQL 慢查询和刷盘。
- Redis AOF/RDB。
- 大文件上传下载。
- 临时文件未清理。

**治理**
- 日志采样和异步。
- 磁盘隔离。
- 限制批任务。
- 告警和清理策略。

**面试追问：磁盘 util 100% 一定代表吞吐满了吗？**
不一定。可能是单盘处理小 IO 延迟高，吞吐不大但繁忙度满了。

### 9. 网络问题怎么排查？

**结论先行：**
网络问题要从 DNS、连通性、延迟、丢包、带宽、连接数和应用超时逐层排查。

**分层展开：**

**基础检查**
- DNS 解析。
- ping 延迟和丢包。
- telnet/nc 端口连通。
- traceroute 路径。

**连接状态**
- `ss -s` 看总览。
- `ss -antp` 看连接状态。
- TIME_WAIT、CLOSE_WAIT、SYN_SENT。

**抓包**
- tcpdump。
- Wireshark 分析。
- 看重传、RST、窗口大小。

**面试追问：CLOSE_WAIT 很多说明什么？**
通常说明对端已关闭连接，但本地应用没有及时 close，可能是代码连接泄漏。

## 四、线上故障场景

### 10. 接口突然大量超时怎么排查？

**结论先行：**
大量超时先确认影响范围和止血，再沿入口、应用、下游、资源和变更定位。

**分层展开：**

**影响范围**
- 单接口还是全站。
- 单机房还是全区域。
- 部分用户还是全部用户。

**入口层**
- QPS 是否异常。
- 网关是否排队。
- 限流是否触发。

**应用层**
- goroutine 是否暴涨。
- 连接池是否耗尽。
- CPU/GC 是否异常。

**依赖层**
- DB 慢 SQL。
- Redis 慢查询。
- MQ 积压。
- RPC 下游超时。

**面试追问：为什么要先看最近变更？**
线上故障高比例与发布、配置、流量切换、数据任务有关，最近变更能快速缩小范围。

### 11. 502、503、504 分别可能是什么？

**结论先行：**
502 通常是网关从上游收到无效响应，503 表示服务不可用，504 表示网关等待上游超时。

**分层展开：**

**502**
- 上游进程崩溃。
- 连接被重置。
- 协议错误。

**503**
- 服务实例不可用。
- 熔断或限流。
- 上游没有健康实例。

**504**
- 上游响应超时。
- 下游依赖慢。
- 应用线程或连接池阻塞。

**面试追问：504 一定是网关问题吗？**
不是。网关只是报超时的位置，根因常在应用或更下游依赖。

### 12. 如何做故障复盘？

**结论先行：**
复盘不是追责，而是还原事实、定位根因、沉淀机制，防止同类问题再次发生。

**分层展开：**

**时间线**
- 故障开始。
- 告警触发。
- 处理动作。
- 恢复时间。

**影响面**
- 用户范围。
- 接口和业务损失。
- 错误率和时长。

**根因**
- 直接原因。
- 深层机制缺陷。
- 为什么监控没提前发现。

**改进**
- 代码修复。
- 监控告警。
- 发布流程。
- 压测和预案。

**面试追问：复盘里最重要的产出是什么？**
可落地的改进项和 owner，而不是一份漂亮文档。

## 五、常用排障命令速记

### 13. Linux 排障常用命令有哪些？

**结论先行：**
常用命令按 CPU、内存、磁盘、网络、进程、日志分类记忆，面试中要说明每个命令解决什么问题。

**分层展开：**

**CPU**
- `top`：整体进程资源。
- `pidstat`：进程和线程 CPU。
- `perf top`：热点函数。

**内存**
- `free`：系统内存。
- `vmstat`：内存和 swap。
- `pmap`：进程内存映射。

**磁盘**
- `df`：磁盘空间。
- `du`：目录大小。
- `iostat`：IO 延迟和利用率。
- `iotop`：进程 IO。

**网络**
- `ss`：连接状态。
- `tcpdump`：抓包。
- `curl`：接口验证。
- `dig`：DNS 解析。

**日志**
- `grep/rg`：搜索日志。
- `tail`：实时查看。
- `journalctl`：系统服务日志。

**面试追问：你排障时会先敲命令还是先看监控？**
先看监控和告警确认范围，再用命令深入到具体机器和进程，避免盲查。
